## 先打个比方

有两种常见的办事方式。第一种像工厂装配线：每一件货都先过 A 工位，再过 B 工位，再过 C 工位，不管来的是什么货，顺序都一样。只要每一道工序都很清楚，这种做法最稳。

第二种像医院分诊台：病人先到前台，被判断是外伤、发烧还是眼科问题，然后分别送去不同科室。每个人不需要走同一条线，但一旦分到某一类，后面走的还是固定流程。

接下来要讲的，就是遇到一堆进来的事时，到底该让它们都排同一条线，还是先分到不同窗口再分别处理。选错了，要么把本来简单的事拖得很长，要么把差别很大的事硬塞进同一套做法里，最后谁也处理不好。

## 第一种拆法：同一批活都走同一串固定工序

当一个任务可以被干净地拆成几道**固定**的小工序，而且每一份输入都要按同样顺序走完整条线时，最适合用 <Term def="链式（Prompt Chaining）：把任务拆成一串固定步骤，让前一步的输出成为后一步的输入。每个输入都走同一条顺序固定的流水线，中途可插程序化检查。">链式（Prompt Chaining）</Term>。它的重点不是「步骤多」，而是**每个输入都走同一条路**。

<ChainingWorkflowDiagram />

盯住这张图看：所有输入都从左往右过同样四道工序，中间没有「看情况换道」这件事。这样拆的好处是，每一步都比「一口气干完整件事」更简单，LLM 更容易做对；你还可以在中途加一道程序化检查，发现缺字段、格式不对就挡下来，不让错误带着一路往后传。Anthropic 把它推荐给那种「能干净拆成固定子任务」的活，比如先写提纲、再检查提纲、再按提纲写正文。

一句话记忆：**链式 = 同一条流水线，所有输入都排队过一遍。** 如果你已经知道每一份输入都该走哪几步，而且先后顺序固定，优先考虑它。

## 第二种拆法：先分诊，再送去不同专科

如果问题不是「大家都走同一条线」，而是「先看它属于哪一类，再把它送去对应的专门流程」，那就该用 <Term def="路由（Routing）：先把输入分到某个类别，再把它送进对应的专用提示、工具或下游流程。各类别后续流程固定，但入口先要做一次分类。">路由（Routing）</Term>。它的重点不是多一步分类，而是**不同类别应该交给不同的专门流程**。

<RoutingWorkflowDiagram />

图里左边所有请求先经过一个 <Term def="路由器（Router）：负责判断当前输入属于哪一类、该送进哪条后续流程的分类器。它可以是 LLM，也可以是规则或传统分类模型。">路由器（Router）</Term>，再被送去退款、技术支持、普通咨询三条不同的后续线。注意两点：第一，分完类之后，每一条支线本身仍是固定流程，所以它依然是工作流；第二，路由成立的前提是**类别边界相对清楚、而且你能比较准确地分出来**。Anthropic 推荐它用于「有明显类别差异、每类更适合用专用提示/工具处理」的任务，比如客服请求分流，或把简单题送小模型、难题送大模型。

一句话记忆：**路由 = 先决定走哪条线，再沿那条专线一路做完。** 它不是每份输入都走全套，而是每份输入只走最适合自己的那一套。

## 它们都还是工作流，不是自主 agent

链式和路由看起来比单次调用复杂很多，但它们都还没有跨到自主 agent 那边。原因只有一个：**路径仍然是人预先设计好的。**

- 链式里，人已经写死了「第 1 步 → 第 2 步 → 第 3 步」。
- 路由里，人已经写死了「可分成哪几类」以及「每一类后面接哪条固定流程」。

所以它们都是工作流，只是拆法不同。真正越界到自主 agent 的时刻，是你发现：既没法把所有输入都塞进同一条固定工序，也没法事先列清楚类别和对应分支，必须让模型边做边判断下一步该怎么走。这也是为什么 Ch11 先讲「上不上 agent」，Ch12 才讲「不上 agent 时，工作流怎么拆更稳」。

## 动手看：该串起来做，还是先分流再做？

上面两种拆法看懂后，真正难的是遇到一个新任务时怎么选。下面这张主动画把决策顺序做成了一个最小判断树：先问是不是所有输入都要走同一串固定工序；如果不是，再问能不能先清楚分成几类，各走自己的固定流程；如果两问都答不上来，你就已经在工作流边界外了。

> 猜一猜：一个「客服总入口」里同时有退款、改地址、账号登录失败三类请求。第一问“所有输入都走同一串固定工序吗”你会答什么？第二问“能否先比较准确地分成几类，再各走固定流程”又该答什么？先想好，再单步播放核对。

<ChainRoutingDecisionDiagram />

把这张图记成两个连续问题就够了：

1. **大家是不是都该走同一串固定工序？** 是 → 优先链式。
2. **如果不是，能不能先稳定分成几类，各走专用固定流程？** 是 → 优先路由。

两问都不成立时，别硬写工作流了。那通常说明任务开放、分支列不全、步骤也会随中间结果变化，这已经是工作流的边界，应该回到 Ch11 的判断框架，考虑是否要往 agent 方向加自主性。

## 容易踩的坑

先看一眼三种最常见的翻车现场。它们几乎把链式、路由以及工作流边界的错误都涵盖了：

<ChainRoutingTrapDiagram />

<Callout type="trap">
  **现象**：一个任务里其实每份输入都要走同样四步，你却先上分类，把每张发票还没识别就硬分成好多类，系统更慢也更难调。
  **原因**：这是把该链式的活误做成路由。分类并没有带来真正的分工，只是多加了一层成本和误分风险。
  **修法**：回到第一问：如果所有输入都走同一串固定工序，就直接用链式，把精力放在每一步的拆分和中途
  gate 上。
</Callout>

<Callout type="trap">
  **现象**：客服总入口里退款、技术支持、普通咨询全被硬塞进同一条长链，简单问题也得把不相关步骤全走一遍，延迟越来越高。
  **原因**：这是把该路由的活误做成链式。类别差异明明很大，却让所有请求共享一条线，导致每一步都不得不「兼顾所有情况」，越写越臃肿。
  **修法**：先做路由，把请求分进专门支线；让每条支线只优化自己那一类，而不是拿一条超长链条硬吃所有请求。
</Callout>

<Callout type="trap">
  **现象**：用户说的是开放任务，比如“帮我调研 3
  个竞品，缺什么自己再查”，你却想靠几条预设类别和固定支线把它写死，结果一遇到新情况就无路可走。
  **原因**：这已经超出链式和路由的适用边界了。你既列不清类别，也写不完后续分支，说明工作流本身不够。
  **修法**：停下来，不要继续堆分支。回到 Ch11 的框架，评估是否该加自主性，往
  agent 或后续的编排模式走，而不是继续硬写死。
</Callout>

## 小结

- 链式：同一路，固定顺序拆步骤
- 路由：先分清类别，再走专线
- 两者都属预先设计好的固定路线
- 先问同不同路，再问能否稳定分流
- 两问都不成立，就别再硬写死

**练习 1（辨析题）** 下面两种任务分别更像链式还是路由？为什么？
（A）把一份会议录音转成纪要：转写 → 提炼要点 → 检查遗漏 → 输出纪要。
（B）客服总入口：用户可能是在问退款、查物流、改密码或技术故障。

<Answer>

（A）更像**链式**。因为每份输入都要走同一串固定工序：先转写、再提炼、再检查、再输出，路径对所有输入都一样。
（B）更像**路由**。因为入口里的请求类别差异很大，先分成退款 / 物流 / 账号 / 技术等类，再把每一类送进自己的固定流程，比分别让所有请求都走同一条长链更稳。

</Answer>

**练习 2（选型题）** 下面三个任务分别该用链式、路由，还是已经更接近自主 agent？说明理由。
（甲）发票图片报销：OCR → 抽取金额 → 校验格式 → 填表。
（乙）客服入口：普通咨询、退款、技术支持三类请求共用一个入口。
（丙）“帮我调研最近三家竞品的定价策略，缺数据时自己补搜、交叉核对后给结论。”

<Answer>

（甲）**链式**：所有输入都走同一串固定工序。
（乙）**路由**：先分成几类，再进对应固定支线。
（丙）更接近**自主 agent**：任务开放、步骤不定、查什么和查几轮都得看中间结果，已经超出链式 / 路由这种预先写死路径的工作流边界。

</Answer>

**练习 3（改 Demo 代码题）** 看本章右边的路由伪代码：它默认 `refund / tech / faq` 三类都能被稳定分出来。假如你发现 `classify(message)` 经常把“订单退款失败”这种请求分错到 FAQ，请改一版伪代码，加一个“分不准时不要硬分”的兜底出口，并说说你为什么这样改。

<Answer>

可以把「低置信度时别硬分」写进路由器：

```python
def handle_ticket(message):
    kind, score = classify(message)
    if score < 0.75:
        return escalate_to_human(message)   # 或交给更强模型 / agent
    if kind == "refund":
        return refund_flow(message)
    if kind == "tech":
        return tech_flow(message)
    return faq_flow(message)
```

关键改动是给路由多加了一道**置信度兜底**：当类别边界不清、分不准时，不要硬塞进某条支线。因为路由成立的前提就是「能比较准确地分类」；一旦这个前提不稳，继续强分只会把错请求送进错流程。更稳的做法是回退到人工、更强模型，或者后续更有自主性的模式。

</Answer>

## 名词解释

<Glossary>
  <GlossaryItem term="链式（Prompt Chaining）">
    把任务拆成一串固定步骤，让前一步的输出成为后一步的输入。每一份输入都走同一条顺序固定的流水线，中途可以插程序化检查。适合「能干净拆成固定子任务」的活，比如先转写、再提炼、再检查。详见本章「第一种拆法」。
  </GlossaryItem>
  <GlossaryItem term="路由（Routing）">
    先把输入分到某个类别，再把它送进对应的专用提示、工具或下游流程。各类别后续流程固定，但入口先要做一次分类。适合「类别明显不同、每类更适合专门处理」的入口型任务。详见本章「第二种拆法」。
  </GlossaryItem>
  <GlossaryItem term="路由器（Router）">
    负责判断当前输入属于哪一类、该送进哪条后续流程的分类器。它可以是
    LLM，也可以是规则或传统分类模型。路由器的前提不是“能分”，而是“能分得比较准”；一旦边界重叠严重，就需要兜底或回退。详见本章「第二种拆法」与练习
    3。
  </GlossaryItem>
</Glossary>
