## 先打个比方

公司里办事，有两种路子。一种是**照流程走**：报销怎么填、走哪几个审批、谁盖章，写在制度里，每次都一样。这种事最省心——照着做就行，不会出岔子，也不用动脑子。

另一种是**请个人来办**：「帮我把这趟出差安排妥当。」具体怎么订、遇到没票了怎么改、临时改行程怎么应对，你说不全，只能交给一个能自己拿主意的人，让他见机行事。

关键在于：**流程固定的事照流程走最稳最省，情况多变、要随机应变的事才值得请个能自己拿主意的人。** 这一章就讲：做这类程序时，怎么判断一件事该「照流程走」还是该「请个人」——杀鸡别用牛刀，牛刀也别拿来杀鸡。选错了，要么白白浪费，要么根本办不成事。

## 再辨析一次：差别不在步数，在「路是谁定的」

第 1 章我们已经认过这几个词：<Term def="工作流（Workflow）：把若干步骤按固定顺序串成的流水线，每一步、走哪条路都是人预先写死的，每次执行都一样、可预测。">工作流（Workflow）</Term> 是路径写死的流水线，智能体是在循环里自己拿主意的程序。这一节只把那条分界线擦得更清楚——很多人会以为「步数多就是智能体、步数少就是工作流」，**这是错的**。

真正的分界线是：**这条路，是人在写代码时就定死的，还是 LLM 在运行时自己边跑边选的？** 一张图把同一件「订高铁票」的事，两种走法并排放出来：

<WorkflowVsAgentDiagram />

看清左右两栏：左边工作流的三步——查车次、下单、发确认——是人**提前写死**的，箭头直来直去、不分叉，没票了也照样往下走；右边智能体没有写死的步骤，只有一个**循环**：想一步、做一步、看一眼，由 LLM **当场**决定这次调哪个工具、要不要换车次、何时收尾。一个工作流可以有几十步却仍是工作流（步步写死），一个 agent 可能只转三圈却是 agent（步步自己选）。**记住这条线，本章后面全靠它做判断。**

## 一个核心原则：先用最简单的，只在需要时加自主性

知道了区别，下一个问题是「那我到底该用哪个」。这里有一条贯穿整个智能体工程的原则，也是本章的灵魂：**先用最简单的方案，只在确实需要时才加自主性。**

这里的 <Term def="自主性（Autonomy）：程序在多大程度上「自己决定下一步」而非按人写死的逻辑走。自主性越高越灵活，能应付越开放的任务，但也越难预测、越难控制、通常越贵越慢。" id="autonomy">自主性</Term> 不是越多越好。给程序加自主性，等于把「决定下一步」的权力交给 LLM——这换来了灵活，但也同时换来了成本、延迟和不可预测性（下一节细看这笔账）。所以正确的姿势不是「先想怎么把它做成最酷的 agent」，而是**反过来**：先问「能不能用最笨最简单的办法（纯代码、写死的工作流）解决」，只有当简单办法确实顶不住时，才一点点往上加自主性。能省则省，这是工程上的稳妥，不是偷懒。

## 同一件事，多种自主性档位：智能体模式光谱

「加自主性」也不是从工作流一步跳到全自主 agent，中间还有一整排档位。把它们按自主性从低到高排开，就是一道 <Term def="智能体模式（Agentic Patterns）：介于「纯工作流」和「全自主 agent」之间的一系列构建方式，如链式、路由、并行、编排-工作者、评估-优化。它们大多仍是路径预先编排好的工作流，只是在不同槽位用上 LLM。" id="agentic-patterns">智能体模式</Term> 光谱，也正好是第四篇接下来要逐章展开的路线图：

<PatternSpectrumDiagram />

从左到右：单次调用 → 链式 → 路由 → 并行 → 编排-工作者 → 评估-优化 → 自主 agent，越往右 LLM 自己拿主意越多。要点有两个：一是**多数实用系统落在中间这些工作流模式上**，自主 agent 只是最右端、最灵活也最贵的那一档，并非默认选项；二是中段那些模式（如 <Term def="编排（Orchestration）：由一个「编排者」按预先设计好的结构，把任务拆给若干「工作者」分头去做、再把结果汇总。控制流是人编排好的，所以它仍属工作流模式，而非完全自主。" id="orchestration">编排</Term>-工作者）虽然用到 LLM，但整体控制流是人预先编排好的，路径不是 LLM 临场决定的——所以它们仍算工作流。链式 / 路由在 Ch12、并行 / 编排-工作者在 Ch13、评估-优化在 Ch14 展开，本章先把全景给你。

逐处对照两边的差异：左边是 `search → 取第一个 → pay` 三个函数**顺序直调**，路径写死，没有循环，也没有「LLM 决定下一步」——所以查到的车次没票时它无路可走，直接崩。右边核心是一个 `while` 循环，每一圈 `llm(ctx)` 让模型看当前情况、**自己决定**调哪个工具或收尾——没票它能自己改查别的车次。这份灵活不是白来的：右边每转一圈都要一次 LLM 调用，**更慢、更贵，而且同样输入下一步走哪不再固定**。这就是自主性的账：左边便宜可控但死板，右边灵活但贵且不定。

## 容易踩的坑

先别急着读三段文字，先看一眼两种最典型的翻车现场。左边是该照章办事却硬请了个爱临场发挥的人，右边是明明状况百出却非要拿死流程去套：

<PatternChoiceTrapDiagram />

<Callout type="trap">
  **现象**：一个明明很固定的任务（按死规则审核报销单：金额超标或缺发票就驳回），非要做成自主
  agent，上线后又慢又贵，还时不时误判。 **原因**：这是「固定可预测任务硬上
  agent」——审核规则是确定、可预先写死的，却把「决定下一步」的权力交给了
  LLM，平白引入随机性、成本和延迟，自主性完全用不上。
  **修法**：按决策框架反向收——第一问「可预测、步数固定吗」答「是」，就止步于工作流（甚至纯代码）：规则命中即驳回
  / 通过，稳、快、便宜。
</Callout>

<Callout type="trap">
  **现象**：一个明明很开放的任务（客户用大白话提的任意运维诉求），硬塞进一个写死的
  if-else 工作流，结果一遇到没预设的情况就卡死、无路可走。
  **原因**：这是反方向的坑——「该用 agent
  的开放任务硬塞进写死工作流」。开放任务的分支你永远列不全，写死的路径碰到没枚举过的情况就走投无路。
  **修法**：这类开放、不可预测的任务本就该往右选自主
  agent，让 LLM 看懂诉求后在循环里自己决定调哪些工具、怎么处理，而不是指望人把所有分支提前枚举。
</Callout>

<Callout type="trap">
  **现象**：「为了通用、省得以后改」，把每个功能都一上来做成最自主的
  agent，结果连最简单的功能也背上了高成本、高延迟和不可预测。
  **原因**：误把自主性当成免费的通用性，把旋钮一拧到底。大多数功能根本不需要运行时决策，自主性纯属浪费。
  **修法**：默认从最简单的方案起步，逐功能按决策框架判断；只有确实开放的功能才加自主性，并优先选光谱中段的工作流模式，而非一步到位全自主。
</Callout>

## 小结

- 工作流 vs 智能体的本质区别：路是人写死的，还是 LLM 边跑边自己选的——不看步数
- 核心原则：先用最简单的方案，只在确实需要时才加自主性，能简单别复杂
- 决策框架：先问「任务可预测、步数固定吗」——是 → 工作流（甚至纯代码）；否 → 才上 agent
- 自主性是把双刃剑：自主性↑ → 灵活↑，但成本 / 延迟 / 不可预测性也一起↑
- 第四篇是一道光谱：链式 / 路由 / 并行 / 编排-工作者 / 评估-优化，自主 agent 只是最右一档

<Exercises>

**练习 1（辨析题）** 下面两句关于「工作流 vs 智能体」的说法，对还是错？为什么？
（A）「一个有 20 个步骤的数据处理流水线，步骤这么多，肯定是智能体。」
（B）「这份灵活是免费的，所以功能能做成 agent 就尽量做成 agent。」

<Answer>

（A）**错**。区别不在步数多少，而在「路是谁定的」。20 步的流水线如果每一步、走哪条路都是人写死的，它仍是**工作流**；判断要看路由是写死的还是 LLM 运行时自己选的，而不是数有几步。
（B）**错**。灵活不是免费的——加自主性等于让 LLM 每步决策，换来灵活的同时也换来成本、延迟和不可预测。正确姿势是「先用最简单的，只在需要时才加自主性」，多数功能根本不需要运行时决策。

</Answer>

**练习 2（选型题 · 给场景选方案）** 下面三个任务，分别该用「工作流（甚至纯代码）」还是「自主 agent」？说明理由。
（甲）每月把数据库里的订单导出成固定格式的 Excel，再上传到指定网盘。
（乙）用户上传一段杂乱的会议录音，要你「整理出待办事项，缺信息时自己想办法补全、必要时上网查」。
（丙）把用户输入的身份证号校验格式、查出归属地、填进固定表单。

<Answer>

（甲）**工作流 / 纯代码**：步骤固定（查库 → 套格式 → 上传），每次一样、可预测，决策框架第一问答「是」，止步于工作流，最稳最便宜。
（乙）**自主 agent**：任务开放、步数不定、要随机应变——整理到哪、缺什么、要不要上网查、查哪些，都得看中间结果临场决定，没法预先写死路径，第一问答「否」，往右上 agent。
（丙）**纯代码 / 工作流**：格式校验和查表是确定规则，写死即可，连 LLM 都未必需要。判别诀窍：能把每一步和分支预先写死的 → 工作流；列不全分支、必须临场决定的 → agent。

</Answer>

**练习 3（改 Demo 代码题）** 看本章「代码对照」里右边那个 agent 写法。它用了 `while True`，万一 LLM 反复改车次却始终订不到票（比如全程没票），会发生什么？请改写这个循环，加一道保护，让它在转太多圈后强制停下并返回兜底回答。说出你改了哪里、为什么。

<Answer>

会**死循环**——agent 无限地「查 → 改车次 → 再查」却永远收不了尾，白白烧钱、卡住不返回。修法是加一个**最大步数上限**：

```python
def book_ticket(goal, max_steps=8):
    ctx = [system_prompt, goal]
    for _ in range(max_steps):     # 不再 while True，最多转 max_steps 圈
        step = llm(ctx)
        if step.done:
            return step.answer
        out = call_tool(step.tool, step.args)
        ctx += [step, out]
    return "抱歉，试了多次仍没订到票，建议换个日期再试。"  # 强制兜底
```

关键改动：把 `while True` 换成 `for _ in range(max_steps)`，转满还没收尾就跳出、返回兜底回答。这正是自主性的代价之一——把决定权交给 LLM 的同时，人必须留一道「最多转几圈」的闸，否则不可预测会变成不可收场。这也反过来说明：能用工作流写死的任务，根本不会有这种「转不停」的风险。

</Answer>

## 名词解释

<Glossary>
  <GlossaryItem term="工作流（Workflow）">
    把若干步骤按固定顺序串成的流水线，每一步、走哪条路都是人提前写死的，每次执行都一样、可预测。和智能体的根本区别：工作流的路是人写代码时定死的，智能体的路是
    LLM 在循环里边跑边选的——不看步数多少。详见本章「再辨析一次」。
  </GlossaryItem>
  <GlossaryItem term="自主性（Autonomy）" id="autonomy">
    程序在多大程度上「自己决定下一步」而不是按人写死的逻辑走。它是一把双刃剑：自主性越高越灵活、能应付越开放的任务，但也越难预测、越难控制、通常越贵越慢——不是越大越好。本章的核心原则就是「只在确实需要时才加自主性」。详见本章「一个核心原则」。
  </GlossaryItem>
  <GlossaryItem term="智能体模式（Agentic Patterns）" id="agentic-patterns">
    介于「纯工作流」和「全自主
    agent」之间的一系列构建方式，如链式、路由、并行、编排-工作者、评估-优化。它们大多仍是路径预先编排好的工作流，只是在不同槽位用上了
    LLM；按自主性从低到高排开就成了一道光谱。第四篇会逐章展开。详见本章「智能体模式光谱」。
  </GlossaryItem>
  <GlossaryItem term="编排（Orchestration）" id="orchestration">
    由一个「编排者」按预先设计好的结构，把任务拆给若干「工作者」分头去做、再把结果汇总。虽然用到了多个
    LLM，但整体控制流是人编排好的、不是 LLM
    临场决定的，所以它仍属于工作流模式，落在光谱中段，而非完全自主的 agent。详见本章「智能体模式光谱」。
  </GlossaryItem>
</Glossary>
