## 先打个比方

想象你在一张桌子上干活。桌子就那么大，东西摆多了就没地方下手了。可活儿还得继续，新资料还在往上堆——怎么办？要么把一摞看过的旧资料**归档成一页纸的摘要**，腾出大半个桌面；要么把那些**压根没用的废纸直接清走**。总之得收拾，桌子才腾得出地方接着干。

模型也有这么一张「桌子」：它一次能同时看见的东西，就那么多，装满了就摆不下新内容了。可对话会一轮轮聊下去，资料会越查越多，桌面迟早被占满。

这一章讲的就是**怎么收拾这张桌子**：桌子有多大、什么东西在占它、满了之后是该「归档成摘要」还是「直接清走」，以及一个反常识的小坑——**摆在桌子正中间的东西，反而最容易被看漏**。

## 第一个概念：上下文窗口，是一笔有限的 token 预算

那张「桌子」翻译成技术语言，就是 <Term def="上下文窗口（Context Window）：模型一次能「同时看见」的全部内容的总容量，以 token 计。它有一个固定的上限，装不下的内容模型根本看不到。" id="context-window">上下文窗口</Term>。它有个固定大小，可以把它想成一笔 <Term def="token 预算（Token Budget）：把上下文窗口的 token 上限看成一笔固定的预算——装进去的每样东西都在花这笔钱，花光就到头了。" id="token-budget">token 预算</Term>：你往里装的每样东西都在花这笔钱，花光就到头。

那这笔预算都被谁花掉了？看下面这根「预算条」，它被几类用途瓜分了：

<ContextBudgetDiagram />

关键不在每一段具体占多少，而在一句话：**它们全从同一根预算里出，加起来不能超上限。** 系统提示、之前每一轮的历史对话、临时检索来的资料、工具调用返回的结果——再加上必须留给模型写回答的「输出余量」，统统挤在这一根固定的条里。其中**历史对话**最危险：它会随着你每聊一轮就往里加一块、只增不减，最容易先把别人挤没地方。

## 第二个概念：窗口装满了，会出什么事

桌子摆满了硬要再放，东西就会掉地上。窗口也一样：当装进去的内容超过 token 上限，超出的部分会被**截断丢掉**（通常是最旧的内容先被丢），模型从此再也看不见它——可能正好把一条关键设定给丢了。

不止丢信息。token 用得越多，每次调用模型就**越贵、也越慢**：你是在为「让模型读完这一大堆」付费和等待。所以放任窗口一直涨，是个又费钱、又变慢、还可能丢信息的三输局面。

那满了到底怎么办？办法就两类，下一节摊开讲。

## 第三个概念：腾地方的两招——压缩 vs 裁剪

回到桌子的比方：腾地方无非两条路——把旧资料**归档成一页纸摘要**，或者把废纸**直接清走**。落到上下文窗口上，就是 <Term def="上下文压缩（summarization）：把最旧的几轮历史交给模型总结成一条短摘要，用摘要替换掉那几轮原文塞回上下文。要点还在，却省下大半 token。" id="context-compression">上下文压缩</Term> 和 <Term def="裁剪（truncation）：直接把最旧的、或与当前任务无关的几轮内容删掉，不做任何处理。零额外成本、最快，但被删的内容彻底没了。" id="context-truncation">裁剪</Term>。下面这张图把两者的做法和取舍并排摆出来：

<CompressionVsTruncationDiagram />

一眼看清差别：**压缩用一次额外的模型调用，换来「信息不全丢」**（要点被总结保留）；**裁剪用「彻底丢失」，换来「零成本、最快」**（一删了事）。所以选哪个，就看那段内容**之后还有没有用**——要点还要用就压缩，确实没用了就裁剪。

裁剪里有一种最常见的简单实现，叫 <Term def="滑动窗口（Sliding Window）：只保留最近的 N 轮对话，每来一轮新的就把最旧的一轮挤出去——像一个固定大小的窗口在历史上往前滑。" id="sliding-window">滑动窗口</Term>：只留最近 N 轮，来一轮新的就挤掉一轮最旧的。它够简单，但有个副作用——更早但重要的设定会被一起滑出去（后面误区里细说）。

## 动手看：一个会涨的窗口，怎么「填满 → 压缩腾空 → 继续」

前面讲的是静态的概念。真跑起来时，窗口是**动态**的：对话一轮轮往里加，条一点点填高，逼近上限，然后靠压缩腾空、再继续。下面这个动画把这个循环演给你看——盯住左边的「窗口容量条」怎么涨、怎么顶到红线、又怎么被压缩腾出空间。

> 猜一猜：对话一直加下去、又不做任何压缩，等窗口顶到 token 上限之后，最旧那几轮会怎样？是被原样留着、还是被悄悄截断丢掉？先想一想，再单步播放到第 ④ 步看答案。

<ContextWindowFillDiagram />

盯住这条循环：条从底往上填（历史在累加）→ 顶到红线、快溢出（再加就要截断丢信息）→ 触发压缩（最旧 4 轮总结成一条摘要塞回底部，腾出大半空间）→ 继续加新轮。**填满 → 压缩腾空 → 继续**——管理上下文窗口，干的就是这件事：不被动等它溢出截断，而是主动出手腾地方，让最该看的内容始终留在模型眼前。

## 第四个概念：lost in the middle——别把关键信息埋在中间

最后一个反常识的坑。你可能以为：只要东西塞进窗口了，模型就一视同仁地看见了。其实不然——存在一个叫 <Term def="lost in the middle：在很长的上下文里，模型对开头和结尾的内容更上心，对正中间的内容最容易看漏；关注度呈两端高、中间凹的 U 形。" id="lost-in-the-middle">lost in the middle</Term> 的现象：长上下文里，模型的注意力**两端高、中间凹**。

<LostInMiddleDiagram />

道理朴素得很：内容太长时，模型对开头和结尾更上心，对正中间最不上心。所以哪怕一条指令明明写进了窗口，只要它被埋在长长的中间段落里，就**大概率被忽略**。结论很实用：**最关键的指令和信息，放在开头或结尾，别埋在中间。**

## 用代码看一个「超长就压缩」的策略长什么样

把上面的循环落到代码里，最常见的就是「设一个阈值：历史一旦超长，就把最旧 N 轮交给 LLM 总结成一条摘要塞回去」。下面是它的骨架：

```python
def manage_history(history, system_prompt):
    # ① 估算当前历史占了多少 token
    used = count_tokens(history)

    # ② 没超阈值就原样返回，不折腾
    if used <= HISTORY_TOKEN_BUDGET:
        return history

    # ③ 超了：把最旧的 N 轮拎出来，交给 LLM 总结成一条短摘要（这是「压缩」）
    oldest = history[:N]
    summary = llm_summarize(oldest)  # 多花一次模型调用

    # ④ 用一条摘要替换掉那 N 轮原文，最近的几轮原文照旧保留
    return [{"role": "system", "content": f"早前对话摘要：{summary}"}] + history[N:]
```

逐块对照看：第 ① ② 步是「水位检测」——没满就别动；第 ③ 步是**压缩**的核心，`llm_summarize` 那一次额外调用，就是压缩比裁剪多花的成本；第 ④ 步用一条摘要顶替掉一大摞原文，窗口当场宽松——同时**保留最近几轮原文**，因为近期对话往往最相关、不该被概括掉。

如果某些内容根本不值得总结（纯属噪声），那就别走压缩，直接**裁剪**掉更划算——比如一个最简单的滑动窗口：

```python
def slide(history):
    # 只保留最近 N 轮，最旧的直接丢弃（这是「裁剪」，零额外成本）
    return history[-N:]
```

注意这两段的取舍正是上一节的主题：`llm_summarize` 那条路保住了要点但多花一次调用；`history[-N:]` 这条路零成本但被切掉的内容彻底没了。**要点还有用就走上面那段，确实没用了就走这段。**

## 容易踩的坑

<Callout type="trap">
  **现象**：「我图省事，每轮都把全部历史一股脑塞进去，结果对话一长，又贵又慢、效果还变差。」
  **原因**：放任历史无限膨胀——迟早顶到 token 上限被截断丢信息，token
  越烧越多更贵更慢，长上下文还触发 lost in the middle 把重点淹没。
  **修法**：设个阈值主动管理——老历史压缩成摘要、无关内容裁剪掉，把窗口控制在合理水位，别被动等它溢出。
</Callout>

<Callout type="trap">
  **现象**：用滑动窗口只留最近 N 轮，聊着聊着模型「忘了」前面交代过的关键设定。
  **原因**：纯滑动窗口是「最旧就删」，更早但重要的设定会被和普通对话一起滑出去——它分不清「旧」和「没用」。
  **修法**：别只靠裁剪。把要长期记住的关键设定**压缩成一条摘要钉在前面**（或写进系统提示），让它不随滑动被丢掉；滑动窗口只用来淘汰近期的普通寒暄。
</Callout>

<Callout type="trap">
  **现象**：一条「绝不能漏」的硬指令明明写进了上下文，模型却老不照做。
  **原因**：这条指令被埋在了一大段长上下文的**正中间**——正好撞上 lost in the
  middle，模型对中间最不上心，等于没看见。
  **修法**：把关键指令挪到上下文的**开头或结尾**（比如放进系统提示开头、或在末尾再强调一遍），躲开中间这个注意力盲区。
</Callout>

## 小结

- 上下文窗口是一笔有限的 token 预算，系统提示 / 历史 / 检索 / 工具结果 / 输出余量都从这一根里出
- 窗口装满会截断丢信息，且 token 越多越贵越慢——不能放任它一直涨
- 压缩把老历史总结成摘要（保信息、多花一次调用），裁剪直接丢掉（省成本、彻底丢失）；要点还有用就压缩，没用了就裁剪
- 管理窗口就是「填满 → 压缩腾空 → 继续」的主动循环，不是被动等溢出
- lost in the middle：长上下文里中间最易被忽略，关键信息放开头或结尾

<Exercises>

**练习 1（选型 · 实践题）** 下面两种内容，分别该用「压缩」还是「裁剪」来腾空间？为什么？
（甲）前 20 轮里，顾客陆续交代了订单号、是 VIP、想退哪件商品——这些后面办退货还要用。
（乙）历史里混了一堆「在吗」「好的谢谢」之类的寒暄和重复确认。

<Answer>

- **甲 → 压缩**：这些是之后还要用的关键信息，直接裁剪会把它们丢掉。应把这几轮总结成一条保留要点的摘要（如「顾客是 VIP，订单号 X，想退某商品」），既腾出空间又不丢信息。
- **乙 → 裁剪**：纯属噪声、之后用不上，直接删掉最划算——省下压缩要花的那次模型调用。不必为没价值的寒暄专门做总结。

要点：选压缩还是裁剪，判据就一条——**这段内容之后还有没有用**。有用就压缩（保住要点），没用就裁剪（省成本）。

</Answer>

**练习 2（设计窗口预算）** 你要给一个会长期聊下去的客服 agent 分配上下文窗口预算。下面四块：系统提示、历史对话、检索资料、留给输出的余量——你会按什么思路决定各自的额度？

<Answer>

一个合理的思路：

- **先扣下「输出余量」**：必须保证模型有地方写回答，否则把输入塞满了它就无法生成——这块要最先留出。
- **系统提示给一块固定且精简的额度**：人设、规矩、红线越精炼越好，它每轮都占预算，臃肿就是持续浪费。
- **历史对话设一个上限（阈值）**：因为它会膨胀，必须压住——一旦超阈值就压缩老历史 / 裁剪无关内容，把它控制在水位内。
- **检索资料按需占用剩下的空间**：用多少给多少，别一次塞太多挤掉别人。

核心一句话：**先留输出、压住会膨胀的历史**，其余按需分配。

</Answer>

**练习 3（思考题）** 「现在模型的上下文窗口越来越大了，所以根本不用管理。」这句话哪里不对？

<Answer>

不对。理由有三：

- 窗口再大也是**有限的**，长对话迟早把它填满，问题只是被推后、没被取消。
- token 越多，每次调用**越贵、越慢**——大窗口塞满的代价更高。
- 窗口越长越容易触发 **lost in the middle**，关键信息被淹没在中间反而更难被注意到。

所以窗口变大不等于不用管理；恰恰相反，在大窗口里更要主动压缩、裁剪、把关键信息摆到两端。

</Answer>

</Exercises>
