## 先打个比方

想象你请人帮你办一件事。你只甩一句「你看着办」，对方多半办得一塌糊涂——不是他笨，是你没说清楚。换个说法：先告诉他**你希望他以什么身份来办**（「就当你是我们店的老客服」），再讲清**到底要办成什么样**，给他**一个办好的范例**照着学，最后让他**别急着拍板、先把思路理一遍再动手**——同一个人，办出来的结果天差地别。

模型也一样。它本身的「脑子」没变，但你**喂给它的那段话**写得好不好，直接决定它办成什么样。把任务交代清楚、给个范例、让它先想再答——这门「把话说到位」的手艺，就是这一章要讲的东西。

它还是这一整篇（上下文工程）里**最容易上手、最立竿见影**的一环：不用改模型、不用接数据库，只要把喂进去的那段话改好，输出当场就不一样。所以我们从它开篇。

## 第一个概念：提示工程，调的是「喂给模型的那段话」

把上面那件事翻译成技术语言，就是 <Term def="提示工程（Prompt Engineering）：通过精心设计「喂给模型的那段输入文字」来引导模型给出更好输出的手艺。它不改模型权重，只改输入——是上下文工程里最直接、最可控的一环。" id="prompt-engineering">提示工程</Term>。注意它的边界：它**不碰模型本身**，只打磨你递进去的那段话。模型是同一个模型，换一段写得更好的话，输出就脱胎换骨——这正是它「最可控」的原因：成本最低、反馈最快、人人能改。

那一段「写得好」的话，长什么样？它不是随口一句，而是由几块固定的部件拼成的：

<PromptAnatomyDiagram />

盯住这六块：**角色**（让它扮谁）、**任务**（到底要它做什么）、**上下文**（做这件事要的背景料）、**约束**（边界与禁区）、**示例**（给一两个范例照着学），以及 <Term def="输出格式（Output Format）：提示里对「答案该长什么样」的明确约定，比如「只输出一个词」「分两行」「用 JSON」。把格式说死能让输出整齐、好被下游程序解析。" id="output-format">输出格式</Term>（答案长什么样）。不是每个提示都要齐六块，但你越缺哪块、模型就越容易在哪块上跑偏。提示工程的日常，就是照着这张清单，看自己的提示**缺了哪块、该补哪块**。

## 第二个概念：清晰具体——同样一句话，模糊和具体差到天上

六块里最容易偷懒、也最致命的，是把**任务**写得含含糊糊。模型不会读心，你说得有多含糊，它就猜得有多离谱。下面拖动滑块，对比同一个需求的「模糊版」和「具体版」分别会得到什么：

> 猜一猜：左边那句模糊的「帮我写个文案」，模型会写出什么？它知道是给谁看、要多长、什么口吻吗？拖到右边看看，把同样的需求说具体之后，输出会规整多少。

差别就一句话：**模糊提示把决策权全甩给了模型，具体提示替它把关键决策都定死了。** 你每多写一个具体约束（受众、长度、口吻、范围），就少给模型一次跑偏的机会。这是提示工程里回报最高的习惯——几乎不花成本，却最管用。

## 第三个概念：少样本——与其费劲描述，不如直接给个范例

有些东西，用话描述特别费劲，给个例子却一秒说清——比如「我要的输出格式」。这就引出 <Term def="少样本（few-shot）：在提示里塞进几个「输入 → 期望输出」的范例，让模型照着这些例子的格式和口径来做新任务。和它相对的是零样本——一个例子都不给，直接让模型做。" id="few-shot">少样本</Term>：在提示里先放几个「输入 → 输出」的范例，模型照着学。它的对照面叫 <Term def="零样本（zero-shot）：一个范例都不给，只描述任务就让模型直接做。简单任务够用，但格式 / 口径稍复杂时模型容易凭自己的理解瞎猜。" id="zero-shot">零样本</Term>——一个例子都不给。

<FewShotDiagram />

看出门道了吗：零样本时模型没见过你要的格式，只能凭自己的理解瞎猜，输出又长又不对版；少样本时它有范例可循，照着套就齐整了。**经验法则：凡是格式 / 口径稍微复杂一点的任务，给两三个范例，往往比堆一大段格式描述更管用、更省事。**

## 第四个概念：思维链——让它先一步步想，再给答案

最后一招专治「复杂推理一上来就答错」。<Term def="思维链（Chain-of-Thought，CoT）：在提示里要求模型把推理过程一步步写出来、最后再给答案，而不是张口就报结论。摊开中间步骤后，每步都可检验，复杂推理 / 多步计算的准确率明显提升。" id="chain-of-thought">思维链</Term> 的做法极简单：在提示里加一句「请一步步推理，再给出答案」，逼模型把中间过程摊开。

<CotDiagram />

道理很朴素：复杂题逼模型「一步到位」，它容易在心算里漏掉一步（左栏漏算了「又买 2 篮」）；让它**先把步骤一条条写出来再下结论**，每一步都看得见、可检验，自然更准（右栏）。代价是输出更长、更慢——所以它用在**真需要推理的复杂题**上，简单事实问答用不着。

## 把四招连起来：系统提示 vs 单次任务提示

上面四招（具体、少样本、思维链，加上六部件）都是在打磨「那段话」。但这段话挂在哪儿，有两种挂法，新手常分不清：

一种是写成 <Term def="系统提示（System Prompt）：在每一轮对话开头都会自动带上的那段固定指令，用来设定 agent 的身份、目标、规矩、能用的工具和说话风格。它是 agent 的「人设与守则」，一次写定、轮轮生效。" id="system-prompt">系统提示</Term>——给 agent 立一次「人设与守则」（你是谁、目标、规矩、口吻），之后**每一轮都自动带上**。另一种是临时为**这一次任务**写的提示，办完就丢，下一次重写。同一套部件，差别只在「写一次轮轮复用」还是「一次性临时拼」。

<Callout type="tip">
  经验法则：**稳定不变的写进系统提示，因任务而变的留给单次提示。**
  角色、口吻、红线、通用规矩——一次写进系统提示，省得每轮重复；而「这次具体要处理的内容、这次特有的少样本范例」则放进单次任务提示。给
  agent
  写系统提示尤其要克制：它每轮都占上下文窗口，写得越臃肿越贵、越容易稀释重点。
</Callout>

## 动手看：一个提示怎么一步步「精化」到能用

前面几招是分开讲的，真做事时它们是**一块块叠上去**的。下面这个动画把这个过程演给你看——任务是「写一句给老顾客的退货提醒」，从一句模糊提示开始，每一步加一块部件，看右边的输出怎么一截截变好。

<PromptRefinementDiagram />

盯住一件事：左边每加一块部件（角色 → 上下文 → 约束 → 示例 → 格式），右边的输出就**好一截**，底下的质量条就**长一段**。提示工程的增益不是某一句「魔法咒语」带来的，而是这样**一块块攒**出来的——这也正是为什么它「最可控」：每一块的效果你都能单独看见、单独调。

## 用代码看一个「带范例 + 定格式」的提示长什么样

上面动画里精化好的提示，落到代码里就是一个 messages 数组：开头是系统提示（人设 + 规矩），中间塞几条少样本范例，最后才是这次真要处理的输入。下面这段就是它的骨架：

```python
messages = [
    # ① 系统提示：每轮都带的人设 + 规矩（含输出格式约束）
    {"role": "system", "content":
        "你是资深电商客服。判断顾客这一单能否退并给出理由。"
        "只依据给定政策；政策没写的就说「需人工」。"
        "输出两行：① 能否退 ② 一句话理由。"},

    # ② 少样本范例：给一组「输入 → 期望输出」让模型照着学
    {"role": "user", "content": "未拆封，下单第 3 天，普通商品。"},
    {"role": "assistant", "content": "① 可退\n② 在 7 天可退期内，且未拆封。"},

    # ③ 这次真要处理的输入（单次任务部分）
    {"role": "user", "content": "已拆封，下单第 9 天，生鲜。"},
]
```

逐块对照看：`system` 那条就是**系统提示**——角色、约束、输出格式都写在这里，**每轮复用**；中间一对 `user` / `assistant` 是一条**少样本范例**，告诉模型「碰到这种输入，就照这个两行格式答」；最后那条 `user` 才是**这一次**真要判断的订单，属于单次任务部分。

如果这道题需要推理（比如政策有多条、要逐条核对），只需在系统提示里加一句，就把**思维链**也接上了：

```python
    # 在系统提示末尾追加一句，要求先推理再下结论
    "...先逐条核对政策再下结论，最后只输出上面规定的两行。"
```

注意这一句的措辞：**先**逐条核对（摊开推理过程），但**最后只输出**规定的两行格式。这样既拿到了思维链的准确率，又不会让一堆推理过程污染最终给用户看的输出——把「想」和「答」分开，是思维链落地时的常用技巧。

## 容易踩的坑

<Callout type="trap">
  **现象**：「我把提示写得老长老长，恨不得把所有要求都塞进去，输出反而更糊、还更贵了。」
  **原因**：把「提示越长越好」当成了真理。其实啰嗦的提示会**稀释重点**——关键约束被淹没在一堆废话里，模型抓不住，还白白多烧
  token。 **修法**：长 ≠
  好，**清晰具体**才好。删掉空话套话，每一句都对准一个明确决策（受众 / 长度 /
  口吻 / 范围 / 格式）；稳定的挪进系统提示复用，别在单次提示里反复堆。
</Callout>

<Callout type="trap">
  **现象**：复杂的输出格式，光用文字描述「我要这样那样的格式」，模型还是时对时错、对不齐。
  **原因**：复杂格式靠「描述」远不如靠「范例」准。文字描述容易有歧义，模型只能猜你到底要哪种；而一个具体的
  **少样本范例**，把格式直接摆在它面前，照着套就行。
  **修法**：与其多写两段格式说明，不如给两三个「输入 →
  期望输出」的范例（few-shot）。复杂格式 / 口径，给例子几乎总比给描述强。
</Callout>

<Callout type="trap">
  **现象**：让模型做需要几步推理的题（多步计算、逻辑判断），它张口就给个结论，经常是错的。
  **原因**：逼它「一步到位」，中间步骤全在心算里走，漏一步就错——它没把过程摊开，你也无从发现哪步错了。
  **修法**：对真需要推理的复杂题，加一句「请一步步推理再给答案」（**思维链**），让它先摊开过程；简单事实问答则不必，免得徒增啰嗦与开销。
</Callout>

{/* ===== 8. 小结与练习｜spec §八：小结 ≤5 条与目标呼应；练习含 1 道「改提示」实践题；答案折叠 ===== */}

## 小结

- 提示工程 = 打磨「喂给模型的那段话」，不改模型、成本最低、反馈最快——是上下文工程里最可控的一环
- 一个好提示由六块拼成：角色 / 任务 / 上下文 / 约束 / 示例 / 输出格式，缺哪块就在哪块跑偏
- 清晰具体替模型定死关键决策；少样本（few-shot）用范例教它格式；思维链让它先想再答、专治复杂推理
- 稳定不变的写进系统提示（轮轮复用），因任务而变的留给单次任务提示
- 提示的增益是一块块「精化」攒出来的，不是某句魔法咒语——每块效果都能单独看见、单独调

**练习 1（改提示 · 实践题）** 下面是一个烂提示，请你按「六部件」清单把它补全成一个像样的提示：

> 「帮我把用户评论分个类。」

提示：想想它缺了哪些部件——它扮谁？分成哪几类？有没有给范例？要什么输出格式？

<Answer>

它几乎只有「任务」一块，且任务本身还很含糊。按六部件补全后，大致长这样：

- **角色**：你是一名电商评论分析助手。
- **任务**：把每条用户评论分类。
- **上下文 / 约束**：类别只有三种——「正面 / 负面 / 中性」，无法判断时归「中性」；只依据评论文字本身。
- **示例（少样本）**：`真香，下次还买 → 正面`；`等了半个月还没到 → 负面`。
- **输出格式**：每条只输出一个词（正面 / 负面 / 中性），不要解释。

要点：原提示把「分成哪几类、什么格式、拿不准怎么办」全甩给了模型去猜。补上约束（限定类别）、示例（给范例定口径）、输出格式（只要一个词）之后，模型才有谱、输出才稳定可用。

</Answer>

**练习 2（辨析题）** 下面三件事，分别该靠「清晰具体」「少样本」还是「思维链」来解决？为什么？
（A）模型总把回答写得太长，你只想要一句话。
（B）模型分类时格式总对不齐，你要它严格输出固定的几个标签之一。
（C）一道要算好几步的应用题，模型张口就报错答案。

<Answer>

（A）靠**清晰具体**——在提示里把约束写死：「只用一句话、≤ 20 字回答」。这是把含糊的期望变成明确约束。
（B）靠**少样本**——给两三个「输入 → 标签」范例，让它照着对版；复杂 / 固定的格式，范例比描述更准。
（C）靠**思维链**——加一句「请一步步推理再给答案」，逼它摊开中间步骤，多步计算才不容易漏错。
判别诀窍：要它「按规矩来」→ 说清楚（具体）/ 给范例（少样本）；要它「别漏算」→ 让它先想（思维链）。

</Answer>

**练习 3（思考题）** 给一个客服 agent 写提示时，下面这些内容，哪些该放进「系统提示」、哪些该放进「单次任务提示」？为什么？
（甲）「你是某电商的客服，语气亲切、绝不承诺政策外的退款。」
（乙）「这位顾客的订单号是 20231001，问的是能不能退。」
（丙）「输出永远分两行：能否退 + 一句话理由。」

<Answer>

- **甲、丙 放系统提示**：角色 / 口吻 / 红线（甲）和固定输出格式（丙）都是**稳定不变、每轮都该生效**的规矩——写进系统提示一次定死、轮轮复用，省得每次重复，也保证一致。
- **乙 放单次任务提示**：具体订单号、这次的问题是**因任务而变**的内容，每次都不一样，属于这一次性的输入。

要点：判断标准就一条——**这条内容会不会每轮都一样**。会，放系统提示；不会，放单次提示。还要记得系统提示每轮都占上下文窗口，别把一次性的东西塞进去，徒增成本又稀释重点。

</Answer>

## 名词解释

<Glossary>
  <GlossaryItem term="提示工程（Prompt Engineering）" id="prompt-engineering">
    通过精心设计「喂给模型的那段输入文字」来引导模型给出更好输出的手艺。它不动模型权重，只打磨输入——所以成本最低、反馈最快、人人能改，是上下文工程里最直接、最可控的一环。详见本章「第一个概念」。
  </GlossaryItem>
  <GlossaryItem term="系统提示（System Prompt）" id="system-prompt">
    每一轮对话开头都会自动带上的那段固定指令，用来设定 agent
    的身份、目标、规矩、可用工具和说话风格——好比给它立一份「人设与守则」，一次写定、轮轮生效。稳定不变的内容才该写进来。详见本章「把四招连起来」。
  </GlossaryItem>
  <GlossaryItem term="少样本（few-shot）" id="few-shot">
    在提示里塞进几个「输入 →
    期望输出」的范例，让模型照着这些例子的格式和口径做新任务。复杂格式靠范例比靠文字描述更准。详见本章「第三个概念」。
  </GlossaryItem>
  <GlossaryItem term="零样本（zero-shot）" id="zero-shot">
    一个范例都不给，只描述任务就让模型直接做。简单任务够用，但格式 /
    口径稍复杂时模型容易凭自己的理解瞎猜——它是少样本的对照面。详见本章「第三个概念」。
  </GlossaryItem>
  <GlossaryItem term="思维链（Chain-of-Thought，CoT）" id="chain-of-thought">
    在提示里要求模型把推理过程一步步写出来、最后再给答案，而不是张口就报结论。摊开中间步骤后每步可检验，多步计算
    /
    复杂推理的准确率明显提升；代价是输出更长更慢，简单问答用不着。详见本章「第四个概念」。
  </GlossaryItem>
  <GlossaryItem term="输出格式（Output Format）" id="output-format">
    提示里对「答案该长什么样」的明确约定，比如「只输出一个词」「分两行：能否退 +
    理由」「用
    JSON」。把格式说死能让输出整齐、好被下游程序解析，是六部件里最实用的一块。详见本章「第一个概念」。
  </GlossaryItem>
</Glossary>
