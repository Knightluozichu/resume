## 先打个比方

还是那位帮你办事的同事。你让他「订明天去上海最便宜的高铁票」——这事他没法一步办完：得先查有哪些车次、看哪趟最便宜，再去下单。

更现实的是，事情常常不顺：他挑中最便宜那趟去订，结果一看**售罄了**。这时靠谱的同事会怎么办？他不会卡死，也不会傻乎乎再订一遍同一趟，而是**改订次便宜的那趟**，订成了才回来交差。

这一章讲的就是：怎么让 AI 像这位同事一样，**试一下、看结果、不行再换个法子，反复转到办成为止**。少了这种「反复转圈」的本事，AI 就只能一锤子买卖——查一次、订一次，一遇到「没票了」就当场卡住，啥也办不成。

## 第一个概念：为什么一次调用不够，得让它转圈

上一章我们让 LLM 学会了在一次调用里查资料、调工具——可那始终是**一次**。问题在于，很多任务的步数是**不定**的：订票要先查再订，查资料可能要查好几个来源，出了错还得重试。一次调用只能想一步、做一件事、看一个结果，办不完这种活。

解决办法就是把「一次调用」放进一个会反复转的环里，这个环就是 <Term def="智能体循环（agentic loop）：智能体反复执行「思考下一步 → 行动（调工具）→ 观察结果」的过程，由 LLM 自己决定每一圈干什么、以及何时结束。一次办不完的任务，靠它一圈圈转着把活干完。" id="agentic-loop">智能体循环</Term>。第 1 章我们已经见过它转一圈的样子（查天气）；这一章要看的是它**反复转很多圈**、且每一圈都比上一圈更接近目标。每转一圈，做一次 <Term def="迭代（Iteration）：循环里的「一圈」。智能体每做一次「思考 → 行动 → 观察」，就完成一次迭代；一次办不完就再迭代一次。" id="iteration">迭代</Term>——一次不够，就再迭代一次。

## 第二个概念：ReAct——把一圈拆成「想、做、看」三段

那循环转的「一圈」里头，到底发生了什么？业界最经典的拆法叫 <Term def="ReAct：Reason（推理 / 思考）+ Act（行动）的合称。指智能体在循环里把「思考」和「行动」交替进行——想一步、做一步、看结果、再想下一步，而不是只想不做或只做不想。" id="react">ReAct</Term>，意思是 **Reason（推理）+ Act（行动）交替着来**。它把一圈拆成清清楚楚的三段：

<ReActStepDiagram />

这三段是循环里每一圈的内部构造：

- 第一段是 <Term def="Thought（思考）：ReAct 一轮里的推理段。LLM 看着当前上下文，盘算「下一步该干啥」——选哪个工具、传什么参数，或判断「够了，可以收尾」。" id="thought">Thought（思考）</Term>：LLM 盘算「下一步该干啥」，比如「先查有哪些车次」。
- 第二段是 <Term def="Action（行动）：ReAct 一轮里的行动段。按 Thought 选定的工具和参数，真去执行一次调用，比如 search_trains(明天, 上海)。" id="action">Action（行动）</Term>：照思考选好的工具真去执行一次，比如调 `search_trains(明天, 上海)`。
- 第三段是 <Term def="Observation（观察）：ReAct 一轮里的观察段。看 Action 返回了什么——工具的结果，比如「G1 已售罄」。这个结果会喂回上下文，成为下一轮思考的依据。" id="observation">Observation（观察）</Term>：看工具返回了什么，比如「G1、G7 都 553 元」。

要害在最后那条**回流的箭头**：这一轮「看」到的结果，会喂回去支撑**下一轮的「想」**。正是这条回流，让循环不是机械重复，而是一圈比一圈聪明。

## 第三个概念：多轮迭代——结果喂回，思考随之改变

把上面这三段反复转好几圈，就是这一章的主角——<Term def="多轮迭代（Multi-turn Iteration）：一次工具调用不够时，智能体反复转圈：每轮做完看结果、把结果喂回上下文，下一轮基于新结果再想再做，一轮接一轮直到任务完成。" id="multi-turn">多轮迭代</Term>。它和前两章最不一样的地方有两点，盯住它们就抓住了循环的精髓：

- **每轮的观察都喂回上下文**——所以上下文**越滚越长**，智能体越转越「懂」当前任务。
- **下一轮的思考基于上一轮的结果**——订 G1 失败了，下一轮才会改订 G7。这份「看了结果再随机应变」，就是循环区别于一锤子买卖的根本。

这个过程光靠文字很难想象，下一节的交互动画会用「订高铁票」把它一圈圈拆开给你看。

## 第四个概念：循环何时该停

循环既然会一圈圈转，就必须有个**刹车**——否则它可能停不下来。智能体循环的 <Term def="终止条件（Termination Condition）：让循环停下来的判据。常见三种：LLM 判定任务完成正常退出、转够 max steps 强制停、出错没法继续兜底退出。" id="termination">终止条件</Term> 通常有三条出口：

<LoopTerminationDiagram />

- **正常退出**：LLM 自己判定任务办完了，主动收尾、给出答案。这是最理想的出口。
- **强制停**：转够了预设的 <Term def="max steps（步数上限）：循环转圈次数的硬性上限。转够这么多圈还没收尾就强制停下、返回兜底回答。作用是防止 LLM 始终判不出「完成」而无限转下去。" id="max-steps">max steps</Term>，不管完没完都硬性叫停、给个兜底回答。
- **出错兜底**：工具报错、没法再继续时，退出循环、返回错误说明。

前一条是「成功出口」，后两条是人留的**硬闸**。为什么不能只靠第一条？因为 LLM 有时**始终判断不出「完成」**——任务办不到、工具反复报错、或它陷入纠结。这时若没有 max steps 这道闸，它就会无限转下去，烧钱又卡死。这一点是下面误区里要专门点的坑。

## 动手看：一个任务怎么转 3 轮才办成

前面讲了「多轮迭代」和「结果喂回」，但只有亲眼看它一圈圈转，才能真正抓住循环的精髓。下面这个动画用「帮我订明天去上海最便宜的高铁票」把它拆成 10 步——这事一次调用搞不定，得转整整 3 轮。

> 猜一猜：动画第 ⑥ 步，第 2 轮订 G1 失败（「G1 已售罄」）。那么第 3 轮 LLM 的思考会变成什么？是再订一次 G1，还是改主意？先想一想——它能看到上一轮的结果吗？再单步播放，看它是不是基于「G1 没票」改订了 G7。

<MultiTurnLoopDiagram />

盯住两件事：一是**左侧的上下文栏每轮加一条、越滚越长**——每轮看到的结果都被喂了回去；二是**右侧的思考每轮都在变**，而且变化都源自上一轮看到的结果（看到 G1 最便宜 → 想订 G1；看到 G1 售罄 → 改订 G7）。是它**自己**根据结果一轮轮调整路线，最后自己判定「订成了」跳出循环。这一圈圈相关地转下去，就是「多轮迭代」的全部秘密。

## 用代码看带刹车的循环骨架

上面的动画，落到代码里就是一个**带步数上限的 `for` 循环**。下面这段精简伪代码，是几乎所有智能体框架的共同骨架——把它读懂，多轮迭代和终止条件就都通了：

```python
def run_agent(user_goal, max_steps=10):
    # 上下文 = 智能体的记忆，先放进系统提示和用户目标
    context = [system_prompt, user_goal]

    for step in range(max_steps):        # 最多转 max_steps 圈（刹车在此）
        # ① 思考（Thought）：LLM 看着当前上下文，决定下一步
        thought = llm(context)

        # ② LLM 自己判断：任务办完了吗？办完就正常退出
        if thought.is_done:
            return thought.answer        # 出口一：LLM 判定完成

        # ③ 行动（Action）：照 LLM 选的工具和参数真去执行
        observation = act(thought.tool, thought.args)

        # ④ 观察（Observation）：把结果喂回上下文，进入下一圈
        context.append(thought)
        context.append(observation)

    # 转满 max_steps 还没收尾 → 强制停，给兜底回答
    return "抱歉，我尝试了多轮仍没能完成这个任务。"  # 出口二：max steps
```

逐句对照动画看：`for step in range(max_steps)` 就是那个会转多圈、但**最多转 max_steps 圈**的环；`llm(context)` 是每一轮的**思考**；`if thought.is_done` 是 LLM **自己**判断何时收尾（出口一）；`act(...)` 是**行动**；`context.append(observation)` 是**观察喂回**——下一圈 `llm(context)` 就能看到它了，这正是「上下文越滚越长、下一轮基于上一轮」的代码落点。

最后那行 `return "抱歉……"` 是第二个出口：`for` 转满 max_steps 还没等到 `is_done`，就强制停、返回兜底回答。两个出口缺一不可——少了第一个它不会主动收尾，少了第二个它可能无限转。至于第三个「出错兜底」出口，只需把 `act(...)` 包进 `try / except`，工具报错时也跳出循环即可，这里为聚焦先省略。

## 容易踩的坑

<Callout type="trap">
  **现象**：一个 agent 跑起来就停不下来，反复调同一个工具、烧了一堆 token
  还不返回——典型的{" "}
  <Term
    def="智能体停不下来、原地打转的状态：要么始终判不出完成而无限转，要么反复调同样的工具拿同样的结果却不收尾。常因没设 max steps 或没把结果喂回上下文导致。"
    id="dead-loop"
  >
    死循环
  </Term>
  。 **原因**：循环没有 max steps 这道闸。LLM
  始终判断不出「任务完成」（任务办不到、或它陷入纠结），又没有步数上限兜底，于是无限地「思考→行动→观察」转下去。
  **修法**：给循环加 max steps（`for step in range(max_steps)` 而非 `while
  True`），转满强制停、返回兜底回答；再加一个出错兜底出口。自主性可以给
  LLM，但「最多转几圈」这道刹车必须由人留着（回看本章终止条件图）。
</Callout>

<Callout type="trap">
  **现象**：agent
  像「失忆」一样原地打转——每轮都做同样的判断、调同样的工具、拿同样的结果，永远不往前走。
  **原因**：每轮的 observation
  没有真正喂回上下文。下一轮的思考看不到上一轮的结果，自然每次都从头来过、重复同一步。
  **修法**：确保每轮都把工具结果 append
  回上下文（`context.append(observation)`），让下一轮的 `llm(context)`
  能看到它。循环能「往前走」的前提，就是结果喂回——漏了它，多轮迭代就退化成无效的重复。
</Callout>

## 小结

- 步数不定的开放任务一次调用搞不定，得靠循环一圈圈转着把活干完
- ReAct 把一圈拆成三段：Thought 想下一步、Action 真去做、Observation 看结果
- 多轮迭代的精髓：观察逐轮喂回（上下文越滚越长）、下一轮思考基于上一轮结果
- 循环靠三条出口停下：LLM 判完成正常退出、转够 max steps 强制停、出错兜底退出
- 没有 max steps / 不喂回结果 → 死循环或原地打转，是循环最常见的两个坑

<Exercises>

**练习 1（改循环 · 思考题）** 本章代码骨架里只实现了两个出口（LLM 判完成、max steps）。现在要补上第三个出口「出错兜底」：当某次 `act(...)` 调工具报错时，不要让整个程序崩溃，而是优雅地退出循环、返回一句错误说明。请改写这段循环。

<Answer>

把 `act(...)` 包进 `try / except`，捕获到异常就跳出循环、返回兜底说明：

```python
def run_agent(user_goal, max_steps=10):
    context = [system_prompt, user_goal]
    for step in range(max_steps):
        thought = llm(context)
        if thought.is_done:
            return thought.answer            # 出口一：完成
        try:
            observation = act(thought.tool, thought.args)
        except ToolError as e:
            return f"工具出错，无法继续：{e}"   # 出口三：出错兜底
        context.append(thought)
        context.append(observation)
    return "抱歉，我尝试了多轮仍没能完成这个任务。"  # 出口二：max steps
```

关键改动：用 `try / except` 包住行动那一步，工具报错时直接 `return` 一句错误说明、跳出循环。这样三个出口就齐了——LLM 判完成（正常）、max steps（强制）、出错兜底（异常）。三道出口配合，循环既能正常收尾，又在最坏情况下停得下来。

</Answer>

**练习 2（辨析题）** 下面两段循环各有什么问题？分别会导致哪种「停不下来」？
（A）`while True:` 里只有「思考→行动→观察」，没有任何退出判断。
（B）`for step in range(max_steps):` 里有 max steps，但每轮**忘了**把 observation 加回 context。

<Answer>

（A）会**无限转**——`while True` 没有出口，LLM 也没机会主动收尾，循环永远不停，典型死循环。修法：换成带 max steps 的 `for`，并加上 `if thought.is_done: return` 的正常出口。

（B）会**原地打转**——虽然有 max steps 兜住、不会真的无限转，但每轮思考都看不到上一轮的结果，于是反复做同样的判断、调同样的工具，白白转满 max_steps 也没进展。修法：每轮都 `context.append(observation)`，让结果喂回，下一轮才能基于它往前走。

判别诀窍：没出口 → 无限转；有出口但结果不喂回 → 转满步数也是空转。两道都补上，循环才既停得下来、又真往前走。

</Answer>

**练习 3（应用题）** 你要给一个「帮用户查并预约最近的核酸检测点」的 agent 设 max steps。你会怎么估这个值？设太小、太大各有什么坏处？

<Answer>

先估「正常需要几轮」：查附近检测点（1 轮）→ 挑一个看有没有空号（可能 1～2 轮）→ 预约（1 轮）→ 约满了换一个再约（多 1～2 轮）。正常 4～6 轮能办完，那 max steps 设个 8～10、留点余量比较稳。

- **设太小**（比如 3）：稍微不顺（第一个点约满了要换）就被强制叫停，本来能办成的事也办不成，属于「误杀」。
- **设太大**（比如 100）：万一 agent 失控原地打转，要烧很多圈、等很久才停，浪费 token 和用户时间。

权衡：按任务复杂度估正常轮数再留余量，复杂任务给大些、简单任务给小些；更进一步还能加「连续几轮无进展就提前停」的策略，比单纯调大调小更聪明。核心是：max steps 是兜底闸，宁可让正常任务有充裕余量，也要保证失控时停得下来。

</Answer>

  <GlossaryItem term="智能体循环（agentic loop）" id="agentic-loop">
    智能体反复执行「思考下一步 → 行动（调工具）→ 观察结果」的过程，由 LLM
    自己决定每圈干什么、什么时候结束。一次办不完的任务，就靠它一圈圈转着把活干完——这是「智能体」会自己干活的根本机制。详见本章「第一个概念」。
  </GlossaryItem>
  <GlossaryItem term="迭代（Iteration）" id="iteration">
    循环里的「一圈」。智能体每做一次完整的「思考 → 行动 →
    观察」，就完成一次迭代；一次办不完，就再迭代一次。详见本章「第一个概念」。
  </GlossaryItem>
  <GlossaryItem term="ReAct" id="react">
    Reason（推理 / 思考）+
    Act（行动）的合称。指智能体在循环里把「想」和「做」交替进行——想一步、做一步、看结果、再想下一步，而不是只想不做或只做不想。它是循环里每一圈的内部构造。详见本章「第二个概念」。
  </GlossaryItem>
  <GlossaryItem term="Thought（思考）" id="thought">
    ReAct 一轮里的推理段。LLM
    看着当前上下文，盘算「下一步该干啥」——选哪个工具、传什么参数，或判断「够了，可以收尾」。简单说：动手之前先想清楚。详见本章「第二个概念」。
  </GlossaryItem>
  <GlossaryItem term="Action（行动）" id="action">
    ReAct 一轮里的行动段。照 Thought 选定的工具和参数，真去执行一次调用，比如
    search_trains(明天,
    上海)。它把「想」落地成一次真实动作。详见本章「第二个概念」。
  </GlossaryItem>
  <GlossaryItem term="Observation（观察）" id="observation">
    ReAct 一轮里的观察段。看 Action 返回了什么——工具的结果，比如「G1
    已售罄」。这个结果会喂回上下文，成为下一轮思考的依据。详见本章「第二个概念」。
  </GlossaryItem>
  <GlossaryItem term="多轮迭代（Multi-turn Iteration）" id="multi-turn">
    一次工具调用不够时，智能体反复转圈：每轮做完看结果、把结果喂回上下文，下一轮基于新结果再想再做，一轮接一轮直到任务完成。它和单轮问答、一次增强调用最大的不同，是「转很多圈、且圈圈相关」。详见本章「第三个概念」。
  </GlossaryItem>
  <GlossaryItem term="终止条件（Termination Condition）" id="termination">
    让循环停下来的判据。常见三种出口：LLM 判定任务完成正常退出、转够 max steps
    强制停、出错没法继续兜底退出。前者是成功出口，后两者是人留的硬闸。详见本章「第四个概念」。
  </GlossaryItem>
  <GlossaryItem term="max steps（步数上限）" id="max-steps">
    循环转圈次数的硬性上限。转够这么多圈还没收尾，就强制停下、返回兜底回答。作用是防止
    LLM
    始终判不出「完成」而无限转下去——这是防死循环、防烧钱的必备闸门。详见本章「第四个概念」。
  </GlossaryItem>
  <GlossaryItem term="死循环（失控循环）" id="dead-loop">
    智能体停不下来、原地打转的状态：要么始终判不出完成而无限转，要么反复调同样的工具拿同样的结果却不收尾。表现是不返回、白烧钱。常因没设
    max steps
    或没把结果喂回上下文导致，是循环最常见的坑。详见本章「容易踩的坑」。
  </GlossaryItem>
</Glossary>
