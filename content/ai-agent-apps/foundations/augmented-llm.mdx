## 先打个比方

上一章我们把只会背书的 AI 比作一个被关在小黑屋里的实习生：聪明，但手里啥也没有。这一章不急着让他「自己干活」，先做一件更基础的事——**给他配齐家伙**。

给他一个能随时翻的**资料柜**：遇到不懂的，去柜子里抽出相关文件再答，而不是凭记忆瞎说。再给他一部**电话**：需要查实时信息时，他能拨出去问一声、把答案记下来。最后给他一个**笔记本**：这次聊到的要点先记着，重要的还能誊进档案，下次见你时翻出来接着聊。

人还是那个人，脑子一点没变——可配齐了资料柜、电话、笔记本之后，他能办的事完全不是一个量级了。这一章讲的就是：怎么给一个 LLM 配上这三样，让它从「只会续写文字」变成一块**真正能用**的积木。少了这一步，后面所有更复杂的智能体，都没有可靠的零件可搭。

{/* ===== 3. 概念讲解｜spec §三：一节一概念；术语首现 <Term def id> 写句中；新核心概念 ≤5 ===== */}

## 第一个概念：增强，加的是外面的壳，不是里面的模型

把上面那个「配齐家伙的实习生」翻译成技术语言，就是 <Term def="增强型 LLM（Augmented LLM）：在一个基础大语言模型（base LLM）外面包上检索、工具、记忆等能力而成的整体。base LLM 的权重一个字节都没改，是输入与可用动作被「外挂」扩展了，使它能查资料、能动手、能记事。" id="augmented-llm">增强型 LLM</Term>。它是搭建一切智能体与工作流的**原子单元**——最小、最基础的那块积木。

要害在「增强」这两个字加在哪里：

<AugmentedLlmDiagram />

注意中央那个 base LLM——它**还是它**，权重没动、没重新训练。变的是它外面那层壳：检索把外部资料喂进它的输入，工具让它的输出能触发真实动作，记忆让它跨轮还记得事。换句话说，我们没有把模型换成一个更聪明的，而是在原模型外面**包了一层能力**。

这件事的威力在于：同一个 base LLM，换上不同的资料柜、不同的工具、不同的笔记本，就能胜任完全不同的任务——退货客服、内部答疑、订单助理……**任务变了，只换外壳，不必重训模型**。这正是「增强型 LLM 是通用原子单元」的含义。

## 第二个概念：检索——先捞对资料，再带着资料回答

第一种增强是 <Term def="检索（Retrieval）：拿用户的问题去外部知识库里找出相关片段，把这些片段拼进上下文，再交给 LLM 作答。它让模型不必凭记忆瞎答，而是基于捞到的真实资料回答。" id="retrieval">检索（Retrieval）</Term>。它补的是裸 LLM「不知道私有 / 最新资料」这一短板——你们公司的退货政策、今天刚改的价目表，模型训练时根本没见过。

它的流程很直白，就四步：

<RetrievalFlowDiagram />

这套「先检索、后生成」的做法有个广为人知的名字，叫 <Term def="RAG（Retrieval-Augmented Generation，检索增强生成）：先从外部知识库检索出相关片段，再让 LLM 基于这些片段生成回答的范式。核心是「让模型答之前先有据可依」，而不是凭训练记忆作答。" id="rag">RAG</Term>。关键不在「有没有接知识库」，而在**捞回来的资料对不对**——捞错了，模型拿着错料照样答错。检索的更深一层（怎么切片、怎么召回、怎么排序）我们留到篇 5 专门讲，这里你只要记住直觉：**检索 = 答之前先去外面查准资料**。

## 第三个概念：工具——让它从「只会说」到「真能做」

第二种增强是 <Term def="工具（Tools）：LLM 能调用的外部函数或接口。LLM 产出一个结构化调用（调哪个、传什么参数），系统真正去执行，再把结果回填进上下文。它让模型能查实时数据、能触发真实动作，而不只是输出文字。" id="tools">工具（Tools）</Term>。它补的是裸 LLM「不能行动」这一短板。

工具和检索常被搞混，但分工很清楚：**检索是「读」**——从现成的知识库里捞静态资料（如政策文档）；**工具是「做」**——主动触发一次执行，去拿动态、实时的结果（如这一单此刻的物流状态）。一句话：检索查的是写好的资料，工具跑的是一次真实调用。

工具这条通路走三步：LLM 产出一个调用（比如 `check_order_status(订单号)`）→ 系统真去执行 → 把结果回填进上下文。这三步我们放进下一节的主动画里一起看；工具的设计与安全（篇 3 深讲）这里先按下不表。

## 第四个概念：记忆——关掉就忘的，和跨会话还记得的

第三种增强是**记忆（Memory）**，补的是裸 LLM「没有记忆」这一短板。但记忆不是一种东西，它分两类，新手最容易混：

<MemoryTypesDiagram />

一类是 <Term def="短期记忆（Short-term Memory）：活在上下文窗口内、限于单次会话的记忆。本轮对话、刚拿到的工具结果都属于它；会话一结束就忘，不会自动留到下次。" id="short-term-memory">短期记忆</Term>：它活在 <Term def="上下文窗口（Context Window）：LLM 一次能「看」的输入文字总量上限。本轮对话、检索片段、工具结果都要塞进这个窗口；塞超了就装不下或被截断。" id="context-window">上下文窗口</Term> 里，限于这一次会话——你这轮说过「我怕冷」，它本轮记得，可会话一关就忘了。另一类是 <Term def="长期记忆（Long-term Memory）：写进外部存储（数据库 / 向量库）、能跨会话保留的记忆。下次同一用户再来，系统能把相关条目调出来载入上下文。" id="long-term-memory">长期记忆</Term>：要专门写进外部存储，下次你再来，它还能把「你是 VIP、上月退过一次货」调出来。

区别就一句话：**短期记忆关掉会话就忘，长期记忆要写进外部存储才能跨会话还记得。** 把两者混为一谈——尤其是把一切历史都硬塞进上下文窗口当记忆——是后面会专门点的坑。记忆的存取细节同样留给篇 5。

## 动手看：三路增强如何汇成一次「强」调用

前面三种增强是分开讲的，但真实场景里它们往往**一起上**。下面这个动画用一个具体例子把它们汇到一处——任务是「根据我们公司的退货政策，回答顾客这单能不能退」。

> 猜一猜：如果跳过第 ② 步检索，第 ④ 步 LLM 凭什么答出「我们公司的退货政策」？先想一想——它训练时见过你家的政策吗？再单步播放，看看少了检索这一路，输入里会缺哪块料。

<AugmentedCallDiagram />

盯住一件事：检索、记忆、工具三条通路从三个方向**汇聚**进同一个 base LLM 的上下文（第②~⑤步），让这一次调用从「凭空乱答」变成「有据可依」，最后给出回答（第⑥步）。这和上一章不一样——第 1 章讲的是同一个环里**反复转圈**（循环迭代），这一章讲的是一次调用里**多路汇合**（汇聚增强）。一个是时间上的反复，一个是一次调用内的富化。把这两件事分清楚，你就抓住了这两章的分界。

## 用代码看一次增强调用的骨架

上面的动画，落到代码里就是「调模型之前先攒料、调完看要不要执行工具」。下面这段精简的伪代码，是几乎所有增强型 LLM 的共同骨架：

```python
def augmented_call(user_question, user_id):
    # ① 检索增强：拿问题去知识库捞相关片段
    docs = knowledge_base.retrieve(user_question)

    # ② 记忆增强：载入这位用户的长期记忆（跨会话还记得的）
    memory = long_term_memory.load(user_id)

    # ③ 把三方信息拼成「富化」上下文，再带着工具清单调一次 LLM
    context = [system_prompt, memory, docs, user_question]
    decision = llm(context, tools=[check_order_status])
    return decision
```

逐句对照动画看：`retrieve(...)` 是**检索**（第②步，捞政策片段）；`long_term_memory.load(...)` 是**记忆**（第③步，载入顾客历史）；这两路结果和系统提示、用户问题一起拼进 `context`，就是动画里三路「汇聚」进上下文的那一刻；`llm(context, tools=[...])` 则是带着工具清单调一次模型（第④步处理富化输入）。注意：到这里**模型只被调了一次**，没有循环——这就是「一次增强调用」。

那工具到底什么时候真执行？看 LLM 这一次的产出：

```python
    # 接上：LLM 这次产出的可能是「先调个工具」
    if decision.is_tool_call:
        # ④ 工具增强：系统真去执行，把结果回填进上下文
        result = run_tool(decision.tool, decision.args)
        context.append(decision)
        context.append(result)
        # 带着回填的结果，再调一次拿最终答案
        decision = llm(context, tools=[check_order_status])
    return decision.answer
```

`is_tool_call` 是 LLM **自己**决定「我需要先查一下订单状态」（对应动画第⑤步）；`run_tool(...)` 是系统真去执行、`context.append(result)` 是把结果回填——这正是工具通路「产出调用 → 执行 → 回填」三步。看出门道了吗：检索和记忆是**调模型前**就备好的料，工具是**调模型后**按它的要求去执行再回填的——三路殊途同归，都是为了把上下文喂得更足。至于「要不要在这外面再套一个 `while` 一圈圈转」，那是上一章的循环，是把多次这样的增强调用串起来的上层结构。

## 容易踩的坑

<Callout type="trap">
  **现象**：「我接了个向量库做检索，怎么答案还是经常不准、甚至胡编？」
  **原因**：把「接了知识库」当成了「答案一定准」，忽略了**检索质量**。捞回的片段如果不相关、不完整、过时，LLM
  拿着错料照样答错——「有没有检索」和「捞到的对不对」是两件事。
  **修法**：去看每次实际检索回了哪些片段、它们是否真覆盖了问题、来源是否最新；先治检索质量（切片、召回、排序），而不是怪模型（细节见篇
  5）。接库只是第一步。
</Callout>

<Callout type="trap">
  **现象**：把每一轮的全部历史都不断追加进上下文当「记忆」，跑久了越来越慢、越来越贵，还时不时答非所问。
  **原因**：混淆了短期记忆和长期记忆，把一切都堆进**上下文窗口**。历史无限增长会撑爆窗口（被截断、变贵变慢），无关内容还干扰模型抓重点。
  **修法**：分清两种记忆——本轮必要的留在短期（上下文）里，需跨会话保留的写进外部存储做长期记忆、用时**按需调取**相关条目，而不是把所有历史全量回灌。
</Callout>

## 小结

- 增强型 LLM = base LLM（权重不变）+ 外包三种能力，是搭一切智能体的原子单元
- 检索补「不知道私有 / 新资料」：先捞对资料再答，这套「先检索后生成」就是 RAG
- 工具补「不能行动」：LLM 产出调用 → 系统执行 → 结果回填；检索是「读」、工具是「做」
- 记忆补「没有记忆」：短期活在上下文窗口、关掉就忘，长期写进外部存储、跨会话还记得
- 一次增强调用是三路信息「汇聚」进一次调用，区别于第 1 章的「循环迭代」

<Exercises>

**练习 1（改增强组合 · 思考题）** 在本章主动画的「退货客服」例子里，三路增强（检索 / 记忆 / 工具）都用上了。现在换一个新需求：做一个「公司内部 wiki 答疑机器人」，只回答 wiki 里写过的内容，不需要查任何实时数据、也不需要记住每个提问者是谁。这个需求该保留哪几路增强、去掉哪几路？为什么？

<Answer>

只保留**检索**，去掉工具和记忆。

- **保留检索**：wiki 是公司私有、模型训练时没见过的资料，必须把相关 wiki 片段检索进上下文，LLM 才能据此回答（典型 RAG 场景）。
- **去掉工具**：需求里没有「触发一次执行去拿实时数据」的动作，纯粹是答现成资料，用不上工具。
- **去掉记忆**：不需要跨会话记住提问者，每次问答相互独立，长期记忆是多余的负担。

要点：增强是**按需加**的，不是越多越好。用不上的增强只会徒增复杂度、成本和出错面——先看任务缺哪块料，再决定上哪几路。

</Answer>

**练习 2（辨析题）** 下面三件事，分别该用「检索」「工具」还是「长期记忆」来解决？为什么？
（A）回答「我们公司的年假政策是几天」。
（B）回答「我现在这一单快递到哪了」。
（C）顾客这次没说，但助手要知道「他是 VIP、且偏好邮件联系」。

<Answer>

（A）用**检索**——年假政策是写好的静态私有资料，捞出相关条款拼进上下文即可。
（B）用**工具**——实时快递状态是动态数据、不在静态库里，得 LLM 产出调用、系统真去查接口取回。
（C）用**长期记忆**——这是需要跨会话保留的用户信息，得从外部存储里把「VIP、偏好邮件」调出来载入上下文。
判别诀窍：要现成的静态资料 → 检索；要触发一次执行拿实时结果 → 工具；要跨会话记住的事 → 长期记忆。

</Answer>

**练习 3（应用题）** 有人说：「我们模型答不准，是不是得换个更大更强的模型重新训练？」按本章的思路，你会先建议他检查什么、为什么？

<Answer>

先别急着换模型 / 重训，先检查**增强外壳**对不对、够不够。

理由：答不准很多时候不是模型本身不行，而是**喂给它的料不对**——本该检索的私有资料没接（它只能凭通用知识瞎答）、检索质量差（捞回错料）、或该长期记住的信息没存。这些都是模型**外面**的问题，换个更大的模型也照样答错。

按本章逻辑，增强改的是「输入与可用动作」，不是模型权重。所以正确的排查顺序是：① 该回答的内容有没有被检索 / 记忆送进上下文？② 送进去的料对不对、全不全？③ 需要执行的动作有没有配上工具？这三步都到位了还不行，才轮到考虑换模型。先治外壳，往往又快又便宜。

</Answer>

</Exercises>

  <GlossaryItem term="检索（Retrieval）" id="retrieval">
    拿用户问题去外部知识库里找出相关片段，把片段拼进上下文，再让 LLM
    带着这份资料作答。一句话：答之前先去外面查准资料，而不是凭记忆瞎说。详见本章「第二个概念」。
  </GlossaryItem>
  <GlossaryItem term="RAG（检索增强生成）" id="rag">
    Retrieval-Augmented Generation
    的缩写，即「先检索、后生成」：先从知识库检索相关片段，再让 LLM
    基于这些片段生成回答。它是把检索接到 LLM
    上的最常见做法，核心是让模型「有据可依」。详见本章「第二个概念」。
  </GlossaryItem>
  <GlossaryItem term="工具（Tools）" id="tools">
    LLM 能调用的外部函数或接口。LLM
    产出一个结构化调用（调哪个、传什么参数），系统真去执行，再把结果回填进上下文。它让模型从「只会说」变成「真能做」——查实时数据、触发真实动作。和检索的区别：检索是「读」现成资料，工具是「做」一次执行。详见本章「第三个概念」。
  </GlossaryItem>
  <GlossaryItem term="短期记忆（Short-term Memory）" id="short-term-memory">
    活在上下文窗口内、限于单次会话的记忆。本轮对话、刚拿到的工具结果都属于它；会话一结束就忘，不会自动留到下次。像便利贴，用完即扔。详见本章「第四个概念」。
  </GlossaryItem>
  <GlossaryItem term="长期记忆（Long-term Memory）" id="long-term-memory">
    写进外部存储（数据库 /
    向量库）、能跨会话保留的记忆。下次同一用户再来，系统能把相关条目调出来载入上下文。像存进档案柜的资料，下次还能翻出来。详见本章「第四个概念」。
  </GlossaryItem>
  <GlossaryItem term="上下文窗口（Context Window）" id="context-window">
    LLM
    一次能「看」的输入文字总量上限。本轮对话、检索片段、工具结果都要塞进这个窗口；塞超了就装不下或被截断。短期记忆就活在这个窗口里——所以不能把所有历史一股脑全往里堆。详见本章「第四个概念」。
  </GlossaryItem>
</Glossary>
