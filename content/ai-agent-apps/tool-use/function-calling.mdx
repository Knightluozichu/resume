## 先打个比方

想象你递给一个新来的师傅一份**工具箱清单**：每一件工具旁边都写着一句话——「这把是拧十字螺丝的」「这把是剪铁丝的」「这把是量长度的」。师傅接到活儿，并不会把工具箱翻个底朝天，而是照着这份清单上的那句话，挑出最趁手的那件。

挑得准不准，几乎全看那句话写得清不清楚。要是两件工具旁边都写着「拧东西用的」，师傅就只能瞎抓一件；要是清单上压根没有能干这活儿的工具，他就会停下来说「这活儿我手头没工具，我直接告诉你怎么办」。

模型用工具——这套让模型自己决定调哪个工具的能力，叫 <Term def="让模型在回答里『提议调用某个外部工具/函数』的能力：你把工具定义喂给它，它据此判断该不该调、调哪个、传什么参数，并以结构化形式给出这个调用意图；真正执行函数的是你的程序。" id="function-calling">Function Calling</Term>——也是一模一样的道理。这一章不讲工具调用的消息格式（那是上一章的事），只盯一件事：模型到底**怎么「看见」那份清单、又凭什么从中挑出一件**。看不懂这一层，你就只会埋怨「模型怎么老挑错」，却不知道病根其实在你写的那句话上。

## 第一个概念：工具是怎么进入模型「视野」的

模型并没有「内置」你的函数——它看到工具的唯一途径，是你把工具的定义当**文本**塞进这一轮的上下文里。这个动作叫 <Term def="工具 schema 注入（Tool Schema Injection）：把每个工具的定义——函数名、参数形状、一句 description——序列化成文本，放进这一轮发给模型的上下文里。模型据此「看见」有哪些工具可用，而不是「内置」了这些函数。" id="tool-schema-injection">工具 schema 注入</Term>：把每个工具的名字、参数、一句描述序列化成一段文字，注入上下文。下面这张图把这条路画了出来：

<ToolsAsMenuDiagram />

盯住从左到右这一程：左边是你写的工具定义，中间被序列化成文本注入上下文，右边模型就「看见」了一份**工具菜单**——每项 = 一个名字 + 一句描述。所以记住一句话：**模型对工具的全部认知，就来自菜单上那几行字。** 这几行写得清不清楚、准不准确，直接决定了模型能不能用对。这也是为什么后面要反复强调那句 description。

## 第二个概念：模型怎么从菜单里挑一个——工具选择

模型「看见」菜单之后，下一步就是 <Term def="工具选择（Tool Selection）：模型读懂用户意图后，从注入上下文的那份工具菜单里挑出最该调用的那个（或判定都不合适、不调）的过程。它靠的是拿用户意图去逐个比对各工具的 description。" id="tool-selection">工具选择</Term>：读懂用户到底想干嘛，再拿这份意图去对每个工具的描述，挑出最贴的那个。这里的关键抓手，就是工具定义里那句 **description**。

description 是专门写给模型看的——它是模型判断「这个工具到底干嘛、什么时候该用」的主要依据。名字（name）只像菜单上的菜名，给模型一个对号的标签；真正说清用途、让模型拿意图去比对的，是那句描述。所以描述写得越清楚、越贴近真实用途，模型挑得越准；含糊或几个工具写得雷同，它就容易点错菜。**这一步具体怎么转，下一节的动画会一帧一帧演给你看。**

## 第三个概念：挑中之后，多个调用是并排发还是排队发

模型挑工具，有时一次只挑一个，有时一口气挑好几个。这就引出一对概念：<Term def="并行工具调用（Parallel Tool Calls）：模型在一轮里一次性提议多个彼此独立、互不依赖的调用，它们可以同时发出、一起拿回结果，省时间。" id="parallel-tool-calls">并行工具调用</Term> 和 <Term def="串行工具调用（Sequential Tool Calls）：后一个调用要用到前一个调用的结果，只能排队——先调 A 拿到结果，再把它当输入去调 B。调用之间存在数据依赖时只能串行。" id="sequential-tool-calls">串行工具调用</Term>。一张图分清：

<ParallelVsSequentialDiagram />

决定走哪条路的，就一件事——**调用之间有没有「数据依赖」**。查订单和查天气是两件互不相干的事，谁也不等谁，模型一轮里同时提议、一起拿回结果，省时间，这就是并行；而「先查到退货政策、再按政策看这单还在不在可退期内」，后一步要吃前一步的结果，就只能先 A 后 B 排成一条链，这就是串行。**彼此独立 → 并行；后者吃前者输出 → 串行。** 这个判断，正是评估一个 agent 设计好不好的常见考点。

<FunctionCallingDecisionDiagram />

盯住这条决策链：① 用户意图 + 整份工具菜单一起进入模型视野 → ② 模型先读懂用户到底想要什么（「快递到哪了」= 想查订单状态）→ ③ 拿这份意图去逐个比对每个工具的 description，估匹配度 → ④ 挑中最贴的 `check_order`，其余压暗（**若都不贴 → 不调任何工具、直接用文字答**）→ ⑤ 产出调用意图「调 `check_order(order_id=…)`」。要害有两个：一是模型挑得准不准，全压在「比对 description」这一步，所以那句描述是你能攥在手里的方向盘；二是它自始至终只「**提议**」，从不自己执行——真去跑函数的是 app 的代码。

## 用代码看那份「菜单」长什么样

A 型章不展开实现（解析、回填那套上一章已经讲透）。这里只贴最能说明本章概念的一段——你交给模型的 `tools` 列表里，两三个工具各自的 `name` 和 `description`：

```python
tools = [
    {
        "name": "check_order",
        # description 才是模型选它的依据：写清「干嘛的 + 何时调」
        "description": "查某一笔具体订单的实时状态（到货进度 / 能否退货）；问到「我的单到哪了 / 能退吗」时调",
    },
    {
        "name": "search_docs",
        # 与 check_order 划清边界：查通用政策条文，不针对具体某一单
        "description": "搜通用帮助文档与政策条文（如退货规则、运费说明）；不针对某一笔具体订单",
    },
    {
        "name": "get_weather",
        "description": "查某城市的实时天气；问到天气、冷热、下不下雨时调",
    },
]
```

逐行对照前面那张菜单图看：模型看到的就是这几行字。`name` 只是标签，**真正让它分得清「这活儿该点哪道菜」的，是每条 `description`**——它写得清不清、几个工具之间边界划得开不开，直接决定模型挑得准不准。注意 `check_order` 和 `search_docs` 都跟「订单/退货」沾边，所以两句描述特意写成职责互斥（一个查「这一单的具体状态」、一个查「通用政策条文」），免得模型挑花眼。这一段没有任何调用解析或结果回填——那是上一章工具调用协议的事，本章只管「模型凭这几行字怎么挑」。

## 容易踩的坑

<Callout type="trap">
  **现象**：给 agent
  配了几十个工具，越配越全，结果模型反而经常挑错工具、或该调的漏调。
  **原因**：工具太多 = 菜单太长，模型要在一大堆 description
  里逐个比对，干扰项一多就「挑花眼」——这正是工具选择那一步被噪声淹没了。
  **修法**：精简工具集，按场景只注入当下相关的子集；合并职责重叠的工具；把每个
  description 写得更可区分。本质是减少模型在「比对 description」时的干扰项。
</Callout>

<Callout type="trap">
  **现象**：两个工具的 description
  写得几乎一样（都像「查订单相关信息」），模型一会儿挑这个、一会儿挑那个，来回犯错。
  **原因**：description 是模型区分工具的唯一抓手，写雷同 =
  这个抓手失效了，两者匹配度难分高下，模型只能随机挑。根子是工具设计上职责重叠、边界不清。
  **修法**：要么把两个工具合并成一个；要么把两句描述改成明确互斥、各写清独有的触发场景，让模型有据可分（如本章代码里
  `check_order` 与 `search_docs` 的划法）。
</Callout>

<Callout type="trap">
  **现象**：默认「只要配了工具，模型每轮都会调一个」，于是代码里不写「模型没调工具」这条分支，线上撞到就崩。
  **原因**：模型逐个比对后，若没有工具的 description
  贴合用户意图，它会**不调任何工具、直接用文字答**——「有工具」不等于「必然调工具」。
  **修法**：把「模型没调工具、直接给了文字答案」当成一条正常分支兜住：每次回复先判断有没有调用意图，没有就直接把它的文字答案返回给用户。
</Callout>

## 小结

- 模型不「内置」工具：靠工具 schema 注入，把名字 + 参数 + 描述当文本塞进上下文，它才「看见」一份工具菜单
- 工具选择 = 读懂用户意图 → 拿意图逐个比对每个工具的 description → 挑最贴的；都不贴就不调、直接答
- description 是模型选工具的主要依据：写得清不清、几个工具边界划得开不开，直接决定挑得准不准
- 调用之间有没有数据依赖决定并行还是串行：彼此独立 → 并行省时间；后者吃前者输出 → 只能串行
- 模型只「提议」调用、从不自己执行；工具太多或描述雷同会让它挑花眼、乱挑

<Exercises>

**练习 1（辨析题）** 下面三句对「模型怎么挑工具」的说法，哪些对、哪些错？说明理由。
（A）「模型是把我所有函数的源代码读了一遍，才知道有哪些工具。」
（B）「模型主要靠工具的 name 来挑，description 只是注释，写不写无所谓。」
（C）「就算配了工具，用户问的事跟谁都不沾边时，模型可以一个都不调、直接答。」

<Answer>

（A）**错**。模型看不到你的源代码，它看到的只是被序列化注入上下文的工具 schema——名字 + 参数 + 一句 description 这几行文本。这就是「工具 schema 注入」。
（B）**错**，而且是常见误解。name 只是个对号的标签，真正说清「这个工具干嘛、何时该用」、让模型拿意图去比对的是 description。描述写不好，名字再漂亮也照样挑错。
（C）**对**。模型逐个比对后若没有工具贴合意图，就不调任何工具、直接用文字答。「有工具」不等于「必然调工具」，这条分支代码必须兜住。

</Answer>

**练习 2（场景判断题 · 判断会不会调 / 调哪个）** 一个客服 agent 的菜单里有：`check_order`（查某一笔具体订单的实时状态：到货 / 退货资格）、`search_docs`（搜通用帮助文档 / 政策条文）、`get_weather`（查天气）。下面三个用户输入，模型分别会调哪个工具，还是不调？为什么？
（甲）「我下周想去三亚玩，那边这几天热不热？」
（乙）「你们家七天无理由退货具体是怎么规定的？」
（丙）「我的订单 A123 现在还能退吗？」

<Answer>

（甲）调 **`get_weather`**——意图是问某地天气，最贴 `get_weather` 的描述；另两个工具跟天气不沾边，匹配度低被压下去。
（乙）调 **`search_docs`**——问的是「通用政策怎么规定」，不针对某一笔具体订单，正落在 `search_docs`「搜政策条文」的描述里；`check_order` 是查「某一单的具体状态」，边界不同。
（丙）调 **`check_order`**——「我这单 A123 能不能退」要的是**这一笔具体订单**的退货资格，正是 `check_order` 的用途；这里能挑准，靠的就是 `check_order` 和 `search_docs` 的 description 把「具体某单」和「通用政策」划开了。
判别诀窍：把用户意图和每条 description 对一遍，挑最贴的；都不贴就不调。

</Answer>

**练习 3（设计题）** 你发现自己的旅行助手老是「查机票」和「查酒店」两个工具用混。结合本章原理，说说病根可能在哪、怎么改才能让模型挑得更准。

<Answer>

病根八成在两个工具的 **description 写得太像、职责边界不清**——因为 description 是模型区分工具的唯一抓手，描述雷同时它的匹配度难分高下，只能乱挑。
改法：把两句描述改成**明确互斥、各写清独有的触发场景**，例如 `search_flights` 写「查城市间航班的班次 / 价格 / 时刻，问到机票、航班、飞过去时调」，`search_hotels` 写「查目的地酒店的房型 / 价格 / 空房，问到住宿、订房、住哪时调」。再顺手做两件事：删掉或合并职责重叠的多余工具，减少干扰项；工具实在多时按场景只注入相关子集。把「比对 description」这一步的噪声降下来，模型挑选行为就稳定可预期，而不是碰运气。

</Answer>

</Exercises>

    模型在一轮里一次性提议多个**彼此独立、互不依赖**的调用（比如同时查订单又查天气），它们可以同时发出、一起拿回结果，省时间。能不能并行，看调用之间有没有数据依赖。详见本章「第三个概念」。
  </GlossaryItem>
  <GlossaryItem
    term="串行工具调用（Sequential Tool Calls）"
    id="sequential-tool-calls"
  >
    后一个调用要用到前一个调用的结果，只能排队：先调 A
    拿到结果，再把它当输入去调
    B。调用之间存在**数据依赖**时（如先查到政策、再按政策判断某单），就只能串成一条链。详见本章「第三个概念」。
  </GlossaryItem>
</Glossary>
