## 先打个比方

还是上一章那个新来的师傅。上一章我们说，他照着工具旁边的那句话挑趁手的家伙——但挑中之后用得顺不顺，是另一回事。你递给他的工具本身，称不称手？

想想一把真正好用的工具：手柄握着称手（**命名**清楚），旁边的说明贴得明明白白（**描述**到位），它干的活儿不大不小刚刚好（**粒度**合适），万一卡住了会「咔」一声提示你「该换刀片了」而不是默默崩坏（**错误信息**友好），干完活儿给你的是一件成品而不是一地零件（**返回值**干净）。换成一把烂工具——名字看不懂、没说明、要么细到拧颗螺丝得换五次头、要么大到一把抵整个工具箱、坏了一声不吭、用完扔给你一堆没归拢的料——再聪明的师傅也得被它拖垮。

agent 用工具也一样：**模型再会挑，工具本身设计得糟，它照样办不成事。** 这一章不讲「模型怎么挑」（那是上一章的事），只讲一件事——**怎么把工具设计得让 agent 用着顺手**。我们会沿着五个维度，把「好工具」和「坏工具」一项一项摆在一起对照着看。

## 先看全景：好工具的五个维度

把「一个工具好不好用」拆开，就落在五件事上：**命名**（名字能不能让人一眼看懂干嘛）、**描述**（那句 description 写得清不清）、**粒度**（一次管多大一件事）、**错误信息**（失败时说不说人话）、**返回值**（给回来的是精料还是一堆原料）。这五项里，每一项都有明确的好坏之分。下面这张表把它们一项一项摆出来对照：

<ToolDesignDimensionsDiagram />

盯住这张表的右列和左列：同一个维度，往左是「坏」、往右是「好」，差别一目了然。命名和描述上一章已经反复讲过（模型靠它们挑工具），所以本章把重头放在后三项——**粒度、错误信息、返回值**，它们才是「挑对之后用不用得顺」的关键，也是新手最容易做砸的地方。下面逐个拆。

## 第一个取舍：工具粒度——不大不小才好用

第一个真正需要权衡的，是 <Term def="工具粒度（Tool Granularity）：一个工具一次到底管多大一件事。太细 = 只做一个原子动作，办成一件事要串很多个工具；太粗 = 一个工具塞进好几件不相干的功能；合适 = 一个工具刚好办成一件完整的事。" id="tool-granularity">工具粒度</Term>——一个工具一次到底该管多大一件事。这件事没有「越细越好」的标准答案，太细和太粗各有各的坑。一张图把三态摆在一起：

<ToolGranularityDiagram />

看这三栏：拆得**太细**（左），办成「订一张票」要让 agent 把 search、pick、pay、issue 一长串调用串起来，链路越长越慢、越容易在中途某一步翻车——这叫组合爆炸；揉得**太粗**（右），一个工具把订票、改签、退票、查天气全包了，参数一大堆，模型分不清这次到底想用它干哪件事，反而更易选错、传错。**合适**的那栏（中）刚刚好：一个 `book_train_ticket` 工具，一次调用办成一件完整的事。这正是 <Term def="职责单一原则（Single Responsibility）：一个工具只负责一件清楚、完整的事，不把多件不相干的功能揉进同一个工具。它让工具粒度合适、模型也容易判断什么时候该用它。" id="single-responsibility">职责单一原则</Term> 在工具上的体现——一个工具，一件事。

## 第二个关键：错误信息——决定 agent 能不能自己爬起来

工具会失败，这很正常。真正拉开差距的是：**失败时你返回给模型的，是一句它能照做的话，还是一个它看不懂的裸异常。** 这就引出 <Term def="可恢复的错误（Recoverable Error）：工具失败时返回的、让 agent 看得懂并能照着改的错误信息——说清错在哪、该怎么改。反面是抛一个裸异常（如 KeyError），agent 看不懂、无从下手。" id="recoverable-error">可恢复的错误</Term>。

为什么这件事对 agent 格外要命？因为 agent 在那一刻**手里只有工具返回的这段文字**——它看不到你的日志、查不了文档、问不了同事。信息够，它就能自己纠正参数、重试成功；信息不够（一个光秃秃的 `KeyError`），它就只能原样再试一次、卡死，或者瞎猜乱试。下一节的动画会把这条岔路一帧一帧演给你看。

## 第三个关键：返回值——给精料，别甩一地原料

最后一项是工具**返回**什么。新手最常犯的错，是把工具拿到的原始数据（整段 API 的 raw JSON、整张数据库表）原样丢回给模型。正确的做法是 <Term def="结构化返回（Structured Return）：工具把结果整理成字段清晰、只留要紧信息的结构（如一个精简对象）返回，而不是把一大坨原始数据原样塞给模型。" id="structured-return">结构化返回</Term>——把结果整理成字段清晰、只留要紧信息的形状。

原样丢一大坨有两宗罪：一是把宝贵的上下文窗口塞满，挤掉真正要紧的信息、还更费钱；二是要紧的几个字段被埋在几十个噪声字段里，模型更难抓重点、更容易答偏。好的返回值像一份**摘要**：替模型把噪声筛掉，只留它和用户真正用得上的那几项。

## 动手看：错误信息的好坏，怎么让 agent 一个爬起来、一个躺平

前面三个取舍里，**「错误信息怎么影响 agent」**最难凭文字想象。下面这个动画把它演出来——同一次工具调用失败（agent 传了个格式不对的 `order_id`），左右两路只有一个差别：工具返回的错误信息写得好不好。

> 猜一猜：左路的工具只抛了一个光秃秃的 `KeyError`，没说哪里错、该怎么改。agent 下一步会怎么做？是看懂了、把参数改对再试，还是看不懂、原样把同一个调用再撞一次？先想一想，再单步播放到第 ③、④ 步对比着看。

<ToolErrorRecoveryDiagram />

盯住这条岔路：① 两路从同一个失败的调用出发 → ② 工具返回——**分叉就在这一步**：左路抛裸异常 `KeyError`，右路返回「格式应为 A + 4 位数字，你传的是 '123'」→ ③ agent 看到后，左路看不懂、只能原样重试，右路读懂了、主动把参数改成 `'A0123'` → ④ 结局：左路还是同一个错、卡死，右路这次成功、自己把错纠了过来。一句话记住：**错误信息不是写给日志看的，是写给一个只能读这段字的执行者看的操作指引。** 写清「错在哪 + 怎么改」，agent 就能自救；甩个裸异常，它就只能躺平。

{/* ===== 7. 常见误区｜spec §七：D 型 ≥2 个，必含 1 个「场景选错方案」坑；固定 现象→原因→修法 三行 ===== */}

## 容易踩的坑

<Callout type="trap">
  **现象**：为了「灵活」，把工具拆到最细——查订单拆成查头、查体、查尾三个工具，结果
  agent 办一件事要串 8 个调用，又慢又老在中途某一步出错。
  **原因**：这是「粒度选错方案」——细 ≠ 灵活。拆太碎会组合爆炸，链路越长，agent
  出错和超时的概率越高。
  **修法**：按「职责单一」收成「一个工具办成一件完整的事」的粒度（如一个
  `get_order_status`
  顶替三个碎工具）；只有确实存在多种独立组合需求时才下沉粒度，别无脑拆到最细。
</Callout>

<Callout type="trap">
  **现象**：工具直接把查到的整张表 / 整段第三方 API 的原始 JSON
  返回给模型，结果上下文几轮就被撑爆、还更贵，模型也抓不住重点。
  **原因**：这是「返回值设计差」——把一大坨 raw
  数据原样透传，宝贵的上下文窗口被噪声字段塞满，要紧信息反而被淹没。
  **修法**：做结构化返回——只留这次真正要用的字段，必要时聚合 / 截断 /
  分页（如「返回前 5 条 + 总数 + 翻页游标」），替模型把噪声筛掉。
</Callout>

<Callout type="trap">
  **现象**：工具失败时直接 `raise` 一个裸异常（`KeyError`、`ValueError`），指望
  agent「自己看着办」，线上一失败就卡死或反复瞎试。 **原因**：agent
  手里只有工具返回的那段文字，裸异常它看不懂、无从修正，只能原样重试。
  **修法**：把失败也设计成「可恢复的错误」——返回一句说清「错在哪 +
  该怎么改」的提示，让 agent 能读懂、纠正参数、重试成功，而不是对着堆栈干瞪眼。
</Callout>

## 小结

- 工具好不好用，看五个维度：命名、描述、粒度、错误信息、返回值——每项都有好坏之分
- 粒度取舍：太细组合爆炸、太粗职责不清；「一个工具办成一件完整的事」（职责单一）通常最好
- 错误信息决定 agent 能不能自救：裸异常让它卡死，「错在哪 + 怎么改」的可恢复错误让它纠正重试
- 返回值要结构化、只留要紧字段：原样甩 raw 数据会撑爆上下文、淹没重点
- 模型再会挑工具，工具本身设计得糟，agent 照样办不成事——好工具是「挑得准」之后用得顺的前提

**练习 1（辨析题）** 下面三句关于工具设计的说法，哪些对、哪些错？说明理由。
（A）「工具拆得越细越灵活，所以一律拆到最细准没错。」
（B）「工具失败时抛个 `KeyError` 就行，反正 agent 自己会看着办。」
（C）「工具查到多少就返回多少，原样给模型最保险，免得漏信息。」

<Answer>

（A）**错**。细 ≠ 灵活。拆到最细会组合爆炸——办一件事要让 agent 串一长串调用，链路越长越慢、越易在中途翻车。粒度的黄金点是「一个工具办成一件完整的事」（职责单一），只在确有多种独立组合需求时才下沉粒度。
（B）**错**。agent 手里只有工具返回的那段文字，裸异常它看不懂、无从修正，多半原样重试、卡死。应返回「可恢复的错误」——说清错在哪、怎么改，它才能自救。
（C）**错**。原样返回一大坨 raw 数据会撑爆上下文、还把要紧字段淹没在噪声里。应做结构化返回，只留要紧字段，必要时聚合 / 截断 / 分页。

</Answer>

**练习 2（选型题 · 给场景选粒度）** 你在给一个外卖客服 agent 设计工具。对「帮用户取消一笔还没出餐的订单」这件事，下面三种切法选哪个？为什么？
（甲）拆成 `find_order` + `check_status` + `lock_order` + `do_cancel` + `notify_user` 五个工具，让 agent 自己串。
（乙）一个 `cancel_order(order_id)` 工具，内部把校验、取消、通知一条龙办完。
（丙）一个 `manage_order(order_id, action)` 工具，取消、改地址、催单、退款都走它，靠 `action` 参数区分。

<Answer>

选 **（乙）**。
（甲）太细：取消一单要 agent 串五个调用，链路长、慢、易在中途某步出错（组合爆炸）。
（丙）太粗：一个工具管取消/改址/催单/退款四件不相干的事，参数语义全压在 `action` 上，模型容易选错、传错（职责不清）。
（乙）粒度刚好——一个工具办成「取消这一单」这件完整的事，一次调用搞定，符合职责单一，agent 用着最顺。判别诀窍：让一个工具对应「一件用户能一句话说清的完整的事」。

</Answer>

**练习 3（改 Demo 代码题）** 把下面这个坏工具，按五个维度改好（重点修命名、错误、返回）。说出你改了哪几处、为什么。

```python
def proc(a):
    return str(db.query(a))   # a 不存在时抛裸异常；返回整段 raw
```

<Answer>

可以改成这样（思路对即可）：

```python
def get_user_profile(user_id):
    """按 user_id 查用户档案；user_id 为 6 位数字，如 100245"""
    if user_id not in db:
        return {"error": f"找不到 user_id={user_id!r}；"
                f"它应为 6 位数字，请核对后重试"}
    u = db.query(user_id)
    return {"name": u.name, "level": u.level, "city": u.city}
```

改了三处，对应三个维度：
① **命名**：`proc(a)` → `get_user_profile(user_id)`，名字和参数一看就懂（含糊 → 清晰）。
② **错误**：从「`a` 不存在直接抛裸异常」→「返回一句说清『找不到什么 + 格式该是什么 + 请重试』的可恢复错误」，让 agent 能自救。
③ **返回值**：从「`str(...)` 整段 raw」→「只留 name / level / city 三个要紧字段的结构化对象」，不撑爆上下文。
（顺手补的 docstring 就是「描述」维度。）

## 名词解释

<Glossary>
  <GlossaryItem term="工具粒度（Tool Granularity）" id="tool-granularity">
    一个工具一次到底管多大一件事。太细 =
    只做一个原子动作，办成一件事得把好几个工具串起来（组合爆炸，慢且易错）；太粗
    = 一个工具塞进好几件不相干的功能（职责不清，模型分不清该用它干哪件）；合适 =
    一个工具刚好办成一件完整的事——通常最好用。详见本章「第一个取舍」。
  </GlossaryItem>
  <GlossaryItem
    term="职责单一原则（Single Responsibility）"
    id="single-responsibility"
  >
    一个工具只负责一件清楚、完整的事，不把多件不相干的功能揉进同一个工具。它是判断工具粒度合不合适的准星：一个工具，对应「一件用户能一句话说清的完整的事」。详见本章「第一个取舍」。
  </GlossaryItem>
  <GlossaryItem term="可恢复的错误（Recoverable Error）" id="recoverable-error">
    工具失败时返回的、让 agent
    看得懂并能照着改的错误信息——说清错在哪、该怎么改。反面是抛一个裸异常（如
    KeyError），agent 看不懂、只能原样重试或卡死。因为 agent
    在那一刻手里只有这段文字，所以这段话要写成「给执行者看的操作指引」。详见本章「第二个关键」与「动手看」。
  </GlossaryItem>
  <GlossaryItem term="结构化返回（Structured Return）" id="structured-return">
    工具把结果整理成字段清晰、只留要紧信息的结构（如一个精简对象）再返回，而不是把一大坨原始数据（整段
    raw
    JSON、整张表）原样塞给模型。好处：不撑爆上下文窗口、不淹没重点、更省钱。详见本章「第三个关键」。
  </GlossaryItem>
</Glossary>
