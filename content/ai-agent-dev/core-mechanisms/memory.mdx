## 小特聊久了会忘事，得有个本子记下来

上一章我们让私人助理**小特**学会了边想边查边干。可你和它聊久了会发现一件烦事：它会忘。开头你交代「我预算 500、喜欢靠窗」，聊到后面，它居然又问你一遍预算多少——前面说过的，它像没听见一样。

这是因为小特的「脑子」装当下这场对话的地方就那么大，聊得越久，越早说的越容易被挤出去、忘掉。人遇到这种事会怎么办？掏出个**本子**，把重要的事记下来，要用的时候再翻回去看。这一章就讲：怎么给小特配上这个「本子」。

没有它会怎样？小特永远只活在「眼前这几句」里——你昨天说的、十分钟前说的，它全不记得，每次都得你从头重讲一遍。这样的助理，根本谈不上「懂你」。

## 两种记忆：脑子里的当下，本子里的长期

小特其实有两种记忆，分工完全不同。

第一种叫 <Term def="Agent 当前这场对话能直接「看进眼里」的内容，就装在上下文窗口里——是第 2 章那张有容量上限的桌子。它装的是「眼前这场对话」：聊得越久越满，塞满后最早的内容被挤出窗口、被遗忘（断片）。所以短期记忆有限、且会忘。">短期记忆</Term>：就是第 2 章讲的**上下文窗口**——小特当前这场对话能直接看进眼里的全部内容。它像一张固定大小的桌子，聊得越久堆得越满，塞爆了最早的几句就被挤下桌、忘掉。这就是它会「断片」的根。**短期记忆 = 脑子里的当下，有限、会满、会忘。**

第二种叫 <Term def="把重要信息存到模型外部（数据库、向量库等）的记忆，不占上下文窗口。它容量几乎不限、且不会因为对话变长而丢失；要用时再「检索」出相关的几条、塞回窗口给大脑看。长期记忆正是突破上下文窗口物理上限的办法。">长期记忆</Term>：把重要信息存到**小特的大脑之外**（一个数据库、一个专门的记忆库），不占那张桌子。它容量几乎不限，也不会因为对话变长就丢——就像写进本子里的字，不会因为你后面又说了很多话而消失。要用的时候，再从本子里把相关的那几条**翻出来**，放回桌上给大脑看。**长期记忆 = 本子里的长期，存到外面、要用再翻回来。**

看出关键了吗？**长期记忆正是用来突破上下文窗口物理上限的**——窗口再大也会满，但只要重要信息存进了外部长期记忆，哪怕它被挤出了窗口，要用时照样捞得回来。下面这张图把「对话进窗口 → 写入长期 → 检索回来 → 窗口满遗忘 → 但长期还在」整条信息流逐环点亮，跟着走一遍：

## 记忆怎么用起来：写入、检索、遗忘三个操作

光有「两层记忆」还不够，得有一套**操作**让它转起来。记忆系统其实就围着三个动作打转，合起来叫 <Term def="操作长期记忆的三个基本动作：① 写入——把短期窗口里值得长期留存的信息存进长期记忆；② 检索——要用时从长期记忆里找出相关的几条、捞回短期窗口；③ 遗忘——淘汰或压缩不重要的内容，别让长期记忆无限膨胀、也别让短期窗口塞爆。三者配合，记忆才既记得住又用得好。">记忆三操作（写入 / 检索 / 遗忘）</Term>：

- **写入**：把短期窗口里**值得长期留存**的信息（你的预算、偏好、办过的事）挑出来，存进长期记忆。不是每句话都存——那是把整本流水账都抄下来，又乱又贵。
- **检索**：以后聊到相关话题，从长期记忆里**找出相关的几条**、捞回短期窗口塞给大脑。注意只捞「相关的」，不是把整个本子全摊上桌——桌子摆不下，也会淹没重点。
- **遗忘**：主动**淘汰或压缩不重要的**内容。长期记忆不能只进不出，否则越积越多、检索越来越慢越不准；短期窗口更得及时清掉过气的旧对话，免得塞爆。

这三个操作正好对应上面那张架构图里的环节：写入是「②」，检索是「③」，遗忘是「④」。第六节我们会把它们落成几行 Python，你会看到它们其实没那么玄。

## 检索的灵魂：按「意思」找回，不是按关键词

三个操作里，**检索**最容易被想当然——很多人以为它就是「关键词搜索」：你查「预算」，就去记忆里找含「预算」二字的条目。可真这么干，会漏掉一大片。

因为人说话是会换词的。你之前说「我**打算花 500 块**」，后来问「我的**预算**是多少」——这两句一个共同的字都没有，关键词搜索直接抓瞎。长期记忆真正的做法叫 <Term def="长期记忆找回相关内容的方式：先把每条记忆和查询都变成一串数字（向量，embedding），再按「意思相近」（向量相似度）排序，捞出最相关的几条——而不是按字面是否含相同关键词来匹配。所以查询「预算」能找回「打算花 500 块」，哪怕两句一个字都不一样，因为它们意思相近。">语义检索</Term>：把每条记忆和你的查询都变成一串数字（这串数字叫**向量**，也叫 embedding），再按「**意思相近**」排序，捞出最相关的那几条——比的是意思，不是字面。

所以「预算」能命中「打算花 500 块」，因为这两句话**意思相近**，转成向量后离得近。下面这个 Demo 让你换不同查询，亲眼看它按意思命中哪条：

## 记忆里都装些啥：情景、语义、程序三类

最后一件事：长期记忆里存的东西，性质并不一样，按内容可以分成三类——记住这个分法，你写记忆系统时才知道「这条该当什么记」。

第一类是 <Term def="一类长期记忆，存的是「发生过的具体事件、对话历史」——什么时候发生了什么，带时间线。比如「上次你订了去上海的票」。Agent 靠它记住你们之前聊过、办过什么，接着上回的茬。">情景记忆（episodic）</Term>：存**发生过的事、对话历史**，比如「上次你订了去上海的票」。第二类是 <Term def="一类长期记忆，存的是「不带时间、长期为真的事实知识和偏好」，比如「你预算 500、喜欢靠窗」。Agent 靠它把你当熟人、按你的设定办事，不用每次重说。">语义记忆（semantic）</Term>：存**事实知识和稳定偏好**，比如「你预算 500、喜欢靠窗」。第三类是 <Term def="一类长期记忆，存的是「怎么做某事的步骤、技能」，比如「订票的标准流程：查→比价→下单→确认」。Agent 靠它把一类任务按固定套路办利索，不用每次现想步骤。">程序记忆（procedural）</Term>：存**怎么做某事的步骤、技能**，比如「订票的标准流程：查 → 比价 → 下单 → 确认」。

一句话区分：情景记**「发生过什么」**、语义记**「关于你的事实」**、程序记**「这事该怎么办」**。下面这个 Demo 把三类并排，点一类看它具体装什么、小特什么时候用得上：

## 代码长什么样：写入、检索、拼进窗口

把上面三个操作落到代码里，其实就几行。下面这段 Python（把真实的 embedding 细节省成一个打分函数）就是一套最简记忆的骨架——`save_memory` 写入、`retrieve` 按「意思相近」捞回 top-k、再把捞回的拼进 `messages` 给大脑看：

```python
LONG_TERM = []   # 长期记忆：存到窗口之外的一个列表（真实里是数据库 / 向量库）

def save_memory(text):
    """① 写入：把一条值得长期留存的信息存进长期记忆。"""
    LONG_TERM.append(text)

def retrieve(query, k=2):
    """② 检索：按「意思相近」给每条记忆打分，返回最相关的 k 条（top-k）。"""
    # similarity 比的是「意思」不是关键词（真实里把两者转成向量算相似度）
    scored = [(similarity(query, m), m) for m in LONG_TERM]
    scored.sort(reverse=True)              # 相似度从高到低
    return [m for _score, m in scored[:k]]  # 只捞最相关的 k 条，不全摊上桌

def build_messages(user_question):
    """把检索回来的相关记忆，拼进短期窗口（messages）给大脑看。"""
    recalled = retrieve(user_question)     # 从长期捞回相关的
    memory_note = "已知用户信息：" + "；".join(recalled)
    return [
        {"role": "system", "content": memory_note},  # 捞回的记忆塞进窗口
        {"role": "user", "content": user_question},
    ]
```

`save_memory("用户打算花 500 块")` 就是**写入**长期记忆；之后用户问「我的预算是多少」，`retrieve` 拿这句去**检索**，靠 `similarity`（比意思）把「打算花 500 块」排到最前、捞回来；`build_messages` 再把它拼进 `messages` 的 system 里——这就是把长期记忆**塞回短期窗口**给大脑看。至于**遗忘**，这里没写：真实系统里你还得定期清掉 `LONG_TERM` 里过气、重复的条目，别让它无限膨胀拖慢检索。

## 容易踩的坑

<Callout type="trap">
  **现象**：「记忆不就是把所有对话历史全塞进窗口吗？我把每一轮都拼进 messages
  不就行了。」结果对话一长就报超长、又慢又贵。**原因**：把「记忆」等同于「把全部历史灌进上下文窗口」——可窗口有物理上限（第
  2
  章），全塞必然塞爆，且窗口越满模型越抓不住重点。**修法**：重要信息**写入长期记忆**存到窗口之外，每轮只**检索**回当前相关的几条塞进窗口，而不是把整本流水账硬怼进去。
</Callout>

<Callout type="trap">
  **现象**：「检索不就是搜关键词嘛——查『预算』就找含『预算』二字的记忆。」结果用户说过「打算花
  500
  块」却怎么也搜不到。**原因**：以为**关键词匹配**就够了，忽略了人说话会换词——「预算」和「打算花多少钱」一个共同字都没有，字面匹配直接漏掉。**修法**：用**语义检索**——把记忆和查询都转成向量按「意思相近」排序，比的是意思不是字面，「预算」才能命中「打算花
  500 块」。
</Callout>

<Callout type="trap">
  **现象**：「长期记忆当然是存得越多越好，只管往里写、从不删。」结果记忆库越积越大，检索越来越慢、还老捞回一堆过时
  /
  重复的条目。**原因**：只做**写入**、不做**遗忘**——长期记忆无限膨胀后，相关信息被海量噪声淹没，检索质量直线下降。**修法**：把**遗忘**当成一等操作——定期淘汰过时、压缩重复、合并冗余，让记忆库保持精简，检索才又快又准。
</Callout>

## 小结

- **短期记忆 vs 长期记忆**：短期 = 上下文窗口里的当前对话（有限、会满、会断片）；长期 = 存到模型外部、要用再检索回来——长期正是突破窗口物理上限的办法（回收第 2 章）
- **记忆三操作**：**写入**（把重要信息存进长期）、**检索**（用时捞回相关的几条塞进窗口）、**遗忘**（淘汰 / 压缩不重要的，别让长期膨胀、别让窗口塞爆）
- **语义检索**：长期记忆靠「**按意思找回**」——把内容和查询转成向量按相似度排序，比意思不比字面，所以「预算」能命中「打算花 500 块」（深的 embedding 机制留到 RAG 那章）
- **记忆三类型**：**情景**（发生过的事 / 对话历史）、**语义**（事实知识 / 偏好）、**程序**（怎么做某事的步骤）——一句话：记「发生过什么」/「关于你的事实」/「这事该怎么办」
- 一套最简记忆 = `save_memory` 写入 + `retrieve` 按相似度返回 top-k + 把捞回的拼进 `messages`，再加上定期遗忘

（c）「上周三帮用户订过一张去杭州的票」；
（d）「给这个用户订票的标准流程：先查班次 → 比价 → 下单 → 发确认」。

<Answer>

- **（a）短期记忆** —— 这是当前这场对话里刚说的话，直接放进上下文窗口给大脑看即可，不必长期留存。
- **（b）长期记忆 · 语义记忆** —— 「预算 500、只坐高铁」是关于用户的、不带时间的**稳定事实 / 偏好**，属于语义记忆，存进长期、以后每次订票都检索回来用。
- **（c）长期记忆 · 情景记忆** —— 「上周三订过去杭州的票」是**发生过的具体事件**、带时间线，属于情景记忆。
- **（d）长期记忆 · 程序记忆** —— 「订票的标准流程：查 → 比价 → 下单 → 确认」是**怎么做某事的步骤 / 套路**，属于程序记忆。

要点：当前对话用短期（窗口）；要长期留存的，按「发生过的事 / 客观事实 / 办事步骤」分别归到情景 / 语义 / 程序。

</Answer>

**练习 2（改 Demo 题）** 打开「动手一」的语义检索 Demo，先选查询「我的预算是多少？」，记下「打算花 500 块」的相似度分；再切到「我要去哪个城市？」，看「打算花 500 块」这条的分数怎么变了。为什么同一条记忆，对不同查询的相似度差这么多？请用「按意思找回」解释。

<Answer>

查询「我的预算是多少？」时，「打算花 500 块」相似度很高（命中）；切到「我要去哪个城市？」后，同一条「打算花 500 块」的相似度掉得很低，命中的变成了「想去上海」。

原因：语义检索比的是**查询和这条记忆的「意思」有多近**，不是看记忆本身。「打算花 500 块」讲的是**钱**，和「预算多少」意思贴得近、和「去哪个城市」几乎不相干——所以对前一个查询分高、对后一个查询分低。换句话说，相似度是「**查询 ↔ 记忆**」这一对算出来的，查询变了，同一条记忆的分自然跟着变。这正是「按意思找回」：捞回哪条，取决于哪条和**当前这个查询**意思最近。

</Answer>

**练习 3（问答题）** 用户之前说过「我打算花 500 块」，后来问「我的预算是多少」——这两句一个共同的字都没有。Agent 凭什么还能从长期记忆里把它捞回来？如果改用「关键词匹配」来检索，会发生什么？

<Answer>

凭的是**语义检索**：长期记忆在检索时，把每条记忆和这个查询都转成**向量**（一串数字），按「**意思相近**」（向量相似度）排序，而不是看字面有没有相同的字。「我打算花 500 块」和「我的预算是多少」虽然一个共同字都没有，但**意思非常近**（都在讲花多少钱 / 预算），转成向量后离得近，于是被排到最前、捞了回来。

如果改用**关键词匹配**：它只会去找含「预算」二字的条目——而「我打算花 500 块」里压根没有「预算」这两个字，于是**一无所获**，Agent 会以为自己从没记过用户的预算，只能再问一遍（又变回那个「聊久了就忘、反复追问」的笨助理）。这正是为什么长期记忆必须用语义检索、而不是关键词匹配。

</Answer>

<GlossaryItem term="短期记忆">
  Agent 当前这场对话能直接「看进眼里」的内容，就装在**上下文窗口**里——是第 2
  章那张有容量上限的桌子。它装的是「**眼前这场对话**」：聊得越久越满，塞满后最早的内容被挤出窗口、被遗忘（断片）。所以短期记忆**有限、会满、会忘**。类比：小特的「脑子」，只装得下当下。详见本章「两种记忆」一节与架构动画。
</GlossaryItem>

<GlossaryItem term="长期记忆">
  把重要信息存到模型**外部**（数据库、向量库等）的记忆，**不占上下文窗口**。它容量几乎不限、且不会因为对话变长而丢失；要用时再**检索**出相关的几条、塞回窗口给大脑看。长期记忆正是**突破上下文窗口物理上限**的办法——窗口会满会忘，但存进长期的东西照样捞得回。类比：小特的「本子」，存到外面、要用再翻回来。详见本章「两种记忆」一节与架构动画。
</GlossaryItem>

<GlossaryItem term="记忆三操作（写入 / 检索 / 遗忘）">
  操作长期记忆的三个基本动作：①
  **写入**——把短期窗口里值得长期留存的信息存进长期记忆；②
  **检索**——要用时从长期记忆里找出**相关的**几条、捞回短期窗口；③
  **遗忘**——淘汰或压缩不重要的内容，别让长期记忆无限膨胀、也别让短期窗口塞爆。三者配合，记忆才既**记得住**又**用得好**。详见本章「记忆怎么用起来」一节与代码。
</GlossaryItem>

<GlossaryItem term="语义检索">
  长期记忆**找回**相关内容的方式：先把每条记忆和查询都变成一串数字（**向量**，embedding），再按「**意思相近**」（向量相似度）排序，捞出最相关的几条——而**不是**按字面是否含相同关键词来匹配。所以查询「预算」能找回「打算花
  500 块」，哪怕两句一个字都不一样，因为它们意思相近。深一层的 embedding
  机制留到《RAG 检索增强生成》那章。详见本章「检索的灵魂」一节与检索 Demo。
</GlossaryItem>

<GlossaryItem term="情景记忆（episodic）">
  一类长期记忆，存的是「**发生过的具体事件、对话历史**」——什么时候发生了什么，带时间线。比如「上次你订了去上海的票」。Agent
  靠它记住你们之前聊过、办过什么，接着上回的茬。对应人脑里「记得昨天去过哪」那种记忆。详见本章「记忆里都装些啥」一节与类型
  Demo。
</GlossaryItem>

<GlossaryItem term="语义记忆（semantic）">
  一类长期记忆，存的是「**不带时间、长期为真的事实知识和偏好**」，比如「你预算
  500、喜欢靠窗」。Agent
  靠它把你当熟人、按你的设定办事，不用每次重说。对应人脑里「知道水会结冰」那种记忆。（注意：和「语义检索」里的「语义」是两回事——这里指**记忆的内容类型**，那里指**检索时按意思比对**。）详见本章「记忆里都装些啥」一节与类型
  Demo。
</GlossaryItem>

<GlossaryItem term="程序记忆（procedural）">
  一类长期记忆，存的是「**怎么做某事的步骤、技能**」，比如「订票的标准流程：查 →
  比价 → 下单 → 确认」。Agent
  靠它把一类任务按固定套路办利索，不用每次现想步骤。对应人脑里「会骑车」那种说不太清、但一上手就会的记忆。详见本章「记忆里都装些啥」一节与类型
  Demo。
</GlossaryItem>
