## 小特接了个大活，不能想哪做哪

你交代私人助理**小特**：「帮我办一场去上海的三天旅行。」这可不是一句话能办完的小事——要订往返的票、订住的地方、还得安排好这三天每天去哪玩。要是小特闷着头、想到哪做到哪，很容易乱套：可能票订好了才发现那几天酒店全满，或者光顾着排景点、把回程的票忘了。

一个靠谱的人接到这种大活，会先**停下来列个计划**：先订往返票、再订酒店、最后排每天的行程——把这一大坨事拆成几件小事，排好先后，再一件件去做。这一章就讲：怎么让 AI 也这样「先想好整盘怎么走，再动手」。

没有它会怎样？小特只会盯着眼前一步瞎做，做着做着就跑偏、漏东西，最后办出一场乱七八糟的旅行。复杂的活，光会「一步步反应」不够，还得会「先谋全局」。

## 为什么要先规划：复杂任务得先谋全局

上一章我们讲了 ReAct，让小特学会「边想边做」——想一步、动手查一步、看到结果再接着想。这招对付「看今天要不要带伞」这种几步就完的事很好使。可一旦活变大（办一整场旅行），光靠「走一步看一步」就不够了：你需要先把**整盘棋**看清楚。

这就是 <Term def="Agent 在动手前，先把一个复杂目标想清楚整体怎么走的过程：把大目标拆成有序的子任务、排好先后、定好整体路线，再逐步执行。规划解决的是「复杂任务一步做不完、闷头做容易跑偏漏步」的问题——它给执行一张「宏观蓝图」。和 ReAct「边想边做」的微观一轮互补：ReAct 管「这一步怎么走」，规划管「整条路线怎么排」。">规划（planning）</Term>：动手之前，先把这个复杂目标想清楚——拆成哪几件事、按什么顺序、整条路线怎么走，然后再一步步执行。它专治「闷头做容易跑偏 / 漏步」：先有了整张蓝图，每一步才走得有方向、不乱套。

注意它和 ReAct 的分工：**ReAct 是「边想边做」的微观一轮**（这一步该查什么、查到了再想下一步），**规划是「先想好整体路线」的宏观蓝图**（整件事分几步、先后怎么排）。两者互补——规划排好大方向，ReAct 在每一步里把它走扎实。

## 任务分解：把大目标拆成一棵任务树

规划的核心动作，就是 <Term def="把一个大目标拆成若干可执行的子任务 / 子目标的过程；子任务太粗还能接着拆，层层拆下去就形成一棵「任务树」。树根是大目标，每个叶子是「能直接动手做」的小步。拆解让一个一步做不完的大活，变成一串好下手的小活。">任务分解（task decomposition）</Term>——把一个大目标拆成若干**可执行的子任务**。

拿「办一场去上海的三天旅行」来说：先拆成三件事——订往返高铁票、订酒店、排三天行程。前两件已经够具体、能直接动手；可「排三天行程」还是太粗，于是接着往下拆：Day1 去哪玩、Day2 去哪玩、Day3 怎么返程。这样一层层拆下去，就长出一棵**任务树**：树根是大目标，往下分叉成子任务，子任务再分叉……拆到每个**叶子**都是「能直接动手做」的小步为止。

## 三种规划策略：CoT、ToT、先规划后执行

「先想好再做」具体怎么想，有好几种**套路**。最常见的三种，结构差别很大、适用的场景也不一样，得分清楚什么时候用哪个。

第一种是 <Term def="思维链（Chain-of-Thought）：让 LLM 把推理一步步线性地写下来，想 1 → 想 2 → 想 3 → 答，一条直线想到底，不分叉、不回头。结构最轻，适合简单、线性、一条路就能想通的问题。第 1 章见过它「只会想、不调工具」的一面；这里关注它的「线性结构」。">CoT（思维链）</Term>：把推理排成**一条直线**，想 1 → 想 2 → 想 3 → 给答案，线性一步步往下推。它最轻便，对付**简单、线性**的问题（一条路想到底就能解）足够了。

第二种是 <Term def="思维树（Tree-of-Thoughts）：不走一条直线，而是从一个想法分出多个分支，每条路各探一探，互相比较后挑出最优的那条、剪掉差的——像下棋时在脑子里推演几步、对比哪步更好。适合有多种解法、需要试错的难题。代价是要探多条路，更慢更贵。">ToT（思维树）</Term>：不走一条道，而是**分出多个分支**去探索——像下棋一样，在脑子里推演好几条路，各走一走、互相比较，挑出**最优的那条**，把差的分支剪掉。它适合**有多种解法、需要试错**的难题。

第三种是 <Term def="先规划后执行（Plan-and-Execute）：先一次性把整个任务拆成一张完整的计划清单（订票→订酒店→排行程→出发），再照着清单逐项打勾执行——计划在前，执行在后。适合步骤明确、能提前拆清楚的复杂多步任务，比如办一场旅行。和「边走边想下一步」相反：它先把整盘排好。">Plan-and-Execute（先规划后执行）</Term>：先一口气把整件事拆成一张**完整的计划清单**，再照着清单**逐项打勾执行**——计划在前、执行在后。它适合**步骤明确的复杂多步任务**（像办旅行：能提前把整张计划排清楚，再一项项做）。

## 反思与重规划：计划不是定死的

最后一件关键事：计划列得再好，执行起来也可能**碰壁**——按计划要订的那家酒店，一查发现订满了。这时候怎么办？

靠谱的做法叫 <Term def="反思（reflection）与重规划（replan）：执行某个子任务时发现计划行不通（这一步失败、或前提变了），Agent 先反思——弄清哪里错了、为什么走不通；再重规划——调整剩余的步骤（换个做法 / 换个顺序 / 补一步），然后接着执行。核心：计划不是一锤定死的，是边执行边修的。少了它，Agent 一步卡死就全盘崩。">反思与重规划（reflection / replan）</Term>：执行受阻时，Agent 先**反思**——弄清哪里行不通（「原计划那家订满了」）；再**改计划**——调整剩下的步骤（换一家、或换个日期）；然后接着执行。它不会在卡住的那一步**一条道走到黑**，而是绕过去、把活办成。

关键就在这句话：**计划不是定死的，是边执行边修的**。真实任务里总有意外，能反思、能重规划，规划才真正可靠。下面这个 Demo 把「顺利」和「受阻重规划」两条路摆一起，切换看小特怎么绕过卡点：

## 代码长什么样：拆解、执行、出错就重规划

把本章串起来，规划的骨架其实就三步：先 `decompose` 把大目标**拆成子任务列表**（任务树拍平成有序的一串），再 `for` 循环逐个**执行**，某步**失败就 `replan`**（反思 + 改计划）再接着走。下面这段 Python（把真实的 LLM 调用省成几个函数）就是它的心脏：

```python
def run_with_plan(goal):
    plan = decompose(goal)        # ① 拆解：把大目标拆成有序的子任务列表（任务树）
    i = 0
    while i < len(plan):
        task = plan[i]
        result = execute(task)     # ② 执行：动手做当前这个子任务
        if result.ok:
            i += 1                  # 这步成了，推进到下一个子任务
        else:
            # ③ 受阻：反思哪错了 + 改剩下的计划，再从当前步接着走
            plan = replan(goal, plan, done=plan[:i], failed=task, why=result.error)
    return "目标达成"
```

三步正好对应前面三样东西：`decompose(goal)` 是**任务分解**（呼应那棵任务树，把大目标拆成一串子任务）；`for` / `while` 逐个 `execute` 是**照计划执行**；而 `replan(...)` 那一支就是**反思与重规划**——`result.ok` 为假（这步失败）时，它带着「已做完的、失败的那步、为什么失败」去**改剩下的计划**，再从当前步继续，而不是直接崩掉。

注意 `replan` 这一步：它让 `plan` 在执行途中**被改写**，这正是「计划不是定死的，是边执行边修的」落到代码上的样子。把这一支删掉（失败就直接 `return` 报错），Agent 就退回「一步卡死、全盘崩」的笨办法。

## 容易踩的坑

<Callout type="trap">
  **现象**：「ToT 那么强，能探多条路选最优，那啥任务都上 ToT 准没错！」于是问「1
  加 1 等于几」也让它分叉探好几条思路。**原因**：把 ToT
  当成「越强越好」，忽略了它要**探多条分支、来回比较**，比 CoT
  慢得多也贵得多。**修法**：看任务**需不需要试错 /
  有没有多种解法**——简单、一条路就能想通的问题，线性的 **CoT**
  最省事；只有**有多解、要反复试错**的难题才值得动用 ToT。简单问题硬上 ToT =
  又慢又贵还没必要。
</Callout>

<Callout type="trap">
  **现象**：「计划一开始就排好了，照着做就行，中途绝不能改。」结果按计划要订的酒店订满了，Agent
  卡在这一步死活推进不下去、整个任务崩了。**原因**：把**计划当成一锤定死**的，没有反思
  /
  重规划这一环——可真实任务总有意外（订满、涨价、前提变了）。**修法**：把**重规划当成标配**——执行某步失败就反思哪错了、改掉剩余步骤再继续。计划是**边执行边修**的蓝图，不是刻死的剧本；一步卡死就全盘崩，往往是缺了这一环。
</Callout>

<Callout type="trap">
  **现象**：拆任务时要么拆得**过粗**（「排三天行程」当一步，根本没法直接动手），要么拆得**过细**（连「打开订票
  App」都拆成一步，碎得没法管）。**原因**：没拿捏好**叶子的粒度**——叶子应是「**能直接动手做**的最小有意义步骤」。**修法**：拆到**每个叶子都能直接执行、又不琐碎**为止：太粗（还没法直接做）就继续拆，太细（一句话能带过）就合并。粒度合适，计划才既清楚又好执行。
</Callout>

## 小结

- **为什么规划**：复杂任务一步做不完，闷头做容易跑偏 / 漏步——得先把大目标拆成有序的子任务、排好先后，给执行一张宏观蓝图（ReAct 管「这一步怎么走」，规划管「整条路线怎么排」，互补）
- **任务分解**：把大目标拆成子任务、子任务太粗再拆，形成一棵**任务树**，拆到每个**叶子**都「能直接动手做」为止
- **三种策略**：**CoT**（一条直线、线性想到底，简单问题用）、**ToT**（分多条分支探索、选最优剪掉差的，有多解 / 要试错的难题用）、**先规划后执行**（先列整张计划清单再逐项执行，步骤明确的复杂多步任务用）
- **反思与重规划**：执行受阻就反思哪错了、改剩下的计划再继续——**计划不是定死的，是边执行边修的**；少了它，一步卡死就全盘崩
- 一套带规划的骨架 = `decompose(goal)` 拆成子任务列表 + `for` 逐个 `execute` + 失败就 `replan`

- **（a）用 CoT（思维链）** —— 这是**简单、线性**的计算，一条路一步步算到底就行，不需要分叉探索，也不用提前列计划。线性的 CoT 最省事。
- **（b）用先规划后执行（Plan-and-Execute）** —— 这是**步骤明确的复杂多步任务**：能提前把整张计划拆清楚（订票 → 订酒店 → 排行程 → 出发），先排好整张清单再逐项执行最合适。
- **（c）用 ToT（思维树）** —— 这题**有多种走法、需要不断试错和回退、比较哪条路更优**，正是 ToT 的主场：分出多条分支探一探、互相比较、选最优剪掉差的。线性的 CoT 撞了墙回不了头，搞不定这种要试错的难题。

要点：简单线性 → CoT；步骤明确的复杂多步 → 先规划后执行；有多解 / 要试错 → ToT。

</Answer>

**练习 2（改 Demo 题）** 打开「动手二」的反思 / 重规划 Demo，先停在「顺利路径」，再切到「受阻重规划路径」——多出来的是哪两步？如果一个 Agent **没有**这两步（执行失败就直接报错停下），办这场旅行会发生什么？请用「计划不是定死的」来解释。

<Answer>

多出来的是**反思**（弄清原计划那家酒店订满了、行不通）和**改计划**（换一家或换日期），再回到执行。

如果 Agent 没有这两步：它按计划去订那家酒店，一发现订满了就**卡在这一步直接报错停下**——后面的排行程、出发全做不了，整场旅行就泡汤了。原因正是它把**计划当成定死的**：计划里写了订「那一家」，订不上就无路可走。而真实任务总有意外（订满、涨价、前提变了），所以必须能**反思 + 重规划**——发现行不通就改掉剩余步骤（换一家）再继续。这就是「**计划不是定死的，是边执行边修的**」：能边走边改，规划才真正可靠。

</Answer>

**练习 3（问答题）** 第六节那段带规划的代码骨架里，如果把 `replan(...)` 那一支删掉、改成「`execute` 失败就直接 `return '失败'`」，这个程序会退化成什么？为什么这样不好？

<Answer>

它会退化成一个**「计划定死、一步卡死就全盘崩」的执行器**：`decompose` 拆好计划后，只会照着原计划一步步往下做，**一旦某步失败就直接放弃整个任务**，没有任何挽救。

为什么不好：真实任务里子任务失败太常见了（酒店订满、票卖光、某接口报错）。删掉 `replan` 后，Agent 失去了「**反思哪错了 + 改剩下的计划再继续**」的能力，等于把灵活的蓝图变成了刻死的剧本——撞上任何一个意外就整盘崩。`replan` 这一支正是「计划边执行边修」的代码体现，它让 `plan` 能在执行途中被改写、绕过障碍，是带规划的 Agent 之所以可靠的关键。

</Answer>

<GlossaryItem term="规划（planning）">
  Agent
  在**动手之前**，先把一个复杂目标想清楚整体怎么走的过程：把大目标拆成有序的子任务、排好先后、定好整条路线，再逐步执行。它专治「**复杂任务一步做不完、闷头做容易跑偏漏步**」——先有一张「宏观蓝图」，每步才走得有方向。和
  ReAct「边想边做」的微观一轮互补：**ReAct
  管「这一步怎么走」，规划管「整条路线怎么排」**。类比：办大活前先列个计划。详见本章「为什么要先规划」一节。
</GlossaryItem>

<GlossaryItem term="任务分解（task decomposition）">
  把一个**大目标拆成若干可执行的子任务 /
  子目标**的过程；子任务太粗还能接着拆，层层拆下去就形成一棵「**任务树**」——树根是大目标，每个**叶子**是「能直接动手做」的小步。拆解让一个一步做不完的大活，变成一串好下手的小活。类比：把「办一场旅行」拆成订票
  / 订酒店 / 排行程，行程太粗再拆成 Day1 / Day2 /
  Day3。详见本章「任务分解」一节与任务树动画。
</GlossaryItem>

<GlossaryItem term="CoT（思维链）">
  Chain-of-Thought：让 LLM 把推理**一步步线性地**写下来，想 1 → 想 2 → 想 3 →
  答，一条直线想到底，**不分叉、不回头**。结构最轻，适合**简单、线性、一条路就能想通**的问题。第
  1
  章见过它「只会想、不调工具」的一面；本章关注它的「线性结构」——三种规划策略里最简单的那个。详见本章「三种规划策略」一节与对比
  Demo。
</GlossaryItem>

<GlossaryItem term="ToT（思维树）">
  Tree-of-Thoughts：不走一条直线，而是**从一个想法分出多个分支**，每条路各探一探，互相比较后挑出**最优的那条、剪掉差的**——像下棋时在脑子里推演几步、对比哪步更好。适合**有多种解法、需要试错**的难题。代价是要探多条路，**更慢更贵**，所以简单问题不该硬上。详见本章「三种规划策略」一节与对比
  Demo。
</GlossaryItem>

<GlossaryItem term="Plan-and-Execute（先规划后执行）">
  先一次性把整个任务拆成一张**完整的计划清单**（订票 → 订酒店 → 排行程 →
  出发），再照着清单**逐项打勾执行**——计划在前、执行在后。适合**步骤明确、能提前拆清楚的复杂多步任务**（如办一场旅行）。和「边走边想下一步」相反：它先把整盘排好，再照着做。详见本章「三种规划策略」一节与对比
  Demo。
</GlossaryItem>

<GlossaryItem term="反思与重规划（reflection / replan）">
  执行某个子任务时发现**计划行不通**（这一步失败、或前提变了），Agent 先
  **反思**——弄清哪里错了、为什么走不通；再**重规划（replan）**——调整剩余的步骤（换个做法
  / 换个顺序 /
  补一步），然后接着执行。核心：**计划不是一锤定死的，是边执行边修的**。少了它，Agent
  一步卡死就全盘崩。类比：订酒店发现订满了，就换一家、改改计划再继续，而不是杵在原地。详见本章「反思与重规划」一节与重规划
  Demo。
</GlossaryItem>

</Glossary>
