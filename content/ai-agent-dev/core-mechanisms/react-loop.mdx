## 小特办事，是边想边查边调整的

你让私人助理**小特**：「看看今天上海要不要带伞。」一个靠谱的小特不会闷头一拍脑袋说「带吧」——他会先想「这得看天气」，然后真的去查一下上海今天的预报，看到「小雨」，才回你「带伞」。

注意他这一套：**想一下 → 动手查 → 看到结果 → 再接着想**，一小步一小步交替着来，而不是关在屋里凭空把整件事想完。每动一步，他都拿到了新信息，下一步才想得更准。

这一章就讲：怎么让一个 AI 也这样「边想边干边看」地办事。没有它会怎样？AI 只能闷头硬想——可它根本不知道上海今天下不下雨，于是只能瞎猜一个，还说得特别笃定。

## ReAct：把「想」和「干」拧成一股绳

上一章我们说，小特的大脑（LLM）有两个软肋：**会幻觉**（记不清就自信地圆一个）、**不知道实时信息**（它不知道上海今天的天气）。那怎么办？答案就是这一章的主角—— <Term def="一种让 LLM「边推理边动手」的工作模式。名字来自 Reasoning（推理）+ Acting（行动），它把「思考 Thought → 行动 Action → 观察 Observation」三步交织成一个循环：想一步、动手查一步、看到结果再接着想，直到办成。">ReAct 范式</Term>。

它的名字就是 **Re**asoning（推理）+ **Act**ing（行动）拼起来的，干的事正是把这两件事**拧成一股绳**：

- 光推理（想）不够——它不知道实时信息，只能瞎猜；
- 光行动（干）不够——它得先想清楚该查什么、再想清楚查到的结果说明什么。

所以 ReAct 让 LLM **想一步、干一步、看一步，再接着想**，循环往复。这正好呼应第 1 章那条 Agent 循环（感知→决策→行动→观察）——只不过 ReAct 是它落到「一段提示 + 一轮轮执行」上的**具体打法**：每一轮，LLM 都显式写出自己的思考、决定下一个动作，系统执行后把结果喂回去，再转下一轮。

## Thought：把「我现在在想什么」写出来

这一圈里，每一轮 LLM 先做的事叫 <Term def="ReAct 一轮里 LLM 显式写出来的推理：「我现在该想什么、下一步该干什么」。它把模型的思考过程从「藏在脑子里」变成「白纸黑字看得见」，方便我们跟踪它的判断，也方便它自己一步步往下推。">Thought（推理轨迹）</Term>——它把「我现在该想什么、下一步干什么」**显式地写出来**。

比如小特写：「要回答带不带伞，得先知道上海今天的天气，我自己不知道，得去查。」这句话不直接解决问题，但它把模型的**判断过程摊到了明面上**。好处有两个：一是我们能跟着看它怎么想的、想错了能发现；二是它自己写下来后，下一步该干嘛就顺理成章了。这就是「让推理看得见」——而不是黑箱里一团乱麻。

## Action 与 Observation：一个去做，一个把结果带回来

Thought 之后，LLM 决定动手，这一步叫 <Term def="ReAct 一轮里 LLM 决定要执行的那个动作，通常是「调用某个工具、并传入参数」，比如查天气、搜资料、算数。Action 由 LLM 自己写出来，系统照着它真去执行。">Action（行动）</Term>：它写出一个**要调用的动作**，通常就是「调哪个工具、传什么参数」。比如小特写 `weather("上海")`——意思是「去调那个查天气的工具，参数是上海」。

系统照着这个 Action 真的把工具跑一遍，再把返回的结果填回来，这一步叫 <Term def="系统执行 Action（工具调用）之后，把工具返回的结果填回给 LLM 当作下一轮输入的那段内容。关键：Observation 是系统填回的真实数据，不是 LLM 自己编的——这是 ReAct 能「实证」而非「空想」的来源。">Observation（观察）</Term>：比如「小雨，降水 60%」。**这条结果是系统真去查回来的，不是大脑编的**——这是 ReAct 全章最关键、也最容易被搞混的一点。

把这一轮三部分连起来看：`Thought`（想）和 `Action`（做什么）是 **LLM 自己写的**，`Observation`（结果）是 **系统填回的**。下个小节的 Demo 你可以点三个标签，亲眼看清这条分界线在哪。

## ReAct vs 纯 CoT：一个去查，一个瞎猜

你可能会问：让 LLM「一步步想」不是早就有了吗？是的，那叫 <Term def="思维链（Chain-of-Thought）：让 LLM 把推理过程一步步写出来再给答案，只靠「想」、不调用任何工具。它能让推理更有条理，但全程闷在模型脑子里——遇到它不知道的实时 / 事实信息，就只能凭记忆瞎猜，可能编出幻觉。">纯 CoT（思维链）</Term>。但 CoT 只会**在脑子里想**，从头到尾不碰外部世界。

差别就在这：问一个**需要实时信息**的问题——「现在从北京去上海最快的高铁几点发车」——纯 CoT 不知道今天的时刻表，只能凭印象编一个（可能是幻觉）；ReAct 则会先想「我不知道，得去查」，然后**真去调时刻表工具**，拿到真实数据再回答。下面这个 Demo 把两条路摆一起，切换看看它们答同一个问题的天差地别：

一句话记牢：**CoT 是闷头推理，ReAct 是推理 + 实证**。凡是需要实时、事实信息的任务（查天气、查时刻表、搜最新资料、算具体数），就不能只靠 CoT 空想，必须用 ReAct 让它去查。

## 代码长什么样：一段 ReAct 循环骨架

上面那张 trace 图，翻译成代码，核心也是一个 `while` 循环：让 LLM 产出 Thought + Action，系统执行 Action 拿到 Observation，把它拼回对话，再转下一轮——直到 LLM 决定收尾。下面这段 Python 骨架（把真实工具细节省了）就是 ReAct 的心脏：

```python
def run_react(question):
    messages = [{"role": "user", "content": question}]
    while True:
        step = llm.think(messages)        # LLM 产出这一轮的 Thought + Action
        if step.action == "finish":       # 它认为信息够了，收尾给答案
            return step.answer
        obs = run_tool(step.action)        # 系统真去执行工具，拿回 Observation
        messages.append(step.thought_text) # 把这一轮的思考+行动拼回对话
        messages.append(f"Observation: {obs}")  # 再把工具结果（系统填的）拼回去
        # 带着新到的 Observation，下一圈 llm.think 会想得更准
```

四个关键点正好对应图里的四样东西：`llm.think` 产出 **Thought + Action**（这两样是 LLM 写的），`run_tool` 执行后得到 **Observation**（这是系统填的），`messages.append` 把它们**拼回对话**当作下一轮的输入，而 `while True` 就是那条「再转一圈」的循环。

注意 `obs = run_tool(...)` 这一步——它会真的去调外部工具、把现实世界的数据带回来。这正是 ReAct 和纯 CoT 的分水岭：CoT 没有这一步，全程闷在脑子里；ReAct 有了它，才能「实证」而不是「空想」。把这个循环拆成只跑一次、且去掉 `run_tool`，ReAct 立刻退化回纯 CoT。

## 容易踩的坑

<Callout type="trap">
  **现象**：「`Observation`
  那行天气结果，不也是模型自己写出来的一段文字吗？」于是以为整条 trace 全是 LLM
  编的。**原因**：把「系统填回的真实数据」和「LLM 生成的推理」混为一谈。Thought
  和 Action 是 LLM 写的，但 Observation
  是**系统真去执行工具后填回**的。**修法**：记住分界线——LLM 每轮只写到 Action
  就停，系统跑完工具把 Observation 拼回去，模型才续写下一轮。Observation
  来自外部世界，不是模型「想」出来的。
</Callout>

<Callout type="trap">
  **现象**：「ReAct 这么强，那啥任务都套 ReAct 准没错！」于是问「1 加 1
  等于几」也让它先 Thought 再调计算器。**原因**：把 ReAct
  当成「越用越好」，忽略了它每轮调工具都有延迟和成本。**修法**：看任务**需不需要外部信息**——简单问答、模型自己就知道的常识，纯回答或
  CoT 就够了；只有需要实时 / 事实 / 算具体数的任务，才值得动用 ReAct 去查。
</Callout>

<Callout type="trap">
  **现象**：「让它每轮多想几段
  Thought，推理越充分越好吧？」结果它在原地反复空想、迟迟不动手查。**原因**：把「Thought
  越多」当成「越靠谱」，可推理再多也变不出它不知道的事实。**修法**：ReAct
  的价值在「想到该查时**真去查**」，而不是堆 Thought。该出 Action 就出
  Action，让 Observation 带回真数据，比多想十句强。
</Callout>

## 小结

- **ReAct = Reasoning + Acting**：把「思考 Thought → 行动 Action → 观察 Observation」交织成一个循环，边想边干边看，直到办成——它是第 1 章 Agent 循环落到提示 / 执行上的具体打法
- **Thought** 是 LLM 把「我现在该想什么、下一步干什么」显式写出来，让推理过程看得见
- **Action** 是 LLM 决定要调的动作（调哪个工具、传什么参数），**Observation** 是系统执行工具后填回的真实结果
- 关键分界：**Thought / Action 是 LLM 自己写的，Observation 是系统填回的**——不是模型编的
- **ReAct vs 纯 CoT**：CoT 只闷头推理、遇到实时 / 事实信息只能瞎猜；ReAct 推理 + 真去查，能实证——需要实时 / 事实信息的任务必须 ReAct

</Answer>

<GlossaryItem term="Thought（推理轨迹）">
  ReAct 一轮里 LLM
  **显式写出来**的推理——「我现在该想什么、下一步该干什么」。它不直接给答案，而是把模型的判断过程从「藏在脑子里」变成「白纸黑字看得见」：方便我们跟着看它怎么想的、想错能发现，也方便它自己顺理成章地推下一步。由
  LLM 自己生成。详见本章「Thought」一节。
</GlossaryItem>

<GlossaryItem term="Action（行动）">
  ReAct 一轮里 LLM
  决定要执行的那个**动作**，通常是「调哪个工具、传什么参数」（比如
  `weather("上海")`）。Action 由 LLM
  自己写出来，系统照着它**真去执行**那个工具。它是 ReAct 区别于「只会想」的纯
  CoT 的关键——靠它去触碰外部世界。详见本章「Action 与 Observation」一节。
</GlossaryItem>

<GlossaryItem term="Observation（观察）">
  系统执行 Action（工具调用）之后，把工具**返回的真实结果**填回给
  LLM、当作下一轮输入的那段内容（比如「小雨，降水
  60%」）。最关键的一点：Observation 是**系统填回的真实数据，不是 LLM
  自己编的**——它正是 ReAct
  能「实证」而非「空想」的来源，也是初学者最容易搞混的地方。详见本章「Action 与
  Observation」一节与拆解 Demo。
</GlossaryItem>

<GlossaryItem term="纯 CoT（思维链）">
  思维链（Chain-of-Thought）：让 LLM
  把推理过程一步步写出来再给答案，**只靠「想」、不调用任何工具**。它能让推理更有条理，但全程闷在模型脑子里——遇到它不知道的实时
  /
  事实信息（今天的天气、时刻表、最新资料），就只能凭记忆瞎猜，可能编出幻觉。这正是
  ReAct 要补的洞：ReAct = CoT 的推理 + 真去查的实证。详见本章「ReAct vs 纯
  CoT」一节与对比 Demo。
</GlossaryItem>

</Glossary>
