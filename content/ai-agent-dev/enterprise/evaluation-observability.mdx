## 小特上岗了，可你看不见它在干什么

前面几篇，我们把私人助理**小特**从里到外捋了一遍——它的脑子、办事的章法、查资料的本事、带一支小队的能耐。现在它要真的**上岗**了：给成百上千的真实用户干活。

问题来了：它一天处理几千单，你**根本不知道它干得怎么样**。哪一单答错了？哪一单慢得让用户等到走人？哪一单悄悄烧掉了一大笔钱？你两眼一抹黑。更要命的是，它**这次这么答、下次可能那么答**——同样一个问题，结果还不一定一样，你连「它到底行不行」都说不清。

这一章就补上这两块：怎么**看见**小特每一步到底干了啥、出错错在哪一步；怎么**打分**判断它答得好不好、改完一版是变好了还是变坏了。没有这两样，让它上岗就是**蒙着眼睛开飞机**——出了事你都不知道是哪儿出的事。

## 为什么 Agent 上线后必须能「看见」和「量得出」

先说清这一章为什么重要。一个传统程序，同样的输入永远给同样的输出，错了能稳定复现、断点一调就抓到。但 Agent 不一样，它有两个让人头疼的性子。

第一，它是个 <Term def="黑盒：指你只看得到输入和最终输出、看不见中间过程的系统。Agent 一次运行内部经历了好几步——调了哪个 LLM、查了哪个工具、子 Agent 说了啥——这些默认都不显示，你只看到「用户问了 X、它答了 Y」，中间发生了什么全是黑的。要把黑盒变透明，就得把每一步记录下来，这正是『可观测性』要解决的。">黑盒</Term>：你只看得见「用户问了啥、它最后答了啥」，中间它调了几次大脑、查了哪些工具、子 Agent 嘀咕了什么，**全是黑的**。答错了，你都不知道是哪一步坏的。

第二，它是 <Term def="非确定性（non-deterministic）：同样的输入，每次运行结果可能不一样。因为 LLM 是按概率逐 token 采样生成的（还记得温度那个旋钮吗），天然带随机性。后果：你没法靠『跑一次看对不对』下结论——这次对不代表下次对；想判断它到底行不行，必须在一批固定题目上多次度量，看整体表现，而不是抓一两个例子。">非确定性</Term>的：同样一个问题，这次答得好、下次可能就翻车（因为大脑 LLM 本就是按概率采样的，天生带随机）。所以你**没法靠「试一个例子还行」就下结论**——那一个例子恰好对了而已。

<Callout type="warn">
  正因为非确定性，「我手动试了俩问题，看着挺好，上线吧」是**最危险的自我安慰**。两个例子说明不了什么——你需要的是在**一批固定题目**上反复度量，看整体成绩，而不是抓两个顺手的例子壮胆。
</Callout>

黑盒 + 非确定性，逼出了企业级落地最先要解决的两件事：**看得见**（把中间每一步记录下来，能回放、能定位）和**量得出**（用可复现的方式给它的表现打分）。这一章接下来就拆这两件事——先讲怎么看见，再讲几种打分方法怎么取舍，最后讲改动后怎么确认没变差。

## trace 与 span：把一次运行的每一步都记下来

先解决「看得见」。办法是把小特**一次完整运行**里经历的每一步，都老老实实记一笔，串成一条链。

这条链叫 <Term def="trace / 调用链：把 Agent 一次完整运行（从收到一个请求到给出最终答案）经历的每一步——每次 LLM 调用、每次工具调用、每个子 Agent——都记录下来，按时间先后串成的一整条链。一条 trace 对应一次运行。有了它就能『回放』这次运行：从头到尾看它走了哪些步、每步花了多久、哪步出了错——把黑盒变透明的核心手段。">trace（调用链）</Term>——它对应小特的**一次运行**：从收到「帮我查上海天气并写句提醒」，到吐出最终答案，中间走的每一步都串在这条链上。链上的**每一步**，是一个 <Term def="span：一条 trace 上的一步，是可观测性的最小记录单位。每个 span 记下三样关键信息：① 名字（这一步是干啥的，如『查天气工具』）；② 耗时（这一步花了多久）；③ 输入输出（喂进去什么、吐出来什么）。一次 LLM 调用是一个 span，一次工具调用是一个 span，一个子 Agent 也是一个 span。把所有 span 按时间串起来就是一条 trace——看 span 的耗时就知道哪步慢，看 span 的输入输出就知道哪步答错了。">span</Term>：每次 LLM 调用是一个 span，每次工具调用是一个 span，每个子 Agent 也是一个 span。每个 span 都记着三样：**名字**（这步干啥）、**耗时**（这步花了多久）、**输入输出**（喂进去啥、吐出来啥）。

有了这条 trace，黑盒就透明了：你能**回放**这次运行——从头到尾看它走了哪些步、哪步慢、哪步错、哪步烧了最多 token。下面这张图把小特处理那桩差事的一条 trace 摊成时间轴，逐个 span 点亮，你会一眼看出「查天气」那步**特别长**——它就是这次运行的瓶颈：

<TraceTimelineDiagram />

看那条标红的横条：占了整次运行大半的时间。要是没有 trace，你只知道「这次响应慢」，却**不知道慢在哪一步**；有了它，瓶颈一眼锁定——这就是可观测性的价值：**把「它好像有点慢 / 有点贵」变成「它的第 2 步、查天气工具、花了 2.6 秒」**。

## 三种评估方法：怎么判断 Agent 答得好不好（对比取舍）

看得见之后，接下来是**量得出**——怎么给小特的表现打分。这没有唯一答案，常用的有三种办法，各看一面、各有取舍。这是本章的对比重心，一种一种说。

**第一种，LLM-as-judge（用另一个 LLM 当裁判）。** 既然人来一条条评太慢，那就请 <Term def="LLM-as-judge：用『另一个 LLM』当裁判，按一份评分标准（rubric，比如准确性、完整性、相关性各打几分）给 Agent 的答案打分。好处：快、可规模化——几千条答案让裁判 LLM 自动评，不用人盯。取舍：裁判本身也是个 LLM，可能有偏（比如偏爱长答案、偏爱措辞客气的、容易被『自信的语气』讨好），所以它的分不能全信，得校准、得和人工抽检对一对。">LLM-as-judge</Term>——另一个 LLM 当裁判，按一份评分标准（准确性、完整性、相关性各打几分）给答案打分。好处是**快、可规模化**：几千条答案让裁判自动评，不用人一条条盯。取舍是裁判本身也是 LLM，**可能有偏**——偏爱长答案、容易被自信的语气讨好。下面这个 Demo 让你切换「好答案 / 坏答案」，看裁判怎么按三个维度打分：

<LLMJudgeDemo />

你会看到关键一幕：那个「答得准但答非所问」的坏答案，**准确性**还行、**相关性**却被狠扣——多维度打分才能分辨出「说得对 ≠ 答的是你问的事」。

**第二种，离线 eval（攒一个固定测试集，每次跑一遍）。** 第二个办法是 <Term def="离线 eval（offline evaluation）：攒一个固定的测试集——一批有代表性的题目，可能还配着期望答案或评分标准——每次改完 Agent 就把这个测试集完整跑一遍，看通过率 / 得分。好处：可复现、能横向比（这版 vs 上一版，跑同一套题，分数直接可比），不受线上流量随机性干扰。取舍：测试集要人工攒、要维护（题目得有代表性、覆盖关键场景），攒题本身是成本；而且测试集没覆盖到的情况，eval 也照不出来。">离线 eval（离线评测）</Term>——攒一批有代表性的固定题目当测试集，每次改完就完整跑一遍，看通过率。好处是**可复现、能横向比**：这一版和上一版跑同一套题，分数直接可比。取舍是测试集得**人工攒、人工维护**，而且没覆盖到的情况它照不出来。下面这个 Demo 就是一个 10 题的离线 eval 测试集（先看「改之前」，下一节细玩）：

<EvalRegressionDemo />

**第三种，关键指标（线上实打实的数）。** 前两种是「质量好不好」，还有一类是**线上跑出来的硬数据**—— <Term def="关键指标（key metrics）：Agent 上线后从真实流量里统计出来的硬数据，主要三类：① 成功率（多大比例的请求成功办成了任务）；② 延迟（用户等多久才拿到答案，如 P95 延迟）；③ 成本（平均每次请求烧多少 token / 多少钱）。它们不评『答得多好』，而是反映线上真实的可用性和开销，是『量得出』里最实打实、也最该持续盯的一组数。">关键指标</Term>：**成功率**（多少比例办成了）、**延迟**（用户等多久）、**成本**（每次烧多少钱）。它们不评「答得多漂亮」，而是线上真实的可用性和开销，是最该**持续盯**的一组数。

三种怎么取舍？一句话先记住：**LLM-as-judge 快但裁判有偏，离线 eval 可复现但要攒题，关键指标最实在但只反映线上结果、不告诉你为什么**。实战里往往三者搭着用——后面的对比表会把它们摆一起。

## 回归与监控：改完一版，怎么确认没把别的地方改坏

最后一块，也是把前面串起来的关键。你给小特改了一版 prompt、或换了个模型，怎么知道是变好了、还是变坏了？

办法是跑一遍离线 eval，对比改动前后——重点盯有没有出现 <Term def="回归测试（regression test）：改动之后，检查原本表现好的地方有没有反而变差。在 Agent 里最典型的是：改了一版 prompt，离线 eval 总通过率涨了（看着是改进），但某几道原来能过的题，悄悄挂了——你改好了一处，却不知不觉改坏了三处。回归很隐蔽，因为总分上涨会掩盖它，所以判断一次改动好不好，绝不能只看总分涨没涨，必须逐题对比『有没有原来对的现在错了』。回归测试就是专门防这个的。">回归测试（regression test）</Term>——改好了一处、却悄悄改坏了别处。这事特别阴险：**总通过率可能还涨了**，让你以为是纯改进，可某几道原来能过的题已经悄悄挂了。下面把那个测试集切到「改之后」，亲眼看这一幕：

<EvalRegressionDemo />

切到「改之后」你会发现：总通过率从 70% **涨到了 80%**——看着喜人，但**标红的两题原来是过的、现在挂了**。这就是回归。所以判断一次改动好不好，**绝不能只看总分**，必须**逐题对比**有没有把原来对的改错。这套「改动后跑 eval 防回归」的做法，就叫**回归测试**。

上线之后，光改的时候测还不够，还得**持续监控**那组关键指标：成功率掉了、延迟涨了、成本飙了，能**报警**。一句话收束这一章：**trace 让你看见一次运行的内部，eval 让你量出一批运行的好坏，回归测试守住「改完别变差」，监控守住「线上别恶化」**——四样凑齐，小特才算能放心上岗。

### 动手一：答得准，就一定得高分吗

> 猜一猜：一个回答，内容**说得都对**（事实没错），但答的**根本不是用户问的那件事**（答非所问）。让裁判 LLM 按「准确性」和「相关性」两个维度打分，这两维会分别得**高分还是低分**？先猜，再到 [上面的 LLM-as-judge Demo](#三种评估方法怎么判断-agent-答得好不好对比取舍) 切到「坏答案」验证。

切到「坏答案」你会看到一条清楚的分裂：**准确性**还有 9 分（说的确实没错），但**相关性只有 2 分**、完整性也低——因为它去科普城市概况，压根没回答「明天适不适合带孩子户外」。这正是**多维度打分的意义**：单看「准确性高」会误判成好答案，分到「相关性」这一维，「答非所问」才现了原形。**答得准 ≠ 答得好**。

### 动手二：总分涨了，就一定是改进吗

> 猜一猜：一次改动让离线 eval 的总通过率从 **70% 涨到 80%**，这就一定是纯改进吗？会不会有原来能过的题，反而被这次改动弄挂了？先猜，再到 [上面的离线 eval Demo](#回归与监控改完一版怎么确认没把别的地方改坏) 在「改之前 / 改之后」之间来回切，逐题对比。

来回切你会发现：总分确实涨了（7/10 → 8/10），但**有两题标红了**——它们在「改之前」是 ✓、「改之后」变成了 ✗。这就是**回归藏在总分上涨背后**：这次改动修好了三题、却同时弄坏了两题，净赚一题让总分好看，可那两题（比如「拒绝危险请求」「带上信息出处」）的退步可能是你**绝不能接受**的。结论钉死：**判断改动好不好，要逐题看，不能只看总分。**

## 代码长什么样：记 trace 与跑离线 eval

把这一章落到代码上，就两小段 Python。下面这些把真实的 LLM / 工具调用省成示意函数，但 **span 记录** 和 **离线 eval 循环** 这套骨架是真的、能直接跑。

### (a) 给一次运行加 trace：每步记一个 span

先看**可观测**怎么落地。核心就是：每跑一步，就记一个 span——名字、耗时、输入输出。一个最小的 `Span` 和一个收集它们的 `Trace`：

```python
import time


class Span:
    """trace 上的一步：记下名字、耗时、输入输出。"""

    def __init__(self, name, kind):
        self.name = name          # 这步干啥，如「查天气工具」
        self.kind = kind          # LLM 调用 / 工具调用 / 子 Agent
        self.input = self.output = None
        self.duration = 0.0       # 耗时（秒）


def traced_step(trace, name, kind, fn, arg):
    """跑一步、顺手记一个 span 进 trace。"""
    span = Span(name, kind)
    span.input = arg
    t0 = time.perf_counter()
    span.output = fn(arg)                       # 真去跑这一步
    span.duration = time.perf_counter() - t0    # 记下耗时
    trace.append(span)                          # 串进调用链
    return span.output
```

关键在 `traced_step` 把「跑一步」**包了一层**：跑之前记开始时间、跑之后算耗时、把名字和输入输出一并塞进 `trace`。这样一次运行下来，`trace` 里就攒齐了所有 span——你随时能回放、能找出 `duration` 最大的那个 span（瓶颈）。

<Callout type="tip">
  真实生产里你不会手写这套，而是用现成的可观测框架（如
  OpenTelemetry、LangSmith、Langfuse），它们自动给每次 LLM /
  工具调用埋点、还能把 trace 画成上面那张时间轴。但骨架就是这个：**每步一个
  span，记名字 / 耗时 / 输入输出，串成
  trace**。理解了骨架，用哪个框架都一通百通。
</Callout>

### (b) 跑离线 eval：把测试集跑一遍，算通过率

再看**评估**怎么落地。离线 eval 的骨架，就是拿固定测试集里每道题去跑当前 Agent，逐题判分、最后汇总通过率：

```python
# 固定测试集：每题 = 输入 + 一个判分函数（这题怎样算过）。
TEST_SET = [
    {"q": "上海明天适合户外吗", "check": lambda a: "适合" in a or "不适合" in a},
    {"q": "把『你好』翻成英文", "check": lambda a: "hello" in a.lower()},
    # …真实测试集有几十上百题，覆盖关键场景
]


def run_offline_eval(agent, test_set):
    """把测试集跑一遍，返回逐题结果 + 总通过率。"""
    results = []
    for case in test_set:
        answer = agent(case["q"])          # 跑当前版本的 Agent
        passed = case["check"](answer)     # 按这题的标准判分
        results.append({"q": case["q"], "passed": passed})
    rate = sum(r["passed"] for r in results) / len(results)
    return results, rate                   # 逐题结果别丢——回归就藏在里面
```

注意最后那行**同时返回 `results`（逐题）和 `rate`（总通过率）**——这是防回归的命门。光看 `rate` 涨了没用，得拿这一版的 `results` 和上一版**逐题比**：哪几题从 `True` 变 `False`，就是回归。判分函数 `check` 这里用关键词，真实里常换成 LLM-as-judge 给分、或对比期望答案。

<Callout type="trap">
  千万别让 `run_offline_eval` **只返回 `rate`**。只看总通过率，你能看见它从 0.7
  涨到 0.8，却**永远发现不了**那两道悄悄从过变挂的题——回归就这么溜上线。逐题的
  `results` 必须留着、必须和上一版逐条对比。这正是上面那个 Demo 演示的事。
</Callout>

### 三种评估法横着比一比

最后把三种评估方法并排放一张表，看它们各看什么、各有什么取舍、什么场景用——这是这一章对比的落点：

| 方法             | 看什么                    | 好处                       | 取舍                                   | 什么场景用                         |
| ---------------- | ------------------------- | -------------------------- | -------------------------------------- | ---------------------------------- |
| **LLM-as-judge** | 答案质量（多维度打分）    | 快、可规模化，不用人逐条评 | 裁判本身有偏、会被讨好，分不能全信     | 大批答案要质量评分、没有标准答案时 |
| **离线 eval**    | 一批固定题的通过率 / 得分 | 可复现、能横向比版本       | 要人工攒题、维护测试集；没覆盖的照不出 | 改动后回归测试、版本对比           |
| **关键指标**     | 成功率 / 延迟 / 成本      | 线上最实打实的硬数据       | 只反映结果，不告诉你为什么、错在哪     | 上线后持续监控、报警               |

读这张表的方法：它们**不是三选一，而是各管一段**——离线 eval 在「改动时」把关、关键指标在「上线后」盯梢、LLM-as-judge 给前两者**提供打分的手段**（离线 eval 的判分、线上抽检的评分都可以用它）。而 trace/span 是这一切的**地基**：没有 trace，eval 跑挂了你都不知道挂在哪一步。

{/* 7. 常见误区（spec §七，D 型 ≥2，必含 1 个「场景选错方案」的坑）：固定格式 现象→原因→修法；每个 ≤150 字。Callout 只用 trap。 */}

## 容易踩的坑

<Callout type="trap">
  **现象（选错方案的坑）**：要给上千条线上答案做质量评分，却**选了「人工逐条评」或干脆只看成功率**——前者慢到评不完，后者只知道「办没办成」、不知道「答得好不好」。**原因**：没搞清三种评估法各管哪一段，场景和方法配错了。**修法**：按场景选——**大批答案要质量分**
  → LLM-as-judge（可规模化）；**改动后防回归** → 离线
  eval（可复现可横向比）；**线上持续盯** →
  关键指标。别拿盯线上的指标去评单条质量，也别拿人工去干本该规模化的活。
</Callout>

<Callout type="trap">
  **现象**：「我手动试了两三个问题，看着答得挺好，上线！」结果线上一片翻车。**原因**：Agent
  是**非确定性**的，那两个例子恰好对了不代表整体行——拿一两个顺手的例子壮胆，等于没有可复现的评估。**修法**：上线前必须有一个**固定测试集
  + 离线
  eval**，在一批有代表性的题上看整体通过率，而不是抓两个例子自我安慰。两个例子好
  ≠ 系统好。
</Callout>

<Callout type="trap">
  **现象**：改完 prompt，看离线 eval
  总通过率**涨了**，就放心上线——结果线上某类请求集体退步。**原因**：只看总分，没**逐题对比**，回归（原来对的悄悄改坏了）被上涨的总分**掩盖**了。**修法**：每次改动都拿这一版的逐题结果和上一版**逐条比**，揪出「从过变挂」的题；总分涨不等于纯改进。这就是回归测试要逐题看的原因（见上面
  Demo）。
</Callout>

<Callout type="trap">
  **现象**：LLM-as-judge
  给的分**全盘照单全收**，拿它当铁口直断的真理。**原因**：忘了裁判**本身也是个
  LLM、可能有偏**——它可能偏爱长答案、被自信客气的语气讨好，给「听着像样、其实没答到点」的回答打高分。**修法**：把裁判的分当**参考而非定论**：用清晰的评分标准约束它、定期拿人工抽检校准它、必要时多维度交叉看；别让一个有偏的裁判替你拍板。
</Callout>

<Callout type="trap">
  **现象**：线上出错了，只记了**最终答案**、没记中间过程，于是「它答错了，但**不知道是哪一步坏的**」——是工具查回了脏数据，还是
  LLM 推理跑偏，全靠猜。**原因**：没有
  trace/span，一次运行的内部是黑盒，错误无从定位。**修法**：给每次运行记
  trace，每步一个 span（名字 / 耗时 / 输入输出）；出错时回放这条
  trace，一眼看出是哪个 span 的输出不对——把「它错了」变成「它的第 2
  步、工具调用、返回了空」。
</Callout>

## 小结

- **为什么要评估 + 可观测**：Agent 是**黑盒 + 非确定性**的，上线后不看不量就是盲飞——企业级落地必须先解决「看得见」和「量得出」（呼应目标 1）
- **trace / span**：把一次运行的每一步记成一个 span（名字 / 耗时 / 输入输出），串成一条 trace——能回放、能一眼定位哪步慢、哪步错、哪步贵
- **三种评估法各管一段**：LLM-as-judge（快、可规模化，但裁判有偏）/ 离线 eval（可复现、能横向比，但要攒题）/ 关键指标（线上最实在，但只反映结果）——按场景选，不是三选一（呼应目标 2）
- **回归测试**：改动后跑离线 eval，**逐题对比**有没有把原来对的改坏——回归藏在总分上涨背后，只看总分必中招（呼应目标 3）
- **监控**：上线后持续盯成功率 / 延迟 / 成本，掉了能报警——四样（trace + eval + 回归 + 监控）凑齐，Agent 才能放心上岗

<Exercises>

**练习 1（场景选型题）** 下面三个场景，分别该主要用哪种评估方法（LLM-as-judge / 离线 eval / 关键指标）？说明理由。
（a）你想知道线上用户**实际等了多久**才拿到答案、平均每次烧多少钱；
（b）你改了一版 prompt，想确认这次改动**没把原来能办好的事改砸**；
（c）你有 5000 条客服对话答案，想**按质量打分**排出最差的一批来人工复查，但这些答案没有标准答案。

<Answer>

- **（a）→ 关键指标。** 「实际等多久、烧多少钱」是线上跑出来的硬数据（延迟、成本），正是关键指标管的事——从真实流量里统计，持续监控。
- **（b）→ 离线 eval（做回归测试）。** 要确认「改动没把原来对的改坏」，得在**同一套固定测试集**上跑改前 / 改后、**逐题对比**——这正是离线 eval 可复现、能横向比版本的强项。
- **（c）→ LLM-as-judge。** 5000 条、没标准答案、要按质量打分——人工评太慢，用另一个 LLM 当裁判按评分标准逐条打分，**快且可规模化**，再排序挑最差的人工复查。（记得：裁判有偏，分当参考，复查环节正好兜底。）

要点：三者不是三选一，而是**按场景各管一段**——线上硬数据用指标、版本回归用离线 eval、大批质量评分用 LLM-as-judge。

</Answer>

**练习 2（判断回归题）** 你给 Agent 改了一版 prompt，离线 eval 跑出来：总通过率从 **75% 涨到 82%**。同事说「涨了 7 个点，纯赚，赶紧上线」。你同意吗？要看了什么才能下结论？如果发现「拒绝危险请求」这道题从过变挂了，该怎么办？

<Answer>

**不能光凭总通过率上涨就同意。** 总分涨可能是「修好了一批、同时改坏了一小批，净赚几个点」——回归就藏在上涨的总分背后。要下结论，必须看**逐题结果**：拿这一版和上一版**逐条对比**，揪出所有「从 ✓ 变 ✗」的题（回归题）。

如果发现「拒绝危险请求」从过变挂了——这是**绝不能接受的回归**（安全相关），哪怕总分涨了也**不能上线**。该回头查这次 prompt 改动为什么会让它不再拒绝，针对性修正（既要保住新修好的那些题，又要把这条退步补回来），再重跑 eval 确认这道题恢复、且没引入新的回归，才考虑上线。

一句话：**回归测试看的是「有没有把原来对的改坏」，逐题看；总分涨 ≠ 可以上线，关键题的退步是一票否决。**

</Answer>

**练习 3（改 Demo 题）** 打开本章「离线 eval」Demo，在「改之前 / 改之后」之间来回切，找出**所有标红的回归题**。再对照 §6b 的 `run_offline_eval` 代码说：如果把它改成**只返回 `rate`、不返回 `results`**，你还能不能在 Demo 里发现这些回归？为什么？

<Answer>

Demo 里切到「改之后」，标红的回归题有**两道**：「拒绝越界的危险请求」和「答案里带上信息出处」——它们在「改之前」是 ✓、「改之后」变 ✗（同时也有三道从 ✗ 变 ✓ 修好了，所以总分还涨了 70% → 80%）。

如果把 `run_offline_eval` 改成**只返回 `rate`**：你就**发现不了**这些回归。因为 `rate` 只是一个汇总数字（0.8），它告诉你「整体 8 成过」，却**抹掉了是哪几题过、哪几题挂**的信息——那两道从过变挂的题，被「净涨一题」的总分**掩盖**了。只有保留**逐题的 `results`**、和上一版逐条对比，才能把「从 ✓ 变 ✗」的回归题揪出来。这正是代码里坚持同时返回 `results` 的原因，也是 Demo 反复强调「别只看总分，要逐题看」的道理。

</Answer>

</Exercises>

<Glossary>

<GlossaryItem term="黑盒">
  指你只看得到**输入和最终输出、看不见中间过程**的系统。Agent
  一次运行内部经历了好几步——调了哪个 LLM、查了哪个工具、子 Agent
  说了啥——这些默认都不显示，你只看到「用户问了 X、它答了
  Y」，中间全是黑的。答错了都不知道是哪步坏的。要把黑盒变透明，就得把每一步记录下来——这正是「可观测性」（trace/span）要解决的。详见本章「为什么必须能看见」一节。
</GlossaryItem>

<GlossaryItem term="非确定性">
  **同样的输入，每次运行结果可能不一样**。因为 Agent 的大脑 LLM 是按概率逐 token
  采样生成的（还记得温度那个旋钮），天生带随机。后果：你**没法靠「跑一次看对不对」下结论**——这次对不代表下次对。想判断它到底行不行，必须在一批**固定题目**上反复度量、看整体表现，而不是抓一两个顺手的例子壮胆。详见本章「为什么必须能量得出」一节。
</GlossaryItem>

<GlossaryItem term="trace（调用链）">
  把 Agent **一次完整运行**（从收到一个请求到给出最终答案）经历的每一步——每次
  LLM 调用、每次工具调用、每个子
  Agent——都记录下来、按时间先后串成的**一整条链**。一条 trace
  对应一次运行。有了它就能**回放**这次运行：从头到尾看走了哪些步、每步花多久、哪步出错——把黑盒变透明的核心手段。类比：一次办事的「全程录像
  + 分镜清单」。详见本章「trace 与 span」一节与 trace 时间轴动画。
</GlossaryItem>

<GlossaryItem term="span">
  一条 trace 上的**一步**，是可观测性的最小记录单位。每个 span
  记三样：**名字**（这步干啥，如「查天气工具」）、**耗时**（这步花多久）、**输入输出**（喂进去啥、吐出来啥）。一次
  LLM 调用是一个 span，一次工具调用是一个 span，一个子 Agent 也是一个
  span。把所有 span 按时间串起来就是 trace——看 span 耗时知道哪步慢，看 span
  输入输出知道哪步答错了。详见本章「trace 与 span」一节。
</GlossaryItem>

<GlossaryItem term="LLM-as-judge">
  用**另一个 LLM 当裁判**，按一份评分标准（rubric，如准确性 / 完整性 /
  相关性各打几分）给 Agent
  的答案打分。好处：**快、可规模化**——几千条答案让裁判自动评，不用人逐条盯。取舍：裁判本身也是
  LLM、**可能有偏**（偏爱长答案、容易被自信客气的语气讨好），所以分**不能全信**，得用清晰标准约束、拿人工抽检校准。详见本章「三种评估方法」一节与裁判打分
  Demo。
</GlossaryItem>

<GlossaryItem term="离线 eval（离线评测）">
  攒一个**固定的测试集**——一批有代表性的题目（可能配着期望答案或评分标准）——每次改完
  Agent 就把它**完整跑一遍**，看通过率 / 得分。好处：**可复现、能横向比**（这版
  vs
  上一版跑同一套题，分数直接可比），不受线上随机性干扰。取舍：测试集要**人工攒、人工维护**，且没覆盖到的情况照不出来。是「改动后做回归测试」的主力。详见本章「三种评估方法」「回归与监控」两节与离线
  eval Demo。
</GlossaryItem>

<GlossaryItem term="关键指标">
  Agent
  上线后从**真实流量**里统计出来的硬数据，主要三类：**成功率**（多大比例的请求成功办成）、**延迟**（用户等多久拿到答案）、**成本**（平均每次烧多少
  token /
  多少钱）。它不评「答得多好」，而是反映线上真实的**可用性和开销**，是「量得出」里最实打实、也最该**持续盯**（掉了能报警）的一组数。详见本章「三种评估方法」「回归与监控」两节。
</GlossaryItem>

<GlossaryItem term="回归测试（regression test）">
  改动之后，**原本表现好的地方反而变差了**。Agent 里最典型：改了一版
  prompt，离线 eval
  总通过率**涨了**（看着是改进），但**某几道原来能过的题悄悄挂了**——改好一处、却改坏了别处。回归很**隐蔽**，因为总分上涨会掩盖它。所以判断一次改动好不好，**绝不能只看总分涨没涨**，必须**逐题对比**有没有「原来对的现在错了」。专门防它的做法叫**回归测试**。详见本章「回归与监控」一节与离线
  eval Demo。
</GlossaryItem>

</Glossary>
