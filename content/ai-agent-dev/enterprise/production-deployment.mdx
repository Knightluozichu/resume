## 小特要从一个人，变成一支正规军

前面整本书，我们把私人助理**小特**从里到外打磨了一遍：有了脑子、会办事、会查资料、能带小队，还装了护栏、会评估。可它一直是**一个人**，服务的也只是你一个老板。

现在它要真正**上岗**了——给成千上万真实用户同时干活。这时候问题全冒出来：这么多人同时找它，一个个排着来根本忙不过来；它干一件事要好几秒（每次都得问大脑），万一某次大脑没回话、卡住了，用户就干等着；你想给它换套更好的工作方式，一上来就让它对所有人用新方式，万一新方式有毛病，所有人当场全遭殃。

这一章就是把小特从「一个能干活的人」升级成「一支**能扛海量用户的正规军**」：有分工的架构、抗得住并发、出错能自己兜底、上线能一点点试。不办这些，它上岗第一天就会**被人潮挤垮、被一次卡顿拖死、被一次坏更新炸穿**——能在你电脑上跑，离能上线服务真实用户，还差着这一整套。

## 生产架构：上线不是一个脚本，是一套架构

先建立最重要的一个直觉：**Agent 上线，不是把你本地那个脚本挂到服务器上就完事**。它得是一套有分工的<Term def="生产架构：Agent 真正上线服务海量用户时，背后那套有分工的系统结构，而不是一个单独的脚本。典型组件：① API 网关——所有请求的统一入口，负责收请求、鉴权（确认你是谁、有没有权限）、限流（限制每个用户的调用频率）；② 队列——突发请求先在这排队，削峰、把『收请求』和『处理请求』解耦成异步；③ Agent 服务——真正干活的地方，从队列取请求、调 LLM / 工具，可以水平扩成多个实例分摊压力；④ 缓存——把问过的、重复 / 相似的请求结果存起来，命中就直接返回、不再调模型；⑤ 可观测——全程记录 trace / 指标（第12章那套），哪步慢 / 错 / 贵都看得见。一句话：上线 = 一套架构，不是一个脚本。">生产架构</Term>——一条请求要穿过好几个各司其职的组件。

这套架构典型长这样，按职责分三段：

- **接入层**：**API 网关**——所有请求的统一入口，负责收请求、**鉴权**（确认你是谁、有没有权限）、**限流**（每人每秒最多发几次，挡住滥用和洪峰）。
- **处理层**：**队列**（突发请求先排队、削峰）→ **Agent 服务**（真干活：调 LLM / 工具，可以多开几个实例分摊压力）。
- **支撑层**：**缓存**（重复请求命中就直接返回、不调模型）+ **可观测**（trace / 指标全程记录，就是第12章那套）。

## 并发与异步：海量请求同时来，一个个串着处理必死

接着看这套架构要解决的第一个硬骨头：**海量请求同时来，而 Agent 一次调用很慢**（问一次大脑就得好几秒）。如果像本地脚本那样一个处理完再处理下一个（串行），后面的人能排到天荒地老。

对策是<Term def="并发与异步：海量请求同时到达、而 Agent 每次处理又很慢（LLM 延迟好几秒）时，让系统能『同时推进很多请求、不互相干等』的做法。三个关键手段：① 异步——『收下请求』和『真正处理请求』解耦：请求先入队、立刻给用户一个『收到了，在处理』，后台再慢慢取出来处理，不让用户的连接一直阻塞等着；② 队列削峰——突发洪峰流量先在队列里排队，后端按自己的处理能力匀速取出来做，避免瞬时洪峰直接压垮 Agent 服务；③ 并发控制——限制『同时在跑』的请求数（比如最多 50 个并发），别一次性把所有请求都丢给 LLM / 工具，否则会把下游打爆、触发对方限流甚至全线雪崩。一句话：异步让请求不互相等，削峰和并发控制让系统不被冲垮。">并发与异步</Term>，三招配合：

- **异步**：把「收请求」和「处理请求」拆开。请求先**入队**、立刻回一句「收到了，在处理」，后台再慢慢取出来干，**不让用户的连接一直挂着干等**。
- **队列削峰**：突发洪峰先在队列里**排队**，后端按自己的处理能力**匀速取**——别让瞬时的流量高峰直接砸到 Agent 服务上。
- **并发控制**：限制**同时在跑**的请求数（比如最多 50 个并发）。别一股脑把所有请求都丢给 LLM / 工具，否则会**把下游打爆**——触发对方限流、甚至全线雪崩。

上面那张架构图里的**队列**和**Agent 服务（可水平扩多个实例）**，正是这三招的落点。代码长什么样，§6 会一段段写出来。

## 超时·降级·重试：生产里，调用一定会失败

生产环境有条铁律：**你依赖的 LLM、工具，迟早会超时、会失败**。网络抖一下、对方服务忙、偶发报错——本地跑十次没事，上线跑十万次必然踩到。所以可靠性要靠「三件套」兜住，这是本章代码的另一个重心：

- **超时**：给每一步设个<Term def="超时（timeout）：给一次调用设一个『最多等多久』的上限，到点还没返回就主动掐断、不再傻等。为什么必须有：LLM / 工具偶尔会卡住很久甚至永远不返回，如果不设超时，用户的请求就会一直挂着干等，体验极差，还可能把服务的连接 / 线程占满拖垮整个系统。设了超时，一步卡太久就掐断，转而走重试或降级。">超时</Term>上限——卡太久就**主动掐断**，别让用户傻等、也别让一个慢请求占着资源拖垮整个服务。
- **重试**：失败了**自动再试几次**（通常配合「等一会儿再试」的退避）。但有个前提——只对<Term def="幂等（idempotent）：一个操作执行一次和执行多次，效果完全一样。比如『查天气』是幂等的——查一次查三次结果都一样、没有副作用；而『下单』『扣款』『发邮件』不是幂等的——重复执行会重复下单、重复扣款。为什么和重试有关：重试 = 失败了自动再做一次，所以只能对幂等操作无脑重试；对非幂等操作（下单 / 扣款）盲目重试，会造成重复下单、重复扣款等严重后果，必须先用唯一请求 ID 等手段做去重，确认这次重试不会被重复执行，才能重试。">幂等</Term>的操作（做一次和做多次效果一样，比如查询）才能无脑重试，对下单 / 扣款这类**重试了会重复执行**的操作要格外小心。
- **降级 / 兜底**：主模型挂了就<Term def="降级 / 兜底（fallback）：当首选方案失败时，自动退而求其次，保证用户至少拿到一个『还能用』的结果，而不是一个错误。在 Agent 里典型有两层：① 降级到备用方案——主模型超时 / 失败时，切换到一个备用模型（可能更小、更便宜但能用）继续把活干完；② 兜底答案——如果连备用方案也失败了（主备都挂），返回一个预先准备好的、安全得体的兜底回复（比如『服务繁忙，请稍后再试』），而不是把内部的报错堆栈直接抛给用户。核心原则：永远不让用户直接撞见原始错误——出错时降级，把失败兜成一个体面的结果。">降级</Term>到**备用模型**，备用也不行就返回一个**安全的兜底答案**（「服务繁忙，请稍后再试」），**绝不把内部报错直接抛给用户**。

## 灰度发布与回滚：新版本别一上来就全量

最后一件，关乎**上线本身**。你改了一版 prompt、换了个新模型，怎么发布到线上？最危险的做法是**一上来就把 100% 流量切过去**——万一新版本有问题，**所有用户同时遭殃**。

稳妥的做法是<Term def="灰度发布 / 金丝雀发布（canary release）：新版本上线时，不一上来就全量，而是先只放一小部分流量（比如 5%）给新版本，盯住第12章那些关键指标（成功率 / 延迟 / 成本），确认没问题，再逐步放量 5% → 50% → 100%。名字来自旧时矿工带金丝雀下井探毒气——用一小撮流量当『探子』先试，有问题也只影响这一小撮人。好处：把『一上来全量、有问题瞬间炸全部用户』的大赌注，拆成一档档可控、可观察的小步，风险大大降低。">灰度发布</Term>（也叫金丝雀发布）：**先放 5% 流量**给新版本，盯住第12章那些指标（成功率 / 延迟 / 成本），没问题再 **50% → 100%** 一档档放量。一旦哪档指标变差，立刻<Term def="回滚（rollback）：发现新版本有问题时，立刻把流量切回稳定的老版本，撤下新版本。它是灰度发布的『安全绳』——灰度时如果某一档指标变差（成功率掉了 / 延迟涨了 / 成本飙了），不必慌乱排查，先一键回滚让线上恢复稳定，再从容定位新版本的问题。有了快速回滚，上线才敢大胆试：因为知道一旦出事，几秒钟就能撤回到已知良好的状态，损失被牢牢封顶。">回滚</Term>——切回老版本，撤下新版本。

灰度的好处：把「一上来全量、有问题瞬间炸全部用户」的大赌注，拆成**一档档可控的小步**；回滚是它的安全绳——出事能秒撤、损失封顶。下面这个 Demo 让你当一回发布负责人，一档档放量、盯着指标决定继续还是回滚（「猜一猜」同样在 §5）：

> 猜一猜：你的 Agent 调主模型时**主模型超时了**。如果它做了降级（超时 + 重试 + 备用 + 兜底），用户会**直接看到一个报错**吗，还是有别的办法兜住？先猜，再到 [上面的降级 Demo](#超时降级重试生产里调用一定会失败) 切「主模型超时 / 全挂」看它走哪条路。

切「主模型超时」你会看到一条清楚的路径：主模型超时 → **掐断** → 自动**重试**一次仍超时 → **降级到备用模型**，备用正常返回——**用户拿到的是备用模型的答案，全程没看到任何报错**。再切「全挂」：主备都失败时，它**返回一句安全的兜底答案**（「服务繁忙，请稍后再试」），而不是把内部错误堆栈甩给用户。结论钉死：**做了降级的 Agent，出错时不是报错，而是一层层把失败接住**——超时掐断、重试、切备用、给兜底，用户体验不崩、内部信息也不外泄。

### 动手二：新版本在 5% 灰度时成功率掉了，该继续放到 50% 还是回滚

> 猜一猜：你的新版本在 **5% 灰度**时一切正常，可放到 **50%** 时成功率突然从 98% 掉到 92%、延迟也飙了。这时该**继续放到 100%**，还是**回滚**？先猜，再到 [上面的灰度 Demo](#灰度发布与回滚新版本别一上来就全量) 一档档点过去验证。

点「指标正常 → 继续放量」从 5% 走到 50% 档，你会看到指标**变差了**（成功率掉、P95 延迟飙）。这时正确动作是点「**指标变差 → 回滚**」：流量立刻切回老版本，**只有那 50% 里的一小撮请求受了点影响，绝大多数用户始终用着稳定的老版本**。要是头铁继续放到 100%，问题就**炸到全部用户**了。这正是灰度 + 回滚的价值：**用一小撮流量当探子先试，出事能秒撤、损失封顶**——上线从一场豪赌，变成一步步可控、可撤的小步。

## 代码长什么样：异步入队、超时重试、降级兜底、限并发

这一章的重心在代码。下面四段把真实的 LLM / 工具调用省成示意函数，但**异步入队**、**超时重试**、**降级兜底**、**限并发**这套骨架是真的、能直接跑。Python 用标准库 `asyncio`，这是生产里写异步 Agent 服务最常见的底子。

### (a) 异步处理：请求先入队，后台慢慢取来跑

先看并发的底子——异步 + 队列。核心是把「收请求」和「处理请求」拆开：收到请求只管丢进队列、立刻返回；后台的 worker 从队列里取出来慢慢跑。这样海量请求来了不会互相干等：

```python
import asyncio

queue: asyncio.Queue = asyncio.Queue()


async def enqueue(request):
    """收到请求：只管丢进队列、立刻返回『收到了』，不阻塞用户。"""
    await queue.put(request)
    return {"status": "accepted"}  # 立刻回应，处理在后台异步进行


async def worker(name):
    """后台 worker：不停从队列取请求来处理（可同时跑多个 worker）。"""
    while True:
        request = await queue.get()       # 队列空就在这等，不空转
        try:
            answer = await handle(request)  # await：等 LLM 时不占住线程
            await deliver(request, answer)
        finally:
            queue.task_done()
```

关键在两点：`enqueue` 收下请求就**立刻返回**（用户不用挂着等几秒），真正的活丢给后台；`worker` 里的 `await handle(...)` 是异步等待——**等 LLM 回话的那几秒，这个 worker 不会占住，能去推进别的请求**。再开几个 `worker` 协程（`worker("w1")`、`worker("w2")`…），就能同时处理多条请求。这就是「队列削峰 + 异步」落到代码上的样子。

<Callout type="tip">
  `async` / `await` 的精髓：Agent 大部分时间在**等**（等 LLM、等工具返回），CPU
  其实闲着。异步让这段「等待」可以被别的请求利用——一个 worker 等着第一条请求的
  LLM 返回时，正好去启动第二条。这正是 Agent 这种「慢 +
  高并发」场景特别适合异步的原因。
</Callout>

### (b) 超时 + 重试：一步卡太久就掐断，失败了再试几次

再看可靠性的第一层——超时 + 重试。给调用包一层：用 `asyncio.wait_for` 设超时，超了就掐断；失败 / 超时就重试几次，每次之间**退避**（等得越来越久，别猛捶对方）：

```python
import asyncio


async def call_with_retry(coro_fn, *, timeout=3.0, retries=2):
    """带超时的调用 + 失败重试。只对幂等操作这么用！"""
    last_err = None
    for attempt in range(retries + 1):
        try:
            # 单次调用设超时：超过 timeout 秒还没返回就掐断
            return await asyncio.wait_for(coro_fn(), timeout=timeout)
        except (asyncio.TimeoutError, TransientError) as err:
            last_err = err
            if attempt < retries:
                await asyncio.sleep(2**attempt)  # 退避：1s, 2s, 4s… 越等越久
    raise last_err  # 重试都用完了还不行，抛出去让上层降级
```

关键在 `asyncio.wait_for(coro_fn(), timeout=timeout)`：它给这次调用套了个**闹钟**，到点没返回就抛 `TimeoutError`、不再傻等。外层 `for` 循环捕获超时 / 瞬时错误后**重试**，并用 `2**attempt` 做**退避**（第 1 次等 1 秒、第 2 次等 2 秒…），避免在对方正忙时反复猛捶。重试用完仍失败，就把错误**抛给上层去降级**（下一段）。

<Callout type="warn">
  重试有个**绝对前提**：只对**幂等**操作无脑重试。`call_with_retry`
  适合「查天气、查订单」这类查询——重试几次结果一样、没副作用。但**绝不能**直接拿它去重试「下单
  /
  扣款」——网络抖一下让你以为失败、实际对方已经执行了，一重试就**重复下单、重复扣款**。非幂等操作要先用唯一请求
  ID 去重，确认这次不会被重复执行，才能重试。
</Callout>

### (c) 降级兜底：主模型挂了切备用，再不行给安全兜底答案

可靠性的第二层——降级。把上面的「带超时重试的调用」叠起来：先试主模型，超时 / 失败就切备用模型，备用也不行就返回一个**安全兜底答案**，全程不让用户撞见原始错误：

```python
async def ask_with_fallback(prompt):
    """主模型 → 备用模型 → 安全兜底答案，逐层降级。"""
    try:
        # 第一层：主模型（带超时 + 重试）
        return await call_with_retry(lambda: call_llm("gpt-main", prompt))
    except Exception:
        pass  # 主模型彻底不行，往下降级（生产里这里要记一条 trace / 告警）

    try:
        # 第二层：降级到备用模型（可能更小 / 更便宜，但能用）
        return await call_with_retry(lambda: call_llm("gpt-backup", prompt))
    except Exception:
        pass  # 备用也挂了，走最后的兜底

    # 第三层：兜底答案——安全、得体，绝不把内部报错抛给用户
    return "抱歉，服务暂时繁忙，请稍后再试。"
```

关键在三层 `try` 的**逐级退让**：主模型失败 → 切 `gpt-backup` 备用 → 备用也失败 → 返回那句安全兜底答案。无论哪一层挂，用户拿到的都是**一个能用的结果或一句体面的话**，而不是一个吓人的报错堆栈——这正是上面 `DegradationDemo` 演示的三条路径。注意每次降级处生产里都该**记一条 trace / 触发告警**（第12章那套），别让降级悄悄发生而你不知道主模型已经挂了。

### (d) 限并发：用 Semaphore 框住同时在跑的请求数

最后看并发控制——别把下游打爆。`asyncio.Semaphore` 是个「令牌池」：同时只发出去 N 个令牌，第 N+1 个请求得等前面的还回令牌才能开跑：

```python
import asyncio

# 全局并发闸：同时最多 50 个请求在调下游（保护 LLM / 工具不被打爆）
llm_semaphore = asyncio.Semaphore(50)


async def handle(request):
    """处理一条请求：进闸 → 调用 → 出闸，把并发数框在 50 以内。"""
    async with llm_semaphore:        # 拿不到令牌就在这等，不会冲垮下游
        return await ask_with_fallback(request["prompt"])
    # 离开 async with，令牌自动归还，下一个等待的请求才能进来
```

关键在 `async with llm_semaphore`：它保证**同时最多 50 个请求**在调下游，第 51 个会乖乖在这排队等。为什么要框？因为下游（LLM API、数据库）有它自己的承受上限——你一次性把上千个请求全砸过去，会触发对方限流、甚至把它打趴，反而谁都别想成功。限并发就是**主动给自己踩个刹车，换来整体的稳**。把 (a)(b)(c)(d) 拼起来，就是一个能扛并发、会重试降级、不打爆下游的生产级 Agent 服务雏形。

## 容易踩的坑

<Callout type="trap">
  **现象**：调 LLM /
  工具时**不设超时**，本地测一切顺，上线后偶尔整个服务卡死、大批用户一起转圈。**原因**：某次
  LLM / 工具卡住很久甚至永不返回，没有超时就一直傻等——这个慢请求占着连接 /
  线程不放，攒多了把服务的资源占满，**一个慢请求拖垮整个服务**。**修法**：给每一次外部调用都设**超时**（如
  `asyncio.wait_for`），到点掐断，转去重试或降级。生产里「没有超时的调用」≈「定时炸弹」。
</Callout>

<Callout type="trap">
  **现象**：Agent 出错时，把捕获到的**原始异常 /
  错误堆栈直接返回给用户**——用户看到一屏看不懂的报错，里面还可能露出内部 API
  key、文件路径、数据库结构。**原因**：没做降级兜底，把「内部错误」当「给用户的回答」抛了出去——既砸体验，又**泄露内部信息**给潜在攻击者。**修法**：永远不让用户撞见原始错误——出错时**降级**（切备用模型）或返回一个**安全、笼统的兜底答案**；详细错误只记进内部
  trace / 日志，不外泄。
</Callout>

<Callout type="trap">
  **现象**：新版本（换了模型 / 改了
  prompt）**不灰度、直接全量上线**，结果新版本有个没测出来的问题，**瞬间炸到
  100%
  用户**，线上一片翻车才发现。**原因**：把整个用户群当成了第一批小白鼠，没有用一小撮流量先探路，出问题时影响面直接拉满。**修法**：**灰度发布**——先放
  5% 流量盯指标（成功率 / 延迟 / 成本），好再 50% →
  100%；并备好**一键回滚**，哪档变差立刻切回老版本，把影响面和损失死死封顶。
</Callout>

<Callout type="trap">
  **现象**：给「下单 / 扣款 /
  发邮件」这类操作**无脑加了重试**——结果网络抖一下让你以为失败、其实对方已执行，一重试就**重复下单、重复扣款、重复发邮件**。**原因**：把**非幂等**操作当幂等的重试了。重试
  =
  自动再做一次，只对「做几次结果都一样」的幂等操作安全。**修法**：重试前先分清幂等性——查询类放心重试；下单
  / 扣款等非幂等操作，先用**唯一请求 ID
  去重**（对方据此识别『这单已处理过』），确认重试不会被重复执行，再重试。
</Callout>

## 小结

- **生产架构**：Agent 上线是**一套有分工的架构**，不是一个脚本——API 网关（收 / 鉴权 / 限流）→ 队列（削峰）→ Agent 服务（干活，可水平扩）→ 缓存（挡重复）→ 可观测（记录），各司其职（呼应目标 1）
- **并发与异步**：海量请求 + 慢调用，靠**异步**（请求入队、不互相干等）、**队列削峰**（洪峰先排队）、**并发控制**（限同时在跑的数，别打爆下游）一起扛
- **超时 · 重试 · 降级兜底**：调用一定会失败——**超时**掐断慢请求、**重试**（只对幂等操作）救偶发失败、**降级**切备用 / 给兜底答案，永远不让用户撞见原始错误（呼应目标 2）
- **灰度发布与回滚**：新版本别一上来全量——先 5% 流量盯指标、好再 50% → 100%，变差立刻**回滚**，把上线从豪赌变成可控、可撤的小步（呼应目标 3）
- **生产级 = 能跑得久**：架构扛并发、可靠性接住失败、灰度控住上线风险——三样齐了，Agent 才从「能在你电脑上跑」变成「能上线服务真实用户、且跑得稳」

**练习 1（独立实现题）** 给定一个会失败的模型调用 `call_llm(model, prompt)`（可能超时、可能抛错），请写一个 `robust_ask(prompt)`：① 调主模型 `"main"`，给 3 秒**超时**；② 超时 / 失败就**重试**一次；③ 还不行就**降级**到备用模型 `"backup"`（同样带超时）；④ 备用也挂了，返回一句**安全兜底答案**。要求：用户在任何情况下都不会撞见原始异常。

<Answer>

把「带超时重试的调用」和「逐层降级」叠起来就是答案：

```python
import asyncio


async def call_with_retry(model, prompt, *, timeout=3.0, retries=1):
    """带超时的单模型调用 + 重试。失败 / 超时都重试，用完仍不行就抛出。"""
    last_err = None
    for attempt in range(retries + 1):
        try:
            return await asyncio.wait_for(
                call_llm(model, prompt), timeout=timeout
            )
        except Exception as err:        # 超时或调用本身报错
            last_err = err
            if attempt < retries:
                await asyncio.sleep(2**attempt)  # 退避
    raise last_err


async def robust_ask(prompt):
    """主模型 → 备用模型 → 安全兜底，用户永远撞不见原始异常。"""
    for model in ("main", "backup"):
        try:
            return await call_with_retry(model, prompt)
        except Exception:
            continue                    # 这个模型彻底不行，换下一个
    return "抱歉，服务暂时繁忙，请稍后再试。"  # 兜底
```

讲清三个要点：① **超时**——`asyncio.wait_for(..., timeout=3.0)` 给每次调用设上限，卡太久掐断；② **重试**——`call_with_retry` 内 `for` 循环失败后重试一次，并退避一下；③ **降级 + 兜底**——`robust_ask` 里 `for model in ("main", "backup")` 主模型不行就切备用，两个都挂了 `return` 那句安全兜底答案。因为最外层用 `try / except ... continue` 接住了每个模型的所有异常、最后一定 `return` 一句话，所以**用户在任何情况下都不会撞见原始异常**——这正是「超时 + 重试 + 降级兜底」三件套的完整落地。

（进阶：真实里还会区分「可重试的瞬时错误」和「不该重试的错误」、给非幂等操作加去重、在每次降级处记 trace / 告警。）

</Answer>

**练习 2（场景判断题）** 你的 Agent 里有两个工具：`get_order_status(order_id)`（查订单状态）和 `place_order(items)`（下单）。如果给**这两个工具的调用都套上「失败自动重试 3 次」**，分别会发生什么？哪个安全、哪个危险？危险的那个该怎么改？

<Answer>

- **`get_order_status` → 安全。** 查订单状态是**幂等**操作——查一次、查三次结果都一样、没有任何副作用。失败重试几次完全没问题，重试只是想多试出一个成功的结果。
- **`place_order` → 危险！** 下单是**非幂等**操作——执行一次就下一单。一旦遇到「网络抖动让客户端以为失败、但服务端其实已经下单成功」的情况，重试 3 次就可能**下出 3 单、扣 3 次钱**，酿成事故。
- **危险的那个怎么改**：不能无脑重试。正确做法是**幂等化**——下单时带一个**唯一请求 ID（幂等键）**，服务端据此识别「这个 ID 的单已经处理过了」，重复请求直接返回上次的结果而不再下单。这样即使客户端重试，也不会重复下单。确认有了这层去重保护，才可以安全重试。

一句话：**重试只对幂等操作无脑安全**；非幂等的（下单 / 扣款 / 发邮件）必须先用唯一请求 ID 做去重，否则重试就是在制造重复下单 / 重复扣款。

</Answer>

  工具偶尔会卡住很久甚至永不返回，不设超时，用户请求就一直挂着干等，还可能把服务的连接
  /
  线程占满、拖垮整个系统。设了超时，一步卡太久就掐断，转而走重试或降级。详见本章「超时·降级·重试」一节。
</GlossaryItem>

<GlossaryItem term="幂等">
  一个操作**执行一次和执行多次，效果完全一样**。比如「查天气 /
  查订单」是幂等的（查几次结果都一样、没副作用）；而「下单 / 扣款 /
  发邮件」**不是**（重复执行会重复下单 / 扣款）。和重试的关系：重试 =
  失败了自动再做一次，所以**只能对幂等操作无脑重试**；非幂等操作盲目重试会造成重复下单
  / 扣款，必须先用唯一请求 ID
  等手段去重，确认不会被重复执行才能重试。详见本章「超时·降级·重试」一节与误区。
</GlossaryItem>

<GlossaryItem term="降级">
  首选方案失败时，**自动退而求其次**，保证用户至少拿到一个「还能用」的结果而不是一个错误。两层：**降级到备用方案**（主模型超时
  / 失败 → 切到备用模型，可能更小 / 更便宜但能用）、**兜底答案**（连备用也挂了 →
  返回一句预先准备好的、安全得体的回复，如「服务繁忙，请稍后再试」）。核心原则：**永远不让用户直接撞见原始错误**——出错时降级，把失败兜成一个体面的结果。详见本章「超时·降级·重试」一节与降级
  Demo。
</GlossaryItem>

<GlossaryItem term="灰度发布">
  新版本上线时**不一上来就全量**，而是先只放一小部分流量（如
  5%）给新版本，盯住第12章那些关键指标（成功率 / 延迟 /
  成本），确认没问题再逐步放量 5% → 50% →
  100%。名字来自旧时矿工带金丝雀下井探毒气——用一小撮流量当「探子」先试，有问题也只影响这一小撮人。好处：把「一上来全量、有问题瞬间炸全部用户」的大赌注，拆成一档档可控、可观察的小步。详见本章「灰度发布与回滚」一节与灰度
  Demo。
</GlossaryItem>

<GlossaryItem term="回滚">
  发现新版本有问题时，**立刻把流量切回稳定的老版本**、撤下新版本。它是灰度发布的**安全绳**——灰度时某档指标变差（成功率掉
  / 延迟涨 /
  成本飙），不必慌乱排查，先一键回滚让线上恢复稳定，再从容定位问题。有了快速回滚，上线才敢大胆试：因为知道一旦出事，几秒钟就能撤回到已知良好的状态，**损失被牢牢封顶**。详见本章「灰度发布与回滚」一节与灰度
  Demo。
</GlossaryItem>

</Glossary>
