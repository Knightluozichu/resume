## 小特的大脑，到底是怎么读你的话的

上一章我们说，私人助理**小特**的「大脑」就是 LLM。你给它发一句「帮我订明天去上海的高铁票」，它好像一下就懂了。但它读你这句话的方式，和你想的很不一样——它不是一个字一个字地认，也不是真的「理解」了上海在哪。

这一章就掀开小特的脑壳看看：它怎么把你的话拆成一块块来读、为什么聊久了会「断片」忘事、为什么有时会一本正经地编出个根本不存在的航班号。搞清楚大脑的脾气，你才知道它哪里靠谱、哪里得防着点。

没搞清会怎样？你会高估这颗大脑，把它当成「无所不知、过目不忘」的神，然后被它信誓旦旦的胡说带进坑里——也想不明白为什么小特光有大脑还不够，非得再配上手和本子。

## 大脑眼里没有「字」，只有 token

你以为小特是按「字」读你的话的，其实不是。大脑做的第一件事，是把你这句话切成一块块叫 <Term def="LLM 处理文字的最小单位。一段文字会被切成一串 token——可能是一个字、一个词，也可能是半个词，每个 token 对应一个固定的整数编号。LLM 眼里没有「字」，只有这串编号序列。">token</Term> 的小块，再把每块换成一个固定的**编号**。这个切的过程，就叫 <Term def="把一段文字切成一串 token、再换成对应整数编号的过程。LLM 读任何文字前都先做这一步——之后它处理的全是编号，不是字。">token 化（Tokenization）</Term>。

关键有两点：一是切的单位**不等于字**——「上海」「高铁」常常是一整块 token，而不是拆成两个字；二是大脑后续处理的全是**编号**，不是你看到的汉字。下面这张图把「我要订去上海的高铁票」切给你看，点播放跟着走一遍：

<TokenizationDiagram />

为什么要懂这个？因为很多后面的事都由 token 决定：模型一次能读多少、按什么收费、能记住多长——量的全是 token，不是字数。**「token ≈ 字」这个想当然，是后面一切误会的源头。**

## 一次能「看进」多少：上下文窗口

大脑不是过目不忘的。它一次能「看进眼里」的 token 是有上限的，这个上限就是 <Term def="LLM 一次最多能「看进」的 token 数量上限（比如 8000、128000）。系统提示、对话历史、你当前的问题全挤在这一个窗口里；一旦塞满，最早的内容会被挤出窗口，模型就再也看不到、等于忘了。">上下文窗口（Context Window）</Term>。你可以把它想成一张**固定大小的桌面**：系统给的开场设定、你俩之前聊的所有历史、还有你当前这句问题，全得摊在这张桌子上，大脑才看得见。

桌子就这么大。聊得越久，历史越堆越多，桌子很快就满了。再往里塞，**最早的那几句就会被挤下桌**——大脑从此看不见它们，等于忘了。这就是为什么和它聊长了，它会突然「断片」，忘了你开头交代过的事。下个小节的 Demo 你可以亲手把历史轮数拖上去，看最早的对话怎么被挤出窗口。

记住这点：窗口**不是越大越好用**就万事大吉——再大的桌子也会满，而且桌上东西越多，大脑越容易看花眼、抓不住重点。

## 它怎么「写」下一个字：采样与温度

大脑生成回答，不是整段想好一次吐出来的，而是**一个 token 一个 token 往外蹦**。每蹦一个，它其实是在算「下一个 token 最可能是谁」——给每个候选都打一个概率，然后**按概率挑一个**。这个「按概率挑」的动作，就叫 <Term def="LLM 生成文字的方式：每写一个 token，它都给所有候选 token 算一个概率，再按这个概率分布随机挑一个，而不是永远挑概率最高的那个。这就是为什么同样的问题，它每次回答可能略有不同。">采样（Sampling）</Term>。

那「按概率挑」有多随机，谁说了算？就是 <Term def="一个调节采样随机性的旋钮。温度低（接近 0），分布变尖，几乎总挑概率最高的那个，输出稳定但呆板；温度高，分布被抹平，冷门候选也可能被挑中，输出多样但容易跑偏甚至胡说。">温度（Temperature）</Term>——一个调随机性的旋钮。温度调**低**，概率分布变得很尖，大脑几乎每次都挑那个最稳的候选，输出稳定但呆板；温度调**高**，分布被抹平，连冷门候选都可能蹦出来，输出更有创意但也更容易跑偏。下个小节的 Demo 里你能拖这根旋钮，亲眼看分布从尖峰变平。

## 它不是在查事实，它在「续写」：幻觉

最后一件必须知道的事：大脑**不是一个查得到标准答案的数据库**，它做的是「根据前文，续写出最像样的下一段」。绝大多数时候这很好用，但当它对某个事实**记不清、或压根没学过**时，它不会说「我不知道」——它会照着「一段话该长什么样」**自信地圆一个出来**。这种一本正经地编造、听起来却很可信的现象，就叫 <Term def="LLM 一本正经地编出看似可信、实则错误或根本不存在的信息的现象。根源是它在做概率续写而非查事实——记不清时它会顺着语感「圆」一个出来，而且语气照样很笃定。">幻觉（Hallucination）</Term>。

举个具体的：你让小特报一下那班高铁的**车次号**，它其实没真去查 12306，但「该有个车次号」这件事它很清楚，于是它顺口续写出一个「G1次」——听起来煞有介事，可能根本是编的。它语气还特别笃定，最坑就坑在这。第五节的温度 Demo 你会看到：**温度越高，它越敢往冷门方向蹦，也就越容易胡说**。

<Callout type="warn">
  幻觉不是「偶尔出的
  bug」，而是「概率续写」这套机制的**固有副作用**——只要它在续写而不是查证，就永远有圆出假信息的可能。所以凡是要求准确的事实（车次、价格、法条、报价），都不能只信大脑的嘴，得让
  Agent 配上工具去**真查一遍**。这正好引出下一节。
</Callout>

## 承上启下：正因为大脑有这些短板，Agent 才要配工具和记忆

把这一章的四件事连起来看，你会发现它们都指向同一个结论——**光有大脑（LLM）远远不够**：

- 窗口**有上限** → 聊长了就会忘事，所以 Agent 要配 **记忆**（第 1 章的「本子」），把重要信息存到窗口之外，要用时再捞回来，突破这张桌子的物理上限。
- 大脑**会幻觉、又不知道实时信息**（它不知道今天的票价、不知道现在几点） → 所以 Agent 要配 **工具**（第 1 章的「手」），让它能真去查 12306、真去算、真去读实时数据，而不是靠脑补。

看出来了吗？第 1 章讲的四要素里，「工具」和「记忆」不是锦上添花，而是**专门来补大脑这两个洞的**。这就是为什么一个能办事的 Agent，从来不是「一个大脑」，而是「大脑 + 手 + 本子」的组合。

## 代码长什么样：调用大脑时，你能拧哪些旋钮

把上面两个旋钮落到代码里，其实就是调用 LLM 时多传两个参数。下面这段 Python（省了真实 SDK 细节）就是小特调用它大脑的样子：

```python
response = llm.chat(
    messages=[
        {"role": "system", "content": "你是订票助理小特"},  # 占用上下文窗口
        {"role": "user", "content": "帮我订明天去上海的高铁票"},  # 也占窗口
    ],
    temperature=0.2,   # ① 温度旋钮：0.2 偏稳，几乎总挑最可能的 token
    max_tokens=500,    # ② 最多生成 500 个 token（不是 500 个字！）就停
)
```

`temperature=0.2` 就是把上面那根旋钮拧到偏低——办正经事（订票、报数字）要稳，就调低；写文案、想点子要花样，才调高。`max_tokens=500` 限制它最多**生成多少 token**——注意是 token 不是字，「上海」「高铁」可能各只算一个 token。而 `messages` 里的每条内容，连同之后所有来回，都得挤进那张有上限的上下文窗口里——这就是为什么对话越长，越要当心窗口被塞爆。

## 容易踩的坑

<Callout type="trap">
  **现象**：「这模型上下文 8000，我这段文章才 6000
  字，肯定塞得下。」结果一提交就报超长。**原因**：把 token 当成了字数。token ≠
  字——中文一个字常算一两个 token，英文一个词可能拆成几个
  token，标点、空格也占。6000 字很可能远超 8000 token。**修法**：永远按 token
  估算，别按字数；不确定就用模型方的 tokenizer 工具数一遍，留足余量。
</Callout>

<Callout type="trap">
  **现象**：「把温度调成
  0，它就绝不会胡说了吧？」可它照样一本正经地编出假车次。**原因**：把「幻觉」当成了「温度高才有的毛病」。温度只管**随机性**，幻觉的根源是「概率续写而非查事实」——温度
  0
  只是让它每次都挑最可能的续写，但那个续写本身可能就是编的。**修法**：要事实准确，靠的不是调低温度，而是给
  Agent 配**工具**去真查一遍；温度低只能让胡说更「稳定」，不能让它变真。
</Callout>

<Callout type="trap">
  **现象**：「上下文窗口当然越长越好，全塞进去最保险！」于是把几十页文档一股脑灌进去，结果它反而抓不住重点、还更慢更贵。**原因**：以为「能看见」就等于「用得好」。窗口越满，关键信息越容易被噪声淹没，模型注意力被稀释（业内叫「迷失在中间」）。**修法**：别一股脑全塞，只放当前真正相关的内容；长信息交给**记忆
  + 检索**按需捞回，而不是硬怼进窗口。
</Callout>

## 小结

- LLM 读你的话先做 **token 化**：切成一块块带编号的 token，它眼里没有「字」只有编号——「token ≈ 字」是头号误会
- **上下文窗口**是大脑一次能看进的 token 上限，像一张固定大小的桌子；塞满了最早的内容被挤出去，于是「断片」忘事
- 它生成回答是逐 token **按概率采样**，**温度**是随机性旋钮：低温稳定呆板、高温多样但易跑偏
- **幻觉**源于「概率续写而非查事实」：记不清就自信地圆一个，温度越高越敢胡说——调低温度治不了根
- 正因窗口有限、会幻觉、不知实时信息，Agent 才必须给大脑配 **记忆**（突破窗口）和 **工具**（真查真算）——回收第 1 章四要素

<Answer>

</Answer>

**练习 2（算 token 型）** 某模型上下文窗口是 8000 token。系统提示固定占 1200 token，你当前的问题占 600 token，每一轮对话历史约占 380 token。请估算：最多能保留多少轮完整历史而不溢出？聊到第 20 轮会怎样？

<Answer>

留给历史的预算 = 8000 − 1200（系统提示）− 600（当前问题）= 6200 token。每轮 380 token，6200 ÷ 380 ≈ 16.3，所以最多保留 **16 轮**完整历史（第 17 轮就会溢出）。聊到第 20 轮：历史想占 20 × 380 = 7600 token，加系统提示和当前问题共 9400 token，超出容量 1400 token——最早的约 3~4 轮对话会被挤出窗口、被大脑「忘掉」，模型再也看不到那几轮里说过的事。这正是第五节 Demo 拖到高轮数时看到的标红溢出。

</Answer>

**练习 3（问答型）** 有人说：「LLM 这么强，直接拿它当 Agent 不就行了，何必再配工具和记忆？」请用本章讲的 token 化、上下文窗口、温度、幻觉里至少两个概念，反驳这个说法。

<Answer>

光有 LLM 这颗大脑撑不起一个能办事的 Agent，至少两个洞要补：**① 上下文窗口有上限** —— 大脑一次只能看进有限 token，对话或资料一长，最早的内容就被挤出窗口、被遗忘，所以必须配**记忆**把重要信息存到窗口之外、按需捞回，突破这张桌子的物理上限。**② 会幻觉、又不知实时信息** —— 大脑是概率续写而非查事实，记不清就自信地编（比如编个假车次），也不知道今天的票价和时刻，所以必须配**工具**去真查 12306、真算、真读实时数据，而不是靠脑补。（还可补一刀：温度调低只能让胡说更稳定，治不了幻觉的根，更说明非靠工具不可。）所以 Agent 从来不是「一个大脑」，而是「大脑 + 手 + 本子」的组合。

</Answer>

  把一段文字切成一串 token、再换成对应整数编号的**过程**。LLM
  读任何文字前都先做这一步，之后它处理的全是编号。关键是切的单位**不等于字**——「上海」「高铁」常是一整块
  token。详见本章「大脑眼里没有字」一节与 Token 化动画。
</GlossaryItem>

<GlossaryItem term="上下文窗口（Context Window）">
  LLM 一次最多能「看进」的 token 数量**上限**（比如
  8000、128000）。把它想成一张固定大小的桌面：系统提示、对话历史、你当前的问题全得摊在这张桌子上；一旦塞满，**最早的内容被挤出窗口**，模型就再也看不到、等于忘了。这就是聊长了会「断片」的原因。详见本章「一次能看进多少」一节与
  Demo。
</GlossaryItem>

<GlossaryItem term="采样（Sampling）">
  LLM 生成文字的方式：它一个 token 一个 token 往外写，每写一个都给所有候选 token
  算一个概率，再**按这个概率分布随机挑一个**，而不是永远挑概率最高的那个。这就是为什么同样的问题，它每次回答可能略有不同。详见本章「它怎么写下一个字」一节。
</GlossaryItem>

<GlossaryItem term="温度（Temperature）">
  一个调节采样**随机性**的旋钮。温度低（接近
  0），概率分布变尖，几乎总挑最可能的那个
  token，输出稳定但呆板；温度高，分布被抹平，冷门候选也可能被挑中，输出多样但容易跑偏甚至胡说。办正经事调低、要创意调高。详见本章「采样与温度」一节与温度
  Demo。
</GlossaryItem>

<GlossaryItem term="幻觉（Hallucination）">
  LLM
  一本正经地编出**看似可信、实则错误或根本不存在**的信息的现象（比如顺口编个假车次号）。根源是它在做**概率续写而非查事实**——记不清时顺着语感「圆」一个出来，而且语气照样很笃定。温度越高越容易触发；调低温度治不了根，得靠工具去真查。详见本章「它不是在查事实」一节。
</GlossaryItem>
