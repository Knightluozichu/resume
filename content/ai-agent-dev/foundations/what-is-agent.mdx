## 为什么我们需要一个会「自己办事」的程序

你给一个特别能干的私人助理——我们叫他**小特**——发条消息：「帮我订明天去上海的高铁票，上午到，预算 500。」然后你就去忙别的了。小特自己去比价、挑车次、下单、确认出票，办完才回你一句「订好了，G1 次 7:30」。

注意小特做的，不是「回答你一个问题」，而是**替你把一件事从头办到尾**。中间他要查信息、做判断、动手操作、看结果对不对——你只交代了目标，过程全是他自己张罗。

这一章就讲清：能像小特这样「自己把差事办完」的程序，就叫 AI Agent。没有它会怎样？你只能得到一个「问一句答一句」的工具，每一步都得你亲自盯着、亲自点确认，它永远不会自己往前走一步。

## 小特是怎么炼成的：Agent 的四要素

我们说的 <Term def="一个能自己感知环境、做出决策、采取行动来达成你交代的目标的程序。它不只回答问题，而是替你把一件事从头办到尾。">AI Agent</Term>，本质上就是把一个「会思考的大脑」装上「能干活的手脚」，再配一本「能记事的本子」，然后丢给它一桩差事。落到小特身上，正好是四样东西：

- **大脑**：就是 <Term def="大语言模型（Large Language Model），像 GPT 这类能读懂文字、做推理、生成回答的模型。在 Agent 里它负责想清楚下一步该干什么。">LLM</Term>，负责理解差事、做判断、决定下一步。小特靠它「想清楚该订哪班车」。
- **手**：就是 <Term def="Agent 能调用的外部能力，比如查天气、搜数据库、下订单、运行代码。LLM 只会输出文字，工具让它能真正在外部世界里动手做事。">工具</Term>，让小特能真的去查 12306、真的去下单——光在脑子里想是订不到票的。
- **本子**：就是 <Term def="Agent 用来存放对话历史、已知事实、中间结果的地方，相当于它的工作笔记。没有记忆，它做完一步就忘了前一步，没法接着往下办。">记忆</Term>，记住「你说过预算 500」「刚才查到 G1 次最合适」，办到一半不会断片。
- **差事**：就是 <Term def="你交给 Agent 要它达成的最终结果，比如「订好明天去上海的高铁票」。Agent 的每一步行动都是为了把目标推进一点点，直到达成。">目标</Term>，是小特一切行动的方向——他做的每件事，都是为了把这桩差事往前推一步。

四样齐了，小特才算一个能办事的 Agent。少哪样都不行：没大脑不会判断，没手只能空想，没本子记不住，没目标不知道往哪使劲。

## 小特办一件事的一圈：感知 → 决策 → 行动 → 观察

小特不是一锤子买卖，他办事是**一圈一圈转着来**的。这一圈叫 <Term def="Agent 反复执行的「感知环境 → 做出决策 → 采取行动 → 观察结果」四个环节构成的循环。它是 Agent 区别于「只答一次话」的根本——办不成就带着新信息再转一圈，直到达成目标。">Agent 循环（Agent Loop）</Term>，四个环节首尾相接：

1. **感知**：小特先把现在的处境看清楚——明天去上海、预算 500、要上午到。
2. **决策**：他用大脑比对几趟车的时刻和票价，决定订 G1 次 7:30 那班。
3. **行动**：他调用订票工具真的去下单——这一步会**改变外部世界**（出了一张票）。
4. **观察**：他读回执，确认出票成功。要是没订上，就带着新信息回到第 1 步，再转一圈。

关键在那个「再转一圈」：纯粹答一次话的程序走完就停了，而 Agent 会**看结果再决定下一步**，办不成就重来，直到把差事办成。这条反馈循环，正是它的灵魂。

## 同样是程序，凭什么它才叫 Agent

光说「会办事」还不够准。把三种常见程序摆在一起对比，差别就清楚了。下面这张图逐列揭示，对比点全在最右列那条循环箭头上：

<AgentParadigmDiagram />

最左是**纯 LLM**：你问一句、它答一句，单向走一趟就结束，像个只会答话的咨询台——它不会自己再去做什么。

中间是 <Term def="人预先把步骤 1、2、3 固定编排好，程序照着一步步跑，每步做什么、跑完跑哪都是写死的，中途不会自主拐弯。">固定工作流</Term>：像「收到邮件 → 提取附件 → 存到网盘 → 发通知」这样人编死的一条链，每步固定、不自主决策。它会办事，但办法是人提前定死的。

最右才是 **AI Agent**：它的大脑自己决策、自己调工具、看结果再决定下一步——那条反馈环，纯 LLM 和固定工作流都没有。**有没有这条「自主决定下一步」的循环，就是 Agent 和前两者的分水岭。**

## 自治性：方向盘交给谁，是可以调的

最后一个概念叫 <Term def="Agent 自己做主、不需要人介入的程度。自治性是一条连续的谱系：低自治只敢给建议、凡事要人点确认；高自治能独立把整件事办完。不是越高越好，要看场景配。">自治性</Term>。它不是「是」或「否」的开关，而是一条从低到高的**谱系**——核心问题是：方向盘多大程度交给 Agent，多大程度还攥在人手里。

从「只敢给你提个建议、做不做你说了算」（小特拟好方案、每步等你点确认），到「整件事丢给它、办完才告诉你结果」（你彻底放手），中间有好几档。下个小节的 Demo 你可以亲手把这根滑杆从 L1 拖到 L5，看决策权怎么一点点从人手里转到 Agent 手里。

要记住的是：自治性**不是越高越好**。关键决策（花大钱、发对外消息）该留个人工确认就得留——把方向盘全交出去，翻车的也是你。

## 代码长什么样：一段 agent loop 骨架

上面那张循环图，翻译成代码，核心就是一个 `while` 循环——感知、决策、行动、观察，转一圈，没办成就再转。下面这段 Python 骨架（把工具细节都省了）就是 Agent 的心脏：

```python
def run_agent(goal):
    done = False
    while not done:                     # 一圈一圈转，直到差事办成
        obs = perceive()                # ① 感知：看清现在的处境
        thought = llm.think(obs, goal)  # ② 决策：大脑想清楚下一步
        action = thought.next_action    # 从想法里取出要做的动作
        result = act(action)            # ③ 行动：调工具真的去做
        done = goal_reached(result)     # ④ 观察：办成了吗？没成就再转一圈
    return result
```

四行核心语句正好对应图里四个环节：`perceive` 是感知、`llm.think` 是决策、`act` 是行动、`goal_reached` 是观察。注意 `act()` 这一步——它会真的去调外部工具、改变世界，这就是 Agent 和「只在脑子里想」的纯 LLM 的分水岭。

而那个 `while not done` 就是图里的「再转一圈」。把它去掉、只跑一次，Agent 立刻退化成「问一句答一句」的纯 LLM。所以这个循环，才是它之所以是 Agent 的根本。

## 容易踩的坑

<Callout type="trap">
  **现象**：「我给 GPT 套了个网页界面，能聊天，这就是 Agent
  了吧？」结果它只会一问一答，从不自己动手办事。**原因**：把「能调用
  LLM」当成了「是 Agent」。套个壳的纯 LLM
  没有工具、没有循环，走完一趟就停。**修法**：检查它有没有那条「感知-决策-行动-观察」的循环——会不会自己调工具、看结果再决定下一步。没有循环，就只是纯
  LLM。
</Callout>

<Callout type="trap">
  **现象**：「Agent
  当然越自治越好，全自治最高级！」于是把花钱、发对外消息的权限全交出去，结果它自作主张办砸了大事。**原因**：把自治性当成单调的「越高越牛」，忽略了关键决策的风险。**修法**：按场景配自治级别——日常小事可以高自治，花大钱、不可逆的操作留个人工确认（L2/L3）。方向盘全交出去，翻车的是你。
</Callout>

<Callout type="trap">
  **现象**：「我写了个 if-else 脚本，能自动把邮件分类，这是
  Agent！」可它换个没见过的邮件就傻眼。**原因**：把人预先编死的**固定工作流**当成了
  Agent。固定脚本每一步都是人写死的，没有 LLM
  自主决策。**修法**：看决策是谁做的——步骤是程序员提前编死的，就是固定工作流；是
  LLM 临场根据情况自己定的，才是 Agent。
</Callout>

## 小结

- AI Agent = 替你把差事从头办到尾的程序，四要素：大脑（LLM）、手（工具）、本子（记忆）、差事（目标）
- 它一圈圈转着办事：**感知 → 决策 → 行动 → 观察**，办不成就带新信息再转一圈
- 纯 LLM 只答一次、固定工作流照编死的链走，**只有 Agent 有那条「自主决定下一步」的反馈循环**
- 自治性是一条 L1→L5 的谱系，方向盘多少交给 Agent 可调；**不是越高越好**，关键决策留人工确认
- 判断一个程序是不是 Agent，就看它有没有「LLM 自主决策 + 调工具 + 看结果再决定」的循环

<Exercises>

**练习 1（场景判断型）** 有人给你看一个程序：「每天早上 8 点，它自动抓取昨天的销售数据，套用固定模板生成一张报表，发到群里。」请判断它算不算 AI Agent？该归到哪种范式？说出你的理由。

<Answer>

**不算 Agent，它是固定工作流。** 理由：它的每一步——抓数据、套模板、发群——都是人提前编排死的，跑的时候没有任何「临场决策」，换个情况也不会自己调整。判断的关键是看决策是谁做的：这里全是程序员写死的流程，没有 LLM 根据当下情况自主决定下一步，也没有「看结果再决定」的反馈循环，所以是固定工作流而非 Agent。如果改成「它会读今天的数据异常，自己决定要不要深挖、调哪些工具去查根因、查不出就换个角度再查」，那才开始有 Agent 的味道。

</Answer>

**练习 2（问答型）** 同样基于一个 LLM，「一个客服问答机器人」和「一个能自己查订单、改地址、发退款的客服 Agent」差在哪？请用本章的「四要素」和「循环」来解释。

<Answer>

问答机器人基本只有**大脑**（LLM）：你问它答，单向一趟就结束，是个纯 LLM。客服 Agent 则四要素齐全：除了大脑，还有**工具**（能真的查订单系统、改地址、调退款接口）、**记忆**（记住这通对话里你是谁、订单号多少）、**目标**（把你的诉求解决掉）。更关键的是它有那条**感知-决策-行动-观察循环**：查到订单状态（感知）→ 判断该退款（决策）→ 调接口发起退款（行动）→ 确认退款成功（观察），办不成还会换法子再来。有没有这条循环和工具，就是「只会说」和「能办事」的分水岭。

</Answer>

**练习 3（问答型）** 第六节那段 agent loop 骨架里，如果把 `while not done:` 改成只跑一次（去掉循环），这个程序会退化成什么？为什么？

<Answer>

会退化成一个**纯 LLM**（最多是「LLM + 一次工具调用」）。原因：去掉循环后，它只能「感知一次、决策一次、行动一次」就返回，没法「看结果再决定下一步」。而「办不成就带着新信息再转一圈」正是 Agent 的灵魂——失去循环，它就不能自我纠正、不能多步推进，只剩一锤子买卖，和问一句答一句的纯 LLM 没有本质区别。

</Answer>
