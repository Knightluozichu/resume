## 桌子就那么大，该摆什么、怎么摆，是门讲究

私人助理**小特**干活时，面前有一张桌子，所有要用的东西——你的吩咐、之前聊过的话、刚查回来的资料、你这一刻的问题——都得摆到桌上它才看得见。可这张桌子就那么大，摆不下所有东西。

前几章其实一直在「往这张桌上放东西」：教它记事、是往桌上放「之前的对话」；教它查资料，是往桌上放「检索回来的相关段落」。可东西越来越多，桌子总有摆满的一天——这时候该摆什么、每样摆多少、摆在桌子的什么位置，就成了一门真讲究的活。

这一章讲的就是这门讲究：在一张摆不下所有东西的桌子上，**怎么把有限的地方用在刀刃上**。没有它会怎样？要么桌子摆爆、重要的东西被挤掉；要么东西都摆上了，可关键那张纸压在最底下，小特愣是没看见——查得再准、记得再全，用不上也是白搭。

## 上下文工程：在有限窗口里安排「放什么、放多少、怎么排」

先把这门讲究正式起个名。<Term def="在有限的上下文窗口里，精心安排「放什么、放多少、怎么排」，让模型把注意力用在刀刃上的功夫。它是统领记忆与 RAG 的总纲：记忆决定「记什么、把什么捞回窗口」，RAG 决定「检索什么资料拼进窗口」，而上下文工程决定「这一窗口的有限空间整体怎么规划」——哪些放、各占多少、怎么排序。">上下文工程（context engineering）</Term> 说的就是：上下文窗口（小特的桌子）容量有限，每次提问能塞进去的东西就那么多，于是要像规划一块寸土寸金的地一样，精心安排——哪些内容值得放、每样占多少、按什么顺序排——好让模型的注意力落在真正重要的地方。

为什么把它单拎出来讲，还放在这一篇收官？因为它是**统领前几章的总纲**。回头看：第 6 章的**记忆**，解决的是「记什么、要用时把什么捞回窗口」；第 8 章的 **RAG**，解决的是「检索什么资料、拼哪几段进窗口」。这两章各自决定「往窗口里放某一类东西」；而上下文工程站得更高一层，决定的是**这一整个窗口的有限空间整体怎么规划**——记忆捞回来的、RAG 检索到的、系统提示、当前问题，全都要在这一块地里抢位置，谁放、谁让、谁排前面，得有人统一拍板。这就是上下文工程。

它具体落到三件可操作的事上：**怎么把预算分给各类内容**（下一节）、**装不下时怎么压缩腾地方**、**怎么排顺序好让模型看得见关键信息**。下面三节逐个讲，先从「分预算」开始——这张图把一笔窗口预算切给五类内容，拖一拖就有感觉：

## 上下文预算分配：窗口是一笔有限预算，分给好几样

把窗口想成一笔**有限的预算**——比如 8000 个 token，这就是你这一次能花的全部额度。<Term def="把有限的上下文窗口（比如 8000 token）当成一笔预算，分配给同时要进窗口的好几类内容：系统提示（角色）、few-shot 示例、对话历史、检索回来的资料、当前问题。每一类占一份额度，加起来不能超过窗口容量——给一类多分，就得从另一类省。所以预算分配本质是个权衡：在固定的总额里决定每样放多少。">上下文预算分配</Term> 要做的，就是把这笔额度分给同时要进窗口的好几样东西：

- **系统提示**（给小特定角色、立规矩）——通常固定的一笔开销；
- **few-shot 示例**（几个「问→答」范例，教它照着答）——放几个、放不放，占额度；
- **对话历史**（之前几轮的来回）——越聊越长，最能吃额度；
- **检索资料**（RAG 从知识库捞回的相关片段）——放几段、每段多长，占额度；
- **当前问题**（用户这一轮真正要问的）——必须留，否则模型不知道要答什么。

关键就在于：**这几样加起来不能超过窗口容量**。给检索资料多分两段，就得从对话历史或 few-shot 那里省出来——这是个躲不掉的**权衡（trade-off）**。这正是上一节那个 Demo 让你拖的：五个滑块此消彼长，总和一旦冒过预算线就标红。预算分配不是「能塞就都塞」，而是「在固定总额里，把每一份额度花得值」。

## 压缩 / 摘要：太长了就浓缩成几句要点

预算不够分时，除了「砍掉某一类」，还有一招更巧的——<Term def="当对话历史或检索资料太长、占了太多窗口时，把一长段内容浓缩成几句要点的做法（常由另一次 LLM 调用「请把以下内容总结成要点」完成）。目的：丢掉冗余、留住关键，腾出窗口空间。代价是可能丢细节——压得越狠省得越多，但越容易把后面可能用得上的信息也一起舍掉，所以压多狠是个权衡。">压缩 / 摘要（compression / summarization）</Term>。比如对话已经来回了十几轮，大半是寒暄闲聊，真正有用的就那么几条事实。与其把整段原文都塞进窗口，不如**先把它浓缩成几句要点**——丢掉冗余、留住关键，一下子省出大半空间。

但压缩有代价：**它可能把细节也一起丢掉**。压得越狠、省得越多，可越容易把「当时看着没用、后面却用得上」的信息也一并舍掉。所以压多狠是个权衡——压太松省不下窗口，压太狠会丢关键。下面这个 Demo 把一段对话「压缩前 vs 压缩后」摆在一起对比，切一切看省了多少、又丢了什么：

<CompressionCompare />

看清那两面：左边原文占 420 token，右边摘要只占 96，省下七成多窗口；但「压缩的代价」那一栏也老老实实写着——有一条本来有用的信息（用户提到「我妈也想买别的」）在压缩里被当闲聊舍掉了。这就是压缩的本质：**用「可能丢细节」换「省窗口」**，值不值得，看你压的是什么、丢得起丢不起。

## lost-in-the-middle：注意力呈 U 形，重要信息要放首尾

就算东西都摆上桌了，还有最后一道暗坑——东西摆上了，不等于模型就**看得一样清**。研究发现一个反直觉的事实：模型对上下文里不同位置的信息，**注意力并不均匀**。这个现象就叫 <Term def="lost-in-the-middle，迷失在中间。指模型对上下文里不同位置信息的注意力并不均匀，而是呈 U 形：放在开头和结尾的信息记得牢、用得上；放在正中间的信息容易被忽略（注意力两头高、中间低）。所以重要信息要放在上下文的首尾，别埋在正中间——这是上下文「怎么排顺序」时最该记住的一条。">lost-in-the-middle（迷失在中间）</Term>：模型的注意力沿上下文位置呈一条 **U 形曲线**——放在**开头和结尾**的信息记得牢、用得上，放在**正中间**的容易被忽略（两头高、中间低）。

这意味着：**「怎么排」和「放什么」一样重要**。同一段关键资料，摆在窗口首尾，模型抓得牢；埋进一大堆内容的正中间，它很可能就「看漏」了——你检索得再准、内容确实在窗口里，模型却没用上。所以排上下文时有条铁律：**把最重要的信息放在首尾，别埋在中间**。下面这个动画把注意力曲线扫一遍，再让你把一条关键信息挪到不同位置，亲眼看它「被抓住的概率」怎么变：

<LostInMiddleDemo />

注意那条 U 形：探针扫到首尾，柱子高（注意力强）；扫到正中间，柱子矮（注意力弱）。把★关键信息拖到正中间，命中概率最低；拖到首尾，最高。这也正好回收了第 2 章那个 trap 的预告——当时说「窗口越满，关键信息越容易被噪声淹没、注意力被稀释，业内叫迷失在中间」，这一章把它掰开了：不只是「满」的问题，**位置**本身就决定了它被不被看见。

## 代码对照：把「放什么 / 放多少 / 怎么排」写进上下文组装

讲了这么多，落到代码就是一个函数：**给定各类素材和一个预算上限，组装出最终喂给模型的 `messages`**。它要同时做三件事——按预算挑（放什么 / 放多少）、太长就压缩（腾地方）、把关键信息排在首尾（怎么排，躲开 lost-in-the-middle）。先看主干：

```python
def build_context(system, few_shot, history, retrieved, question,
                  budget=8000):
    """在预算内组装上下文：按优先级挑内容、超了就压缩历史，
    并把最重要的（检索资料 / 当前问题）放在首尾，别埋中间。"""
    # ① 必留项：系统提示钉在最前，当前问题钉在最后（首尾 = 注意力最高处）
    head = [{"role": "system", "content": system}]
    tail = [{"role": "user", "content": question}]
    spent = ntokens(system) + ntokens(question)

    # ② 次高优先级：检索资料也尽量靠尾部（紧挨问题，模型最看得见）
    retrieved = retrieved[:top_k_within(retrieved, budget - spent)]
    spent += ntokens(retrieved)

    # ③ 历史塞中间；放不下就「压缩」成摘要，腾出窗口（呼应动手二）
    if spent + ntokens(history) > budget:
        history = summarize(history)        # 一长段 → 几句要点
    middle = few_shot + history             # 不那么关键的，放中段

    # 顺序：system → few-shot/历史(中) → 检索资料 + 问题(尾)
    return head + middle + as_messages(retrieved) + tail
```

读这段的关键不在语法，在它**把本章三招都落到了一处**。`head` / `tail` 体现「**怎么排**」——系统提示和当前问题钉死在首尾（注意力最高处），检索资料也尽量靠尾，**最关键的东西绝不埋进中间**（躲开 lost-in-the-middle）。`top_k_within` 和那行 `if … summarize(history)` 体现「**放多少**」——预算不够时，先按额度截检索资料，再把吃额度最狠的对话历史压缩成要点（这正是动手一的预算权衡 + 动手二的压缩）。`middle` 体现「**放什么**」——把不那么关键的 few-shot 和历史放中段，把宝贵的首尾留给真正重要的。

<Callout type="tip">
  **一句话抓住这段**：上下文工程不是写死一个固定模板，而是**按这次任务的预算和优先级，动态决定每样放多少、压不压、排哪儿**。`build_context`
  里每一行都在回答「放什么 / 放多少 /
  怎么排」中的一个——这就是把前几章「往窗口塞东西」统一规划起来的总纲。
</Callout>

## 容易踩的坑

<Callout type="trap">
  **现象**：「重要资料只要塞进窗口就行，放哪儿无所谓。」于是把关键那段检索资料埋在一大堆历史的**正中间**，结果模型答题时像没看见它一样。**原因**：忽视了
  **lost-in-the-middle**——模型注意力呈 U
  形，正中间是**注意力洼地**，埋在那儿的信息最容易被忽略。**修法**：把最重要的信息排在上下文的**首尾**（紧挨系统提示或紧挨当前问题），中段只放不那么关键的内容；检索资料尤其要靠近问题放。
</Callout>

<Callout type="trap">
  **现象**：「窗口越大越好，能塞的全塞满最保险！」于是把几十段资料、几十轮历史一股脑全灌进去。结果又慢又贵，模型还抓不住重点。**原因**：以为「能看见」就等于「用得好」——窗口越满，关键信息越被噪声淹没、注意力越稀释（迷失在中间更严重）。**修法**：**别贪满**，只放这次任务真正相关的内容；额度紧就**压缩**长历史、**少而精**地放检索资料，把窗口留给刀刃上的东西，而不是塞到撑。
</Callout>

<Callout type="trap">
  **现象（选错招的坑）**：对话历史太长、窗口快满了，于是**无脑把整段历史狠狠压成一句话**，图省窗口。结果后面用户追问一个早先提过的细节（订单号、型号），小特答不上来——那条信息在压缩里被丢了。**原因**：**该用检索的场景误用了压缩**——压缩是「浓缩留要点、丢细节」，细节一旦压没就找不回；而「按需捞回某个具体细节」本该交给**记忆
  +
  检索**。**修法**：分清场景——**压缩**用于「整体太长、只需保留大意」；**检索**用于「细节多、用时再按需精准捞回」。别把该存进记忆库按需检索的细节，一压了之。
</Callout>

## 小结

- **上下文工程**：在**有限窗口**里安排「**放什么、放多少、怎么排**」，让注意力用在刀刃上——它是统领记忆（记什么 / 捞什么）与 RAG（检索什么）的**总纲**
- **预算分配**：窗口是一笔有限预算，分给系统提示 / few-shot / 历史 / 检索资料 / 问题——加起来不能超，给一类多分就得从另一类省，是个**权衡**
- **压缩 / 摘要**：太长就浓缩成几句要点，省下大半窗口；代价是**可能丢细节**，压多狠取决于你丢得起什么
- **lost-in-the-middle**：注意力呈 **U 形**（首尾高、中间低），重要信息要放在**首尾**、别埋正中间——「怎么排」和「放什么」一样重要
- **选招**：整体太长只需大意 → **压缩**；细节多、用时再精准捞 → **检索 / 记忆**；位置不对没被看见 → 把关键信息**挪到首尾**

<Exercises>

**练习 1（场景选型题）** 下面三个场景，各该用「预算分配 / 压缩 / 挪信息位置」里的哪一招（可不止一招）？说明理由。
① 一段 30 轮的客服对话，绝大多数是寒暄，窗口快满了，但你只需要保留「用户要退什么、订单号多少」这几条事实。
② 你刚检索回 6 段相关资料，可窗口只够再放 3 段，必须取舍。
③ 关键资料明明在窗口里，模型答题时却总像没看见它——你检查发现它被排在了一长串历史的正中间。

<Answer>

① 用**压缩**。30 轮对话大多是无用寒暄、你又只需要「大意 + 几条关键事实」，正适合把整段历史**摘要成几句要点**，省出大半窗口、留住该留的。（若那几条事实是后面会被反复追问的细节，还可以顺手把它们**存进记忆库**，需要时再检索回来，避免压缩丢失。）

② 用**预算分配**。窗口只够 3 段，就得在 6 段里**取舍**——按相似度 / 相关性挑最相关的 top-3 放进去，把额度花在最该花的资料上。这就是预算权衡：放不下全部，就只放最值得放的。

③ 用**挪信息位置**。资料在窗口里却没被用上，是 **lost-in-the-middle**——它被埋在了注意力洼地（正中间）。修法是把这段关键资料**挪到首尾**（紧挨当前问题或系统提示），别让它压在历史中段。

要点：压缩解决「太长」，预算分配解决「放不下、要取舍」，挪位置解决「在窗口里却没被看见」——三招对应三类不同的病。

</Answer>

**练习 2（改 Demo 题）** 打开「动手一」的预算分配 Demo，先把所有滑块拖到接近预算线但不超。现在你要给「检索资料」再加 1500 token：① 不动其他滑块，直接把检索资料 +1500，会发生什么？②要不超预算，你打算从哪一类省出这 1500，为什么选它而不选别的？

<Answer>

① 直接 +1500 会让总和**冒过预算线、整条标红**（超预算）——因为五样已经接近占满，再加任何一类都会超。这正说明预算是死的：**没有「凭空多放」，多一处就得少一处**。

② 最该省的通常是**对话历史**或 **few-shot 示例**，而不是系统提示或当前问题：

- **系统提示**（角色 / 规矩）和**当前问题**（用户这一轮要问的）是必留项，砍了模型会跑偏或不知道答什么——不能省。
- **对话历史**最吃额度、且越早的越可能无关，可以**压缩**或**截断**早期轮次来腾地方（顺带呼应压缩那一招）。
- **few-shot 示例**如果这次任务格式已经稳定，少放一两个范例影响也不大。

所以一般从「历史 / few-shot」里省，把腾出的 1500 给更关键的检索资料。这就是预算分配的权衡逻辑：**先保必留项，再从最不吃亏的那类里割肉**。

</Answer>

**练习 3（问答题）** 你的窗口已经快满，又想多塞两段刚检索回来的相关资料。请完整回答：这两段该塞吗？要塞的话，从哪里腾空间、腾出来的空间怎么用、这两段该放窗口的什么**位置**？

<Answer>

**该不该塞**：看这两段是不是这次任务**真正相关**的。是 → 值得塞（检索资料正是让模型「开卷答」的依据）；只是「沾点边」→ 别贪，塞进去反而稀释注意力、还占额度。

**从哪里腾空间**：窗口已满，塞新的就得腾旧的。优先动**最吃额度、又最不关键**的——通常是早期的**对话历史**：把它**压缩**成几句要点（省下大半），或截掉太早、已无关的轮次。系统提示和当前问题是必留项，不动。

**腾出的空间怎么用**：用来放这两段检索资料（这正是「预算分配」的权衡——从历史省出来的额度，转给更相关的检索资料）。

**放什么位置**：放在**靠近当前问题的尾部**（或紧跟系统提示的头部），**绝不埋进中段**——因为 lost-in-the-middle，正中间是注意力洼地，关键资料埋那儿等于白塞。检索资料紧挨问题放，模型最看得见、最用得上。

一句话串起来：**该塞（若相关）→ 压缩历史腾空间（预算 + 压缩）→ 把这两段排在紧挨问题的首尾（躲开 lost-in-the-middle）**。这三步正好是本章三招的合用。

</Answer>

</Exercises>

<GlossaryItem term="上下文工程（context engineering）">
  在**有限的上下文窗口**里，精心安排「**放什么、放多少、怎么排**」，让模型把注意力用在刀刃上的功夫。它是统领**记忆**与
  **RAG** 的**总纲**：记忆决定「记什么、把什么捞回窗口」，RAG
  决定「检索什么资料拼进窗口」，而上下文工程决定「这一窗口的有限空间**整体怎么规划**」——哪些放、各占多少、怎么排序。类比：小特的桌子就那么大，该摆什么、摆多少、摆哪儿，是门讲究。详见本章「上下文工程」一节与预算分配
  Demo。
</GlossaryItem>

<GlossaryItem term="上下文预算分配">
  把有限的上下文窗口（比如 8000
  token）当成一笔**预算**，分配给同时要进窗口的好几类内容：系统提示（角色）、few-shot
  示例、对话历史、检索回来的资料、当前问题。每一类占一份额度，**加起来不能超过窗口容量**——给一类多分，就得从另一类省。所以预算分配本质是个**权衡**：在固定总额里决定每样放多少。详见本章「上下文预算分配」一节与动手一
  Demo。
</GlossaryItem>

<GlossaryItem term="压缩 / 摘要（compression / summarization）">

  LLM
  调用「请把以下内容总结成要点」完成）。目的：丢掉冗余、留住关键，**腾出窗口空间**。代价是**可能丢细节**——压得越狠省得越多，但越容易把后面可能用得上的信息也一起舍掉，所以压多狠是个**权衡**。详见本章「压缩

</GlossaryItem>
