## 闭卷凭记忆瞎说，不如开卷先翻资料再答

你问私人助理**小特**：「咱们公司的退货政策是几天？」小特张口就来：「一般是 30 天。」——可你们公司明明写的是 7 天。它不是故意骗你，是它「脑子里」压根没存过你们公司的规定，又被逼着非答不可，只好按「一般印象」编了一个。

这就像一场**闭卷考试**：题目超纲了，可还不准交白卷，于是凭着模糊记忆硬凑一个答案，听着挺像那么回事，其实是错的。换成人，遇到没把握的题会怎么办？**翻书**。手边要是有本写着公司规定的手册，先翻到那一页、照着上面的话答，自然就不会瞎编了。

这一章就讲怎么把小特从「闭卷」改成「**开卷**」：给它配一个能查的资料库，答之前先翻进去找到相关那几段，再照着资料回答。没有它会怎样？小特只会拿训练时那点旧记忆硬撑，碰到它没学过的私有内容、最新消息，就只能一本正经地胡说。

## 为什么要 RAG：把「闭卷」改成「开卷」

先说清这章到底在解决什么。大语言模型有两个绕不开的硬伤。第一，它的知识**在训练那一刻就冻住了**——训练之后发生的事、你们公司内部的私有资料，它统统没见过。第二，就算它不知道，被问到时也往往**不会说「我不知道」，而是编一个**听起来很顺的答案——这就是上一篇反复提到的<Term def="模型在不知道答案时，不肯认错，而是一本正经地编造一个听起来合理、实则错误的回答。它不是故意撒谎，而是被「必须答点什么」逼出来的——尤其在被问到训练时没见过的私有 / 最新内容时。RAG 通过先检索真实资料再让模型照着答，能大幅减少幻觉。">幻觉</Term>。

把这两条合起来：模型像个**只能闭卷、还不许交白卷**的考生，超纲题只能瞎蒙。<Term def="Retrieval-Augmented Generation，检索增强生成。思路是给模型外挂一个可检索的外部知识库：回答前先「检索」出和问题相关的资料段落，再把这些段落「拼进提示」让模型基于它们生成答案——相当于把模型从「闭卷凭记忆答」改成「开卷先翻资料再答」。能让模型用上训练时没见过的私有 / 最新知识，并大幅减少幻觉。">RAG（检索增强生成）</Term> 的解法很直接：给模型配一个**可检索的外部知识库**（你的产品手册、FAQ、内部文档），回答前先从里面**检索**出和问题相关的几段，再把这些段落**拼进提示**让模型照着答。一句话——**先翻资料、再答题，把闭卷改成开卷。**

这么一改，前面两个硬伤同时解决：私有 / 最新的内容只要进了知识库就能被查到（不再受「训练时冻住」的限制），模型照着检索回来的真实资料答（幻觉大大减少）。下面这张图把一整条 RAG 管线逐阶段点亮——注意它分成两大段，跟着走一遍，先建立整条管线的结构直觉：

## Embedding 与向量：把「意思」变成一串数字

RAG 检索的灵魂，是上一篇《记忆系统》欠你的那个坑——「按意思找回，而不是按关键词」，我们说**留到这章掰开**。现在掰开：靠的是 <Term def="把一段文本变成一串固定长度的数字（一个向量）的过程，这串数字也叫 embedding。关键性质：意思相近的文本，它们的向量在空间里离得近；意思无关的，离得远。所以比较两段文本「意思有多近」，就变成了比较两个向量「离得有多近」。这串数字由一个专门的 embedding 模型算出来，维度通常几百到几千。">embedding（嵌入）</Term>。

它做的事一句话能说清：把一段文本喂给一个专门的 embedding 模型，吐出一串固定长度的数字，这串数字就叫这段文本的 <Term def="一串固定长度的数字（比如 1536 个），是 embedding 模型给一段文本算出的「坐标」。可以把它想成高维空间里的一个点：意思相近的文本，点离得近；意思无关的，点离得远。RAG 把每个 chunk 和每个问题都变成向量，靠「向量离得近不近」来判断「意思相关不相关」。">向量</Term>。它的魔法在于：**意思相近的文本，算出来的向量在空间里离得近；意思八竿子打不着的，离得远。** 所以「两段话意思有多近」这个虚的问题，就变成了「两个向量离得有多近」这个能算的问题。

这就是为什么查询「预算」能命中「打算花 500 块」——它俩一个共同的字都没有，但**意思近**，向量就离得近。下面这个 Demo 把高维向量压成 2D 给你看：意思相近的短语聚成簇，换个查询★，看它命中哪些最近的邻居：

<VectorSpaceDemo />

注意那条主线：**位置近 = 意思近**。检索一个问题，就是把问题也变成一个点，在空间里找离它最近的几个点。真实 embedding 是几百~几千维（图里压成 2D 只为看个意思），但「按距离找最近邻」的道理一模一样。

## Chunk 切分：长文档先切小块，再各自向量化

知道了「文本能变向量」，下一个现实问题是：一份文档动辄几千上万字，**整篇算成一个向量**行不行？不行——整篇的「意思」被糊成一个点，你问一个具体问题，它既不精准（命中的是整篇而不是相关那段），又塞不进上下文窗口。所以建库第一刀是 <Term def="把一份长文档切成一个个小片段（chunk）的过程，每个 chunk 再各自 embed 成向量。切分有讲究：块太大，一块塞进太多内容，检索不精准、还占上下文窗口；块太小，一句话被拦腰切断，丢了上下文。所以要选个合适的块大小，并常让相邻块带一点重叠（overlap），保住跨块句子的连贯。">chunk 切分（分块）</Term>：把长文档切成一个个小块，每块再各自 embed 成向量。

切多大有讲究：切**太大**，一块里塞了一大段，检索时命中里混进无关内容、不精准，还更占窗口；切**太小**，一句话被拦腰切断，丢了上下文。还常让相邻块**带一点重叠（overlap）**，免得一句话刚好卡在两块的接缝处被切散。下面这个 Demo 让你拖两个旋钮——块大小、重叠——实时看文档被切成几块、重叠区在哪：

<ChunkingDemo />

玩一玩你就有手感了：块大小和重叠不是随便定的，它直接决定「每块塞多少上下文、跨块连不连贯、检索精不精准」——**这是 RAG 检索质量的第一道闸**。

## 检索 + 拼接生成：找到相关段落，拼进提示让模型照着答

前三步把知识库建好了（文档→切块→每块向量→存库），这都是**离线**一次性做的。真正每次提问才走的是最后这段 <Term def="RAG 在线问答的核心两步。检索（Retrieval）：把用户问题也 embed 成向量，在向量库里找离它最近的 top-k 个 chunk（= 意思最相关的几段资料）。拼接生成（Augmented Generation）：把检索到的 chunk 拼进给模型的提示里——「基于以下资料回答：…，问题：…」，让模型基于这些真实资料生成答案，而不是凭记忆瞎编。这两步合起来就是 Retrieval-Augmented Generation。">检索 + 拼接生成</Term>：

第一步**检索**——把用户问题也 embed 成向量（和 chunk 用**同一套** embedding，才能在同一空间里比远近），在库里找离它最近的 <Term def="检索时取「最相似的前 k 个」结果。在 RAG 里就是：算出问题向量和每个 chunk 向量的相似度，按相似度从高到低排序，取前 k 个 chunk（比如 k=3）作为「检索命中」拼进提示。k 太小可能漏掉相关资料，k 太大会塞进无关内容、还占窗口，要适中。">top-k</Term> 个 chunk——离得近 = 意思最相关。第二步**拼接生成**——把这几个 chunk 拼进给模型的提示里：`基于以下资料回答：{相关资料}，问题：{用户问题}`，让模型照着这些真实资料生成答案。

这两步合起来，就是 RAG 全名的来历：**Retrieval（检索相关资料）+ Augmented（拿资料增强提示）+ Generation（基于资料生成）**。回头看上面那张管线图的紫色段，⑤⑥⑦⑧⑨ 走的正是这条路。下面第六节，我们把这一整条管线落成真实可跑的 Python。

## 代码逐段拆解：从零写一个最简 RAG

讲了这么多，落到代码到底长什么样？我们一步步把「问退货政策、让模型照手册答」这条管线用 Python 写出来。先是**离线建库的两件事：embed 和 chunk**。`embed` 把一段文本变成向量——真实里调 embedding 模型（如 OpenAI 的 `text-embedding-3-small`），这里为了能离线跑通，先用一个**确定性的 mock**示意（同样的文本永远得到同样的向量）：

```python
import hashlib

def embed(text: str) -> list[float]:
    """把文本变成一个固定长度的向量。
    真实里调 embedding 模型；这里用 hash 做确定性 mock，同文本同向量。"""
    h = hashlib.sha256(text.encode("utf-8")).digest()
    # 取 8 个字节映射到 [-1, 1]，凑成一个 8 维「向量」示意
    return [(b - 127.5) / 127.5 for b in h[:8]]

def chunk(doc: str, size: int = 20, overlap: int = 4) -> list[str]:
    """把长文档切成带重叠的小块（按字示意；真实里按 token / 句子切）。"""
    step = size - overlap
    pieces = []
    for start in range(0, len(doc), step):
        pieces.append(doc[start : start + size])
        if start + size >= len(doc):
            break
    return pieces
```

`embed` 的真实实现会换成一次 API 调用，但**接口不变**：进去一段文本，出来一串数字。`chunk` 就是动手一里那两个旋钮的代码版——`size` 是块大小、`overlap` 是重叠，`step = size - overlap` 决定下一块从哪起。

<Callout type="warn">
  **关键点**：检索时，问题和 chunk **必须用同一个 `embed`
  函数**（同一套坐标系），算出的向量才能比远近。要是建库用一个 embedding
  模型、查询时换了另一个，两套向量不在同一空间里，距离就毫无意义、检索全乱。
</Callout>

有了这两块，**建索引**就是把文档切块、每块 embed、把 `(chunk, 向量)` 一条条存好。真实里存进向量数据库（FAISS、Chroma、pgvector 等），这里用一个简单的列表示意「索引」长什么样：

```python
# 示意知识库：一段公司退货政策文档
DOC = (
    "本店退货政策如下。自收货之日起七天内，"
    "商品完好可无理由退货。生鲜与定制商品不支持退货。"
    "退款将在审核通过后三个工作日内原路退回。"
)

# 离线建库：切块 → 每块 embed → 存成 (chunk, 向量) 列表
INDEX = [(c, embed(c)) for c in chunk(DOC)]
```

到这儿，离线建库就做完了——`INDEX` 里每一条是「一个 chunk + 它的向量」，等着被查。注意这整段是**一次性**的：文档不变，索引就不用重建。

下面是**在线问答**的第一步——**检索**。给一个问题，把它也 embed 成向量，对索引里每个 chunk 算一下「问题向量和 chunk 向量有多近」，排序取最近的 top-k。衡量「多近」最常用的是 <Term def="衡量两个向量「方向有多一致」的相似度，等于两向量夹角的余弦值，范围 -1 到 1：越接近 1 越相似（方向几乎一致 = 意思越近），越接近 0 越不相关。算法上等于「两向量点积 ÷ 各自长度的乘积」。RAG 检索几乎都用它来给 chunk 和问题的相关度打分，再取分数最高的 top-k。">余弦相似度（cosine similarity）</Term>：

```python
import math

def cosine(a: list[float], b: list[float]) -> float:
    """余弦相似度：两向量越「同方向」越接近 1，越不相关越接近 0。"""
    dot = sum(x * y for x, y in zip(a, b))
    na = math.sqrt(sum(x * x for x in a))
    nb = math.sqrt(sum(y * y for y in b))
    return dot / (na * nb) if na and nb else 0.0

def retrieve(query: str, k: int = 2) -> list[str]:
    """检索：问题 embed 成向量，对每个 chunk 算相似度，返回最相关的 top-k。"""
    qv = embed(query)
    scored = [(cosine(qv, vec), c) for c, vec in INDEX]
    scored.sort(reverse=True)            # 相似度从高到低
    return [c for _score, c in scored[:k]]   # 只取最相关的 k 个
```

`retrieve` 就是 RAG 全名里的 **R（Retrieval）**：算相似度、排序、取 top-k。它和上一篇《记忆系统》里那个 `retrieve` 长得几乎一样——区别只在这章把 `similarity` 真正落成了 `cosine`，把「比意思」掰成了「算向量夹角」。

<Callout type="tip">
  **关键点**：`k` 要适中。`k` 太小，相关资料可能没被捞全；`k` 太大，会把不相关的
  chunk 也塞进提示，既干扰模型、又占上下文窗口。常见做法是先
  `k=3~5`，再按效果调。
</Callout>

最后一步——**拼接生成**，也就是 RAG 全名里的 **AG（Augmented Generation）**。把检索到的 chunk 拼进提示，组成「基于以下资料回答：…，问题：…」，再调 LLM 让它照着资料答：

```python
def rag_answer(query: str, k: int = 2) -> str:
    """完整一轮 RAG：检索相关 chunk → 拼进提示 → 调 LLM 基于资料生成。"""
    chunks = retrieve(query, k)                 # R：检索相关资料
    context = "\n".join(f"- {c}" for c in chunks)
    prompt = (
        "基于以下资料回答问题，只用资料里的信息，资料里没有就说不知道。\n"
        f"【资料】\n{context}\n"
        f"【问题】{query}"
    )
    # A + G：拿资料增强提示，调 LLM 基于资料生成答案
    resp = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": prompt}],
    )
    return resp.choices[0].message.content

print(rag_answer("退货政策是几天？"))   # 「自收货起七天内，商品完好可无理由退货。」
```

这就是一个**最简但完整**的 RAG：`chunk` + `embed` 建库（离线），`retrieve` 按 cosine 取 top-k（在线检索），`rag_answer` 把命中拼进提示调模型（在线生成）。注意那句提示里的「**只用资料里的信息，资料里没有就说不知道**」——它是把模型摁在「开卷」上的关键，没有这句，模型还是容易脱离资料自由发挥。

## 容易踩的坑

<Callout type="trap">
  **现象**：「chunk 随便切多大都行吧，反正都 embed
  成向量了。」结果检索老是命中不准——要么捞回一大段里只有一句相关，要么一句话被切散根本捞不全。**原因**：**忽视了块大小
  + 重叠的取舍**——切太大不精准、还占窗口，切太小丢上下文。chunk
  是检索质量的第一道闸，不是随便切。**修法**：选个让每块大致是「一两句完整的话」的块大小，并加一点**重叠**（overlap），让跨块的句子在接缝处不被切断；按检索效果调这两个旋钮。
</Callout>

<Callout type="trap">
  **现象**：「检索回来的 top-k
  直接全塞进提示让模型答就行。」结果模型被一堆不相关的内容带偏，答得驴唇不对马嘴。**原因**：**检索回来不等于都相关**——top-k
  只是「最近的 k 个」，`k`
  设大了、或库里压根没有相关资料时，照样会捞回一堆离题的 chunk。**修法**：`k`
  取适中；给相似度设个**阈值**，太低的命中干脆丢掉；提示里写明「**资料里没有就说不知道**」，别逼模型硬答。检索是「找相关的」，不是「找最近的就一定信」。
</Callout>

<Callout type="trap">
  **现象**：「RAG
  不就是把整篇文档塞进提示里让模型读吗？」于是把几万字文档原样怼进
  prompt，结果超长报错、又慢又贵。**原因**：**把 RAG 误当成「整文档塞
  prompt」**——其实 RAG 的核心恰恰是**只检索相关的几个
  chunk**，而不是把全部资料灌进去。**修法**：记住 RAG =
  先**切块建库**、提问时只**检索 top-k
  相关片段**拼进提示。整篇塞进去既塞不下、又淹没重点；检索相关片段才是 RAG
  省窗口、又精准的关键。
</Callout>

<Callout type="trap">
  **现象**：「建库用一个 embedding 模型，查询时图省事换了另一个，反正都是
  embedding 嘛。」结果检索结果一团乱，相关的根本捞不回来。**原因**：**问题和
  chunk 用了不同的 embedding
  模型**——两套向量不在同一坐标系里，算出的距离毫无意义。**修法**：建库和查询**必须用同一个
  embedding 模型 / 同一个 `embed` 函数**；换了 embedding 模型，整个索引要**重新
  embed 重建**，不能新旧混用。
</Callout>

## 小结

- **为什么 RAG**：LLM 知识训练时就冻住、不知私有 / 最新、还会**幻觉**（闭卷凭记忆瞎说）；RAG 给它外挂可检索的知识库，先翻资料再答——把**闭卷改成开卷**
- **Embedding 与向量**：embedding 把一段文本变成一串数字（向量），**意思近则向量近**；所以「比意思」就变成「比向量距离」，这正是第 6 章欠的语义底层
- **Chunk 切分**：长文档先切成小块再各自 embed；块太大不精准 / 占窗口、太小丢上下文，常带**重叠**——这是检索质量的第一道闸
- **检索 + 拼接生成**：问题也 embed，按 **cosine 相似度**取 **top-k** 相关 chunk（R），拼进提示「基于以下资料回答」调 LLM（AG）——这就是 Retrieval-Augmented Generation
- 一个最简 RAG = `chunk` + `embed` 建库（离线一次性）+ `retrieve` 按 cosine 取 top-k + `rag_answer` 把命中拼进提示调模型（在线每次提问才走）

**练习 1（独立实现题）** 给定下面三段「文档」和一个问题，仿照本章写出一个最简 RAG 的核心代码（用 mock `embed` 即可，不必真调模型）。要求：① 把三段文档各当成一个 chunk、各 `embed` 成向量存进索引；② 写 `retrieve(query, k)` 用 cosine 相似度取 top-k；③ 写 `build_prompt(query)` 把检索到的 chunk 拼成「基于以下资料回答：…，问题：…」的提示串（不必真调 LLM，打印提示即可）。

文档：`["自收货起七天内可无理由退货", "生鲜与定制商品不支持退货", "退款三个工作日内原路退回"]`；问题：`"退货时限是多久？"`

<Answer>

```python
import hashlib, math

def embed(text: str) -> list[float]:
    h = hashlib.sha256(text.encode("utf-8")).digest()
    return [(b - 127.5) / 127.5 for b in h[:8]]

def cosine(a, b):
    dot = sum(x * y for x, y in zip(a, b))
    na = math.sqrt(sum(x * x for x in a))
    nb = math.sqrt(sum(y * y for y in b))
    return dot / (na * nb) if na and nb else 0.0

DOCS = [
    "自收货起七天内可无理由退货",
    "生鲜与定制商品不支持退货",
    "退款三个工作日内原路退回",
]
# ① 每段当一个 chunk，embed 后存进索引
INDEX = [(d, embed(d)) for d in DOCS]

def retrieve(query, k=1):
    # ② 问题 embed，对每个 chunk 算 cosine，取 top-k
    qv = embed(query)
    scored = [(cosine(qv, vec), c) for c, vec in INDEX]
    scored.sort(reverse=True)
    return [c for _s, c in scored[:k]]

def build_prompt(query, k=1):
    # ③ 把检索到的 chunk 拼进提示
    chunks = retrieve(query, k)
    context = "\n".join(f"- {c}" for c in chunks)
    return (
        "基于以下资料回答，资料里没有就说不知道。\n"
        f"【资料】\n{context}\n【问题】{query}"
    )

print(build_prompt("退货时限是多久？"))
```

要点回收：三段文档各 `embed` 进 `INDEX`（建库）；`retrieve` 用 `cosine` 排序取 top-k（检索）；`build_prompt` 把命中拼成「基于以下资料回答」的提示（拼接）。这正是本章 `chunk`+`embed` 建库 → `retrieve` cosine top-k → 拼接提示的最简落地。（注意 mock `embed` 是 hash，命中不一定语义最准；真实 embedding 模型才会让「退货时限」命中「七天内可退货」。）

</Answer>

**练习 2（改 Demo 题）** 打开「动手二」的语义空间 Demo，把查询切到「出门要带伞吗？」，把 top-k 从 k=1 调到 k=3。命中的点从几个变成几个？多出来的那个点，是天气簇的，还是别的簇的？为什么它会被捞进来？这说明 `k` 调大有什么风险？

<Answer>

命中从 1 个变成 3 个。「出门要带伞吗？」落在天气簇里，k=1、k=2 时命中的都是天气簇的点（「带不带伞」「今天下雨吗」这类，意思最近）；k=3 时，因为天气簇就那么几个点，第 3 近的点很可能**已经不在天气簇里**了，而是隔壁簇里离查询相对没那么远的一个——它和查询其实没那么相关，只是「排第 3 近」而已。

**练习 3（问答题）** 用户问「退货政策是几天？」，知识库里有一句「自收货起 7 天内可无理由退货」——这两句没有一个共同的关键词靠得上（一个说「政策几天」，一个说「七天内可退货」）。RAG 凭什么能把这句检索回来、再让模型照它答？请说清「检索」这一步具体比的是什么，以及如果换成「关键词匹配」会怎样。

<Answer>

凭的是 **embedding + 余弦相似度的语义检索**。检索时，RAG 把问题「退货政策是几天？」和库里每个 chunk 都用**同一个 embedding 模型**变成**向量**（一串数字），再用 **cosine 相似度**算「问题向量和 chunk 向量有多近」。「退货政策是几天」和「自收货起 7 天内可退货」虽然字面不重叠，但**意思非常近**（都在讲退货的时限），转成向量后离得近、cosine 分高，于是被排进 top-k 捞了回来。「检索」这一步比的是**意思（向量距离）**，不是字面有没有相同的字。

拿回来之后，RAG 把这句 chunk 拼进提示「基于以下资料回答：自收货起 7 天内可退货，问题：退货政策是几天？」，模型照着这句真实资料答「7 天」——这就是「开卷」。

</Answer>

<Glossary>

<GlossaryItem term="幻觉">
  模型在**不知道答案时不肯认错、反而编一个**听起来合理实则错误的回答。它不是故意撒谎，而是被「必须答点什么」逼出来的——尤其被问到训练时没见过的私有
  /
  最新内容时最容易发生。类比一场闭卷考试里超纲了还不准交白卷，只好凭模糊印象瞎蒙。RAG
  通过**先检索真实资料、再让模型照着答**，能大幅减少幻觉。详见本章「为什么要
  RAG」一节。
</GlossaryItem>

<GlossaryItem term="RAG（检索增强生成）">
  Retrieval-Augmented Generation
  的缩写，检索增强生成。给模型外挂一个**可检索的外部知识库**：回答前先**检索**出和问题相关的资料段落，再把它们**拼进提示**让模型基于这些段落生成答案——相当于把模型从「闭卷凭记忆答」改成「**开卷先翻资料再答**」。能让模型用上训练时没见过的私有
  / 最新知识，并大幅减少幻觉。详见本章「为什么要 RAG」一节与管线动画。
</GlossaryItem>

<GlossaryItem term="embedding（嵌入）">
  把一段文本变成一串固定长度数字（一个**向量**）的过程，这串数字也叫
  embedding。关键性质：**意思相近的文本，向量在空间里离得近**；意思无关的，离得远。所以「两段文本意思有多近」就变成了「两个向量离得有多近」这个能算的问题。由专门的
  embedding 模型算出，维度通常几百到几千。详见本章「Embedding
  与向量」一节与语义空间 Demo。
</GlossaryItem>

<GlossaryItem term="向量">
  一串固定长度的数字（比如 1536 个），是 embedding
  模型给一段文本算出的「坐标」。可以想成**高维空间里的一个点**：意思相近的文本点离得近，意思无关的离得远。RAG
  把每个 chunk
  和每个问题都变成向量，靠「向量离得近不近」判断「意思相关不相关」。（图里压成
  2D 只为看个意思，真实是几百~几千维。）详见本章「Embedding 与向量」一节。
</GlossaryItem>

<GlossaryItem term="chunk 切分（分块）">
  把一份长文档切成一个个小片段（chunk）的过程，每个 chunk 再各自 embed
  成向量。切分有讲究：**块太大**，一块塞太多内容、检索不精准还占窗口；**块太小**，一句话被拦腰切断、丢了上下文。所以要选合适的块大小，并常让相邻块带一点**重叠（overlap）**，保住跨块句子的连贯。它是
  RAG 检索质量的第一道闸。详见本章「Chunk 切分」一节与切分 Demo。
</GlossaryItem>

<GlossaryItem term="检索 + 拼接生成">
  RAG 在线问答的核心两步。**检索（Retrieval）**：把用户问题也 embed
  成向量，在向量库里找离它最近的 **top-k** 个
  chunk（意思最相关的几段资料）。**拼接生成（Augmented
  Generation）**：把检索到的 chunk
  拼进给模型的提示——「基于以下资料回答：…，问题：…」，让模型基于真实资料生成答案，而不是凭记忆瞎编。两步合起来就是
  Retrieval-Augmented Generation。详见本章「检索 + 拼接生成」一节与代码。
</GlossaryItem>

<GlossaryItem term="top-k">
  检索时取「**最相似的前 k 个**」结果。在 RAG 里就是：算出问题向量和每个 chunk
  向量的相似度，按相似度从高到低排序，取前 k 个 chunk（比如
  k=3）作为命中拼进提示。`k`
  太小可能漏掉相关资料，太大会塞进无关内容、还占窗口，要**适中**。详见本章「检索
  + 拼接生成」一节与代码 `retrieve`。
</GlossaryItem>

<GlossaryItem term="余弦相似度（cosine similarity）">
  衡量两个向量「**方向有多一致**」的相似度，等于两向量夹角的余弦值，范围 -1 到
  1：越接近 1 越相似（方向几乎一致 = 意思越近），越接近 0
  越不相关。算法上等于「两向量**点积 ÷ 各自长度的乘积**」。RAG 检索几乎都用它给
  chunk 和问题的相关度打分，再取分数最高的 top-k。详见本章代码 `cosine` 与
  `retrieve`。
</GlossaryItem>

</Glossary>
