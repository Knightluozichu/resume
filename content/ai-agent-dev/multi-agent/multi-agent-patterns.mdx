## 一个小特忙不过来，就组一支小队

之前几章里，私人助理**小特**一直是单枪匹马：自己规划、自己记笔记、自己调工具，把一件件事办妥。对付「看今天要不要带伞」「订一张高铁票」这种活，一个小特绰绰有余。

可有些活又大又杂，跨好几个领域。比如「帮我写一篇某款相机的深度测评」——既要去查一堆资料和参数，又要把它写成流畅的文章，还得回头校对挑错。让**一个**小特把查、写、校全包了，他会顾此失彼：脑子里塞满资料腾不出手写作，刚要严谨核对又得切回发散行文，最后哪样都做不利索。

靠谱的做法，是组一支**小队**：找个擅长查资料的、一个擅长写的、一个擅长挑错的，各管一摊；再派个**主管**把活拆开、分下去、最后汇总。这一章就讲：什么时候该把活拆给多个 AI、又该怎么把它们编成一队。

## 为什么要多智能体：一个 Agent 会撑不住

先说清楚动机。把一桩复杂、跨多领域的大活全压给单个 Agent，会撞上三堵墙：

- **上下文塞爆**：查资料、写作规范、校对清单……啥都往一个 Agent 的上下文窗口里塞，很快就塞满，关键信息被淹没（上一篇「上下文工程」讲过这个病）。
- **角色打架**：同一个 Agent 既被要求「严谨核对、别瞎编」，又被要求「文笔发散、写得生动」——两套相反的人设挤在一个脑子里，互相拉扯，哪头都做不好。
- **难分工**：所有步骤搅在一团，出了错很难定位是哪一环坏的，也没法对某一环单独优化。

这就是 <Term def="multi-agent / 多智能体系统：把一个复杂任务拆给多个各有专长的 Agent 分工协作来完成，而不是压给单个全能 Agent。它解决的是「一个 Agent 包揽复杂、跨多领域的大活会撑不住」的问题——上下文塞爆、角色打架、难分工。隐喻：一个特工忙不过来，就组一支特工小队，研究员、写手、审查各司其职。">多智能体（multi-agent）</Term> 要解决的问题：与其让一个 Agent 硬扛，不如把活**拆成多个各有专长的 Agent**，分工协作。隐喻就是从「一个特工」变成「一支特工小队」——研究员、写手、审查各司其职，还有人当主管统筹。

至于这些 Agent 怎么编队、谁指挥谁、活怎么流转，就是下面三种**协作拓扑**要回答的。

## 三种协作拓扑：supervisor、swarm、pipeline

把多个 Agent 连起来干活，有三种最常见的**编队结构**。它们的形状、谁指挥谁、活怎么流转都不一样，适用的场景也不同——这是本章对比的核心。

第一种是 <Term def="supervisor / 主管制拓扑：一个「主管」Agent 负责把大任务拆成子任务、分派给下面几个专精 worker Agent，等它们各自干完、汇报结果，再由主管汇总成最终结果。是中心化指挥——拆、派、汇都归主管统筹。适合需要一个统筹者把活拆开分下去、再合起来的复杂任务。">supervisor（主管制）</Term>：一个**主管** Agent 在上，把大任务拆成子任务、分派给下面几个专精 worker，等它们各自干完、汇报结果，主管再**汇总**成最终结果。这是**中心化指挥**——拆、派、汇都归主管管。

第二种是 <Term def="swarm / 对等群拓扑：多个 Agent 地位对等，没有中心主管；它们之间互相传话、接力，谁该接手下一步就由 Agent 之间自己商量决定。是去中心化协作。适合任务边界不清、需要 Agent 之间灵活接力、动态决定下一步交给谁的场景。">swarm（对等群）</Term>：多个 Agent **地位对等**，没有谁当头儿；它们互相传话、接力，**谁该接手谁接手**，由 Agent 之间自己商量。这是**去中心化**。

第三种是 <Term def="pipeline / 流水线拓扑：把多个 Agent 串成一条线，上一个 Agent 的输出就是下一个 Agent 的输入，活沿着这条线单向依次流过（研究 → 写作 → 审查 → 出稿）。像工厂流水线，每个工位只做自己那一道工序。适合步骤固定、有清晰先后顺序的任务。">pipeline（流水线）</Term>：把 Agent **串成一条线**，上一个的输出就是下一个的输入，活沿着线**单向依次流过**——研究 → 写作 → 审查 → 出稿，像流水线，每个工位只做自己那一道工序。

## 角色专精：每个 Agent 只管一摊

多个 Agent 之所以比一个全能 Agent 强，关键在 <Term def="role specialization / 角色专精：给多智能体系统里的每个 Agent 一个专门的角色，配上专属的提示词（人设、规矩）和专属的工具，让它只管一摊事（研究员只管查、写手只管写、审查只管挑错）。各管一摊比一个又查又写又校的全能 Agent 更稳、更可控、也更好调试。">角色专精（role specialization）</Term>：给每个 Agent 一个**专门的角色**，配上**专属的提示词和工具**，让它只干一件事。

拿那篇相机测评来说：研究员 Agent 配上搜索工具、人设是「严谨、只摆事实、不臆测」；写手 Agent 不给搜索工具、人设是「文笔流畅、把素材组织成好读的文章」；审查 Agent 人设是「挑刺、核对事实和措辞」。**各管一摊**，互不打架——比起逼一个 Agent 同时严谨又发散，这样每个角色都能做到位，整体也更稳、更好查错。下面这个 Demo（下一节细玩）就能看到三个专精 worker 各领一件子任务的样子。

## 单还是多：别为多而多

最后一件要紧事：**多 Agent 不是越多越好**。把活拆成多个 Agent 是有代价的——

- **协调开销**：Agent 之间要来回传话、对齐上下文，沟通本身就花时间花 token。
- **更慢更贵**：跑好几个 Agent，调用次数翻倍，延迟和成本都上去。
- **更难调试**：链路变长、状态分散在多个 Agent 里，出了错更难定位。

所以选 <Term def="单 vs 多 Agent 选型：判断一个任务该用一个 Agent 还是多个 Agent。判据：任务简单、单领域、一步能完成 → 用单 Agent（更省更快更好调）；任务复杂、跨多领域、需要不同专长或视角 → 才值得用多 Agent。核心原则是「别为多而多」——多 Agent 带来协调开销、更慢更贵更难调试，只有收益大于这些开销时才划算。">单 vs 多 Agent 选型</Term> 有个判据：任务**简单、单领域、一步能完成**（翻译一句话、答个常识问题），用**单 Agent** 最省；只有任务**复杂、跨多领域、需要不同专长或视角**（写深度报告、跨部门审批）时，多 Agent 的收益才盖得过它的开销。一句话——**别为多而多**。下一节的选型 Demo 让你拿几个场景亲手判一判。

## 代码长什么样：一段 supervisor 骨架

把「动手一」那套流程落到代码上，supervisor 的骨架其实很短：先定义一组**带角色的 worker**（每个配专属提示），主管把大目标**拆成子任务**逐个**分派**给对应 worker，**收集**各自结果，最后**汇总**成最终输出。下面这段 Python（把真实的 LLM 调用省成几个函数）就是它的心脏：

```python
# ① 一组专精 worker：每个有自己的角色 + 专属提示（角色专精）
workers = {
    "researcher": Agent(role="研究员", prompt="只管查资料，严谨、只摆事实"),
    "writer": Agent(role="写手", prompt="只管把素材写成流畅的文章"),
    "reviewer": Agent(role="审查", prompt="只管挑错、核对事实与措辞"),
}

def supervisor(goal):
    plan = decompose(goal)          # ② 主管拆活：大目标 → [(worker 名, 子任务), ...]
    results = {}
    for name, subtask in plan:      # ③ 逐个分派给对应的专精 worker，收集结果
        results[name] = workers[name].run(subtask)
    return synthesize(goal, results)  # ④ 主管汇总各 worker 的产出 → 最终结果
```

四步正好对应前面讲的几样东西。`workers` 那个字典是**角色专精**——每个 worker 一个角色、一份专属提示，各管一摊。`decompose(goal)` 是主管**拆活**（呼应「动手一」第①步）。`for` 循环逐个 `workers[name].run(subtask)` 是**分派 + 收集**（第②③步并行干、第④步汇报）。最后 `synthesize(...)` 是主管**汇总**（第⑤步定稿）。

注意这正是 **supervisor 拓扑**：所有的拆、派、汇都由 `supervisor` 这个中心函数统筹，worker 之间互不直接对话——这就是「中心化指挥」。换成 **pipeline**，代码会变成把上一个 worker 的输出直接喂给下一个（`out = reviewer.run(writer.run(researcher.run(goal)))`，串成一条线）；换成 **swarm**，则是 worker 之间自己决定把活交给谁、没有这个中心 `supervisor`。同一支小队，编队方式不同，代码骨架就不同。

## 容易踩的坑

<Callout type="trap">
  **现象（选错方案的坑）**：「多 Agent 那么强、能分工，那啥任务都拆成多 Agent
  准没错！」于是连「翻译一句话」也派一支研究员 + 写手 +
  审查的小队上。**原因**：把多 Agent 当成「越多越好」，忘了它要 Agent
  之间**来回传话、对齐上下文**，协调开销大、更慢更贵更难调试。**修法**：看任务**复不复杂、跨不跨领域、要不要多种专长**——简单、单领域、一步能完成的活，**单
  Agent** 最省；只有**复杂、跨多领域、要不同视角**的活才值得多 Agent。
  **别为多而多**。
</Callout>

<Callout type="trap">
  **现象**：拆成了多个
  Agent，可它们各干各的，研究员查到的资料写手没收到、写手写的稿审查没看见，最后拼出来的东西对不上、自相矛盾。**原因**：Agent
  之间**没传清上下文**——只顾拆分，忘了协作的前提是把该共享的信息、各自的产出在
  Agent 间**传递到位**。**修法**：明确每个 Agent
  的**输入和输出**，让上一环的产出确实流到下一环（pipeline 串好、supervisor
  收齐再汇总）；别让拆开的 Agent 变成一盘各说各话的散沙。
</Callout>

<Callout type="trap">
  **现象**：给多个 Agent
  派活时角色没分清——研究员也被要求顺手写两句、写手也被要求核对事实，**职责重叠**，结果两个
  Agent
  对同一件事给出打架的结论，谁也不服谁。**原因**：没做好**角色专精**，职责边界模糊，等于把「一个
  Agent 角色打架」的毛病搬到了多个 Agent 之间。**修法**：给每个 Agent
  一个**清晰、不重叠**的角色和专属提示，一人只管一摊（研究员只查、写手只写、审查只挑错），边界划清，才不会互相打架。
</Callout>

## 小结

- **为什么多智能体**：一个 Agent 包揽复杂、跨多领域的大活会**撑不住**——上下文塞爆、角色打架、难分工；解法是拆成**多个各有专长的 Agent**分工协作（一个特工 → 一支特工小队）
- **三种协作拓扑**：**supervisor**（一个主管拆派汇总，中心化）、**swarm**（一群对等 Agent 互相接力，去中心化）、**pipeline**（串成一条线，上一个的输出是下一个的输入，依次过工序）——结构、控制方式、适用场景各不同
- **角色专精**：给每个 Agent 一个专门角色 + 专属提示和工具，各管一摊，比一个全能 Agent 更稳、更可控、更好调试
- **单 vs 多选型**：不是越多越好——多 Agent 有协调开销、更慢更贵更难调；简单/单领域用**单**，复杂/跨多领域/要多视角才用**多**，**别为多而多**
- 一段 supervisor 骨架 = 一组带角色的 named worker（角色专精）+ 主管 `decompose` 拆活分派 + 收集结果 + `synthesize` 汇总（中心化协调）

（b）「做一份新产品上市方案：先做市场调研、再写策划文案、最后由风控审一遍合规，三步有明确先后」。

<Answer>

- **（a）用单 Agent** —— 翻译一段话是**简单、单领域、一步能完成**的活，一个 Agent 直接办了最省。拆成多 Agent 纯属增加协调开销，更慢更贵还没收益。记住「别为多而多」。
- **（b）用多 Agent，且适合 pipeline（流水线）** —— 这是**复杂、跨多领域**（调研 / 文案 / 风控是三门手艺）的活，值得拆成多个专精 Agent。而且三步**有明确先后顺序**、上一步的产出是下一步的输入（调研结果喂给文案、文案稿喂给风控审），正是 **pipeline** 的主场：串成一条线依次过工序。（若没有固定先后、而是要一个统筹者把活拆开分派再合起来，则更适合 supervisor。）

要点：先判单还是多（简单/单领域→单；复杂/跨多领域→多）；再判拓扑（有清晰先后顺序→pipeline；要中心统筹拆派汇总→supervisor；要对等灵活接力→swarm）。

</Answer>

**练习 2（改 Demo 题）** 打开「动手一」的主管分派 Demo，走到「③ 各自并行干」这一步——三个 worker 是同时开工、各管一摊的。设想把这个系统改回**单个 Agent**包揽查、写、校三件事，对照本章「为什么要多智能体」那三堵墙，说说这个单 Agent 会比小队差在哪三处。

<Answer>

改回单 Agent，会撞回那三堵墙：

- **上下文塞爆**：查资料的素材、写作的规范、校对的清单全得塞进**同一个**上下文窗口，很快塞满、关键信息被淹没；而小队里每个 worker 上下文只装自己那一摊，干净得多。
- **角色打架**：同一个 Agent 既要「严谨核对、别瞎编」又要「文笔发散、写得生动」，两套相反人设互相拉扯；小队里研究员只严谨、写手只发散、审查只挑刺，各角色都能做到位。
- **难分工 / 难调试**：查写校搅成一团，出了错难定位是哪一环坏的；小队里三环分开，哪环出问题一眼看出、还能单独优化。

所以这类**复杂、跨多领域**的活，多 Agent（角色专精 + 中心化协调）比单 Agent 更稳更可控。

</Answer>

**练习 3（问答题）** 第六节那段 supervisor 骨架里，`workers` 字典给每个 worker 配了不同的 `role` 和 `prompt`。如果偷懒，把三个 worker 的 `prompt` 都写成同一句「啥都干，又查又写又校」，这个多 Agent 系统会退化成什么？为什么这样不好？

<Answer>

它会退化成**三个一模一样的全能 Agent 在重复干活**——本质上和「一个 Agent 包揽」没区别，甚至更糟：既没省下单 Agent 的那三堵墙（每个 worker 仍然上下文塞爆、角色打架），又白白多付了多 Agent 的协调开销（来回传话、更慢更贵）。

为什么不好：多 Agent 的价值来自**角色专精**——每个 Agent 有清晰、不重叠的角色和专属提示，各管一摊才更稳更可控。把 prompt 都写成「啥都干」就抹掉了专精，三个 worker 没有分工、职责重叠，还可能对同一件事给出打架的结论。要让多 Agent 真正划算，前提是把每个 Agent 的角色划清、专属提示配到位。

</Answer>

## 名词解释

<Glossary>

<GlossaryItem term="多智能体（multi-agent）">
  把一个复杂任务**拆给多个各有专长的 Agent 分工协作**来完成，而不是压给单个全能
  Agent。它专治「一个 Agent
  包揽复杂、跨多领域的大活会**撑不住**」——上下文塞爆（啥都往一个窗口塞）、角色打架（既要严谨又要发散）、难分工。隐喻：一个特工忙不过来，就组一支特工小队，研究员、写手、审查各司其职，还有人当主管统筹。详见本章「为什么要多智能体」一节。
</GlossaryItem>

<GlossaryItem term="supervisor（主管制）">
  一种协作拓扑：一个「**主管**」Agent
  负责把大任务**拆成子任务、分派**给下面几个专精
  worker，等它们各自干完、**汇报**结果，再由主管**汇总**成最终结果。是**中心化指挥**——拆、派、汇都归主管统筹，worker
  之间互不直接对话。适合需要一个统筹者把活拆开分下去、再合起来的复杂任务。类比：项目经理把活分给组员，再把大家的产出合成交付物。详见本章「三种协作拓扑」一节与拓扑动画。
</GlossaryItem>

<GlossaryItem term="swarm（对等群）">
  一种协作拓扑：多个 Agent
  **地位对等**，没有中心主管；它们之间**互相传话、接力**，谁该接手下一步就由
  Agent 之间**自己商量**决定。是**去中心化**协作。适合任务边界不清、需要 Agent
  之间灵活接力、动态决定下一步交给谁的场景。类比：一群平级同事在群里互相 @
  接力把事推进，没有领导统一调度。详见本章「三种协作拓扑」一节与拓扑动画。
</GlossaryItem>

<GlossaryItem term="pipeline（流水线）">
  一种协作拓扑：把多个 Agent **串成一条线**，上一个 Agent
  的输出就是下一个的输入，活沿着这条线**单向依次流过**（研究 → 写作 → 审查 →
  出稿）。像工厂流水线，每个工位只做自己那一道工序。适合**步骤固定、有清晰先后顺序**的任务。类比：流水线上工人各守一个工位，半成品一站站往后传。详见本章「三种协作拓扑」一节与拓扑动画。
</GlossaryItem>

<GlossaryItem term="角色专精（role specialization）">
  给多智能体系统里的每个 Agent
  一个**专门的角色**，配上**专属的提示词**（人设、规矩）和**专属的工具**，让它只管一摊事——研究员只管查、写手只管写、审查只管挑错。**各管一摊**比一个又查又写又校的全能
  Agent **更稳、更可控、更好调试**（上下文干净、人设不打架、出错好定位）。是多
  Agent 之所以比单个全能 Agent 强的关键。详见本章「角色专精」一节与主管分派
  Demo。
</GlossaryItem>

<GlossaryItem term="单 vs 多 Agent 选型">
  判断一个任务该用**一个** Agent 还是**多个**
  Agent。判据：任务**简单、单领域、一步能完成**（翻译一句话、答常识问题）→
  用**单
  Agent**（更省更快更好调）；任务**复杂、跨多领域、需要不同专长或视角**（写深度报告、跨部门审批）→
  才值得用**多 Agent**。核心原则是「**别为多而多**」——多 Agent
  带来协调开销、更慢更贵更难调试，只有收益大于这些开销时才划算。详见本章「单还是多」一节与选型
  Demo。
</GlossaryItem>

</Glossary>
