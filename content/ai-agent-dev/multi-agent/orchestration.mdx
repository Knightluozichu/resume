## 小队组好了，可它们怎么真正配合起来

上一章里，我们把私人助理**小特**从一个人，扩成了一支**特工小队**——研究员、写手、审查各管一摊，还排好了队形（主管制、对等群、流水线）。可光把人凑齐、排好队，活并不会自己干起来。

队员之间得**传话**：研究员查到的资料怎么递给写手？写手的稿子怎么交给审查？还得有人定**节奏**：谁先动手、谁接着上、一件事来回过几遍。最要命的是，得知道**什么时候收工**——不然两个 Agent 你说「再改改」、我说「你改」，能没完没了地踢皮球，把钱（token）烧光还没出活。

这一章就补上这三块：队员之间**怎么传消息**、怎么共用一块**共享的板子**、由谁来**编排轮次**，以及怎么给小队设个**收工的闸**，别让它转个不停。

## 通信：Agent 之间靠「消息」交流

多个 Agent 要配合，第一件事是**互相传话**。它们之间交流的基本单位，是一条 <Term def="message / 消息：多个 Agent 之间通信的基本单位，是结构化的一条，至少包含三样——谁发的（发送者 / from）、发给谁（接收者 / to）、内容是什么（content）。Agent 之间所有的「传话」（主管分派子任务、worker 汇报结果、流水线上一道工序把产出传给下一道）本质都是在收发消息。结构化是关键：标好 from/to/content，对方才知道这是谁的、给谁的、说的啥。">消息（message）</Term>。

一条消息不是随便一句话，而是**结构化的一条**，至少带三样：**谁发的**（发送者，from）、**发给谁**（接收者，to）、**内容是什么**（content）。标清楚这三样，对方才知道「这是谁递过来的、是给我的、说的什么」。

承接上一章的拓扑，你会发现各种协作其实都是在收发消息：**supervisor** 里，主管把子任务作为消息**分派**给 worker（主管 → 研究员：「去查资料」），worker 干完把结果作为消息**汇报**回主管（研究员 → 主管：「查到 3 篇评测」）；**pipeline** 里，上一道工序把产出作为消息**传给**下一道（研究员 → 写手：「这是资料」）。下面这张图把主管和小队之间一来一回的消息逐条点亮，每条都标着 from → to 和内容——亲眼看「一条消息 = 发送者 + 接收者 + 内容」：

## 共享状态：给所有 Agent 一块黑板

点对点传消息很直接，但有个麻烦：人一多，消息就满天飞——研究员要分别发给写手、审查，写手改完又要分别通知一圈，谁该把什么发给谁，容易乱、容易漏。

另一种办法，是给所有 Agent 一块 <Term def="shared state / blackboard，共享状态 / 黑板：给多智能体系统里所有 Agent 一块共享的状态板（黑板），谁都能往上写、谁都能读。各 Agent 不必两两点对点传消息，而是都冲着同一块板——研究员把资料写上去，写手一读黑板就拿到资料、再把初稿写上去，审查一读就看到资料和初稿。适合多个 Agent 围着同一份逐步完善的产物协作。代价是大家写同一块板，得有人协调写入顺序，否则会冲突。">共享状态 / 黑板（blackboard）</Term>——一块**谁都能写、谁都能读**的板子。研究员把查到的资料写到黑板上；写手不用等研究员专门发给它，自己**一读黑板**就拿到资料，写完初稿又写回黑板；审查再读黑板，一次性看到资料和初稿，写上修改意见。黑板内容**逐步累积**，像一份大家轮流接力、越填越完整的草稿。

这和点对点传消息是两种思路：**消息传递**是各发各的、点对点；**共享黑板**是大家**围着同一块板**、读写同一份产物。后者特别适合「多个 Agent 协作打磨同一个东西」的场景（一起写一篇报告、一起调一份方案）。下面这个 Demo 让你点着走一遍，看三个 Agent 怎么轮流读写、把黑板填满：

## 编排：谁先谁后、转几轮，得有人调度

有了传消息和共享黑板，还差一个总指挥。因为多个 Agent**不会自己有序起来**——你不告诉它们顺序，三个 Agent 可能一起抢着说话、或者都在等别人先动。

得有个 <Term def="orchestration / 编排，轮次控制：多智能体系统里负责调度的那一环——决定谁先谁后、这一轮该轮到哪个 Agent、整个流程转几轮。承担调度的叫编排器（orchestrator）。它一般是一个循环：每一轮选出下一个该干活的 Agent、让它跑、用它的产出更新共享状态，再进入下一轮，直到满足终止条件。编排决定了这支小队按什么节奏推进。supervisor 的主管、pipeline 固定的工序顺序，本质都是一种编排。">编排（orchestration）</Term> 来调度：谁先动手、这一轮轮到谁、整件事转几轮。承担这活的叫**编排器（orchestrator）**。它通常就是一个**循环**：每一轮选出**下一个该干活的 Agent**、让它跑、用它的产出**更新共享状态**，再进入下一轮。

编排决定了这支小队**按什么节奏推进**。上一章的 supervisor 里，「主管」就是编排器（它定下「先派给研究员、再派给写手」的顺序）；pipeline 里，固定的工序顺序（研究 → 写作 → 审查）就是编排。下一节的 Demo 里，你点「再转一轮」，背后就是编排循环在推进一轮——它还藏着本章最后一个、也是最关键的概念。

## 终止：什么时候必须强制收工

多 Agent 最危险的坑，不是干不好，而是缺少 <Term def="termination condition / 终止条件，死循环防护：编排循环必须设的「收工的闸」——满足某个条件就强制停下来，别让多个 Agent 无限转下去。常见的终止条件有三种：① 达成目标（比如审稿 Agent 放行了）；② 到最大轮数（max_rounds，转够 N 轮就兜底收工）；③ 连续几轮没进展（卡住了就停）。不设终止条件，两个 Agent 可能互相踢皮球无限循环——你让我改、我让你改，轮次无限涨、token 烧光还出不了活。终止条件是多智能体系统的安全阀，尤其 max_rounds 这个兜底几乎是必备的。">终止条件（termination condition）</Term>，导致系统停不下来。

设想让一个写作 Agent 和一个审稿 Agent 互相配合：写手出稿、审查挑错、写手再改、审查再挑……如果**不设终止条件**，而审查又永远能挑出点毛病，这俩就会**无限踢皮球**——你让我改、我让你改，轮次一路涨，token 一路烧，活还没出。

所以编排循环**必须设终止条件**——一道收工的闸，满足就强制停。常见三种：**达成目标**（审稿 Agent 放行了就停）、**到最大轮数**（`max_rounds`，转够 N 轮就兜底收工，哪怕没谈拢）、**连续几轮没进展**（卡住了就别耗了）。其中 `max_rounds` 这个兜底几乎是必备的保险丝。下面这个 Demo 把两条路摆一起：左边是**有终止条件**——到点稳稳收工（标绿）；切到**无终止条件**——轮次失控涨、烧 token（标红）。亲手切一下对比：

## 代码长什么样：消息、黑板、编排循环

把这一章的四个概念落到代码上，其实就三小段 Python。下面这些用普通函数把真实的 LLM 调用省成示意，但**消息结构、共享黑板、编排循环 + 终止条件**这套骨架是真的、能直接跑。

### (a) 一条消息：from / to / content

先看**消息**。它就是一个带三个字段的小对象——发送者、接收者、内容。Agent 之间所有「传话」，本质都是造一条这样的消息递出去：

```python
class Message:
    """一条结构化消息：谁发的、发给谁、内容是什么。"""

    def __init__(self, sender, receiver, content):
        self.sender = sender      # from：谁发的
        self.receiver = receiver  # to：发给谁
        self.content = content    # content：内容

    def __repr__(self):
        return f"[{self.sender} -> {self.receiver}] {self.content}"


# 主管把一件子任务作为消息分派给研究员：
msg = Message("supervisor", "researcher", "去查这款相机的评测和参数")
```

关键就在那三个字段：**标清楚 from / to / content**，对方才知道这条消息是谁的、给谁的、说的啥。

<Callout type="tip">
  真实框架里 `Message` 往往还带 `role`（system / user /
  assistant）、时间戳、`tool_call` 等字段，但**发送者 + 接收者 +
  内容**这三样是骨干。别把消息写成一句光秃秃的字符串——没标 from / to，多个 Agent
  一多就对不上号。
</Callout>

### (b) 共享黑板：一个谁都能读写的 dict

再看**共享黑板**。最朴素的实现就是一个 `dict`（或一个对象），所有 Agent 都拿着它、都往里读写同一份状态：

```python
# 共享黑板：一个 dict，各 Agent 读写同一份。
blackboard = {"draft": None, "approved": False, "notes": []}


def writer_agent(bb):
    """写作 Agent：读黑板已有意见 -> 改稿 -> 写回黑板。"""
    rounds = len(bb["notes"])
    bb["draft"] = f"初稿 v{rounds + 1}（已吸收 {rounds} 条意见）"
    return Message("writer", "reviewer", bb["draft"])


def reviewer_agent(bb):
    """审稿 Agent：读黑板 -> 给意见；改到第 2 版就放行。"""
    rounds = len(bb["notes"])
    if rounds >= 2:                 # 收敛了 -> 达成目标
        bb["approved"] = True
        return Message("reviewer", "writer", "可以了，通过")
    bb["notes"].append(f"意见{rounds + 1}：再改改")
    return Message("reviewer", "writer", bb["notes"][-1])
```

注意两个 Agent 都收同一个 `bb` 参数、都读写它的同一批 key（`draft` / `notes` / `approved`）。写手把稿子写进 `bb["draft"]`，审查直接从 `bb` 读到它再追加意见——**谁都不用专门把东西发给谁，黑板就是那个共享的中转站**。这正是「围着同一块板协作」。

<Callout type="trap">
  共享黑板很方便，但**大家写同一块板**得有人协调写入顺序。这段代码靠下面的编排循环让它们**一个一个轮流写**，所以安全；要是放任两个
  Agent **同时**改
  `bb["draft"]`，就会互相覆盖、冲突——共享状态没人协调，是个真坑（下面误区还会再说）。
</Callout>

### (c) 编排循环 + 终止条件：本章的心脏

最后是**编排器**——一个循环，每轮选下一个 Agent、跑它、更新黑板，并在满足条件时**收工**。本章最关键的 `break` 就在这：

```python
AGENTS = {"writer": writer_agent, "reviewer": reviewer_agent}


def orchestrate(bb, max_rounds=8):
    """编排循环：轮流调度 Agent，带终止条件防死循环。"""
    turn_order = ["writer", "reviewer"]
    for r in range(max_rounds):                  # ① 到 max_rounds 兜底收工
        name = turn_order[r % len(turn_order)]   # ② 选下一个该轮到的 Agent
        msg = AGENTS[name](bb)                    # ③ 跑它、更新共享黑板
        print(msg)
        if bb["approved"]:                        # ④ 达成目标 -> 正常收工
            return f"达成目标，第 {r + 1} 轮收工"
    return f"到 max_rounds={max_rounds}，强制收工"  # 兜底：防死循环
```

四步对上前面讲的：`for r in range(max_rounds)` 是**编排 + 到最大轮数兜底**（终止条件之一）；`turn_order[r % len]` 是**选下一个该轮到的 Agent**（轮次控制）；`AGENTS[name](bb)` 是**跑它、更新共享黑板**；`if bb["approved"]` 是**达成目标就收工**（终止条件之二）。

最要命的是这两道闸**缺一不可**：只靠 `if bb["approved"]`、不写 `max_rounds`，万一审查永不放行，循环就**永远跑下去**——这就是死循环。`for r in range(max_rounds)` 这个兜底，是无论如何都能停下来的**保险丝**。

<Callout type="tip">
  这段 `orchestrate` 实际跑出来：写手出 v1 → 审查给意见1 → 写手出 v2 →
  审查给意见2 → 写手出 v3 → 审查**放行**，第 6 轮触发 `bb["approved"]` 收工，**没碰到
  max_rounds 就正常停了**。把 `reviewer_agent` 里的 `if rounds >= 2`
  改成永远不放行，就会一路跑到 `max_rounds=8` 被兜底闸强制收工——这正是终止条件在兜底。
</Callout>

  哗哗烧光，活还没出。**原因**：编排循环**没设终止条件**（或只设了「达成目标」、没设「最大轮数」兜底），一旦双方谈不拢就无限踢皮球。**修法**：编排循环**必须**带终止条件——`达成目标`
  / `到 max_rounds` / `连续几轮没进展`，三者至少留一道，且 `max_rounds`
  这个兜底保险丝几乎必加。亲眼看翻车：把 Demo 切到「无终止条件」连点几次。
</Callout>

<Callout type="trap">
  **现象**：用了共享黑板，可几个 Agent
  **同时**往黑板上写，结果你写的被我覆盖、最后黑板上的内容残缺或自相矛盾。**原因**：**共享状态没人协调**——多个
  Agent 并发读写同一份 `bb`，写入互相打架（race
  condition）。**修法**：让编排器**串行调度**（一轮只让一个 Agent 写，像上面
  `orchestrate` 那样轮流来），或给关键 key 加锁 / 版本号；别放任多个 Agent
  不排队就抢着改同一块板。
</Callout>

<Callout type="trap">
  **现象**：消息满天飞，但 Agent
  收到后**对不上号**——不知道这条是谁发的、是不是给自己的，把别人的活当成自己的干，或者干脆漏处理。**原因**：**消息没结构**——传的是一句光秃秃的字符串，没标

  §6a 的 `Message`）；接收方按 `to` 过滤出该自己处理的，按 `from` 知道是谁来的。
</Callout>

## 小结

- **通信 / 消息**：Agent 之间靠**消息**交流，一条消息是结构化的一条——**谁发的（from）+ 发给谁（to）+ 内容（content）**；分派、汇报、流水线传递本质都是收发消息
- **共享状态 / 黑板**：给所有 Agent 一块**谁都能读写的黑板**，围着同一份逐步完善的产物协作；vs 点对点传消息，后来的 Agent 一读黑板就拿到前面所有人的产出
- **编排 / 轮次控制**：多个 Agent 不会自己有序，得有**编排器**调度——谁先谁后、这轮轮到谁、转几轮；它通常就是一个「选 Agent → 跑 → 更新状态」的循环
- **终止条件 / 死循环防护**：编排循环**必须**设收工的闸——**达成目标 / 到最大轮数（max_rounds 兜底）/ 没进展**，否则两个 Agent 会无限踢皮球烧光 token
- **一段编排循环** = `for r in range(max_rounds)`（编排 + 兜底）+ 选下一个 Agent + 跑它更新黑板 + `if 达成目标: break`（双闸缺一不可）

<Exercises>

**练习 1（独立实现题）** 自己写一个**带终止条件的编排循环** `run(bb, agents, max_rounds)`：每一轮轮流从 `agents`（一个有序列表，元素是 `(名字, 函数)`）里取下一个 Agent，调用它（传入共享状态 `bb`）、更新 `bb`，并在两种情况下停下来——`bb` 里出现 `bb["done"] == True`（达成目标），或者转满 `max_rounds` 轮（兜底）。返回停下来的原因。要求**两道终止闸都要有**。

<Answer>

```python
def run(bb, agents, max_rounds=8):
    """轮流调度 agents，带双终止闸：达成目标 or 到 max_rounds。"""
    for r in range(max_rounds):              # 闸一：到最大轮数兜底
        name, fn = agents[r % len(agents)]   # 选下一个该轮到的 Agent
        fn(bb)                               # 跑它，更新共享状态 bb
        if bb.get("done"):                   # 闸二：达成目标就收工
            return f"达成目标，第 {r + 1} 轮收工"
    return f"到 max_rounds={max_rounds}，强制收工（兜底防死循环）"
```

要点：

- `for r in range(max_rounds)` 既是**编排循环**又是**最大轮数兜底**——无论如何都能停（防死循环的保险丝）。
- `agents[r % len(agents)]` 实现**轮次控制**：一轮选一个，轮流来。
- `if bb.get("done")` 是**达成目标**这道闸。
- **两道闸缺一不可**：只留 `done` 不写 `max_rounds`，万一某个 Agent 永远不把 `done` 置 True，循环就永远跑——这正是死循环。只写 `max_rounds` 不看 `done`，则达成目标后还会空转到轮满，浪费 token。

</Answer>

**练习 2（改 Demo 题）** 打开本章「终止条件」Demo，把开关切到**「无终止条件」**，连点几次「再转一轮」。对照 §6c 的编排循环代码，说说：这个失控的状态，对应代码里**少了哪一行**？把那一行加回去，又是怎么让它停下来的？

<Answer>

失控状态对应代码里**少了 `for r in range(max_rounds)` 这道最大轮数兜底闸**（等价于把循环写成 `while not bb["approved"]:` 却不限轮数）。

- 「无终止条件」就是：只要审稿 Agent 一直挑得出毛病、`bb["approved"]` 永远是 `False`，循环就**永远转下去**，轮次计数器一路涨、token 一路烧——Demo 里你看到的正是这个（转了好几轮还在「再改改 / 你改」，到一定轮数标红示警）。
- 把 `for r in range(max_rounds)` 加回去（或在 `while` 里加 `if r >= max_rounds: break`），就给了循环一道**无论如何都会触发的兜底闸**：哪怕双方永远谈不拢，转满 `max_rounds` 轮也会强制收工。切回 Demo 的「有终止条件」，它到第 3 轮稳稳停住，就是这道闸在起作用。

一句话：**死循环 = 缺了那道兜底的最大轮数闸**；补上它，系统就有了保证能停下来的安全阀。

</Answer>

**练习 3（问答题）** §6b 的共享黑板 Demo / 代码里，两个 Agent 都读写同一个 `bb`。如果把它们改成**同时并发**跑（而不是编排器让它们一轮一个轮流来），可能出什么问题？为什么「编排器串行调度」能避免这个问题？

<Answer>

可能出**写入冲突（race condition）**：两个 Agent 同时改 `bb["draft"]`，后写的把先写的覆盖掉，或者一个 Agent 读到的是另一个 Agent 改了一半的中间状态，导致黑板内容残缺、自相矛盾——这就是「共享状态没人协调」那个坑。

为什么串行调度能避免：**编排器一轮只让一个 Agent 动**（像 `orchestrate` 里 `turn_order[r % len]` 那样轮流取一个来跑），任一时刻只有一个 Agent 在读写 `bb`，自然不会两个写操作打架。也就是说，**编排（谁先谁后、一轮一个）顺手就解决了共享状态的协调问题**——这正是「通信 / 共享状态 / 编排」三者要配合着用的原因：共享黑板提供了协作的介质，编排器提供了安全读写它的秩序。（如果确实要并发，就得给关键 key 加锁 / 版本号，但那比串行复杂得多。）

</Answer>

<GlossaryItem term="消息（message）">
  多个 Agent
  之间**通信的基本单位**，是结构化的一条，至少包含三样——**谁发的**（发送者 /
  from）、**发给谁**（接收者 / to）、**内容是什么**（content）。Agent
  之间所有的「传话」（主管分派子任务、worker
  汇报结果、流水线上一道工序把产出传给下一道）本质都是在收发消息。结构化是关键：标好
  from / to /
  content，对方才知道这是谁的、给谁的、说的啥。详见本章「通信」一节与消息传递动画。
</GlossaryItem>

<GlossaryItem term="共享状态 / 黑板（blackboard）">
  给多智能体系统里**所有 Agent
  一块共享的状态板**（黑板），谁都能往上写、谁都能读。各 Agent
  不必两两点对点传消息，而是都冲着同一块板——研究员把资料写上去，写手一读黑板就拿到资料、再把初稿写上去，审查一读就看到全部。适合多个
  Agent
  **围着同一份逐步完善的产物**协作。代价是大家写同一块板，得有人协调写入顺序，否则会冲突。类比：一块大家轮流接力填的白板。详见本章「共享状态」一节与黑板
  Demo。
</GlossaryItem>

<GlossaryItem term="编排（orchestration）">
  多智能体系统里负责**调度**的那一环——决定**谁先谁后、这一轮该轮到哪个
  Agent、整个流程转几轮**。承担调度的叫**编排器（orchestrator）**。它一般就是一个循环：每轮选出下一个该干活的
  Agent、让它跑、用它的产出更新共享状态，再进入下一轮，直到满足终止条件。编排决定了这支小队**按什么节奏推进**（supervisor
  的主管、pipeline 的固定工序顺序，本质都是编排）。详见本章「编排」一节。
</GlossaryItem>

<GlossaryItem term="终止条件（termination condition）">
  编排循环**必须设的「收工的闸」**——满足某个条件就强制停下来，别让多个 Agent
  无限转下去。常见三种：① **达成目标**（比如审稿 Agent 放行了）；②
  **到最大轮数**（`max_rounds`，转够 N 轮就兜底收工）；③
  **连续几轮没进展**（卡住了就停）。不设它，两个 Agent
  可能**互相踢皮球无限循环**——你让我改、我让你改，轮次无限涨、token
  烧光还出不了活。是多智能体系统的**安全阀**，尤其 `max_rounds`
  这个兜底几乎必备。详见本章「终止」一节与终止条件 Demo。
</GlossaryItem>

</Glossary>
