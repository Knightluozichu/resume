## 拆开那位失忆天才的脑袋

上一章我们认识了那位「关在没窗小屋里、绝顶聪明却严重失忆」的天才顾问。这一章，我们把他的脑袋拆开看看：他到底是怎么读你塞进去的纸条、怎么写回复的？只有看清他天生会什么、不会什么，才知道后面该给他补哪些东西。

你会发现三件有点反直觉的事：他不是一个字一个字读纸条的，他一次能读进的纸条长度是有上限的，而且他写回复不是「想好整句再下笔」，而是一个词一个词地往后接、接到觉得该停为止。

弄懂这三件事，本章的目的就达到了：你会明白他为什么有时一本正经地胡说、为什么转头就忘、为什么干不了实事——而这正好对应后面几篇要给他添的东西。不弄懂，你就会对着这些「毛病」干着急，却不知道病根在哪、该补哪一块。

## 第一件事：模型不按字读，按 token 读

我们直觉以为模型像人一样一个字一个字读，其实不是。它先把你的文字切成一个个小块，再以这些小块为单位处理——这些小块就叫 <Term def="模型处理文字的最小单位，是「切碎的文字块」。一个 token 可能是一个汉字、一个英文单词，也可能是一个单词的一段（子词）或一个标点。模型读你的纸条、写回复，都是以 token 为单位，而不是以「字」为单位。">token</Term>。把整段文字切成 token 的这个过程，叫 <Term def="把一段文字切成一串 token 的过程（也叫「分词」）。不同语言切法不同：中文常一个字切成一个 token，英文常按单词或单词的一段（子词）切——所以同样长的一句话，token 数可能差很多。">tokenization（token 化 / 分词）</Term>。

为什么要切？因为模型内部只认这些「块」的编号，不认原始文字。关键的反直觉点在这：**一个 token 不一定等于一个字**。中文常常一个字就是一个 token；英文则常按单词、甚至单词的一段来切——一个长单词 `unbelievable` 可能被切成 `un`、`believ`、`able` 三个 token。所以「token 数」和你眼睛数的「字数」往往对不上，而且中英文还对不上得不一样。

这件事为什么要紧？因为后面所有「能记多长」「调一次模型多少钱」「会不会超长」的账，算的都是 token 数，不是字数。下面这个小工具让你亲手感受一下——打一句话进去，看它被切成几个 token：

> 猜一猜：同样是「五个符号」，纯中文的「今天天气真」和一个英文长词，哪个切出来的 token 更多？

<AaTokenizerPlayground />

多打几句你会摸到规律：中文大致「一字一 token」，而英文一个长词常被拆成好几块——**token 是模型的「字」，但它和我们说的字不是一回事**。

## 第二件事：他一次只读得进这么长

失忆天才的门缝只有那么宽，你一次能塞进去的纸条长度是有上限的。这个上限就是 <Term def="模型一次能「读进去」的 token 数量上限，也叫上下文长度。你塞给它的所有内容——系统提示、之前的对话、当前问题——加起来的 token 数不能超过这个上限。超出的部分会被挤出窗口、模型读不到，表现出来就像「忘了」。">上下文窗口（context window）</Term>：模型一次能读进去的 token 总数是固定的。

注意，你塞进去的可不只是「当前这句问题」。系统给它的设定（系统提示）、之前来回的对话历史、再加上你这次的问题，全都要一起塞进同一张「纸条」，一起占用这个窗口。纸条越塞越长，一旦超过窗口容量，**最前面的内容就被挤出窗口、模型读不到了**——表现出来，就像它把前面说过的话「忘了」。这正是上一章说的「失忆」在机制上的来由：不是它不想记，是门缝就这么宽，塞不下的就掉出去了。

下面这张解剖图把「纸条—窗口—被挤掉」画清楚，顺带也把上一节的 token 化中英对照摆在一起看：

<AaContextWindowDiagram />

记住这张图的画面：**上下文窗口 = 门缝能塞进的纸条长度**。它有限，超了就「忘」。这也解释了为什么聊天聊久了模型会前言不搭后语——前面的对话被挤出窗口了。怎么在有限窗口里塞下更多有用信息、让它「记得住」，正是后面「记忆与知识」那一篇要解决的事。

## 第三件事：他写回复，是一个个 token 接出来的

最反直觉的来了。我们以为模型「想好一整句」再写出来，其实不是。它干的是一件小到不可思议的事，叫 <Term def="模型生成文字的根本机制：看着已经有的全部文字，对「下一个 token 应该是什么」算出一个概率分布，从中挑一个（通常挑概率最高的），接到已有文字后面；然后把刚接上的也算进「已有文字」，再预测下一个——如此反复。所谓「生成一段话」，就是把这个动作做很多很多次。">下一个 token 预测（next-token prediction）</Term>。

它的完整动作是这样的：看着已经有的全部文字 → 对「下一个 token 该是什么」算出一个概率分布（每个候选 token 配一个概率）→ 从中挑一个（通常挑概率最高的）→ 接到已有文字后面 → 然后把刚接上的也算进「已有文字」，再预测下一个。**所谓「生成一段话」，就是把这个小动作反复做很多很多次**，一个 token 接一个 token，直到挑中一个表示「结束」的 token 才停笔。

这件事是整个大模型的核心，也是后面很多现象的根。下面这张可控动画，把这个「预测—挑选—接上—再来」的循环一步步演给你看：

> 猜一猜：模型写「今天天气真」之后那个字，是「想好后面要写什么」才下笔，还是先给好几个候选字算概率、再挑一个？

<AaNextTokenDiagram />

播完你会明白：模型并不「知道」自己整句要说什么，它只是**每一步都在赌「下一个 token 最可能是谁」**。这件事顺手就解释了它的一个著名毛病——既然它只管「接一个最顺的 token」，而不管「这内容是不是真的」，那当它没有可靠依据时，照样能一脸认真地接出一串通顺却不实的内容。这种「没依据也敢编、还编得很像真的」的现象，就叫 <Term def="大模型一本正经地编造不实内容的现象：它生成文字靠的是「接一个最顺的下一个 token」，而不是「核对事实」，所以在没有可靠依据时，照样能输出通顺但错误的答案（比如编造不存在的引用、数据、人名）。它不是故意撒谎，而是机制使然。">幻觉（hallucination）</Term>。

## 把三件事连起来：他天生的三块短板

把上面三件事连起来，那位失忆天才天生的三块短板就一目了然了，而这三块恰好就是本书后面几篇要逐一给他补的：

- **会幻觉**——没可靠依据也敢一脸认真地编。因为他只管接「最顺的下一个 token」，不管内容真假。补法：后面「记忆与知识」篇给他一座可检索的资料库（RAG），让他答之前先去查、有依据再说。
- **记不住**——关了就忘、聊久了也忘。因为上下文窗口有限，超出的被挤掉。补法：同样在「记忆与知识」篇给他笔记本（短期记忆）和长期记忆，把该记的存下来、要用时再取回塞进窗口。
- **动不了手**——出不了屋，不能上网、算账、查实时数据。因为他天生只会「读 token、写 token」。补法：「让智能体行动」篇给屋子装一部「电话」（工具调用），让他能在回复里写「请帮我做 X」，由外面替他做完、把结果塞回窗口。

看清这三块短板的病根，你就拿到了整本书的地图：所谓「从零构建一个智能体」，本质就是围着这位失忆天才，一块块补上他天生缺的能力。

## 容易踩的坑

<Callout type="trap">
  **现象**：按「字数」估算能塞多少内容、要花多少钱，结果一上英文或代码就严重超预算
  / 超长报错。**原因**：模型一切都按 **token**
  算，不是按字算；英文一个长词常被切成好几个 token，代码、标点更碎，token
  数远多于你数的字数。**修法**：凡涉及「能记多长 / 调用成本 /
  会不会超窗口」，一律用 token 数衡量；拿不准就用本章的分词交互件先切一遍看看。
</Callout>

<Callout type="trap">
  **现象**：模型一本正经地报出一个不存在的引用、数据或人名，你以为它「撒谎」或「坏了」。**原因**：这是**幻觉**，是机制使然——它生成文字靠「接最顺的下一个
  token」，并不会去核对事实，没依据时照样能接出通顺却不实的内容。**修法**：别指望靠「叮嘱它别乱说」根治；要害是给它**可靠依据**——后面用检索（RAG）让它先查到资料再作答，把「凭印象编」换成「照资料说」。
</Callout>

## 小结

- 模型不按字、按 **token** 处理文字；**token ≠ 字数**，中文常一字一 token、英文常一词切成多个子词
- **上下文窗口**是模型一次能读进的 token 上限（= 门缝能塞进的纸条长度）；系统提示 + 历史 + 当前问题共用它，超出的被挤掉 =「忘」
- 「生成」的本质是 **下一个 token 预测**：看着已有文字给下一个 token 算概率、挑一个接上、再来——反复做这件事，直到挑中「结束」
- 大模型天生三块短板——会**幻觉**（接最顺的 token、不核对事实）、**记不住**（窗口有限）、**动不了手**（只会读写 token）
- 这三块短板正是后面篇章要补的：记忆 / RAG 治「幻觉 + 记不住」，工具调用治「动不了手」

**练习 1（判断题）** 判断下面四句话对错，并简述理由：
（a）一个 token 就是一个字，所以「token 数」永远等于「字数」。
（b）只要把对话历史一直发给模型，它就永远记得住所有内容。
（c）模型生成一段话，是先在心里想好整句、再一次性写出来的。
（d）大模型出现幻觉，是因为它故意撒谎。

<Answer>

（a）**错**。token 是模型切碎文字的最小块，不一定等于一个字——英文一个长词常切成多个子词 token，所以 token 数通常不等于字数，中英文还不一样。
（b）**错**。上下文窗口有限，对话越来越长，最前面的内容会被挤出窗口、模型读不到，表现为「忘了」。
（c）**错**。它是「下一个 token 预测」：一个 token 接一个 token地往后接，每一步只赌「下一个最可能是谁」，直到挑中「结束」才停。
（d）**错**。幻觉是机制使然——它靠「接最顺的下一个 token」生成，并不核对事实，没依据时照样能接出通顺却不实的内容，不是故意撒谎。

</Answer>

**练习 2（问答题）** 用一句话说清「上下文窗口」和「失忆天才小屋」这个比喻的对应关系，并解释为什么聊天聊久了模型会「前言不搭后语」。

<Answer>

上下文窗口对应「门缝能塞进的纸条长度」——模型一次只读得进固定长度的纸条。聊天越久，系统提示 + 越攒越长的对话历史 + 当前问题加起来的 token 越多，一旦超过窗口容量，最前面的对话就被挤出窗口、模型读不到了，所以会忘掉前面说过的话，表现为前言不搭后语。

</Answer>

**练习 3（问答题）** 大模型的三块短板（幻觉、记不住、动不了手）分别对应它「读—记—写」机制里的哪一点？后面分别要靠什么来补？

<Answer>

- **记不住** ← 上下文窗口有限（「记」这一环）：超出窗口的被挤掉。补法：记忆（短期笔记本 + 长期记忆），把该记的存下来、要用时取回塞进窗口。
- **幻觉** ← 下一个 token 预测只管「接最顺的 token」、不核对事实（「写」这一环）：没依据也敢编。补法：检索增强（RAG），先查到可靠资料再作答。
- **动不了手** ← 模型天生只会「读 token、写 token」、出不了屋（「读 / 写」之外做不了别的）。补法：工具调用，给屋子装一部「电话」，让它能调外部工具真去做事。

</Answer>

  把一段文字切成一串 token
  的过程，也叫分词。不同语言切法不同：中文常一个字切成一个
  token，英文常按单词或单词的一段（子词）来切，所以同样长的一句话，token
  数可能差很多。详见本章「第一件事」一节。
</GlossaryItem>

<GlossaryItem term="上下文窗口（context window）">
  模型一次能「读进去」的 token
  数量上限，也叫上下文长度。你塞给它的所有内容——系统提示 + 之前的对话 +
  当前问题——加起来的 token
  数不能超过这个上限；超出的部分会被挤出窗口、模型读不到，表现出来就像「忘了」。本书把它比作「门缝能塞进的纸条长度」。详见本章「第二件事」一节。
</GlossaryItem>

<GlossaryItem term="下一个 token 预测（next-token prediction）">
  模型生成文字的根本机制：看着已经有的全部文字，对「下一个 token
  应该是什么」算出一个概率分布，挑一个（通常挑概率最高的）接到后面，再把刚接上的算进「已有文字」继续预测下一个——如此反复。所谓「生成一段话」，就是把这个小动作做很多很多次，直到挑中「结束」。详见本章「第三件事」一节。
</GlossaryItem>

<GlossaryItem term="幻觉（hallucination）">
  大模型一本正经地编造不实内容的现象。它生成文字靠「接一个最顺的下一个
  token」，而不是「核对事实」，所以在没有可靠依据时，照样能输出通顺但错误的答案（比如编造不存在的引用、数据、人名）。它不是故意撒谎，而是机制使然——后面用检索（RAG）给它可靠依据来缓解。详见本章「第三件事」一节。
</GlossaryItem>

</Glossary>
