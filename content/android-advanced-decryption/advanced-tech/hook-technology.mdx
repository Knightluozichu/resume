## Hook——拦截系统行为的"钩子"

Hook 在英文里是"钩子"的意思——在程序执行的某个环节挂上一个钩子，拦截或修改原本的行为。Android 里 Hook 无处不在：热修复用它替换 bug 方法、插件化用它欺骗 AMS 的检查、无埋点统计用它拦截所有页面的 onCreate/onClick。

## Hook 的两种实现方式

### 静态代理（编译期 Hook）

在编译阶段修改字节码，把代理逻辑编织进目标方法。代表工具：
- **AspectJ**：面向切面编程（AOP）框架——在编译期把切面代码"织入"目标类
- **ASM**：直接操作 .class 字节码——更底层、更灵活

```java
// AspectJ 示例——在所有 Activity.onCreate 前后织入统计代码
@Aspect
public class ActivityAspect {
    @Around("execution(* android.app.Activity.onCreate(..))")
    public void aroundOnCreate(ProceedingJoinPoint joinPoint) {
        long start = System.currentTimeMillis();
        joinPoint.proceed(); // 执行原始 onCreate
        long cost = System.currentTimeMillis() - start;
        // 上报这个 Activity 的 onCreate 耗时
        reportPerformance(joinPoint.getThis().getClass().getName(), cost);
    }
}
```

优点：对源代码零侵入、编译器检查类型安全。缺点：需要重新编译打包、不能动态修改。

### 动态代理（运行时 Hook）

在运行时通过反射替换对象或方法。Android 里最常见的动态代理工具：
- **Java 原生 DynamicProxy**：`Proxy.newProxyInstance()` → 只能代理接口
- **反射 + 替换字段**：拿到目标对象的引用 → 反射获取它的某个字段 → 替换为一个代理对象

经典案例：**Hook AMS 拦截所有 startActivity**：

```java
// 1. 拿到 AMS 的 Binder 代理对象
Object amsProxy = getAMSBinderProxy(); // 反射获取

// 2. 用自己的 DynamicProxy 包裹它
Object hookedAMS = Proxy.newProxyInstance(
    amsProxy.getClass().getClassLoader(),
    new Class[]{IActivityManager.class},
    new InvocationHandler() {
        @Override
        public Object invoke(Object proxy, Method method, Object[] args) {
            if ("startActivity".equals(method.getName())) {
                // 在这里可以拦截/修改/记录 startActivity 的调用
                Log.d("Hook", "Activity starting: " + args[1]);
                // 还可以修改 Intent 参数（比如替换目标页面做 A/B test）
            }
            return method.invoke(amsProxy, args); // 调回真实 AMS
        }
    }
);

// 3. 反射替换掉系统持有的 AMS 引用——让系统用的"AMS"变成我们的代理
reflectiveReplaceAMSProxy(hookedAMS);
```

之后整个 App 的所有 startActivity 调用都会被我们的代理拦截——可以做全局页面跳转统计、页面 A/B test 替换、甚至防页面劫持。

## Hook 的应用场景

| 场景 | Hook 什么 | 目的 |
|---|---|---|
| 热修复 | Hook ClassLoader 的 findClass / dexElements | 替换 bug 类 |
| 插件化 | Hook AMS 的 startActivity / Instrumentation | 绕过 Manifest 校验，启动未注册的 Activity |
| 无埋点统计 | Hook View.onClick / Activity.onCreate | 自动统计所有页面和点击事件 |
| 性能监控 | Hook 主线程 Handler / Looper | 监控卡顿、收集方法耗时 |
| A/B Test | Hook startActivity 的 Intent | 动态替换跳转页面 |

## Hook 与 AOP 的关系

Hook 是**手段**，AOP（面向切面编程）是**目的**。Hook 通过在某个执行点"挂钩子"来实现横切关注点（日志、权限检查、统计）的织入——这就是 AOP 的思想。

## 容易踩的坑

<Callout type="trap">
  **现象**：Hook AMS 后 App 运行一段时间崩溃——`ClassCastException: $Proxy cannot be cast to android.os.Binder`。**原因**：DynamicProxy 只能代理接口——某些系统调用需要的是 Binder 对象而不是接口。**修法**：改用 BinderHook（继承并替换 Binder 的实现类）或使用 cglib/ASM 直接修改字节码创建代理子类。
</Callout>

<Callout type="trap">
  **现象**：Hook 了系统的某个方法，但只在 debug 包生效、release 包无效。**原因**：ProGuard 混淆了被 Hook 的目标类或方法名——反射找不到。**修法**：proguard-rules.pro 里 keep 被 Hook 的目标类和方法。
</Callout>

## 小结

- Hook = 拦截/修改系统行为的技术——在程序执行路径上挂"钩子"
- 静态代理：编译期修改字节码（AspectJ/ASM）——安全但不够灵活
- 动态代理：运行时反射替换（DynamicProxy + 反射）——灵活但需要注意兼容性
- Android 常见 Hook 目标：AMS/ClassLoader/Instrumentation/Handler
- Hook 是实现 AOP 的手段——无侵入地织入横切关注点（统计/监控/修复）

<Exercises>

**练习 1（代码型）** 写出 Hook ActivityThread 的 H Handler 来拦截所有 Activity 生命周期回调的伪代码思路。

<Answer>
① 反射获取 ActivityThread 实例 → ② 拿到它的 mH（Handler）字段 → ③ 用自定义 Callback 包裹 mH 的 handleMessage → ④ Callback 里拦截 LAUNCH_ACTIVITY 等消息，插入自己的逻辑（如统计启动耗时）→ ⑤ 最后调回原始 Handler 的处理逻辑。
</Answer>

**练习 2（问答型）** Hook 技术怎么实现无侵入式埋点？改变了 Activity 的代码吗？

<Answer>
通过 Hook Instrumentation 或 ActivityThread 的 Handler，拦截所有 Activity 的生命周期回调（onCreate/onResume/onDestroy 等），在这些回调前后插桩自动上报页面访问数据。App 的业务代码完全不需要改动——这就是"无侵入"。
</Answer>

</Exercises>

<Glossary>

<GlossaryItem term="Hook">
  拦截系统行为的技术——在程序执行的某个环节挂"钩子"，修改或增强原生行为。Android 中用反射 + DynamicProxy 实现运行时 Hook。
</GlossaryItem>

<GlossaryItem term="AOP（面向切面编程）">
  把横切关注点（日志、统计、权限）从业务代码中分离出来独立管理。Hook 是实现 AOP 的一种手段。
</GlossaryItem>

<GlossaryItem term="DynamicProxy">
  Java 标准库提供的动态代理——运行时创建实现了指定接口的代理对象，所有方法调用经过 InvocationHandler 转发。只能代理接口，不能代理类。
</GlossaryItem>
