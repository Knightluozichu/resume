## 为什么 Android 开发者要学 JVM？

Android 上层用 Java/Kotlin 写，所有代码最终跑在**虚拟机**上。Android 使用的虚拟机从最初的 Dalvik 到现在的 ART，它们在设计上参考了标准 JVM 的很多理念——内存分区、垃圾回收、类加载。理解了 JVM，才能对比理解 ART 为什么更快、更省内存。

## JVM 运行时内存分区

JVM 把运行时的内存分为五块：

| 区域 | 存放内容 | 线程共享？ | 会 OOM 吗？ |
|---|---|---|---|
| **堆** | 所有 new 出来的对象和数组 | 是 | **会**——最常见 |
| **方法区** | 类信息、常量、静态变量 | 是 | 会 |
| **虚拟机栈** | 方法调用的局部变量表、操作数栈 | 否（每线程一个） | 会（StackOverflow） |
| **本地方法栈** | native 方法调用 | 否 | 会 |
| **程序计数器** | 当前线程执行的字节码行号 | 否 | 不会 |

### 堆——GC 的主战场

堆是 JVM 管理的最大一块内存，所有 `new` 出来的对象都在这里。现代 JVM 的堆采用**分代回收**设计——根据对象"寿命"把堆分为两个大区：

- **新生代（Young Generation）**：刚出生的对象放在这。分成 1 个 Eden 区 + 2 个 Survivor 区（S0/S1）。对象先在 Eden 出生 → 经过一次 Minor GC 还活着的搬到 Survivor → 在 Survivor 之间来回搬多次后 → 搬到老年代
- **老年代（Old Generation）**：活得够久的对象晋升到这里。Major GC（Full GC）清理老年代，但速度慢、STW（Stop-The-World）时间长

### GC 算法

| 算法 | 原理 | 适用 |
|---|---|---|
| **标记-清除** | 标记活对象 → 清除死对象 | 简单但产生内存碎片 |
| **复制** | 把活对象复制到另一块区域，整块清理原区域 | 新生代（对象少、死得多） |
| **标记-整理** | 标记活对象 → 移到一端 → 清理边界外 | 老年代（碎片整理） |

新生代用**复制**算法（快——因为大部分对象活不久），老年代用**标记-清除/整理**算法。

### 怎么判断对象"死了"？

JVM 使用**可达性分析**：从一组"GC Roots"（栈上引用、静态变量、JNI 引用等）出发，沿着引用链能到达的对象 = 活的，到不了的 = 死的。这就是为什么单例持有 Activity 会导致泄漏——单例作为 GC Root 一直引用着 Activity，GC 认为它"还是活的"。

## 标准 JVM vs Android 虚拟机

| | HotSpot JVM（桌面/服务器） | Dalvik（Android 4.4-） | ART（Android 5.0+） |
|---|---|---|---|
| 字节码格式 | .class（每个类一个文件） | .dex（多个类合并） | .dex（同上） |
| 执行方式 | 解释 + JIT 编译 | JIT 编译 | AOT + JIT 混合 |
| 编译器 | C2/JIT | JIT（每次运行编译） | AOT（安装时预编译） |
| 内存占用 | 较大 | 小（设备内存限制） | 中等 |
| 启动速度 | 中等 | 慢（JIT 编译耗时） | 快（已预编译） |
| 安装速度 | - | 快 | 慢（预编译耗时） |

Android 为什么不用 HotSpot？因为移动设备的特点完全不同——内存小、CPU 弱、注重省电。Dalvik 和 ART 为移动设备量身打造：.dex 格式比 .class 更紧凑（多个类合并去重），驻留内存的堆更小，GC 更频繁但每次停顿更短。

## 小结

- JVM 内存五区：堆（对象）、方法区（类信息）、虚拟机栈（方法调用）、本地方法栈、程序计数器
- 堆分新生代（Eden+2 Survivor）和老年代——分代回收
- 新生代用复制算法（快），老年代用标记-清除/整理
- GC 通过可达性分析判断对象死活——GC Roots 出发，不可达 = 可回收
- Android 不用 HotSpot 而自研 ART：.dex 更紧凑、AOT 预编译启动快、内存占用更低

**练习 2（问答型）** 新生代为什么用复制算法而不是标记-清除？

<Answer>
新生代的对象大部分"朝生夕死"——一次 Minor GC 后存活对象很少。复制算法只需要复制少量活对象，然后整体清空原区域——非常快且无碎片。标记-清除要遍历整个区域标记 + 清除，在活对象少的场景不如复制高效。
</Answer>

<Glossary>

<GlossaryItem term="GC Roots">
  垃圾回收的根集合——栈上引用、静态变量、JNI 全局引用。从 GC Root 沿着引用链能到的对象 = 活的，到不了的 = 可回收。单例持有 Activity 引用导致泄漏的原理：单例是 GC Root → 引用了 Activity → GC 认为 Activity 还活着。
</GlossaryItem>

<GlossaryItem term="Minor GC vs Major GC">
  Minor GC：只清理新生代——快、频繁。Major GC（Full GC）：清理整个堆包括老年代——慢、STW 时间长。
</GlossaryItem>

<GlossaryItem term=".dex 格式">
  Dalvik Executable——Android 的字节码格式。多个 .class 文件合并成一个 .dex，共享常量池去重，比 .class 更紧凑更适合移动设备。
</GlossaryItem>
