## 为什么需要理解"任务"？

你用过手机 App 的场景：从桌面点开微信 → 进聊天列表 → 点一个联系人 → 进去聊天。现在按返回键，你会逐层退出：回到聊天列表 → 再回到桌面。这个"一层层进去、一层层退出"的东西，就是 Android 的**任务返回栈**。

用**工厂流水线**类比：每个 App 就像一个独立车间，车间里有一叠待处理的工单（Activity 栈）。最上面的工单就是你现在看到的页面。来一个新工单（打开新页面），就压到最上面。处理完（按返回），就把最上面那张抽走，露出下一张。如果没有这套"工单叠放"机制，页面跳转会乱套——你不知道按返回键会回到哪里，页面可能在后台无限堆积占满内存。

这章要解决的核心问题：**当多个 App、多个页面交织跳转时，Android 怎么决定哪个页面进哪个栈？你作为开发者怎么精确控制这个行为？**

## Task 和返回栈是怎么工作的

### Task 是什么

<Term def="Task 是 Android 用来组织一组相关 Activity 的容器，每个 Task 维护一个独立的返回栈（back stack）。用户感知到的就是一个「应用」——但实际上一个 App 可能跨越多个 Task，一个 Task 也可能包含来自不同 App 的 Activity。">Task（任务）</Term> 不是一个类或对象——它是一种抽象概念，是 Android 用来把一组 Activity 串成一条「用户操作路径」的容器。

每个 Task 内部维护一个**返回栈（Back Stack）**：后进先出（LIFO）。新 Activity 启动时被推到栈顶；用户按返回键时，栈顶 Activity 被弹出销毁，前一个 Activity 重新显示。

```
Task A 的返回栈：
┌────────────────────┐
│ ActivityC  (栈顶)   │ ← 用户当前看到的
├────────────────────┤
│ ActivityB           │
├────────────────────┤
│ ActivityA  (栈底)   │ ← Task 的根 Activity
└────────────────────┘
```

<Term def="启动模式（launch mode）是一种在 AndroidManifest 中配置的属性，控制 Activity 实例如何与 Task 和返回栈交互——是每次都新建、还是复用已有实例、还是独占一个 Task。">启动模式（Launch Mode）</Term> 和 <Term def="Intent 标志是一组在代码中动态设置的 flag（如 FLAG_ACTIVITY_NEW_TASK），可以更精细地控制 Activity 的启动行为，效果比 manifest 中声明的 launchMode 更灵活。">Intent 标志（Flag）</Term> 就是你控制这个"工单叠放"规则的两把钥匙——一个在清单文件中声明（静态），一个在代码中设置（动态）。

### 四种启动模式

Android 提供了四种启动模式，在 `AndroidManifest.xml` 中通过 `<activity>` 标签的 `android:launchMode` 属性指定：

| 启动模式 | 行为 | 典型场景 |
|---|---|---|
| `standard`（默认） | 每次启动都创建新实例，压入启动它的 Task 的栈顶 | 绝大多数普通页面 |
| `singleTop` | 如果要启动的 Activity 已在栈顶，则不创建新实例，直接复用（回调 `onNewIntent`）；否则行为同 standard | 搜索页面（避免重复打开） |
| `singleTask` | 系统中只保留一个实例。启动时清空它上面的所有 Activity，让它露出来成为栈顶。若不存在则创建 | App 主页面（MainActivity） |
| `singleInstance` | 跟 singleTask 类似，但更极端——这个 Activity 独占一个 Task，该 Task 中只有它一个 | 来电界面、闹钟响铃界面 |

## 动手看：四种启动模式的行为区别

> 猜一猜：从 MainActivity(singleTask) 启动 DetailActivity(standard)，再从 DetailActivity 启动 MainActivity——返回栈里会有几个 MainActivity 实例？切到第 3 步看答案。

<Stepper>
  <Step title="① standard：每次都新建">
    假设当前栈是 `[Main → Detail]`。从 Detail 再次启动 Detail（standard）：系统创建一个新的 Detail 实例，栈变成 `[Main → Detail → Detail]`。按两次返回才能回到 Main。**这是最常见也最容易导致栈无限膨胀的模式。**
  </Step>
  <Step title="② singleTop：栈顶复用">
    栈为 `[Main → Search]`。从 Search 再次启动 Search（singleTop）：系统不创建新实例，而是调用 Search 的 `onNewIntent()`。栈保持 `[Main → Search]` 不变。**注意：如果 Search 不在栈顶（比如栈为 `[Main → Search → Detail]`）启动 Search，singleTop 跟 standard 一样——会创建新实例。**
  </Step>
  <Step title="③ singleTask：清栈复用">
    栈为 `[Main(singleTask) → Detail → Edit]`。从 Edit 启动 Main：系统把 Detail 和 Edit 全部出栈销毁，Main 回到栈顶。栈变成 `[Main]`。Main 的 `onNewIntent()` 被调用。**singleTask 是"整个 Task 里我只存在一个，且我永远在栈底附近"的保证。**
  </Step>
  <Step title="④ singleInstance：独占 Task">
    假设 App A 的栈是 `[A1 → A2]`。A2 启动来电界面 CallActivity（singleInstance）。系统创建一个全新的 Task B，CallActivity 独占其中。最近任务列表会出现两张"卡片"。按返回键：从 CallActivity 返回 → 回到 App A 的 A2（Task 切换，不是栈内返回）。
  </Step>
</Stepper>

## 代码逐段拆解：控制 Activity 的入栈

### 在 Manifest 中声明启动模式

```xml
<!-- AndroidManifest.xml -->
<activity
    android:name=".MainActivity"
    android:launchMode="singleTask">
    <intent-filter>
        <action android:name="android.intent.action.MAIN" />
        <category android:name="android.intent.category.LAUNCHER" />
    </intent-filter>
</activity>
```

这段代码把 MainActivity 设为 `singleTask`——意味着无论在 App 里的哪个页面，只要再次启动 MainActivity，它上面的所有 Activity 都会被清理掉。MainActivity 永远是这个 Task 的"根"或"唯一入口"。

### 用 Intent Flag 动态控制

有时你不想在 Manifest 里写死启动模式，而是在启动那一瞬间根据上下文决定行为。这时用 Intent flag：

```kotlin
// 从任何 Activity 中启动一个"新 Task"里的 Activity
val intent = Intent(this, TargetActivity::class.java).apply {
    flags = Intent.FLAG_ACTIVITY_NEW_TASK or Intent.FLAG_ACTIVITY_CLEAR_TOP
}
startActivity(intent)
```

`FLAG_ACTIVITY_NEW_TASK` 告诉系统：把 TargetActivity 放到一个新的 Task 里。`FLAG_ACTIVITY_CLEAR_TOP` 告诉系统：如果目标 Activity 已经存在于当前 Task 的栈中，把它上面的所有 Activity 清掉，然后复用这个实例。这两个 flag 组合起来的效果类似于 `singleTask`，但更灵活。

### taskAffinity：指定 Activity 在哪个 Task 里

```xml
<activity
    android:name=".SpecialActivity"
    android:taskAffinity="com.example.special"
    android:launchMode="singleTask" />
```

<Term def="taskAffinity 是 Activity 的一个属性，表示它「倾向于」属于哪个 Task。默认值等于应用的包名。当配合 FLAG_ACTIVITY_NEW_TASK 或 singleTask 使用时，taskAffinity 决定了 Activity 被放入哪个 Task——相同 affinity 的在一起，不同的则分到不同 Task。">`taskAffinity`（任务亲和性）</Term> 决定了 Activity "愿意"进入哪个 Task。默认情况下所有 Activity 的 taskAffinity 等于应用的包名，所以它们都在同一个 Task 里。

但如果你想让某个 Activity 进入一个独立的 Task（比如在最近任务列表里显示为一张单独的卡片），就用 `singleTask` + 独特的 `taskAffinity`。

## 容易踩的坑

<Callout type="trap">
  **现象**：把 MainActivity 设为 `singleTask`，发现从通知栏点通知进入 App 时，之前的页面全没了。
  **原因**：`singleTask` 的"清栈"行为不仅在你从 App 内部启动时生效，从外部（通知、deep link）启动同样会清空它上面的所有 Activity。用户正在填的表单、正在编辑的内容全部丢失。
  **修法**：如果目的是"不希望 MainActivity 有多个实例"而不是"每次都清空栈"，应该用 `singleTop` + `FLAG_ACTIVITY_SINGLE_TOP` 组合。或者在 `onNewIntent()` 中判断 intent 来源，决定是否真的需要清栈。
</Callout>

<Callout type="trap">
  **现象**：设置了 `FLAG_ACTIVITY_NEW_TASK` 但 Activity 还是进了当前 Task。
  **原因**：`FLAG_ACTIVITY_NEW_TASK` 单独使用时，只有当目标 Activity 的 `taskAffinity` 与当前 Task 不同时，才会真正创建新 Task。如果 taskAffinity 相同（默认都是包名），系统会把它放进当前 Task——等于没设。
  **修法**：真正需要新 Task 时，组合使用 `FLAG_ACTIVITY_NEW_TASK` + 为目标 Activity 设置不同的 `taskAffinity`。单独用 NEW_TASK 几乎没效果。
</Callout>

<Callout type="trap">
  **现象**：在 Service 或 BroadcastReceiver 等非 Activity 上下文中调用 `startActivity()` 时崩溃，错误为 `Calling startActivity() from outside of an Activity context`。
  **原因**：Android 要求从非 Activity 上下文启动 Activity 时，必须加 `FLAG_ACTIVITY_NEW_TASK`。因为 Service 没有自己的 Task，系统不知道该把新 Activity 放进哪个栈。
  **修法**：在 Service/BroadcastReceiver 中启动 Activity 时，永远加上 `intent.addFlags(Intent.FLAG_ACTIVITY_NEW_TASK)`。
</Callout>

## 小结

- Task 是组织一组相关 Activity 的抽象容器，内部用返回栈（LIFO）管理页面跳转顺序
- 四种启动模式控制 Activity 实例的复用策略：`standard` 每次新建、`singleTop` 栈顶复用、`singleTask` 全栈唯一、`singleInstance` 独占 Task
- Intent flag 可动态覆盖启动模式：`NEW_TASK` 进入新 Task、`CLEAR_TOP` 清栈复用、`SINGLE_TOP` 等价于 singleTop
- `taskAffinity` 配合 `singleTask` 或 `NEW_TASK` 使用，决定 Activity 归属哪个 Task
- 从 Service/BroadcastReceiver 启动 Activity 必须加 `FLAG_ACTIVITY_NEW_TASK`

<Exercises>

**练习 1（改代码型）** 新建一个 Android 项目，创建三个 Activity（A、B、C）。把 A 设为 `singleTask`，B 和 C 保持 `standard`。从 A → B → C → A，用 `adb shell dumpsys activity activities` 观察栈的变化，验证 B 和 C 是否被自动销毁。

<Answer>
从 C 启动 A 时，因为 A 是 singleTask 且已存在于栈底，B 和 C 会被弹出并销毁。栈从 `[A → B → C]` 变成 `[A]`，A 的 `onNewIntent()` 被调用。你可以在 B 和 C 的 `onDestroy()` 中打 log 确认它们确实被销毁了。
</Answer>

**练习 2（改代码型）** 把练习 1 中的 A 改成 `singleTop`，重复 A → B → C → A 流程。这次 B 和 C 会被销毁吗？栈变成什么样？

<Answer>
`singleTop` 只在目标 Activity 已在栈顶时才触发复用。从 C 启动 A，A 不在栈顶，所以行为跟 standard 一样——创建一个新的 A 实例压入栈顶。栈变成 `[A → B → C → A']`（A' 是新实例）。连按四次返回才能退出 App。**关键区别：singleTask 管"全局唯一"，singleTop 只管"栈顶复用"。**
</Answer>

**练习 3（探索型）** 在最近任务列表里长按一张卡片，你会看到什么信息？尝试用 `adb shell am stack list` 命令，看看你能获取到哪些关于任务栈的信息。

<Answer>
长按最近任务卡片可以进入应用信息页或分屏模式。`adb shell am stack list`（Android 7.0+ 的话可能用 `adb shell dumpsys activity activities | grep "Stack #"` 代替）可以列出当前系统的所有 Activity Stack，包括每个 Stack 的 ID、包含哪些 Task、每个 Task 里有哪些 Activity。这是排查任务栈问题的利器。
</Answer>

</Exercises>

## 名词解释

<Glossary>
<GlossaryItem term="Task（任务）">
  把一组相关的 Activity 串成一条「用户操作路径」的容器。你可以把它理解成一叠工单——最上面的工单是你当前看到的页面，按返回键就把最上面那张抽走。它是 Android 对"应用"这一用户概念的系统级抽象。详见本章"Task 是什么"一节。
</GlossaryItem>
<GlossaryItem term="启动模式（Launch Mode）">
  在 AndroidManifest 中声明的一种属性，控制 Activity 实例跟 Task、返回栈怎么互动。有四种：standard（每次都新建）、singleTop（栈顶复用）、singleTask（全栈唯一）、singleInstance（独占一个 Task）。详见本章"四种启动模式"一节。
</GlossaryItem>
<GlossaryItem term="Intent 标志（Flag）">
  在代码中通过 `intent.setFlags()` 设置的属性，可以动态控制 Activity 的启动行为。常见的如 FLAG_ACTIVITY_NEW_TASK（放进新 Task）、FLAG_ACTIVITY_CLEAR_TOP（清栈复用）。代码中设置的 flag 优先级高于 manifest 中声明的 launchMode。详见本章"Intent Flag"一节。
</GlossaryItem>
<GlossaryItem term="taskAffinity（任务亲和性）">
  Activity 的一种"归属感"属性，表示它倾向于进入哪个 Task。默认等于应用的包名。配合 singleTask 或 FLAG_ACTIVITY_NEW_TASK 使用时，决定 Activity 被分到哪个 Task。详见本章"taskAffinity"一节。
</GlossaryItem>
</Glossary>
