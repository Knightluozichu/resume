## 为什么需要"可靠的后台任务"？

你的 App 需要在后台做一件事：把用户编辑的笔记同步到服务器。如果用 `Thread { ... }` 来做，用户刚点完保存就关了 App、切了网络——线程直接被杀，同步中断，笔记丢失。你怎么办？自己写一套"重试机制"、"任务持久化存储"、"网络状态监听"？

Android 生态历史上出现过一堆解决方案——JobScheduler、AlarmManager、Firebase JobDispatcher、甚至老掉牙的 AsyncTask——它们各有各的坑：有的 API 版本限制、有的不保证执行、有的只对 Google Play Services 用户有效。

<Term def="WorkManager 是 Android Jetpack 中的后台任务调度库，专门解决"保证最终一定执行"这个硬需求。它能处理进程被杀后自动重试、根据网络/充电等条件决定执行时机、以及串联多个任务按顺序执行。">WorkManager</Term> 就是为了终结这场混乱而生的——它把"可靠地安排后台任务该什么时候做"这件事统一封装。你的任务不丢、不重复、在条件满足时按时执行——即使 App 被杀死、设备重启后也能恢复。

## WorkManager 的工作机制

### 核心三角：Worker + WorkRequest + WorkManager

```kotlin
// 1. Worker：定义"做什么"
class SyncWorker(context: Context, params: WorkerParameters)
    : Worker(context, params) {
    override fun doWork(): Result {
        return try {
            syncNotesToServer()          // 执行同步逻辑
            Result.success()             // 成功 → 任务结束
        } catch (e: Exception) {
            Result.retry()               // 失败 → WorkManager 按退避策略自动重试
        }
    }
}

// 2. WorkRequest：定义"什么时候做"
val constraints = Constraints.Builder()
    .setRequiredNetworkType(NetworkType.CONNECTED)   // 需要网络
    .setRequiresCharging(true)                        // 需要充电
    .build()

val syncRequest = OneTimeWorkRequestBuilder<SyncWorker>()
    .setConstraints(constraints)
    .setInitialDelay(10, TimeUnit.MINUTES)           // 10 分钟后开始
    .build()

// 3. WorkManager：执行调度
WorkManager.getInstance(context).enqueue(syncRequest)
```

三者关系：`Worker` 定义任务逻辑（doWork），`WorkRequest` 定义任务的条件和时机（约束 + 延迟 + 重试），`WorkManager` 做统一调度（入队 + 选择执行时机 + 通知结果）。

<Term def="ONE_TIME 任务只执行一次；PERIODIC 任务按固定时间间隔重复执行——两者本质是 WorkRequest 的不同子类。">WorkRequest 有两种</Term>：`OneTimeWorkRequest`（执行一次）和 `PeriodicWorkRequest`（周期性重复，最小间隔 15 分钟）。

### 观察任务执行状态

WorkManager 提供 LiveData 接口让你观察任务进展：

```kotlin
WorkManager.getInstance(context)
    .getWorkInfoByIdLiveData(syncRequest.id)
    .observe(lifecycleOwner) { workInfo ->
        when (workInfo?.state) {
            WorkInfo.State.ENQUEUED -> showStatus("排队等待中...")
            WorkInfo.State.RUNNING -> showStatus("正在同步...")
            WorkInfo.State.SUCCEEDED -> showStatus("同步成功！")
            WorkInfo.State.FAILED -> showStatus("同步失败")
            WorkInfo.State.BLOCKED -> showStatus("等待前置任务完成...")
            else -> {}
        }
    }
```

## 动手看：一条 WorkRequest 的完整生命周期

> 猜一猜：如果用户在 SyncWorker 的 `doWork()` 执行期间杀掉了 App，同步会中断吗？

<Stepper>
  <Step title="① 入队（ENQUEUED）">
    `WorkManager.getInstance(context).enqueue(syncRequest)` 把任务注册到内部数据库。**这一步后任务已经持久化**——即使 App 进程被杀、设备重启，WorkManager 在下次启动时会从数据库中恢复任务。
  </Step>
  <Step title="② 等待约束（ENQUEUED）">
    任务被持久化后在后台等待。WorkManager 持续监测约束条件：`NetworkType.CONNECTED` 不满足 → 等待；`RequiresCharging` 不满足 → 继续等待。所有约束满足后才进入执行。
  </Step>
  <Step title="③ 开始执行（RUNNING）">
    WorkManager 内部使用一个后台线程池来执行 `Worker.doWork()`。`doWork()` 在后台线程运行——你不需要自己开线程。**关键：`doWork()` 的返回值必须在 10 分钟内返回，否则 Worker 被认为超时并重试。**
  </Step>
  <Step title="④ 成功（SUCCEEDED）或失败重试（FAILED → ENQUEUED）">
    `Result.success()` → 任务标记为 SUCCEEDED，从数据库中清理（或保留记录用于调试）。`Result.retry()` → 任务重新进入 ENQUEUED 状态，WorkManager 按退避算法（初始 30 秒，指数增长，最多 5 次）重新尝试执行。
  </Step>
  <Step title="⑤ 猜一猜的答案">
    用户在 Worker 执行期间杀进程：任务状态仍保存在 WorkManager 的数据库中。下次 App 启动时，WorkManager 会自动恢复未完成的 ENQUEUED/RUNNING 任务并重新执行——**这就是 WorkManager "可靠"的真正含义。**
  </Step>
</Stepper>

## 代码逐段拆解：任务链与数据传递

### 场景：先压缩图片，再上传到服务器

```kotlin
// 第 1 个 Worker：压缩图片
class CompressWorker(context: Context, params: WorkerParameters)
    : Worker(context, params) {
    override fun doWork(): Result {
        val imageUri = inputData.getString("IMAGE_URI") ?: return Result.failure()
        val compressedUri = compressImage(imageUri)

        // 输出压缩后的图片路径供下一个 Worker 使用
        val output = workDataOf("COMPRESSED_URI" to compressedUri)
        return Result.success(output)
    }
}

// 第 2 个 Worker：上传
class UploadWorker(context: Context, params: WorkerParameters)
    : Worker(context, params) {
    override fun doWork(): Result {
        val uri = inputData.getString("COMPRESSED_URI") ?: return Result.failure()
        return try {
            uploadToServer(uri)
            Result.success()
        } catch (e: Exception) {
            Result.retry()
        }
    }
}

// 串联执行：先压缩，成功后再上传
val inputData = workDataOf("IMAGE_URI" to photoUri.toString())

val compressRequest = OneTimeWorkRequestBuilder<CompressWorker>()
    .setInputData(inputData)
    .build()

val uploadRequest = OneTimeWorkRequestBuilder<UploadWorker>()
    .setConstraints(
        Constraints.Builder()
            .setRequiredNetworkType(NetworkType.CONNECTED)
            .build()
    )
    .build()

WorkManager.getInstance(context)
    .beginWith(compressRequest)     // 第一步：压缩
    .then(uploadRequest)            // 第二步：等压缩成功后上传
    .enqueue()
```

### 唯一任务：防止重复入队

```kotlin
WorkManager.getInstance(context)
    .enqueueUniqueWork(
        "daily-sync",                          // 唯一任务名
        ExistingWorkPolicy.REPLACE,            // 如果已存在同名任务，替换它
        syncRequest
    )
```

`ExistingWorkPolicy` 有三种策略：
- `REPLACE`：取消已有任务，用新的替代
- `KEEP`：如果已有任务在排队或执行，新的被丢弃
- `APPEND`：把新任务加在已有任务后面（形成隐式链）

## 容易踩的坑

<Callout type="trap">
  **现象**：`PeriodicWorkRequest` 设了 5 分钟的间隔，测试时发现根本不按这个时间执行。
  **原因**：`PeriodicWorkRequest` 的**最小间隔是 15 分钟**——这是 WorkManager 为防止过度耗电设置的下限。你设置 5 分钟，系统自动提升到 15 分钟。此外，`PeriodicWorkRequest` 不保证精确时间——WorkManager 在间隔窗口内选择一个合适时机执行，而不是闹钟式的精确触发。
  **修法**：需要精确时间触发 = 这不是 WorkManager 的场景，用 `AlarmManager`。WorkManager 适合不严格要求精确时间的定期任务（如每 24 小时同步一次数据）。
</Callout>

<Callout type="trap">
  **现象**：Worker 的 `doWork()` 里执行了超过 10 分钟的任务，任务被标记为超时根本不重试。
  **原因**：WorkManager 2.3+ 要求 `doWork()` 在 10 分钟内返回。超时的 Worker 被视为失败并停止——不会像正常失败那样重试。
  **修法**：长时间任务用 `ListenableWorker`（Worker 的父类，支持异步），或用 `CoroutineWorker` + `setForeground()` 方法把任务提升为前台服务——后台限制更严格但支持长时间执行。超过 10 分钟的任务通常不是 WorkManager 的最佳场景。
</Callout>

<Callout type="trap">
  **现象**：用 WorkManager 做"立即执行"的任务，发现有时执行了几秒后才开始。
  **原因**：WorkManager 的设计目标是"可靠的延迟执行"，不是"立即后台执行"。任务的入队、约束检查、线程调度都有开销。它的优势是持久化重试和跨进程存活——而不是响应速度。
  **修法**：区分场景用工具：①需要立即后台执行 → `Coroutines` + `Dispatchers.IO`；②需要"保证执行，即使进程被杀" → WorkManager；③需要"长时间后台运行，用户可见" → `ForegroundService`。WorkManager 不替代 Thread/Coroutines，它解决的是"可靠性调度"问题。
</Callout>

## 小结

- WorkManager 解决"可靠调度"：任务持久化在数据库中，进程被杀或设备重启后自动恢复
- 核心三角：`Worker`（定义做什么）→ `WorkRequest`（定义时机和条件）→ `WorkManager`（统一调度）
- `OneTimeWorkRequest` 一次执行，`PeriodicWorkRequest` 周期执行（最小间隔 15 分钟）
- `beginWith().then()` 构建任务链，`setInputData()` + `workDataOf()` 传递数据
- `enqueueUniqueWork()` 防止重复任务，`LiveData` + `getWorkInfoByIdLiveData()` 观察状态

<Exercises>

**练习 1（改代码型）** 创建一个 `NotificationWorker`，它的 `doWork()` 发送一条本地通知。用 `OneTimeWorkRequest` 设置 30 秒延迟后执行。要求设置约束：仅在设备充电时执行。
<Answer>
核心代码：`Constraints.Builder().setRequiresCharging(true).build()` + `OneTimeWorkRequestBuilder<NotificationWorker>().setConstraints(constraints).setInitialDelay(30, TimeUnit.SECONDS).build()`。在 `doWork()` 中用 `NotificationManager.notify()` 发送通知。别忘了 Android 8.0+ 需要通知渠道 `NotificationChannel`——在 `Application.onCreate()` 或 Worker 中创建一次即可。
</Answer>

**练习 2（改代码型）** 构建一条三步骤任务链：① CleanupWorker（清理旧缓存文件）→ ② DownloadWorker（下载新数据，需要网络）→ ③ ParseWorker（解析数据写入数据库）。要求第 2 步的输出数据传给第 3 步。
<Answer>
关键：`DownloadWorker` 的 `doWork()` 返回 `Result.success(workDataOf("RAW_DATA" to downloadedJson))`，`ParseWorker` 的 `doWork()` 中 `inputData.getString("RAW_DATA")` 获取。使用 `beginWith(cleanup).then(download).then(parse).enqueue()` 串联。如果 CleanupWorker 返回 `Result.failure()`，后续的 download 和 parse 都不会执行。
</Answer>

**练习 3（问答型）** 面试官问："我的 App 要把用户相册中选中的 10 张照片上传到服务器，用 Thread？WorkManager？还是 Service？" 你怎么建议？为什么？
<Answer>
建议 WorkManager。原因：①上传 10 张照片可能耗几分钟，用户可能在等待期间退出 App——普通 Thread 会被杀。②需要网络，WorkManager 可以设 `NetworkType.CONNECTED` 约束。③上传可能因网络中断失败——WorkManager 的自动重试机制比手写重试可靠得多。④如果要求"必须最终全部上传成功"，这正是 WorkManager 的"保证执行"场景。⑤如果用 Thread，你需要自己实现重试、持久化、网络监听、排队——你其实在重新发明 WorkManager。
</Answer>

</Exercises>

## 名词解释

<Glossary>
<GlossaryItem term="WorkManager">
  Android Jetpack 中专门做"可靠后台任务调度"的库。任务存储在内部数据库中，进程被杀或重启后自动恢复。支持约束条件（网络、充电、空闲状态）、任务链、唯一任务、状态观察。是 JobScheduler / AlarmManager / AsyncTask 等老方案的系统性替代。详见本章"WorkManager 的工作机制"一节。
</GlossaryItem>
<GlossaryItem term="Worker">
  定义后台任务逻辑的抽象类。你需要继承 Worker 并实现 `doWork()` 方法，返回 `Result.success()`、`Result.failure()` 或 `Result.retry()`。`doWork()` 默认在后台线程执行，最长 10 分钟。详见本章"核心三角"一节。
</GlossaryItem>
<GlossaryItem term="OneTimeWorkRequest">
  只执行一次的后台任务请求。可以设置延迟（`setInitialDelay`）、约束条件（`setConstraints`）、输入数据（`setInputData`）。队列入队（enqueue）后由 WorkManager 统一调度执行时机。详见本章"核心三角"一节。
</GlossaryItem>
<GlossaryItem term="PeriodicWorkRequest">
  按固定时间间隔周期性重复执行的后台任务请求。**最小间隔 15 分钟**——这是 WorkManager 为防止过度耗电设置的系统下限。不保证精确执行时间，只在间隔窗口内选择一个合适时机执行。详见本章"容易踩的坑"一节。
</GlossaryItem>
</Glossary>
