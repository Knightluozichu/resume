## 你的 App 要跑在十年跨度上的设备

2015 年的手机和 2025 年的折叠屏，系统 API 差很多。<Term def="Gradle 中声明的最低可安装 Android 版本，低于此版本的设备无法从商店安装。">minSdk</Term> 决定「多老的机子能装」；<Term def="编译时使用的 SDK 版本，决定你能调用哪些 API 符号。">compileSdk</Term> 决定「你能写哪些新 API」；<Term def="向系统声明已适配的行为变更版本，影响权限、后台限制等策略。">targetSdk</Term> 告诉系统「请按哪一代规则管我」。

## 三个版本号

在 `build.gradle.kts`：

```kotlin
android {
    compileSdk = 35
    defaultConfig {
        minSdk = 26
        targetSdk = 35
    }
}
```

| 版本 | 含义 | 设低了会怎样 |
|---|---|---|
| compileSdk | 编译用 SDK | 无法用新 API / 编译警告 |
| minSdk | 最低安装版本 | 老设备装不了 |
| targetSdk | 行为兼容目标 | 系统用旧兼容模式，可能缺新安全特性 |

{/* 三版本钉在同一条 API Level 数轴上：min 装得上、target 按谁规则跑、compile 能写到多新 */}

## 运行时版本检查

```kotlin
if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.TIRAMISU) {
    registerForActivityResult(ActivityResultContracts.RequestPermission()) { granted ->
        if (granted) pickImage()
    }.launch(Manifest.permission.READ_MEDIA_IMAGES)
} else {
    // API 32 及以下用 READ_EXTERNAL_STORAGE
}
```

<Stepper>
  <Step title="① 查文档确认 API 级别">
    查阅 Android 官方文档，确认目标权限（如 `READ_MEDIA_IMAGES`）从哪个 API 级别引入（API 33 / TIRAMISU）。这决定了运行时分支的最低 SDK_INT 阈值。
  </Step>
  <Step title="② 根据 minSdk 写分支">
    如果你的 minSdk 低于权限引入的 API 级别（如 minSdk=26），必须用 `if (Build.VERSION.SDK_INT >= ...)` 做运行时分支，或使用 AndroidX 兼容 API。

    ```kotlin
    if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.TIRAMISU) {
        // API 33+：细粒度媒体权限
        requestPermissionLauncher.launch(Manifest.permission.READ_MEDIA_IMAGES)
    } else {
        // API 32 及以下：旧版存储权限
        requestPermissionLauncher.launch(Manifest.permission.READ_EXTERNAL_STORAGE)
    }
    ```
  </Step>
  <Step title="③ targetSdk 升至新权限要求">
    将 `targetSdk` 升级到至少等于新权限对应的 API 级别，并在 `AndroidManifest.xml` 中声明新权限、在代码中加入运行时申请逻辑。targetSdk 不够高时系统不会按新规则执行权限检查。
  </Step>
  <Step title="④ 在 minSdk 设备上回归">
    用 minSdk 版本的真机或模拟器跑完整流程——确保低版本走 else 分支时 UI 正常、权限弹窗显示、功能完整。
  </Step>
</Stepper>

<Callout type="tip">
  **AndroidX** 把新 API 行为 backport 到旧系统（如 `ContextCompat`、`ActivityCompat`）——优先用兼容 API，少写 if/else。
</Callout>

<Callout type="trap">
  targetSdk 长期不升 → Google Play 要求过期、用户设备上行为与新版系统不一致（后台限制、分区存储等突然「像坏了」）。
</Callout>

<Callout type="trap">
  compileSdk &lt; 使用的库要求 → 编译失败。升级 compileSdk 不等于必须升 minSdk，但需测 minSdk 设备。
</Callout>

## 小结

- compileSdk ≥ 你用的 API；minSdk = 业务决定的最低设备；targetSdk 尽量跟最新稳定版
- 新 API 用 SDK_INT 分支或 AndroidX
- 升 targetSdk 前读「行为变更」文档并全量测试

<Exercises>

**练习 1（概念）** minSdk=24、targetSdk=35 的手机运行 API 28，系统按哪套规则限制后台？

<Answer>
按 targetSdk 35 声明的行为（若已适配），但只能调用 ≤ API 28 存在的平台 API；minSdk 只影响能否安装，不单独决定运行时策略。
</Answer>

**练习 2（实操）** 要用 NotificationChannel（API 26+），minSdk 21 怎么办？

<Answer>
在 SDK_INT >= 26 时创建 Channel；低版本用旧通知 API 或 AndroidX 封装。在 API 21–25 模拟器上验证通知仍能展示。
</Answer>

</Exercises>

<Glossary>

<GlossaryItem term="minSdk">
  应用支持的最低 Android 版本。低于此版本无法安装。
</GlossaryItem>

<GlossaryItem term="targetSdk">
  声明已测试的行为变更版本。影响权限模型、后台执行、存储访问等系统策略。
</GlossaryItem>

<GlossaryItem term="compileSdk">
  编译期 SDK 版本，决定可用的 API 与 lint 规则，不影响运行时「最低系统」。
</GlossaryItem>

</Glossary>
