## 你改了 3 行代码，怎么确认没有破坏其他功能？

每次发版前手动点一遍所有页面？App 有 100 个页面怎么办？**自动化测试**是你最好的 QA。这章教你两个关键技术：MediaPlayer（最常用的音频播放）和 JUnit 单元测试（保证代码质量）。

## MediaPlayer 音频播放

```kotlin
class AudioPlayer(context: Context) {
    private var mediaPlayer: MediaPlayer? = null

    fun play(resId: Int) {
        release()  // 先释放旧的
        mediaPlayer = MediaPlayer.create(context, resId).apply {
            setOnCompletionListener { release() }
            start()
        }
    }

    fun pause() = mediaPlayer?.pause()
    fun stop() = mediaPlayer?.apply { stop(); prepareAsync() }

    fun release() {
        mediaPlayer?.release()
        mediaPlayer = null
    }
}
```

关键规则：**用完必须 release()**——MediaPlayer 持有系统音频资源，不释放会导致资源泄漏甚至其他 App 无声音。

对于短促音效（按钮点击声、通知铃声），用 **SoundPool** 替代 MediaPlayer：

```kotlin
val soundPool = SoundPool.Builder().setMaxStreams(3).build()
val soundId = soundPool.load(context, R.raw.click, 1)
soundPool.play(soundId, 1.0f, 1.0f, 1, 0, 1.0f)
```

SoundPool 预加载音频到内存，低延迟播放短音效——MediaPlayer 适合长音频（音乐/播客）。

<Stepper>
  <Step title="① create">
    `MediaPlayer.create(context, resId)` 是最简捷的创建方式——内部已调用 `setDataSource` + `prepare()`，创建后即可 `start()`。这也意味着它在 `create()` 阶段就可能因为资源不存在或格式不支持而抛异常。
  </Step>
  <Step title="② setDataSource + prepare">
    如果不用 `create()`，则手动 `setDataSource(path)` 指定音频来源（本地文件/网络 URI），然后调用 `prepare()`（同步，阻塞当前线程）或 `prepareAsync()`（异步，推荐）。异步准备完毕后通过 `OnPreparedListener` 回调通知。
  </Step>
  <Step title="③ start">
    调用 `start()` 开始播放。播放完成后触发 `OnCompletionListener`——在此回调中决定是循环播放（`seekTo(0)` + `start()`）还是释放资源（`release()`）。
  </Step>
  <Step title="④ pause / stop">
    `pause()` 暂停播放，之后可以 `start()` 从暂停位置恢复。`stop()` 停止并重置到 Idle 状态——之后若想再播必须重新 `prepare()`。用 `isPlaying()` 检查当前到底在播还是暂停。
  </Step>
  <Step title="⑤ release">
    调用 `release()` 彻底释放 MediaPlayer 持有的系统音频资源。**活动 onStop/onDestroy 中必须调用**——不释放会导致 AudioFlinger 资源泄漏，其他 App 可能无声音。每次 `create()` 新实例前也应先 `release()` 旧实例。
  </Step>
</Stepper>

## 单元测试基础

### JUnit 4 结构

```kotlin
class CalculatorTest {
    private lateinit var calculator: Calculator

    @Before
    fun setUp() {
        calculator = Calculator()
    }

    @Test
    fun `addition should return sum of two numbers`() {
        val result = calculator.add(2, 3)
        assertEquals(5, result)
    }

    @Test(expected = IllegalArgumentException::class)
    fun `division by zero should throw`() {
        calculator.divide(10, 0)
    }
}
```

### Mockito 模拟依赖

```kotlin
@Test
fun `loadData success should update LiveData`() {
    // 模拟 Repository
    val mockRepo = mock(Repository::class.java)
    `when`(mockRepo.fetchData()).thenReturn(listOf("a", "b"))

    val viewModel = MainViewModel(mockRepo)
    viewModel.loadData()

    // 验证 LiveData 结果
    assertEquals(listOf("a", "b"), viewModel.data.getOrAwaitValue())
}
```

Mockito 的 `mock()` 创建一个假对象，`when().thenReturn()` 控制它返回什么——让你在测试中隔离外部依赖（数据库、网络），只测 ViewModel 自身逻辑。

## 容易踩的坑

<Callout type="trap">
  **现象**：MediaPlayer 播完一首歌再播下一首——无声或崩溃。**原因**：没有先 `release()` 旧的 MediaPlayer 就创建新的——系统音频资源耗尽。**修法**：每次 play 前先 `release()` 确保旧实例被清理。
</Callout>

<Callout type="trap">
  **现象**：测试跑到 Room DAO 那一层要 5 秒——因为真的在写数据库。**原因**：单元测试应该隔离数据库依赖。**修法**：Room DAO 的测试用 `Room.inMemoryDatabaseBuilder`（内存数据库），或直接 mock DAO 接口。
</Callout>

<Callout type="trap">
  **现象**：MediaPlayer 在 Activity destroy 后仍占音频焦点。**原因**：未在 onStop/onDestroy 调用 release()。**修法**：生命周期绑定点释放，或改用 ExoPlayer/Media3 更好管理。
</Callout>

## 小结

- MediaPlayer 适合长音频（音乐/播客），SoundPool 适合短音效（点击声）
- MediaPlayer 用完必须 release()，播放新音频前先释放旧实例
- JUnit：@Test 标记测试方法，@Before 准备环境，assertEquals 验证结果
- Mockito 模拟依赖 → 隔离外部因素，只测被测代码本身
- 单元测试测纯逻辑（ViewModel/Repository），Instrumentation 测试测 UI 交互

<GlossaryItem term="SoundPool">
  短音效播放器——预加载音频到内存，低延迟播放（<10ms）。适合按钮点击声、通知铃声、游戏音效。不适合长音频（音乐用 MediaPlayer）。
</GlossaryItem>

<GlossaryItem term="Mockito">
  Java/Kotlin 的 mock 测试框架。`mock()` 创建假对象，`when().thenReturn()` 控制行为，`verify()` 验证方法是否被调用。隔离依赖、专注测试被测代码。
</GlossaryItem>
