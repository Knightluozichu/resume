## 为什么你敲的代码，电脑能看懂？

你打开编辑器、敲了几行代码、点了运行，屏幕上就出现了一行字。这中间其实经过了一条**工厂流水线**——你的代码是「原料」，经过好几道工位一道道加工，最后从流水线另一头出来一个能直接运行的「成品」。

这条流水线有**四道工位**：第一道把原料里杂七杂八的标注清理干净、把折叠好的部分摊开；第二道把人类能读的文字翻译成机器的"方言"；第三道把方言逐字译成二进制编码；第四道把分散的零件拼成一个完整的、能直接跑的程序。除此之外，你还要知道**怎么规划**一个程序——就像造房子之前要先画图纸，编程也一样，不是坐下来就敲键盘。

这一章就带你走一遍这条流水线的四道工位，顺便看看「编程」这件事到底是怎么从"我想让电脑帮我算个数"变成一段能跑通的成品。没有这条流水线会怎样？你写的代码永远只是一堆人类能读的文字，电脑一个字也不认识。

## C 语言从哪来？

在你动手写第一条 C 代码之前，先花几分钟看看这门语言是怎么来的——它对理解 C 语言"长什么样"很有帮助。

### 从贝尔实验室到全世界

<Term def="1972 年由 Dennis Ritchie 在贝尔实验室创造的一门通用编程语言。它以简洁、高效、可移植著称，是操作系统（Unix/Linux）、嵌入式系统、游戏引擎等领域的基石语言。">C 语言</Term> 是 Dennis Ritchie 于 1972 年在贝尔实验室创造的。最初是为了重写 Unix 操作系统——在此之前，Unix 是用汇编写的，换个机器就要全部重来。Ritchie 想要一门「能跨机器运行的高级语言」，于是 C 诞生了。

C 语言的发展和标准化走了很长的路。下面是它的演进时间线：

<CHistoryTimelineDiagram />

C 语言的关键节点：
- **1972**：Ritchie 在贝尔实验室创造了 C，用来重写 Unix
- **1978**：Brian Kernighan 和 Dennis Ritchie 合著的《The C Programming Language》出版，定义了一个事实上的标准——人称「K&R C」
- **1989**：ANSI 发布了第一个正式的 C 语言标准——C89（也叫 ANSI C），这是 **第一个让不同公司编译器生产出来的程序都能正确运行的标准**
- **1999**：C99 标准发布，新增了 `//` 单行注释、变长数组等特性
- **2011**：C11 标准发布，引入多线程支持、泛型选择器等
- **2018**：C18 标准发布，本质上是 C11 的修正勘误版，没有新特性

C 语言在整个编程流程里的位置是**起点**——你用它写下的每一行代码，都是那条工厂流水线的「原料」。

### 编译器：把人话翻译成机器话

写了 C 代码，电脑怎么认识它？答案是靠一个叫 <Term def="把一种编程语言写的高级代码翻译成机器能执行的二进制指令的工具。C 语言常用的编译器有 GCC、Clang、MSVC 等。">编译器</Term> 的翻译官。你写的 C 代码是人类能读的文字，编译器做的事就是把它们翻译成机器能执行的二进制指令。

编译器在整个流程里，对应的是工厂流水线的**第二道和第三道工位**（编译+汇编）。它先检查你的语法有没有写错（括号对上了吗？分号没忘吧？），然后把 C 语句翻译成汇编语言，最后转成机器码。

### 编译与链接：四道工位

你现在知道了「写完代码到什么结果」中间有四个工位。下面把它们挨个摊开讲清楚。

<Term def="编译器把源码翻译成机器码的过程。对于 C 语言，通常分为四步：预处理（展开头文件和宏）、编译（C→汇编）、汇编（汇编→二进制机器码）、链接（拼接多个目标文件和库成可执行文件）。">编译过程</Term> 就是把你的 .c 文件一步步变成能运行的程序。而 <Term def="编译器工作的最后一道工位。把你编译出来的 .o 目标文件，和你用到的库里别人已经写好的机器码，拼在一起，变成一个能双击就跑的可执行文件。">链接</Term> 是最后一步——把分散的「零件」（.o 文件）拼在一起，对接所有符号引用，变成可以跑的可执行文件。下面这张图把这四步一清二楚地画出来了：

<CCompilationDiagram step={0} />

每一步的解释：

- **第一道工位·预处理**：预处理器做的都是「文本级」的工作。看见 `#include <stdio.h>`？把整个 stdio.h 文件的内容原样粘贴到这行；看见注释？删掉；看见 `#define`？把用到的宏全部替换成对应的文字。处理完后生成了后缀名为 `.i` 的「纯净 C 代码」。

- **第二道工位·编译**：编译器拿到 `.i` 文件，把 C 语言的每一个语句翻译成汇编语言——一种贴近机器、但仍然用人类能读的助记符写的"方言"。如果语法有错（忘记分号、括号不配对、类型不匹配），就在这一步报错停下。输出 `.s` 文件。

- **第三道工位·汇编**：汇编器把 `.s` 文件里的每一条汇编助记符一对一翻译成二进制机器码，生成 `.o` 目标文件——这是「零件」，还不是能跑的东西。

- **第四道工位·链接**：链接器把你的 `.o` 文件和你用到的标准库（比如 `printf` 的实现就住在库文件里）拼在一起。所有对 `printf` 的引用都找到对应的地址，组装成一个能双击就跑的可执行文件。

这四步在编译过程里的顺序是固定的——**预处理→编译→汇编→链接**。每一道都依赖前一道的产出，漏掉任何一步都到不了最后。

### 可移植性：一次编写，到处跑

C 语言最厉害的特性之一：写完一份代码，换个操作系统、换台电脑，只要用那个平台的编译器重新编译，就能生成在那个平台上能跑的程序。这就是 <Term def="写在标准 C 语言里的一份源码，用不同的编译器重新编译，就能在不同操作系统和 CPU 架构上生成行为一致的可执行文件——不需要为每个平台重写代码。">可移植性</Term>。

可移植性靠的是 C 标准——C89、C99、C11、C18。只要你写的是标准 C（不用编译器特有的扩展），不管在哪个系统上编译，行为都是可预期的。下面这张图直观地画出了可移植性的含义：

<PortabilityDiagram />

同一份 `hello.c`，用 GCC 编译 → Linux 上能跑；用 MSVC 编译 → Windows 上能跑；用 Clang 编译 → macOS 上也能跑。结果一致，无需改一行代码。

## 动手：一步步走完编程的全过程

现在你知道了 C 语言怎么来、代码怎么变程序。下面跟着这条七步流水线，一步步看「编程」这件事到底从哪开始、到哪结束——**每切一步，上面的循环图都会高亮当前这一步，下方说明这一步具体干什么**。

> 猜一猜：七个步骤里哪一步最费时间？是敲代码、调试 bug、还是改需求？动手切一切，想想你过去的经历再说答案。

<Stepper>
  <Step title="① 定义目标：想清楚程序要干什么">
    <DevelopmentCycleDiagram step={1} />
    这是编程的**起点**——不是打开编辑器，而是拿支笔把「我想让程序干什么」写清楚。是算一个数学题？是展示一张图片？是管理一个通讯录？目标越具体越好。如果你跳过这一步直接开始敲代码，写到一半发现方向偏了——推倒重来。这一步决定了后面六步的方向。
  </Step>
  <Step title="② 设计：怎么实现这个目标">
    <DevelopmentCycleDiagram step={2} />
    有了目标，设计「怎么到达那里」。选择什么数据结构（用数组还是链表？）、用什么算法、模块怎么划分、用户界面长什么样。这一步产出的是**设计蓝图**——类似盖房子前的施工图纸。好的设计让写代码顺手；差的设计让写代码像在迷宫瞎走。
  </Step>
  <Step title="③ 编写：把设计变成 C 代码">
    <DevelopmentCycleDiagram step={3} />
    到了真正敲键盘的阶段。按照设计蓝图，把一个个逻辑用 C 语言实现出来。注意**可读性**——你六个月后再看这段代码，还能看懂吗？变量名起清楚、关键地方写注释。这一步产出的是 `.c` 源文件——工厂流水线的原料。
  </Step>
  <Step title="④ 编译：让编译器把 C 代码翻译成机器码">
    <DevelopmentCycleDiagram step={4} />
    敲完代码，让编译器上场！一条 `gcc myprogram.c -o myprogram`，编译器帮你跑完**预处理、编译、汇编、链接**四道工位。如果编译器报了一大堆红字错误——别慌，那是它在帮你发现漏掉的分号、写错的拼写。修完错再编译，直到出现那个闪亮的可执行文件。
  </Step>
  <Step title="⑤ 运行：执行程序，看效果">
    <DevelopmentCycleDiagram step={5} />
    可执行文件出来了，跑起来！`./myprogram` 一敲回车，屏幕上出现输出——如果一切顺利。但大概率会发现：结果跟你预期的不完全一样？很正常——你还没有做测试。
  </Step>
  <Step title="⑥ 测试：验证程序对不对">
    <DevelopmentCycleDiagram step={6} />
    测试不是「跑一次看看没报错就完了」。你要准备**一系列输入**——正常数据、边界值、非法输入——每一组都能输出预期结果吗？如果不行，就是 bug，回到步骤③修改，再编译、再运行、再测试。这一步决定了你程序的真实质量。
  </Step>
  <Step title="⑦ 维护：修 bug、加新功能">
    <DevelopmentCycleDiagram step={7} />
    程序上线了不等于结束。用户会发现你没测到的 bug；需求会变——有人会说「这里加个按钮行不行」。维护是编程最长的一步：它可能占到整个程序生命周期 70% 以上的时间。每次改完代码，都要重新走一遍步③到步⑥——测试过才发布。这七个步骤是一个**循环**：从维护得到反馈，回到定义目标开始下一轮。
  </Step>
</Stepper>

<Callout type="tip">
  实际开发中，这七个步骤是**反复迭代**的——很少能一条直线走到底。
  写代码时发现设计有漏洞？回到步②改设计。测试时发现 bug？回到步③修代码。
  把七步画成环就是因为这个原因。
</Callout>

## 你的第一个 C 程序

不废话了，先看代码——这是 C 语言里最经典的一小段：

```c
#include <stdio.h>

int main(void) {
    printf("Hello, World!\n");
    return 0;
}
```

每一行在说什么？`#include <stdio.h>` 把标准输入输出头文件的内容粘贴进来，让你能使用 `printf`。`int main(void)` 是整个程序的入口——操作系统运行你的程序时，第一件事就是钻进 `main`。`printf("Hello, World!\n");` 往屏幕上输出一行文字，`\n` 是换行。`return 0;` 告诉操作系统「一切正常，我跑完了」。

编译和运行它，一条命令就够了：

```bash
# Linux / macOS
gcc hello.c -o hello
./hello
# 输出：Hello, World!
```

在 Windows 上，Visual Studio 里按 Ctrl+F5 运行，Dev-C++ 里点编译运行按钮。背后做的事完全一样——预处理、编译、汇编、链接。

## 容易踩的坑

<Callout type="trap">
  **现象**：编译报错 `undefined reference to 'printf'`，明明写了 `#include <stdio.h>`。
  **原因**：忘了告诉链接器要链接 C 标准库（libc）。这在用某些 IDE 或手动分步编译时会出现——头文件只是做了声明，`printf` 的实现在库里面，链接时没指定就不认识。
  **修法**：用 `gcc` 而不是分步 `gcc -c` + 手动 `ld`。gcc 会自动帮你链接标准库。如果分步做，记得链接时加上 `-lc`。
</Callout>

<Callout type="trap">
  **现象**：程序编译通过了，双击运行窗口一闪就没了，什么都看不到。
  **原因**：程序确实运行了，`printf` 也输出到了控制台——但是控制台窗口在输出后立刻关了。这是 Windows 下直接双击 .exe 的常见行为。
  **修法**：在 `return 0;` 之前加一句 `getchar();`（让程序等你敲一个回车再退出），或者直接从命令行里运行程序。
</Callout>

<Callout type="trap">
  **现象**：把 C 程序保存成了 `.cpp` 后缀，用 `gcc` 编译正常，但用 C++ 编译器报错。
  **原因**：C 和 C++ 是不同的语言。虽然 C++ 兼容大部分 C 语法，但存在细微差异（比如 `int main()` 和 `int main(void)` 在两种语言里含义不同）。保存成 `.cpp` 会让编译器按 C++ 规则处理。
  **修法**：C 程序后缀用 `.c`，C++ 程序用 `.cpp`。让编译器根据后缀自动选择正确的语言规则。
</Callout>

## 小结

- C 语言从 1972 年诞生，经历了 K&R C → C89 → C99 → C11 → C18 五个关键标准版本
- 编程有七个步骤——定义目标、设计、编写、编译、运行、测试、维护——它们是一个反复迭代的循环
- 编译过程分四道工位：预处理（.i）→ 编译（.s）→ 汇编（.o）→ 链接（可执行文件），`gcc` 一条命令全包
- C 标准的每一次更新都在提升代码的可移植性——一份标准 C 代码，用 GCC/Clang/MSVC 在不同系统上编译，行为一致
- `int main(void)` 是程序入口，`printf` 通过 `<stdio.h>` 声明、在链接阶段从标准库拿到实现

<Exercises>

**练习 1（改代码型）** 把上面 Hello World 程序里的 `printf("Hello, World!\n");` 改成两行——第一行输出你的名字，第二行输出今天的日期。在 `return 0;` 之前加 `getchar();` 让窗口停住（Windows 用户尤其需要这行）。跑起来，解释 `\n` 在你新改的程序里起了什么作用。

<Answer>

改完后的程序：

```c
#include <stdio.h>

int main(void) {
    printf("我的名字是张三\n");
    printf("今天是2026年6月14日\n");
    getchar();  /* 等用户敲回车再退出 */
    return 0;
}
```

每个 `\n` 让输出换一行——所以两个 `printf` 各占一行，不会挤在一起。`getchar()` 暂停程序等你敲回车，能防止控制台窗口一闪而逝。

</Answer>

**练习 2（问答型）** 编程七个步骤里，哪一步你最容易想跳过？跳过它之后最可能导致什么问题？

<Answer>

大部分人最容易跳过「设计」——觉得问题简单就直接开始敲。跳过设计的代价是：写到一半发现数据结构选错了、需要推倒重来，浪费了写代码的时间。也有人跳过「测试」——「跑一次没报错就是对的」≠「程序正确」。没有准备多组测试数据，很多边界情况的 bug 等你上线了才会被发现。

</Answer>

**练习 3（问答型）** 编译器分别报了以下三种错误信息，各出在哪道工位？`fatal error: 'stdoi.h' file not found`、`error: expected ';' before 'return'`、`undefined reference to 'print'`。

<Answer>

① `fatal error` 出在**预处理**阶段——编译器找不到你 include 的头文件（可能是拼写错误，`stdio.h` 写成了 `stdoi.h`）。② `expected ';'` 出在**编译**阶段——语法有错，可能前面少了分号。③ `undefined reference` 出在**链接**阶段——你用了 `print` 这个函数，编译器在声明里找到了它，但链接时在所有 .o 和库都找不到它的实现（可能是拼写错误，`printf` 写成了 `print`）。

</Answer>

<Glossary>

<GlossaryItem term="C 语言">
  1972 年由 Dennis Ritchie 在贝尔实验室创造的一门通用编程语言。它以简洁、高效、可移植著称，是操作系统（Unix/Linux）、嵌入式系统、游戏引擎等领域的基石语言。详见本章「C 语言从哪来」一节。
</GlossaryItem>

<GlossaryItem term="编译器">
  把一种编程语言写的高级代码翻译成机器能执行的二进制指令的工具。C 语言常用的编译器有 GCC（Linux/macOS 默认）、Clang（macOS 默认）、MSVC（Visual Studio 自带）。详见本章「编译器」一节。
</GlossaryItem>

<GlossaryItem term="编译过程">
  编译器把源码翻译成机器码的过程。对于 C 语言，通常分为四步：预处理（展开头文件和宏 → .i）、编译（C→汇编 → .s）、汇编（汇编→二进制机器码 → .o）、链接（拼接多个 .o 和库成可执行文件）。详见本章「编译与链接」一节。
</GlossaryItem>

<GlossaryItem term="链接">
  编译器工作的最后一道工位。把你编译出来的 .o 目标文件，和你用到的库里别人已经写好的机器码，拼在一起，变成一个能双击就跑的可执行文件。详见本章「编译与链接」一节。
</GlossaryItem>

<GlossaryItem term="可移植性">
  写在标准 C 语言里的一份源码，用不同的编译器重新编译，就能在不同操作系统和 CPU 架构上生成行为一致的可执行文件——不需要为每个平台重写代码。可移植性靠 C 标准（C89/C99/C11/C18）来保证。详见本章「可移植性」一节。
</GlossaryItem>

</Glossary>
