## 你写的代码，是怎么变成电脑能跑的程序的？

## 最小 C 程序的结构解剖

把你看到的第一个 C 程序拿出来，逐段标清楚：头文件是干什么的、main 函数什么角色、花括号管哪块、`printf` 怎么输出、`return` 给谁看。

### 一个完整的 C 程序，长这样

先看一眼完整的：

```c
#include <stdio.h>

int main(void)
{
    printf("Hello, World!\n");
    return 0;
}
```

下面用一张图把这段程序的每一个关键点标出来。**每一处气泡都是后面会展开讲的知识点——带着这张图往下读。**

<CProgramAnatomyDiagram />

现在逐行解释：

1. `<Term def="程序最开头的一行 #include 指令，把标准库头文件的内容粘贴到这里，让你能用库里的函数。">#include &lt;stdio.h&gt;</Term>`——告诉预处理器：「把 stdio.h 这个文件的内容原样粘贴到我这一行所在的位置」。stdio.h 是 C 标准库的头文件，里面声明了 `printf` 等常用函数。没有这行，编译器不认识 `printf`。

2. `<Term def="每个 C 程序的入口函数。操作系统加载你的可执行文件后，第一件事就是跳进 main 函数开始执行。int 是返回值类型，void 表示这个函数不接受命令行参数。">int main(void)</Term>`——程序入口。`int` 是返回值类型（交一个整数给操作系统），`void` 表示 `main` 不接受命令行参数。操作系统运行你的程序时，第一件事就是钻进来找 `main`。

3. `{ }` 花括号——把函数体裹住。左花括号 `{` 表示「main 函数从这里开始」，右花括号 `}` 表示「main 函数到此结束」。中间的所有语句是 main 函数要干的活。

4. `<Term def="C 标准库的输出函数。把格式字符串里的普通文字原样输出，把 % 打头的转换说明替换成后面参数的值，然后送到标准输出（通常是屏幕）。">printf()</Term>`——C 语言里最常用的输出函数。括号里是一段字符串，双引号裹住。`\n` 是个特殊的「转义序列」，代表换行——让光标跳到下一行开头。

5. `<Term def="函数结束时的语句，把 return 后面整数交给操作系统。0 表示「正常结束」，非零通常表示出了某种错误。">return 0;</Term>`——返回语句。`0` 是一个整数值，交给操作系统，告诉它「程序一切正常，跑完了」。

### 变量与赋值：程序怎么「记住」东西

程序需要记住数据——用户的年龄、圆周率、一个名字。这些数据住在内存里，你通过 `<Term def="一块有名字、有类型、占固定大小字节的内存。你用名字来读写这块内存里的值。">变量</Term>` 来操纵它们。

在 C 语言里使用变量要分两步：先「声明」，再「赋值」。

```c
int num;        /* 声明：告诉编译器 num 是 int 类型 */
num = 10;       /* 赋值：把 10 存进 num 那块内存 */
```

也可以一步到位——声明的同时给一个初始值，这叫 `<Term def="在声明变量的同时就给它一个初始值。int a = 5; 让 a 从出生就是 5，避免读到不确定的垃圾值。">初始化</Term>`：

```c
int num = 10;       /* 声明 + 初始化：一步到位 */
float pi = 3.14;
char grade = 'A';
```

声明只是告诉编译器「有这么一个变量，长 XX 类型」，但内存里那块空间可能还残留着上次别人用过的垃圾数据。**永远在声明时就初始化**——这是 C 程序员的好习惯。

下面这张图画了三个变量在内存里的样子：

<VariableMemoryDiagram />

每一种类型占用的字节数不同、能存的东西不同。`int` 存整数，`float` 存小数，`char` 存单个字符。选错类型就像拿称米的秤去称书——能秤，但结果可能不是你想要的。

### printf()：往屏幕上印东西

`printf` 的核心是一个「模板 + 填空」的机制。双引号里的文字大部分原样输出，但遇到 `%` 打头的地方——那是「转换说明」——就用后面参数的值来替换。

<PrintfFormatDiagram />

三个最常用的转换说明：

| 转换说明 | 替换类型 | 示例 | 输出 |
|---|---|---|---|
| `%d` | int 整数 | `printf("%d", 42)` | `42` |
| `%f` | float/double 浮点 | `printf("%f", 3.14)` | `3.140000` |
| `%s` | char* 字符串 | `printf("%s", "Hi")` | `Hi` |

除了 `%` 转换说明，还有一类叫 `<Term def="反斜杠 \ 开头的特殊字符组合，在输出中代表一个不可见的控制字符——比如 \n 换行、\t 制表符、\ 打出反斜杠本身。">转义序列</Term>`。`\n` 不是把 `\` 和 `n` 两个字印出来，而是告诉 printf：「这里换一行」。

```c
printf("Line1\nLine2\n");
// 输出：
// Line1
// Line2
```

### 注释：写给人类看的标记

C 语言有两种注释方式。编译器在预处理的末尾就把它们全删了——注释只给人看，绝不进最终的二进制程序。

```c
/* 这是多行注释——可以跨行，裹住的部分全部被忽略 */

int x = 10;  // 这是单行注释——从 // 到本行末尾全部被忽略
```

写注释的三条建议：① 写**为什么这样做**而不是**代码干了什么**（代码本身已经说了干什么）。② 变量名起好就是最好的注释——`int daysPerYear = 365` 不需要再注释「这是每年的天数」。③ 过时的注释比没注释更危险——把注释当代码的一部分维护。

## 动手：把两个 .c 文件编译成一个可执行文件

真实项目里，你不会把所有代码塞进一个大文件——你得拆成多个 `.c`，每个负责一块功能，最后拼起来。下面用 Stepper 一步步看这个「拆开→各编各的→拼起来」的过程。

> 猜一猜：编译 `main.c` 时，如果它调用了 `helper.c` 里的函数——但 `main.c` 没有 `#include "helper.c"`——编译器会报错吗？切到第①步找答案。

<Stepper>
  <Step title="① 编译 main.c：只翻译自己，不关心别人">
    <MultiFileCompilationDiagram step={1} />
    把 `main.c` 单独编译成 `main.o`：
    ```bash
    gcc -c main.c -o main.o
    ```
    `-c` 意思是「只编译不链接」——预处理、编译、汇编都干完，产出 `main.o` 目标文件（二进制机器码）。这一步关键：如果 main.c 里调用了 helper.c 的某个函数（如 `greet()`），main.c 里必须有一行**该函数的声明**（告诉编译器「这个函数存在，以后链接会找到它」），否则编译器会警告「隐式函数声明」。但就算有警告，`-c` 这一步通常还是能生成 `main.o`——真正报错的是下一步的链接阶段。
  </Step>
  <Step title="② 编译 helper.c：同样的流程，产出另一个零件">
    <MultiFileCompilationDiagram step={2} />
    同样的命令，对 `helper.c` 再来一遍：
    ```bash
    gcc -c helper.c -o helper.o
    ```
    `helper.o` 和 `main.o` 是平等的——都是独立的 `.o` 目标文件零件。谁都没有比谁高级。每个 .o 里各自装着本文件所有函数的机器码。
  </Step>
  <Step title="③ 链接：拼接零件，对号入座">
    <MultiFileCompilationDiagram step={3} />
    现在有两个零件：`main.o` 和 `helper.o`。链接器把它们拼装起来：
    ```bash
    gcc main.o helper.o -o app
    ```
    链接器做的事：扫描 `main.o`，遇到一个对 `greet` 的函数调用——它在 `main.o` 里留了一个「待填地址」的空洞。然后去 `helper.o` 里面搜索——找到了 `greet` 的实现，拿到内存地址。把空洞填上这个地址。全部符号（函数名、全局变量名）都对上号了，生成最终的可执行文件 `app`。
  </Step>
</Stepper>

<Callout type="tip">
  上面是分三步走的教学习惯写法。实际开发中，一句命令就够了：
  `gcc main.c helper.c -o app`——gcc 自动帮你跑完三道 .c→.o 编译 + 一道链接，省心且快。
  但要知道背后发生了什么——因为报 `undefined reference` 的时候，你得知道问题出在链接那一步。
</Callout>

## 代码逐段拆解：手写你的第一个多文件 C 程序

下面从头开始，手写一个完整的多文件程序。**这段程序没有在虚讲——每一行都能在你的电脑上跑。** 拆成两块：`main.c` 负责程序入口和主流程，`helper.c` 负责提供一个辅助函数。

### 主程序：main.c

```c
#include <stdio.h>

void greet(void);    /* 函数声明——告诉编译器「后面有 greet 这么个函数」 */

int main(void)
{
    int count = 3;
    float price = 9.99;

    printf("count = %d\n", count);
    printf("price = %.2f\n", price);

    greet();         /* 调用另一个文件里定义的函数 */

    return 0;
}
```

逐行解释：

- `#include <stdio.h>`：让 `printf` 可用。
- `void greet(void);`：**函数声明（原型）**。告诉编译器「存在一个名叫 greet 的函数，它没有返回值（void）、不需要参数（void）」。你在本章就要开始理解「声明」和「定义」是两件事——声明只是承诺有这个东西，定义才是真把它写出来。这里的声明告诉编译器不要因为没有 greet 的定义就报错——链接的时候会找到它。
- `int count = 3;`：声明 + 初始化一个 int 变量。
- `float price = 9.99;`：声明 + 初始化一个 float 变量。float 和 int 在内存里存法完全不同——float 用 IEEE 754 浮点数格式，小数点可以漂移。
- `printf("count = %d\n", count);`：`%d` 换成 count 的值（3），输出 `count = 3` 然后换行。
- `printf("price = %.2f\n", price);`：`%.2f` 把 price 保留 2 位小数输出——`price = 9.99`。
- `greet();`：调用另一个文件里的函数。链接器会去 helper.o 里找 greet 的实现。
- `return 0;`：程序正常结束。

### 辅助函数：helper.c

```c
#include <stdio.h>

void greet(void)
{
    printf("Hello from helper.c!\n");
}
```

这个文件很简单——只定义了一个函数 `greet`。注意两点：① 它自己也 `#include <stdio.h>`（每个 .c 文件独立编译，不共享别人的 include）。② 函数名后面是 `{ }` 里写具体干的事（定义），而不是分号结尾（声明才用分号）。声明只承诺名字和类型，定义才把具体做的事写出来。

### 编译并跑起来

```bash
# 方法一：一步到位（推荐日常用）
gcc main.c helper.c -o app
./app

# 方法二：分步来（理解流程用）
gcc -c main.c -o main.o
gcc -c helper.c -o helper.o
gcc main.o helper.o -o app
./app
```

输出：

```
count = 3
price = 9.99
Hello from helper.c!
```

## 容易踩的坑

<Callout type="trap">
  **现象**：编译报错 `error: 'printf' was not declared in this scope`，明明写了 `printf`。
  **原因**：忘了加 `#include <stdio.h>`。没有这行，编译器完全不认识 printf 这个名字——它只知道 C 语言自带的关键字，不知道标准库里有什么函数。
  **修法**：C 文件最开头加 `#include <stdio.h>`。如果仍然报错，检查拼写有没有错（是 `stdio.h` 不是 `studio.h` 或 `stdoi.h`）。
</Callout>

<Callout type="trap">
  **现象**：编译通过，但运行时 printf 输出的是奇怪的随机数字。
  **原因**：声明了变量但没有初始化（例如 `int count; printf("%d", count);`）。那块内存里残留着上次别人用过的数据——这次可能是 0，下次可能是 832749，完全不可预期。
  **修法**：永远在声明时初始化：`int count = 0;`。养成习惯——声明即初始化，从不裸声明变量。
</Callout>

<Callout type="trap">
  **现象**：把 `%d` 和 `%f` 用反了——`printf("%f", 42)` 输出 0.000000，`printf("%d", 3.14)` 输出一个奇怪的整数。
  **原因**：`%d` 期望的参数是 int 类型，`%f` 期望的是 double 类型。int 和 double 在内存和 CPU 寄存器里的存储/传参方式完全不同。printf 按错类型去读二进制位，读出来的是垃圾。
  **修法**：`%d` → int，`%f` → float/double，`%s` → char*（字符串）。转换说明的类型必须和后面参数的实际类型一一对应。编译器不一定会警告你。
</Callout>

<Callout type="trap">
  **现象**：多文件编译时报 `undefined reference to 'greet'`，但 greet 函数明明写在 helper.c 里了。
  **原因**：编译命令里漏了 helper.c——`gcc main.c -o app` 只编译链接了 main.c，链接器找不到 greet 的实现。main.c 里的 `void greet(void);` 声明只告诉编译器「有这个东西」，链接时编译器不管这事了——该链接器来找 greet 的实际地址。
  **修法**：编译命令里加上 helper.c：`gcc main.c helper.c -o app`。或者如果你先 `gcc -c` 各编各的，最后链接时两个 .o 都得给：`gcc main.o helper.o -o app`。
</Callout>

## 小结

- 一个最小 C 程序 = `#include <stdio.h>` + `int main(void) { ... }` + `printf` 语句 + `return 0;`
- `printf` 用格式字符串——`%d`→int、`%f`→float、`%s`→字符串，`\n` 换行，普通文字原样输出
- 变量先声明（定类型和名字），再初始化（给初值）。用之前必须已经有值——垃圾值不可预期
- `//` 和 `/* */` 两种注释在预处理阶段被删掉，只给人看，不进二进制
- 多文件编译：`gcc -c` 只编译不链接（出 .o），最后 `gcc *.o -o app` 链接拼接。`undefined reference` 出在链接阶段

```c
    printf("price = %+.2f\n", price);   /* %+f: 显示正负号 → 输出 "+9.99" */
```
`:03` 是最小宽度 3 且用 0 填充（所以 3 变成 003），`+` 强制显示正负号（正数也打 `+`）。

</Answer>

**练习 2（独立实现型）** 自己从零建一个两文件项目：
- 文件一 `calc.c`——定义两个函数：`int add(int a, int b)` 返回 a+b，`int multiply(int a, int b)` 返回 a*b。
- 文件二 `main.c`——`main` 函数里调用 `add(7, 8)` 和 `multiply(4, 5)`，分别用 `printf` 输出结果。输出格式如 `7 + 8 = 15` 和 `4 * 5 = 20`。
- 编译成 `calculator` 程序并运行。
- **额外挑战**：把 `add` 和 `multiply` 的声明抽到一个头文件 `calc.h` 里，然后在 `main.c` 里 `#include "calc.h"`。体会一下头文件怎么让小项目变干净。

<Answer>

完整代码（带头文件版）：

`calc.h`：
```c
#ifndef CALC_H    /* 防止重复包含的头文件保护 */
#define CALC_H

int add(int a, int b);
int multiply(int a, int b);

#endif
```

`calc.c`：
```c
#include "calc.h"

int add(int a, int b) {
    return a + b;
}

int multiply(int a, int b) {
    return a * b;
}
```

`main.c`：
```c
#include <stdio.h>
#include "calc.h"

int main(void) {
    int sum = add(7, 8);
    int product = multiply(4, 5);
    printf("7 + 8 = %d\n", sum);
    printf("4 * 5 = %d\n", product);
    return 0;
}
```

编译：`gcc main.c calc.c -o calculator`。头文件 `calc.h` 用了 `#ifndef` 保护，防止被重复 `#include` 时出现重复声明错误——这是一个重要的 C 语言惯例。

</Answer>

**练习 3（问答型）** 前面讲了两种注释：`//` 和 `/* */`。如果要用 `/* */` 临时屏蔽一大段代码（里面可能已经包含 `/* ... */` 注释），会发生什么危险？你有什么替代方案？

<Answer>

C 语言的多行注释**不能嵌套**。如果你的代码块里已经有 `/* ... */` 注释，在外面再套一层 `/* */`——第一个 `*/` 就会提前闭合外层注释，导致之后的代码全暴露出来，编译器报一大堆莫名其妙的错误。

替代方案：
- 用 `//` 逐行注释（更安全，推荐）
- 用 `#if 0 ... #endif` 条件编译指令屏蔽整块代码。`#if 0` 表示「永远不编译这一段」，#if 块内的任何注释都不会干扰——这是真正的安全大法。
```c
#if 0
/* 这里的所有代码和注释都不会被编译 */
int old_buggy_function(void) { /* 旧的实现 */ return -1; }
#endif
```

</Answer>
