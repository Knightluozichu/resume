## 多请几个厨师，菜就一定上得更快吗？

前八章你学会了让多个厨师同时干活：怎么招厨师、怎么分账本、怎么排队取菜、怎么不上锁也不打架。可真到了忙时，老板会发现一件怪事——明明多请了人，出菜却没快多少，甚至更慢了。这一章就回答：人多了，活该怎么分，才真的更快？

先想清楚「怎么分活」。一种分法是按工序：洗菜的、切菜的、掌勺的各管一段，菜在他们手上流水般传过去。另一种是按菜量：同一道菜要做二十份，就分给四个厨师每人做五份。分法选错，人再多也白搭。

更隐蔽的坑还有两个。一是两个厨师各写各的便签、本来互不相干，可两张便签恰好写在同一张得来回传的纸上，于是这张纸被反复抢来抢去，两人都被拖慢——明明没共享内容，却互相绊脚。二是不管活多活少，总有几道菜必须一个人从头做到尾，这几道的耗时再多厨师也省不掉，它才是出菜快慢的真正天花板。

## 怎么把活分给多个线程：任务划分三策略

设计并发代码的第一步，是 <Term def="把一份工作拆开、分给多个线程去做的方式。怎么拆决定了能不能并行、并行得好不好——拆错了，线程之间反复等待、反复传数据，人多也快不起来。常见三策略：数据划分、递归划分、流水线。">任务划分</Term>：把一份工作拆成多块，交给不同线程。怎么拆？常见有三条路，下面这张对比图把它们并排摆出来——先看图，再逐个说：

<TaskDivisionDiagram />

第一条是 <Term def="把同一份工作的数据切成 N 份，每个线程独立处理其中一份，做的是同一件事、只是数据不同。适合数据量大、各份处理彼此不依赖的场景（如逐元素变换、并行求和）。也叫数据划分。">数据并行（数据划分）</Term>：一大块数据切成 N 份，每个线程领一份、做同样的处理。比如把一个百万元素的数组分成四段，四个线程各算各段的和。它适合数据量大、各份之间互不依赖的活。

第二条是递归划分（分治）：把问题递归拆成两半，子问题再并行。并行快排就是典型——选个基准把数组分成两边，左右两边交给不同线程各自再快排下去。它适合天然能分治的递归算法。

第三条是 <Term id="任务并行" def="把工作按工序拆成若干阶段，数据依次流过各阶段，不同阶段由不同线程负责，做的是不同的事。像工厂流水线：第一个工位干完交给第二个工位。适合数据连续到达、工序固定的场景。它属于任务并行。">流水线（任务并行）</Term>：按工序分阶段，数据一件件流过。视频处理常这样——解码、滤镜、编码三段各派一个线程，第一帧解完码就交给滤镜线程，自己接着解第二帧。它和数据并行的根本区别是：数据并行是「每个线程做**同样**的事、数据不同」，<Term def="把不同的工作（不同的任务/工序）分给不同线程，各线程做的事不一样。流水线是典型——每个阶段是一种不同的工序。与数据并行（各线程做同样的事、只是数据不同）相对。">任务并行</Term>是「每个线程做**不同**的事」。

三者没有最优，只有最配场景的那一种——这正是本章 §8 选型题要练的判断力。

## 为什么人多了反而更慢：影响并发性能的因素

分法对了，性能还可能被几个隐蔽因素拖垮。最阴险的一个叫 <Term def="两个线程各自访问不同的变量，本不该互相影响；但这两个变量恰好落在同一条缓存行（CPU 在核间搬数据的最小单位，通常 64 字节）上，于是任一线程写自己的变量，都会让整条缓存行在两个核之间来回失效、传输，互相拖慢——明明没有真正共享数据，却付出了共享的代价。所以叫「伪」共享。靠 alignas / 填充把变量对齐到不同缓存行可根治。">false sharing（伪共享）</Term>。它的根子在于：CPU 在核之间搬数据不是一个字节一个字节地搬，而是以一整条 <Term def="CPU 缓存与内存之间、以及核与核之间搬运数据的最小单位，通常 64 字节。读写任何一个变量，都会把它所在的整条缓存行一起搬动。两个变量若落在同一条缓存行，写其中一个就会牵动另一个。">缓存行（cache line）</Term>（通常 64 字节）为单位。两个线程各写各的变量，本该井水不犯河水，可一旦这两个变量挤在同一条缓存行上，任一线程写一下，都会让整条行在两核间被抢来抢去、反复失效——这种来回弹叫 <Term def="同一条缓存行被多个核反复争抢、来回传输失效的现象。每次一个核写了该行，其他核手里的副本就失效、得重新取，于是缓存行像乒乓球一样在核间弹来弹去，开销巨大。false sharing 是它最常见的诱因。">缓存乒乓（cache ping-pong）</Term>。下面这张动画把这场无谓的拉锯演给你看，先猜一猜：

盯住第三、四拍：core0 写 a、core1 写 b，本是两件不相干的事，可整条缓存行却被反复抢来抢去。修复办法在第六拍——用 `alignas` 把两个变量各塞进一条独立缓存行，乒乓立刻消失（代码见 §6）。

除了伪共享，还有两个常被忽略的因素。一个是 <Term def="数据接近度（data proximity / locality）：一个线程要用到的数据是否在内存里挨得近、能否落进同一条缓存行被一次取出。挨得近则缓存命中率高、快；散落各处则频繁缓存未命中、慢。把线程各自要用的数据安排得彼此靠近、又和别的线程的数据分开，是并发性能调优的常见手段。">数据接近度</Term>：一个线程频繁用到的数据若在内存里挨得近，就能被缓存一次取出、命中率高；散落各处则反复缓存未命中。另一个是 <Term def="超额订阅（oversubscription）：创建的线程数远多于硬件能真正并行的核数。多出来的线程并不能真的同时跑，反而让操作系统在它们之间频繁切换上下文——保存/恢复寄存器、刷新缓存——这些切换开销可能吃掉并行带来的全部收益，使程序不升反降。对策是按 std::thread::hardware_concurrency() 提示的核数来定线程数。">超额订阅（oversubscription）</Term>：开的线程数远超核数。多出来的线程并不能真的同时跑，只会让操作系统在它们之间疲于切换——上下文切换的开销可能把并行的好处吃光。对策很简单：线程数按 `std::thread::hardware_concurrency()` 给的核数来定，别瞎开。

## 为并发设计数据结构与异常安全

把活分好、避开性能坑之后，还有一层设计要操心：数据结构本身要为并发而设计。核心原则是**让线程尽量在自己的数据上独立干活，少碰共享状态**——共享越少，需要同步的地方越少，缓存乒乓的机会也越少。数据并行里给每个线程一块专属的累加器、最后再合并，就是这个思路（§6 有代码）。

并发还放大了异常安全的难度。单线程里一个函数抛了异常，栈一路解开、析构函数收尾，相对好控制；可在并发里，一个线程抛异常时，别的线程还在跑、还可能在等它的结果。如果不小心，抛异常的线程会让等结果的线程**永远等下去**（比如它本该 set 的 future 没 set 成），或者让共享状态停在半成品。所以并发代码里凡是「我算完要把结果交给别人」的地方，异常路径上也得有交代——这也是为什么 `std::async` / `std::packaged_task` 这类工具会把异常自动存进 future、让等待方 `get()` 时重新抛出，替你兜住这条路。

## 可伸缩性与 Amdahl 定律：加多少核到头？

设计并发代码绕不开一个问题：这段代码 <Term def="可伸缩性（scalability）：往程序里加更多处理器（核）时，性能能跟着提升多少。理想是加一倍核就快一倍（线性可伸缩）；现实里因为有必须串行的部分，加速比会越来越逼近一个上限、加核的收益递减。Amdahl 定律就是量化这个上限的公式。">可伸缩性</Term>好不好——多给几个核，它能快多少？凭直觉很多人以为「核翻倍、速度翻倍」，但只要程序里有哪怕一小段必须一个人顺序做的活，这个直觉就会大错特错。把它量化的就是 <Term def="Amdahl 定律（Amdahl's Law）：估算并行加速上限的公式。设可并行部分占比为 p、处理器数为 N，则加速比 S = 1 / ((1−p) + p/N)。核心结论是：当 N 趋于无穷大，S 的上限是 1/(1−p)——由串行部分 (1−p) 决定。串行部分越大，再多核也快不过这个天花板。">Amdahl 定律</Term>。

**符号表**（本节首次出现的符号，各一行）：

- $S$：加速比（speedup）——并行后比单核快了几倍
- $p$：可并行部分占总工作的比例（$0 \le p \le 1$）
- $1-p$：必须串行、只能一个人顺序做的部分占比
- $N$：处理器（核）数

设单核跑完整个任务耗时记为 1（一个单位）。其中可并行的那部分占 $p$、串行的那部分占 $1-p$。把任务交给 $N$ 个核：串行部分谁也帮不上，仍耗 $1-p$；可并行部分被 $N$ 个核分摊，从 $p$ 缩到 $p/N$。所以总耗时是：

$$
T(N) = (1-p) + \frac{p}{N}
$$

这个式子在说：用 $N$ 个核跑，总时间 = 那段甩不掉的串行时间 $(1-p)$，加上被 $N$ 个核分摊后的并行时间 $p/N$。

加速比就是「单核时间 ÷ 多核时间」，单核时间为 1，于是：

$$
S = \frac{1}{(1-p) + \dfrac{p}{N}}
$$

这个式子在说：加速比 = 1 除以多核总耗时。分母里 $p/N$ 这一项会随核数 $N$ 增大而变小、贡献越来越少，而 $(1-p)$ 这一项岿然不动——它就是卡住分母不让它继续变小的那只手。

让核数无限多（$N \to \infty$），$p/N \to 0$，加速比逼近：

$$
S_{\max} = \frac{1}{1-p}
$$

这个式子在说：哪怕给你无穷多个核，加速比也突破不了 $1/(1-p)$ 这个天花板——而它**只由串行部分 $1-p$ 决定**。串行部分占 10%（$p=0.9$），上限就是 10 倍；串行部分占 50%（$p=0.5$），上限就只有 2 倍，加再多核都白搭。这就是「一桌菜里总有几道得一个人顺序做，再多厨师也快不过那几道的总时长」的数学版本。

## 代码对照：伪共享翻车版 vs 修复版

type D 的代码对照要把「两个方案的差异行」摆在一起看。先看伪共享的**翻车版**——两个计数器挨着声明，正好落进同一条缓存行：

```cpp
#include <atomic>
#include <cstddef>

// 翻车版：两个计数器紧挨着 → 大概率同在一条 64 字节缓存行
struct Counters {
    std::atomic<long> a{0};   // core0 反复 ++a
    std::atomic<long> b{0};   // core1 反复 ++b
    // a 和 b 相邻，几乎必然共享同一条缓存行 → false sharing
};

void worker_a(Counters& c) {
    for (long i = 0; i < 100000000; ++i) {
        c.a.fetch_add(1, std::memory_order_relaxed);  // 写 a → 整行被抢、core1 失效
    }
}

void worker_b(Counters& c) {
    for (long i = 0; i < 100000000; ++i) {
        c.b.fetch_add(1, std::memory_order_relaxed);  // 写 b → 整行被抢回、core0 失效
    }
}
```

两个线程各跑各的循环，逻辑上 `a` 和 `b` 毫不相干。可因为它们在内存里紧挨着、共享同一条缓存行，每次 `fetch_add` 都触发一次跨核的缓存行争抢——这就是 §5 动画里那场乒乓，吞吐被狠狠压低。

修复只需一个改动：用 `alignas` 把每个计数器对齐到独立的缓存行。下面是**修复版**，差异全在 `alignas` 那两行：

```cpp
#include <atomic>
#include <new>       // std::hardware_destructive_interference_size

#ifdef __cpp_lib_hardware_interference_size
constexpr std::size_t kCacheLine = std::hardware_destructive_interference_size;
#else
constexpr std::size_t kCacheLine = 64;   // 常见缓存行大小，退化默认
#endif

// 修复版：每个计数器各占一条独立缓存行 → 两核各持一行，不再乒乓
struct alignas(kCacheLine) PaddedCounter {
    std::atomic<long> v{0};
};

struct Counters {
    alignas(kCacheLine) PaddedCounter a;   // ← 差异行：对齐到独立缓存行
    alignas(kCacheLine) PaddedCounter b;   // ← 差异行：对齐到独立缓存行
};
```

<Callout type="tip">
  差异就在那两行 `alignas(kCacheLine)`。它强制 `a` 和 `b` 各自起始于一条新的缓存行，于是
  core0 写 `a`、core1 写 `b` 落在**不同**的行上，互不失效——乒乓消失，吞吐常能回升数倍。C++17
  起 `<new>` 提供 `std::hardware_destructive_interference_size`，正是「为了避免伪共享、两个对象至少该隔多远」的标准答案；编译器不支持时退回常见的
  64 字节。代价是每个计数器多占了些填充字节——用一点内存换掉一场乒乓，几乎总是划算的。
</Callout>

## 代码对照：数据划分的 per-thread 累加

数据并行里「让每个线程在自己的数据上独立干活、最后再合并」的典型写法，是给每个线程一个**私有**的局部累加器，全程不碰共享状态，循环跑完才汇总一次：

```cpp
#include <numeric>
#include <thread>
#include <vector>

// 数据划分：把 [first,last) 切成 nthreads 段，每段一个线程算私有局部和，最后合并
long parallel_sum(const std::vector<long>& data) {
    const std::size_t n = data.size();
    if (n == 0) return 0;
    // 按硬件核数定线程数：避免超额订阅（线程数远超核数反而更慢）
    unsigned hw = std::thread::hardware_concurrency();
    unsigned nthreads = hw ? hw : 2;
    if (nthreads > n) nthreads = static_cast<unsigned>(n);

    std::vector<long> partials(nthreads, 0);   // 每线程一个私有累加器槽
    std::vector<std::thread> pool;
    const std::size_t chunk = n / nthreads;

    for (unsigned t = 0; t < nthreads; ++t) {
        std::size_t b = t * chunk;
        std::size_t e = (t == nthreads - 1) ? n : b + chunk;
        // 每个线程只写自己那个 partials[t]——但小心：相邻槽可能 false sharing！
        pool.emplace_back([&, b, e, t] {
            partials[t] = std::accumulate(data.begin() + b, data.begin() + e, 0L);
        });
    }
    for (auto& th : pool) th.join();
    return std::accumulate(partials.begin(), partials.end(), 0L);  // 合并
}
```

注意两处设计：① 线程数取 `hardware_concurrency()`，避免超额订阅；② 每个线程只写自己的 `partials[t]`、全程不加锁。

<Callout type="trap">
  上面的 `partials`
  看似优雅，却埋了一个伪共享的雷：`partials[0]`、`partials[1]`……是连续的
  `long`，多个相邻槽很可能落在**同一条缓存行**上，于是不同线程写各自的槽又触发了缓存乒乓。这正是
  §5 那张图讲的坑，只是换了个马甲。修法和上一段一样：把每个槽 `alignas`
  /填充到独立缓存行，或让线程先在**自己栈上的局部变量**里累加、循环结束才写一次
  `partials[t]`（写一次的开销远小于循环里反复写）。
</Callout>

## 容易踩的坑

<Callout type="trap">
  **现象**：把单线程累加改成多线程后，吞吐不升反降，`perf`
  显示缓存未命中暴增。**原因**：**false
  sharing（伪共享）**——多个线程各写各的变量，但这些变量挤在同一条缓存行上，每次写都让整条行在核间反复失效、传输（缓存乒乓），明明没共享却互相拖慢。**修法**：用
  `alignas(std::hardware_destructive_interference_size)` /
  填充把各线程频繁写的数据对齐到**不同缓存行**；或让线程先在局部变量里算、最后写一次共享槽。
</Callout>

<Callout type="trap">
  **现象**：开了远超核数的线程（如 8 核机器开 200 个线程），程序比开 8
  个还慢、CPU
  看着满载却没干多少活。**原因**：**超额订阅（oversubscription）**——线程数远超核数，多出来的线程不能真并行，只让操作系统在它们之间疲于上下文切换，切换开销吃光了并行收益。**修法**：线程数按
  `std::thread::hardware_concurrency()`
  给的核数来定（或用线程池复用固定数量线程），别一个任务开一个线程。
</Callout>

<Callout type="trap">
  **现象**：以为「核翻一倍、速度就翻一倍」，加了一堆核却发现加速比早早就不动了。**原因**：**误读
  Amdahl 定律**——只盯着可并行部分，忽略了那段必须串行的 $(1-p)$。加速上限是
  $1/(1-p)$，由串行部分卡死：串行占 10% 上限就只有 10
  倍，加再多核也撞这堵墙。**修法**：先用 Amdahl
  估上限、判断「值不值得加核」，把优化重心放在**缩小串行部分**（减锁、减依赖）上，而不是一味堆核。
</Callout>

<Callout type="trap">
  **（场景选错方案）现象**：把一个「每一步都依赖上一步结果」的强顺序任务（如逐帧依赖的状态递推）硬塞进数据并行，结果线程之间反复等待、加锁，比单线程还慢。**原因**：**给强顺序依赖的任务选错了划分策略**——数据并行要求各份**互不依赖**才能各算各的；这种前后强依赖的活根本切不开。**修法**：换思路——若是流式的连续数据，改用**流水线**让不同阶段重叠；若依赖确实无法打破，就承认它该串行（Amdahl
  里的 $1-p$），把并行用在别处。
</Callout>

## 小结

- **任务划分三策略**：数据并行（同一份处理切成 N 份各算各的）、递归划分（分治再并行，如并行快排）、流水线（按工序分阶段、数据流过）——没有最优，只有最配场景的那一种
- **false sharing（伪共享）**：两线程各写各的变量却挤在同一条缓存行，触发缓存乒乓、互相拖慢；用 `alignas` / 填充对齐到不同缓存行根治
- **超额订阅**：线程数远超核数 → 上下文切换吃光并行收益；按 `hardware_concurrency()` 定线程数
- **Amdahl 定律** $S = 1/((1-p)+p/N)$：加速上限 $1/(1-p)$ 由串行部分卡死——串行占 10% 上限就 10 倍，再多核也撞墙
- **并发设计总则**：让线程多在自己的数据上独立干活、少碰共享；异常路径上也要给「交结果」一个交代

**练习 1（场景选型题）** 下面三个任务，各该用哪种任务划分策略（数据并行 / 递归划分 / 流水线）？说明理由。
(a) 给一张 4000×4000 的图片整体调亮：每个像素独立地乘上一个系数。
(b) 一个实时摄像头流：每帧要依次经过「去噪 → 目标检测 → 画框叠加」三道工序，帧源源不断到来。
(c) 对一个一千万元素的乱序数组做排序，且希望充分利用多核。

<Answer>

(a) **数据并行（数据划分）**。每个像素的处理完全独立、互不依赖，是典型「同样的事、数据不同」——把图片切成若干区块，每个线程处理一块即可，几乎线性可伸缩。

(b) **流水线（任务并行）**。三道工序各不相同、且对每帧有前后顺序（去噪完才能检测），但**帧是源源不断的**：让三个线程分别专做一道工序，第 1 帧进入检测时第 2 帧已能开始去噪，各阶段重叠起来。这是「不同的事分给不同线程」的任务并行，硬上数据并行（同一帧切块）反而会被工序间的依赖卡住。

(c) **递归划分（分治）**。排序天然可分治——并行快排/归并：选基准把数组分两半，左右两半交给不同线程各自再递归排序。数据量大、能不断二分，正适合递归划分。

</Answer>

**练习 2（Amdahl 计算题）** 某任务 90% 的工作可并行、10% 必须串行（即 $p=0.9$）。
(a) 用 16 个核，加速比约是多少？
(b) 核数无限多时，加速比的上限是多少？
(c) 若想把加速上限提到 20 倍，串行部分最多只能占多少？

<Answer>

(a) 代入 $S = \dfrac{1}{(1-p)+p/N}$，$p=0.9$、$N=16$：分母 $= 0.1 + 0.9/16 = 0.1 + 0.05625 = 0.15625$，所以 $S = 1/0.15625 \approx 6.4$ 倍。注意——16 个核只换来约 6.4 倍，远不是 16 倍，被那 10% 串行部分拖住了。

(b) 上限 $S_{\max} = \dfrac{1}{1-p} = \dfrac{1}{0.1} = 10$ 倍。哪怕给你无穷多核也突破不了 10 倍。

(c) 要 $S_{\max} = 1/(1-p) \ge 20$，即 $1-p \le 1/20 = 0.05$。所以串行部分最多只能占 **5%**（即 $p \ge 0.95$）。这说明：想提高可伸缩性，关键是**缩小串行部分**，而不是堆核。

</Answer>

**练习 3（改 Demo / 分析题）** 打开本章 `<FalseSharingDiagram />`，单步到第 ⑤ 拍（「性能暴跌」）。请用自己的话说清：这一拍里两个线程各自在做什么、它们共享了哪个数据吗、性能为什么还是暴跌？再说出第 ⑥⑦ 拍的修复改了什么，使乒乓消失。

<Answer>

第 ⑤ 拍：core0 在反复写变量 `a`、core1 在反复写变量 `b`，两个线程**各写各的、并不共享任何数据**（`a` 和 `b` 是两个不同的变量）。性能仍暴跌的原因是：`a` 和 `b` 恰好落在**同一条缓存行**上，而 CPU 在核间以整条缓存行为单位搬数据——core0 写 `a` 会把整行抢到 core0 并让 core1 的副本失效，core1 写 `b` 又把整行抢回去让 core0 失效，整条行像乒乓球一样在两核间反复弹来弹去（缓存乒乓）。这就是 false sharing：没有真正共享数据，却付出了共享的代价。

第 ⑥⑦ 拍的修复：用 `alignas(64)` /填充把 `a`、`b` 各对齐到**两条独立的缓存行** `La`、`Lb`。此后 core0 独占 `La`、core1 独占 `Lb`，一个核写自己的行不再让另一个核失效——缓存行不再跨核争抢，乒乓消失，吞吐回升。代价只是各多占了一点填充字节。

</Answer>

<Glossary>

<GlossaryItem term="任务划分（task partitioning）" id="任务划分">
  把一份工作拆开、分给多个线程去做的方式。怎么拆，决定了能不能并行、并行得好不好——拆错了，线程之间反复等待、反复传数据，人再多也快不起来。常见三条路：**数据划分**（同一份数据切成
  N
  份各算各的）、**递归划分**（分治，拆成子问题再并行）、**流水线**（按工序分阶段，数据流过）。像后厨分活：是按菜量分给几个厨师，还是按工序（洗、切、炒）分给不同人。详见本章「任务划分三策略」一节。
</GlossaryItem>

<GlossaryItem term="数据并行（数据划分）">
  把同一份工作的数据切成 N
  份，每个线程独立处理其中一份——做的是**同样的事，只是数据不同**。适合数据量大、各份处理彼此不依赖的场景，比如把大数组分段求和、把图片分块逐像素处理。像把「同一道菜做二十份」分给四个厨师，每人做五份。与任务并行（各线程做不同的事）相对。详见本章「任务划分三策略」一节。
</GlossaryItem>

<GlossaryItem term="任务并行（流水线）" id="任务并行">
  把**不同的**工作（不同的工序/任务）分给不同线程，各线程做的事不一样。流水线是典型：按工序拆成若干阶段，数据依次流过，每个阶段一个线程专做一种工序（像工厂流水线：第一个工位干完交给第二个工位）。适合数据连续到达、工序固定的场景，比如视频的解码→滤镜→编码。与数据并行（各线程做同样的事、只是数据不同）相对。详见本章「任务划分三策略」一节。
</GlossaryItem>

<GlossaryItem term="false sharing（伪共享）">
  两个线程各自访问**不同的**变量，本不该互相影响；可这两个变量恰好落在**同一条缓存行**上，于是任一线程写自己那个变量，都会让整条缓存行在两个核之间来回失效、传输，两个线程互相拖慢——明明没有真正共享数据，却付出了共享的代价，所以叫「伪」共享。像两个厨师各写各的便签，但两张便签写在同一张必须来回传的纸上，纸被反复抢来抢去。用
  `alignas` /
  填充把变量对齐到不同缓存行就能根治。详见本章「影响并发性能的因素」一节。
</GlossaryItem>

<GlossaryItem term="缓存行（cache line）">
  CPU 在缓存与内存之间、以及核与核之间搬运数据的**最小单位**，通常 64
  字节。读写任何一个变量，都会把它所在的整条缓存行一起搬动。所以两个变量若挤在同一条缓存行里，写其中一个就会牵动另一个——这正是
  false sharing 的物理根源。详见本章「影响并发性能的因素」一节。
</GlossaryItem>

<GlossaryItem term="缓存乒乓（cache ping-pong）">
  同一条缓存行被多个核反复争抢、来回传输失效的现象。每次一个核写了这条行，其他核手里的副本就作废、得重新去取，于是缓存行像乒乓球一样在核间弹来弹去，开销巨大。false
  sharing
  是它最常见的诱因——两个线程各写各的变量，却因共行触发乒乓。详见本章「影响并发性能的因素」一节。
</GlossaryItem>

<GlossaryItem term="数据接近度（data proximity / locality）" id="数据接近度">
  一个线程要用到的数据，是否在内存里挨得近、能否被缓存一次取出。挨得近则缓存命中率高、快；散落各处则频繁缓存未命中、慢。并发性能调优常做两件事：把一个线程自己要用的数据安排得彼此靠近（提高命中率），同时又和别的线程的数据**分开**（避免
  false sharing）。详见本章「影响并发性能的因素」一节。
</GlossaryItem>

<GlossaryItem term="超额订阅（oversubscription）">
  创建的线程数**远多于**硬件能真正并行的核数。多出来的线程并不能真的同时跑，反而让操作系统在它们之间频繁切换上下文（保存/恢复寄存器、刷新缓存），这些切换开销可能吃掉并行带来的全部收益，使程序不升反降。对策：线程数按
  `std::thread::hardware_concurrency()`
  给的核数来定，或用线程池复用固定数量的线程。详见本章「影响并发性能的因素」一节。
</GlossaryItem>

<GlossaryItem term="可伸缩性（scalability）" id="可伸缩性">
  往程序里加更多处理器（核）时，性能能跟着提升多少。理想是加一倍核就快一倍（线性可伸缩）；现实里因为总有必须串行的部分，加速比会越来越逼近一个上限、加核的收益递减。一段代码「可伸缩性好不好」，问的就是它多给核能不能多换来速度。Amdahl
  定律就是量化这个上限的公式。详见本章「可伸缩性与 Amdahl 定律」一节。
</GlossaryItem>

<GlossaryItem term="Amdahl 定律（Amdahl's Law）" id="Amdahl 定律">
  估算并行加速上限的公式。设可并行部分占比为 $p$、处理器数为 $N$，则加速比 $S =
  \dfrac{1}
  {1 - p + p / N}$。核心结论是：当核数趋于无穷大，加速比的上限是 $\dfrac{1}
  {1 - p}$——**只由串行部分 $(1-p)$
  决定**。串行部分越大，再多核也快不过这个天花板（串行占 10% 上限就 10 倍、占
  50% 就只有 2
  倍）。像一桌菜里总有几道必须一个人顺序做，这几道的总时长，再多厨师也省不掉。详见本章「可伸缩性与
  Amdahl 定律」一节。
</GlossaryItem>

</Glossary>
