## 后厨那些「时有时无」的乱子

整本书你学会了让多个厨师高效地一起干活。可真到了忙时，后厨偶尔会冒出怪事：两个厨师同时改同一张单子，菜出错了；两人各攥着对方要的工具僵在原地，谁也动不了；又或者两人在窄过道里反复对向让路，一起往左、一起往右，谁也过不去。这一章就回答：这些乱子怎么分门别类，又怎么把它们揪出来？

最让人头疼的是：这些乱子时有时无。你忙的时候它犯，你一停下来盯着看，它又乖乖不犯了——你越想抓现行，它越不出现。靠「多跑几次碰运气」根本不可靠。

所以得换思路：与其等乱子自己冒头，不如**故意把后厨逼到最忙**，让乱子无所遁形；再给后厨装一套**监控**，自动盯着「有没有两个人同时碰了同一份料、却没打招呼」。这一章讲的就是怎么逼、怎么监控、以及看到症状怎么对号入座。

## 先认脸：并发 bug 分门别类

调试的第一步不是动手改，而是**看症状对号入座**——不同的并发 bug 长相不同，查法也不同。下面这张速查图把全书出现过的几类摆在一起，每类配「症状 / 常见成因 / 往哪查」，先扫一眼记住它们的脸：

逐个点名（都呼应前面各章，本章把它们统一当「要测试、要调试的 bug 源」）：

<Term def="条件竞争（race condition，又称竞态条件）：程序的正确性依赖多个线程操作的相对时序，而这个时序又无法保证。表现为结果时对时错、重跑会变。后厨版：两个厨师同时去改同一张单子，谁后写谁覆盖。第 3 章讲过它，本章把它当作要测试调试的一类 bug。">
  条件竞争
</Term>
：正确性依赖线程之间的相对时序，而时序不受你控制——结果时对时错、换台机器或加点负载就变样。它常常是更具体问题（如数据竞争、破坏不变量）的统称。

<Term def="死锁（deadlock）：多个线程各自持有一把锁、又都在等对方手里的锁，形成环形等待，谁也不肯先放手，于是全部卡死、毫无进展。CPU 通常很闲（线程被挂起）。后厨版：两厨师各攥一件对方要的工具僵在原地。第 3 章讲过其成因与「统一加锁顺序」的破法。">
  死锁
</Term>
：多个线程各持一锁、又互等对方手里的锁，形成环形等待——**全部冻结不动**，CPU
通常很闲。

<Term def="活锁（livelock）：多个线程并没有被锁住，而是都在不停地动（检测到冲突就退避、重试），但因为同步动作总是同时发生，反复礼让、谁也没真正前进。和死锁的关键区别：死锁是都不动、CPU 闲；活锁是都在动、CPU 忙却零进展。后厨版：两人在窄过道反复对向让路，一起左、一起右，谁也过不去。">
  活锁
</Term>
：线程没被锁住、反而都在忙——检测到冲突就退避重试，可退避动作总同时发生，于是反复礼让、**谁也没真正前进**。它和死锁是这一章重点要分清的一对，下一节专门对比。

<Term def="数据竞争（data race）：两个或更多线程并发访问同一块内存，其中至少有一个是写，且这些访问之间没有 happens-before 关系（没有任何同步把它们排出先后）。在 C++ 里数据竞争是未定义行为。它是「条件竞争」最底层、最常见的一种，能被 ThreadSanitizer 自动抓出。第 5、6 章讲过原子类型与 happens-before。">
  数据竞争
</Term>
：两线程并发访问同一内存、至少一个是写、且无任何同步把它们排出先后——在 C++
里这是**未定义行为**。它是最底层、最该用工具自动抓的一类（见后文 TSan）。

<Term def="破坏不变量（broken invariant）：一个对象本应始终满足某个「不变量」（如「链表的 size 等于结点数」「这两个字段要么都更新、要么都不更新」）。当一个线程在中途（只更新了一半）时，另一个线程看到了这个半成品的非法状态，或异常打断了更新，就破坏了不变量。后厨版：半成品的菜被当成成品端走。">
  破坏不变量
</Term>
：对象本该始终满足某条「不变量」，可一个线程更新到一半的「半成品」状态被别的线程看见（或被异常打断），不变量就破了——表现为断言失败、链表断裂这类非法状态。

记住这五张脸，遇到诡异现象就能快速缩小范围。下面把最容易混的死锁与活锁单独拎出来对比。

## 一对孪生陷阱：死锁 vs 活锁

死锁和活锁都让程序「卡住、不出结果」，新手极容易混。它们的本质区别只有一句话：**死锁是「都不动」（线程被挂起、停在等锁上），活锁是「都在动、却零进展」（线程不停退避重试，反复礼让谁也没前进）**——一个不动、一个白动。下面这张并排对比图把两者演给你看，左死锁冻结、右活锁来回弹——先猜一猜，再单步：

> 猜一猜：死锁和活锁，哪个 CPU 占用率高？

<DeadlockVsLivelockDiagram />

盯住两边的**进度条**：无论死锁还是活锁，进度都死死停在 0%——这是它们的共同点（程序卡死）。再看两边线程的**动静**：左边死锁的两个线程在环闭合后定格不动，CPU 多半被挂起、占用很低；右边活锁的两个线程一直在左右跳（退避、重试），CPU 反而跑满。所以猜一猜的答案是：**活锁 CPU 占用率更高**（忙等空转），死锁通常占用低。这也给你一条实用线索——「CPU 闲着，程序却卡住」常常是死锁的信号；「CPU 跑满，却没有任何产出」则更像活锁。

死锁的破法第 3 章讲过：让所有线程按同一固定顺序加锁，环就成不了。活锁的破法是**打破「总是同时退避」**——给退避加一点随机化（随机退避时间），让两个线程错开节奏，就不会永远撞在一起。

## 怎么把乱子揪出来：审查、结构化测试、消毒器

定位并发 bug 有三件趁手的工具，从「人看」到「机器自动抓」：

第一是**代码审查清单**：人脑过一遍——这块共享数据被几个线程碰？都加锁了吗？加锁顺序一致吗？某个函数抛异常时，它该交的结果交了没（不然等它的线程会永远等下去）？审查抓的是「设计层」的疏漏，便宜又有效。

第二是**结构化测试**：并发 bug 难测，但别一上来就开一堆线程瞎跑。正确姿势是分两步——**先把逻辑单线程测对**（不开线程，纯验证算法本身），逻辑都过了，**再开多线程压测**并发部分。这样一旦挂了，你立刻知道是「逻辑错」还是「并发错」，而不必在一团乱麻里猜。代码骨架见 §6。

第三是**消毒器**，本章主角 <Term def="ThreadSanitizer（TSan）：编译器（GCC / Clang）自带的动态分析工具，用 -fsanitize=thread 开启。它在每次内存访问处插桩，记录访问的线程、锁集和向量时钟，再用 happens-before 关系判断两次访问之间有没有同步；一旦发现两次访问无 happens-before、又至少一个是写，就判定为数据竞争并报告冲突的两处代码。它把「时有时无」的数据竞争变成「每跑必抓」。后厨版：给后厨装监控，自动抓「两人同时碰一份料却没打招呼」。">ThreadSanitizer（TSan）</Term>。它不靠运气复现，而是给程序装上「监控」自动抓数据竞争。它怎么做到的？下面这张动画拆给你看——核心是它靠 <Term def="happens-before（先于发生）：一种描述两个操作之间「有明确先后保证」的关系。若操作 A happens-before 操作 B，则 A 的效果一定对 B 可见。两次内存访问之间若存在 happens-before（被某种同步排出了先后），就不算数据竞争；若毫无 happens-before、又至少一个是写，就是数据竞争。第 6 章详细讲过它。">happens-before</Term> 分析来判定（呼应第 6 章）：

<TsanDetectionDiagram />

读懂第 ③④ 拍就懂了 TSan 的精髓：它把「线程 A 写 x」和「线程 B 读 x」两条访问记录摆到一起，问「它们之间有没有 happens-before 关系把它们排出先后」？如果没有、又至少一个是写，就判定数据竞争并报告冲突的两处代码——把一个「时有时无」的 bug 变成「每跑必抓」的确定结果。代价是程序变慢（约 5～15 倍）、内存涨，所以 TSan 在测试 / CI 里跑，不进生产。

## 写代码时就为「可测试」着想

最省事的调试，是让 bug 一开始就难以藏身。<Term def="为可测试性设计（design for testability）：在写并发代码时就让它好测——核心是尽量减少共享状态、把对并发设施（线程、锁、时钟）的依赖抽成可替换/可注入的接口，并让纯逻辑能脱离线程单独跑、先在单线程下测对。共享越少、依赖越能替换，要测的并发面就越小。">为可测试性设计</Term>有几条朴素原则：

- **减少共享状态**：能不共享就不共享。让线程多在自己的数据上独立干活、最后再合并，需要同步的地方就少，能出竞态的地方也就少。
- **把并发依赖抽出来、可注入 / 可替换**：别把「开线程」「取当前时间」「调度」这些硬编码进业务逻辑。抽成接口，测试时换成可控的假实现（比如让「时钟」按你的剧本走），就能确定性地复现特定时序。
- **让逻辑能单线程先测对**：把「算什么」和「怎么并行地算」拆开。纯逻辑（如「累加器加 1」「容器插入」）应该能脱离线程、在单线程下被穷尽测试。逻辑都对了，并发测试才只需盯「同步」这一层。

这三条的共同主线：**缩小要测的并发面**。共享越少、依赖越能替换、逻辑越能单独验证，留给并发 bug 的藏身之处就越小。

## 故意把它逼出来：压力测试与 heisenbug

并发 bug 的偶发性，是测试它最大的敌人。你会遇到一种气人的现象——<Term def="heisenbug（海森堡 bug）：一旦你试图观察它（加打印语句、挂调试器、单步执行），它就因为时序被改变而消失，让你抓不到现行。名字取自海森堡测不准原理：观察行为本身改变了被观察的系统。并发 bug 常是 heisenbug，因为加 print/断点会拖慢或重排线程时序，恰好让原本会撞上的竞争窗口错开。对策是别靠手动复现，改用 TSan（不改语义地插桩）和压力测试。">heisenbug</Term>：你一加 `printf`、一挂调试器单步，bug 就神秘消失，因为这些动作改变了线程的时序，恰好让原本会撞上的竞争窗口错开了。

对付偶发 bug 的正面手段是 <Term def="压力测试（stress testing）：故意把程序逼到极端繁忙的状态来逼出偶发的并发 bug——开远超 CPU 核数的线程、跑海量迭代、并主动扰动时序（如随机让出、随机延时），制造高竞争环境，让原本极小概率才撞上的竞争窗口频繁发生。配合 ThreadSanitizer 一起跑效果最好。后厨版：故意把后厨逼到最忙，让乱子无所遁形。">压力测试</Term>：与其守株待兔，不如主动制造极端繁忙——开远超核数的线程、跑海量迭代、再用随机让出 / 延时主动扰动时序，把那个「极小概率才撞上」的竞争窗口逼成「频繁撞上」。压力测试 + TSan 是黄金搭档：压力测试负责把窗口撞开，TSan 负责把撞上的那一下确定地抓住。

## 代码对照：有数据竞争的版本 vs 加锁修复版

type D 的代码对照要把「前后两个方案的差异行」摆在一起看。先看**有数据竞争的翻车版**——四个线程同时对一个无保护的全局变量做读-改-写：

```cpp
#include <thread>
#include <vector>

// 有数据竞争：多个线程并发对同一个 int 做读-改-写，且无任何同步
int g_counter = 0;  // 共享、无保护

void add_many(int times) {
    for (int i = 0; i < times; ++i) {
        ++g_counter;  // 读 g_counter、加 1、写回——非原子，三步可被打断
    }
}

int run_unsafe() {
    std::vector<std::thread> pool;
    for (int t = 0; t < 4; ++t) {
        pool.emplace_back(add_many, 100000);  // 4 个线程同时 ++ 同一个变量
    }
    for (auto& th : pool) th.join();
    return g_counter;  // 期望 400000，实际常常更小：丢了若干次自增
}
```

`++g_counter` 看似一步，其实是「读出、加一、写回」三步。两个线程可能都读到同一个旧值、各自加一、又写回同一个新值——一次自增就这么丢了。结果时对时错、重跑会变，正是条件竞争里最底层的**数据竞争**。

这种 bug 不该靠肉眼盯、靠运气复现，而该交给 TSan。用 `-fsanitize=thread` 编译运行后，TSan 会确定地报出类似：

```text
WARNING: ThreadSanitizer: data race (pid=12345)
  Write of size 4 at 0x... by thread T2:
    #0 add_many(int)  race.cpp:8
  Previous write of size 4 at 0x... by thread T1:
    #0 add_many(int)  race.cpp:8
  Location is global 'g_counter' of size 4
```

它直接点名冲突的两处访问（都在 `race.cpp:8` 的 `++g_counter`）和涉及的两个线程——把「时有时无」变成「每跑必抓」。修复只需一处：给共享变量配一把互斥量，进临界区前加锁。下面是**修复版**，差异全在 `g_mtx` 和 `lock_guard` 两行：

```cpp
#include <mutex>
#include <thread>
#include <vector>

int g_counter = 0;     // 仍是共享变量
std::mutex g_mtx;      // ← 差异：用一把互斥量保护它

void add_many_safe(int times) {
    for (int i = 0; i < times; ++i) {
        std::lock_guard<std::mutex> lk(g_mtx);  // ← 差异：进临界区前加锁
        ++g_counter;                            // 受锁保护，读-改-写不再被打断
    }
}

int run_safe() {
    std::vector<std::thread> pool;
    for (int t = 0; t < 4; ++t) {
        pool.emplace_back(add_many_safe, 100000);
    }
    for (auto& th : pool) th.join();
    return g_counter;  // 稳定得到 400000；TSan 也不再报告数据竞争
}
```

<Callout type="tip">
  差异只有两行：声明 `std::mutex g_mtx` 和在自增前加 `std::lock_guard`。加锁让「读-改-写」三步重新变得不可打断，多个线程对
  `g_counter` 的访问被排出先后（建立了 happens-before），数据竞争消失——再用
  `-fsanitize=thread` 跑，TSan 也不再报警。注意：本例用锁修复最直白；若只是计数，更轻量的做法是把
  `g_counter` 换成 `std::atomic<int>` 并用 `fetch_add`（无需锁，见第 5
  章）。两条路都能消除数据竞争，区别只是开销与适用场景。
</Callout>

## 代码对照：结构化测试骨架（先逻辑、再压测）

光修好还不够——要有测试守住它，且测试本身要「结构化」：**先单线程测逻辑、再多线程压测**。下面是一个可直接编译运行的骨架（被测对象用原子计数器，逻辑简单到一眼能验对）：

```cpp
#include <atomic>
#include <cassert>
#include <thread>
#include <vector>

// 被测对象：一个线程安全计数器（用原子，逻辑很简单）
struct Counter {
    std::atomic<int> n{0};
    void inc() { n.fetch_add(1, std::memory_order_relaxed); }
    int get() const { return n.load(std::memory_order_relaxed); }
};

// ① 先单线程测「逻辑」对不对：不开线程，纯验证 inc/get 的行为
void test_logic_single_thread() {
    Counter c;
    assert(c.get() == 0);
    c.inc();
    c.inc();
    assert(c.get() == 2);  // 逻辑错（如忘了 +1）在这里就该挂，省得去并发里猜
}
```

第一步刻意**不开线程**：如果连单线程下 `inc` 两次都不等于 2，那是逻辑写错了，跟并发毫无关系——在这一步揪出来，比在多线程的一团乱麻里猜要省心得多。逻辑过了，第二步才上**高压**，逼出偶发的并发 bug：

```cpp
#include <atomic>
#include <cassert>
#include <thread>
#include <vector>

struct Counter {  // 同上一段被测对象：线程安全计数器
    std::atomic<int> n{0};
    void inc() { n.fetch_add(1, std::memory_order_relaxed); }
    int get() const { return n.load(std::memory_order_relaxed); }
};

// ② 逻辑过了，再「压力测试」并发：开远超核数的线程、海量迭代逼出偶发 bug
void test_concurrency_stress() {
    constexpr int kThreads = 64;        // 故意远超核数，制造高竞争
    constexpr int kPerThread = 100000;  // 海量迭代，放大偶发窗口
    Counter c;
    std::vector<std::thread> pool;
    for (int t = 0; t < kThreads; ++t) {
        pool.emplace_back([&c] {
            for (int i = 0; i < kPerThread; ++i) c.inc();
        });
    }
    for (auto& th : pool) th.join();
    assert(c.get() == kThreads * kPerThread);  // 少一个都说明有竞争丢更新
}
```

第二步故意开 64 个线程（远超核数）、每个跑十万次——制造高竞争把偶发窗口放大。最后断言总数严格相等：**少一个都说明有竞争丢了更新**。

<Callout type="tip">
  把这两个测试用 `-fsanitize=thread` 一起编译运行（如 `g++ -std=c++17
  -fsanitize=thread -g test.cpp`），就是「压力测试 +
  消毒器」的黄金搭档：压力测试负责把竞争窗口频繁撞开，TSan
  负责把撞上的那一下确定地抓住、报到具体代码行。注意 TSan
  会让程序慢几倍、吃更多内存，所以它在 CI / 测试环境跑，不进生产构建。
</Callout>

## 容易踩的坑

<Callout type="trap">
  **现象**：偶发的并发 bug，一加 `printf`
  调试或挂上调试器单步，就再也复现不出来了。**原因**：**heisenbug**——打印 / 断点
  /
  单步改变了线程的时序，恰好把原本会撞上的竞争窗口错开了，观察行为本身改变了被观察的程序。**修法**：别靠手动加
  print / 断点复现。改用
  **ThreadSanitizer**（不改语义地插桩，把数据竞争确定地抓出）
  和**压力测试**（高压把窗口撞开）；要记录就用对时序影响极小的手段（如环形缓冲事后转储）。
</Callout>

<Callout type="trap">
  **现象**：并发代码跑了一次、甚至跑了几十次都没出错，就认定它是对的、合入了主线。**原因**：**「跑一次过
  ≠ 正确」**——并发 bug
  是偶发的，是否触发取决于线程的相对时序；测试环境（核少、负载低）下竞争窗口可能根本撞不上，到了生产高负载才爆。**修法**：并发正确性不能靠「碰巧没错」来证明。必须**高压多次**（压力测试：远超核数的线程
  + 海量迭代 + 扰动时序）并配 **TSan**
  自动抓；能形式化推理的关键路径（如内存序）还要人脑过一遍。
</Callout>

<Callout type="trap">
  **（场景选错 /
  判断坑）现象**：程序卡住不出结果，开发者一口咬定是「死锁」，于是埋头去查加锁顺序、找环形等待——可怎么查都没有锁形成的环。**原因**：**把活锁误判成了死锁**。两者都「卡住」，但死锁是线程都被挂起、**CPU
  很闲**；活锁是线程都在忙着退避重试、**CPU
  跑满却零进展**。盯着锁找环，方向就错了。**修法**：先看 **CPU
  占用**这条关键线索——CPU 闲着卡住→多半死锁（查加锁顺序）；CPU
  跑满却无产出→多半活锁（查退避 /
  重试逻辑，给退避加随机化错开节奏）。对照本章「死锁 vs
  活锁」图先分清脸，再对症下药。
</Callout>

## 小结

- **五类并发 bug 的脸**：竞态（结果时对时错）、死锁（卡死·CPU 闲）、活锁（卡死·CPU 忙）、数据竞争（偶发脏数据，含写且无 happens-before）、破坏不变量（半成品被看见）——看症状对号入座再查
- **死锁 vs 活锁**：都卡死、进度归零；死锁是「都不动、CPU 闲」，活锁是「都在动、CPU 忙却零进展」。CPU 占用是分清两者最快的线索
- **三件定位工具**：代码审查（抓设计疏漏）、结构化测试（先单线程测逻辑、再多线程压测）、ThreadSanitizer（靠 happens-before 分析自动抓数据竞争，每跑必抓）
- **为可测试性设计**：减少共享、把并发依赖抽成可注入 / 可替换、让逻辑能单线程先测对——缩小要测的并发面
- **压力测试 + TSan 是黄金搭档**：高压（远超核数线程、海量迭代、扰动时序）把偶发窗口撞开，TSan 确定地抓住；别靠加 print 手动复现（会触发 heisenbug），更别拿「跑一次过」当正确

**练习 1（症状 → 判断是哪类 bug · 选型题）** 下面四个现象，各最可能是哪类并发 bug（竞态 / 死锁 / 活锁 / 数据竞争 / 破坏不变量）？说明判断依据。
(a) 程序运行中突然完全卡住、不再有任何输出，此时 CPU 占用几乎为 0。
(b) 一个并行求和程序，结果每次跑都不一样，有时对、有时偏小；用 `-fsanitize=thread` 一编译运行就报 data race。
(c) 程序卡住不出结果，但 CPU 四个核全部跑满 100%，风扇狂转。
(d) 一个并发链表，偶尔遍历到一半崩溃，检查发现某个结点的 `next` 指向了已被释放的内存，而 `size` 字段和实际结点数对不上。

<Answer>

(a) **死锁**。判断依据：程序卡死 + **CPU 占用接近 0**——线程都被挂起、停在等锁上，谁也不动。下一步去查是不是有多个线程各持一锁、又互等对方的锁（环形等待），以及加锁顺序是否一致。

(b) **数据竞争**（属于条件竞争里最底层的一种）。判断依据：结果时对时错、重跑会变（条件竞争的典型症状）+ **TSan 直接报 data race**（确凿证据：存在无 happens-before 的并发访问且至少一个是写）。修法是给共享累加器加锁或改用原子/per-thread 局部累加再合并。

(c) **活锁**。判断依据：程序卡死 + **CPU 全核跑满**——线程没被挂起，反而都在忙着退避 / 重试，却因为动作总同时发生而谁也没前进。和 (a) 的死锁正好相反：死锁 CPU 闲、活锁 CPU 忙。修法是给退避加随机化，错开两个线程的重试节奏。

(d) **破坏不变量**。判断依据：对象停在「半成品 / 非法」状态——`next` 指向已释放内存、`size` 与实际结点数不符，说明某次「更新多个相关字段」的操作只做了一半就被别的线程看见（或被异常打断），不变量（如「size == 结点数」「next 指向有效结点」）被破坏。修法是让整组相关更新落在同一个临界区里原子地完成，并保证异常路径回滚到一致状态。

</Answer>

**练习 2（方法论判断题）** 同事写了个无锁队列，本地开 4 个线程跑了 100 次测试全过，就提 PR 说「测过了、没问题」。请指出这个测试的两个不足，并给出更靠谱的测试方案。

<Answer>

两个不足：

1. **「跑过 = 正确」是错觉**：并发 bug 偶发，是否触发取决于线程时序。本地 4 线程、低负载下竞争窗口可能根本撞不上，100 次全过不能证明无 bug（到生产高负载才爆）。
2. **没用消毒器、也没做高压**：只靠运气复现，等于没主动去抓数据竞争；4 个线程（多半 ≤ 核数）也谈不上压力，偶发窗口被放大的程度有限。

更靠谱的方案：

- **压力测试**：开远超核数的线程（如 64+）、每线程海量迭代，并主动扰动时序（随机让出 / 随机延时），把偶发窗口频繁撞开。
- **配 ThreadSanitizer**：用 `-fsanitize=thread` 编译运行，让无 happens-before 的并发访问被「每跑必抓」，定位到具体代码行。
- **先单线程测逻辑**：把队列的纯逻辑（push/pop 的顺序语义）先在单线程下穷尽测对，再上并发压测，挂了能立刻分清是逻辑错还是并发错。
- 关键内存序路径再人脑 / 形式化推理一遍（无锁结构尤其需要）。

</Answer>

**练习 3（分析题）** 打开本章 `<DeadlockVsLivelockDiagram />`，单步到最后一拍。请用自己的话回答：死锁和活锁的「相同点」是什么、「关键区别」是什么？为什么说「CPU 占用率」是现场快速区分两者的最好线索？

<Answer>

**相同点**：两者都让程序**卡死、毫无进展**——图里两栏的进度条始终停在 0%，谁也没真正前进。从「程序的产出」看，死锁和活锁是一样的：都不出结果。

**关键区别**：死锁的线程**都不动**（被挂起，停在等锁上），活锁的线程**都在动**（不停地退避、重试），只是因为退避动作总同时发生而反复礼让、谁也没前进。一句话：死锁是「不动」，活锁是「白动」。

**为什么 CPU 占用是最好线索**：因为「动还是不动」直接反映在 CPU 上——死锁线程被挂起、不占 CPU，现场表现为**程序卡住但 CPU 几乎闲着**；活锁线程在忙等空转，现场表现为**程序卡住但 CPU 跑满**。所以一看 CPU 占用就能立刻分流：闲→往死锁方向查（加锁顺序、环形等待），忙→往活锁方向查（退避 / 重试逻辑），不必盲目地都去找锁的环。

</Answer>

<Glossary>

<GlossaryItem term="条件竞争（race condition / 竞态条件）">
  程序的正确性依赖多个线程操作的**相对时序**，而这个时序又无法保证——于是结果时对时错、重跑会变、换台机器或加点负载就翻车。它常是更具体问题（数据竞争、破坏不变量）的统称。像两个厨师同时去改同一张单子，谁后写谁覆盖。第
  3 章讲过它，本章把它当作要测试调试的一类 bug。详见本章「先认脸」一节。
</GlossaryItem>

<GlossaryItem term="死锁（deadlock）">
  多个线程各自攥着一把锁、又都在等对方手里的那把锁，形成**环形等待**，谁也不肯先放手，于是全部冻结、毫无进展。线程被挂起，CPU
  通常很闲。像两个厨师各攥一件对方要的工具僵在原地。破法：让所有线程按同一固定顺序加锁。详见本章「死锁
  vs 活锁」一节。
</GlossaryItem>

<GlossaryItem term="活锁（livelock）">
  线程并没有被锁住，而是**都在不停地动**——检测到冲突就退避、重试；可因为退避动作总是同时发生，反复礼让、谁也没真正前进。和死锁的关键区别：死锁是**都不动·CPU
  闲**，活锁是**都在动·CPU
  忙却零进展**。像两人在窄过道反复对向让路，一起左、一起右，谁也过不去。破法：给退避加随机化，错开节奏。详见本章「死锁
  vs 活锁」一节。
</GlossaryItem>

<GlossaryItem term="数据竞争（data race）">
  两个或更多线程**并发访问同一块内存**、其中**至少一个是写**、且这些访问之间**没有
  happens-before 关系**（没有任何同步把它们排出先后）。在 C++
  里数据竞争是**未定义行为**。它是条件竞争最底层、最常见的一种，能被
  ThreadSanitizer 自动抓出。第 5、6 章讲过原子类型与
  happens-before。详见本章「先认脸」一节。
</GlossaryItem>

<GlossaryItem term="破坏不变量（broken invariant）">
  一个对象本应**始终**满足某条「不变量」（如「链表的 size
  等于结点数」「这两个字段要么都更新、要么都不更新」）。当一个线程更新到**一半**的非法「半成品」状态被别的线程看见，或更新被异常打断，不变量就破了——表现为断言失败、链表断裂等。像半成品的菜被当成成品端走。修法：把整组相关更新放进同一临界区原子完成，异常路径回滚到一致状态。详见本章「先认脸」一节。
</GlossaryItem>

<GlossaryItem term="happens-before（先于发生）">
  描述两个操作之间「有明确先后保证」的关系：若 A happens-before B，则 A
  的效果一定对 B 可见。两次内存访问之间**若存在**
  happens-before（被某种同步排出了先后），就不算数据竞争；**若毫无**
  happens-before、又至少一个是写，就是数据竞争。ThreadSanitizer
  正是靠它来判定。第 6 章详细讲过。详见本章「怎么把乱子揪出来」一节。
</GlossaryItem>

<GlossaryItem term="ThreadSanitizer（TSan）">
  编译器（GCC / Clang）自带的动态分析工具，用 `-fsanitize=thread`
  开启。它在每次内存访问处**插桩**，记录访问的线程、锁集和向量时钟，再用
  happens-before 关系判断两次访问之间有没有同步；一旦发现两次访问无
  happens-before、又至少一个是写，就**判定数据竞争**并报告冲突的两处代码。它把「时有时无」的数据竞争变成「每跑必抓」。代价是程序变慢几倍、吃更多内存，所以在测试
  / CI
  里跑，不进生产。像给后厨装监控，自动抓「两人同时碰一份料却没打招呼」。详见本章「怎么把乱子揪出来」一节。
</GlossaryItem>

<GlossaryItem term="为可测试性设计（design for testability）">
  在写并发代码时就让它**好测**——核心是尽量**减少共享状态**、把对并发设施（线程、锁、时钟）的依赖抽成**可替换
  /
  可注入**的接口、并让纯逻辑能**脱离线程单独跑**、先在单线程下测对。共享越少、依赖越能替换、逻辑越能单独验证，要测的并发面就越小，bug
  的藏身之处也越少。详见本章「写代码时就为可测试着想」一节。
</GlossaryItem>

<GlossaryItem term="heisenbug（海森堡 bug）">
  一旦你试图**观察**它（加打印语句、挂调试器、单步执行），它就因为时序被改变而**消失**，让你抓不到现行。名字取自海森堡测不准原理——观察行为本身改变了被观察的系统。并发
  bug 常是 heisenbug，因为加 print /
  断点会拖慢或重排线程时序，恰好让原本会撞上的竞争窗口错开。对策：别靠手动复现，改用
  TSan（不改语义地插桩）和压力测试。详见本章「故意把它逼出来」一节。
</GlossaryItem>

<GlossaryItem term="压力测试（stress testing）">
  故意把程序逼到**极端繁忙**来逼出偶发的并发 bug——开**远超 CPU
  核数**的线程、跑**海量迭代**、再用随机让出 /
  随机延时**主动扰动时序**，制造高竞争，让原本极小概率才撞上的竞争窗口频繁发生。配
  ThreadSanitizer 一起跑效果最好：压力测试把窗口撞开、TSan
  把撞上的那一下确定抓住。像故意把后厨逼到最忙，让乱子无所遁形。详见本章「故意把它逼出来」一节。
</GlossaryItem>

</Glossary>
