## 一个人的后厨，忙不过来

想象你一个人开了家小餐厅，后厨只有你一个厨师、一个灶台。来了三桌客人，每桌点一道菜。你只能一道一道做：做完第一道，再做第二道，再做第三道。客人等得越来越急——明明灶台还空着大半时间（比如等水烧开时你只能干站着），却没人帮你。

要是能再雇两个厨师、多搭两个灶台呢？三道菜同时开火，等水烧开的工夫，另一个厨师已经在切菜了。出菜快了三倍，客人也满意了。这就是这一章要解决的问题：**怎么让一台电脑像多厨师后厨一样，把活儿同时铺开干，而不是死等一件做完再做下一件。**

没有这套本事会怎样？你的程序就永远是「一个厨师一个灶台」——下载文件时界面卡死、多核 CPU 闲着一大半、该秒开的事要转好几秒圈。这一章就带你把第一个「多厨师」程序跑起来。

## 并发与并行：一个厨师 vs 两个厨师

先把两个最容易混的词分清楚，它们是整本书的地基。

<Term def="同一段时间里有多个任务都在「推进中」，但在单核机器上任一时刻其实只做一件——靠在多个任务之间飞快地来回切换，造出「同时进行」的错觉。像一个厨师在一个灶台上轮流做几道菜。">
  并发（concurrency）
</Term>
说的是：**多个任务都在推进，但不一定真的同时在动手。**
一个厨师守着一个灶台，先翻两下 A 菜、转头切两刀 B 菜的料、再回去给 A
菜加盐……每道菜都在往前走，可任意一个瞬间，他的手其实只在一道菜上。切换够快，客人看上去就像「两道菜一起在做」。

<Term def="同一时刻真的有多个任务在同时动手——靠多个 CPU 核（或多台机器）各干各的。像两个厨师各占一个灶台，真正在同一秒一起炒菜。并行一定是并发，但并发不一定并行。">
  并行（parallelism）
</Term>
则更强：**同一时刻真的有多件事在同时发生。** 两个厨师、两个灶台，一个炒
A、一个炒 B，同一秒钟两口锅都在响。这只有「灶台够多」——也就是 CPU
有多个核——时才办得到。

一句话记牢：**并行一定是并发，但并发不一定并行。** 单核机器也能并发（一个厨师轮流切换），但要并行就得有多个核（多个厨师各占灶台）。下面这张动画把两者掰开演给你看：

看懂了上区那条扫描线在 A、B 之间来回跳——这就是并发的真相：宏观上两道菜都在推进，微观上一个灶台任一刻只做一道。下区两条灶台泳道始终并肩亮，才是真正的并行。

## 为什么要用并发：两个理由

知道了是什么，再说为什么值得费这个劲。用并发主要图两件事。

第一是 **性能**。它又分两种：一种是把一件大活儿拆成几块、几个厨师分头干，比如把一篮子菜分给三个厨师同时洗——这叫 **任务并行**；另一种是同一种处理套在一大堆数据上，比如十个厨师每人洗一筐一模一样的菜——这叫 **数据并行**。两种都是为了「人多力量大，早点出菜」。

第二是 **关注点分离**：把性质不同的活儿交给不同的「厨师」，让代码更清楚。比如一个厨师专门盯着前台招呼客人（响应界面点击），另一个厨师在后厨埋头炒大菜（跑耗时计算）。这样前台厨师永远不会因为后厨那锅炖了半小时的汤而僵在原地——界面就不会卡死。即使只有一个灶台、并不能真并行，这种「各管一摊」的拆法也让程序更好懂、更好维护。

## 进程与线程：两家餐厅 vs 一家餐厅的两个厨师

那「多个厨师」在电脑里到底是什么？这就要分清 <Term def="操作系统分配资源的基本单位，拥有一整套自己独立的内存地址空间。两个进程的内存彼此隔离，谁也读不到对方的数据，要交流得专门「打电话」（进程间通信）。像两家各自独立的餐厅。">进程（process）</Term> 和 <Term def="进程内部的执行单元，是 CPU 实际调度运行的对象。同一进程里的多个线程共享这个进程的代码、全局数据和堆，但每个线程各有自己独立的栈。像同一家餐厅里的多个厨师。">线程（thread）</Term>。

两者最根本的区别在于**内存是不是隔着一堵墙**。一个进程就像一家**独立的餐厅**：有自己的厨房、自己的库存、自己的账本，另一家餐厅完全看不见。要让两家餐厅协作（进程间通信），得专门打电话、传消息，麻烦又慢。

而**同一家餐厅里的两个厨师**就是同一进程里的两个线程：他们用的是**同一个厨房、同一批库存、同一块砧板**——也就是共享同一份代码、全局数据和堆。沟通基本靠喊一嗓子，又快又方便。代价是：两个厨师同时去够同一块砧板，就可能撞手。下面这张解剖图把两种模型的内存布局并排画出来：

## 代码：你的第一个双线程程序

先看一个**单线程**的「Hello World」作对照——一个厨师，顺着做：

```cpp
#include <iostream>

int main() {
    std::cout << "Hello, World!\n";
    return 0;
}
```

这就是普通程序：从 `main` 进门，一行接一行跑到底。现在请来「第二个厨师」——用 `std::thread` 另起一个线程，让它去跑一个函数：

```cpp
#include <iostream>
#include <thread> // 线程库，std::thread 在这里

void hello() { // 交给新线程去跑的「一道菜」
    std::cout << "你好，来自新线程！\n";
}

int main() {
    std::thread t(hello); // 起一个新线程，立刻开始跑 hello()
    t.join();             // 等这个线程跑完，再继续往下
    return 0;
}
```

逐行看：`#include <thread>` 引入线程库——`std::thread` 这个类型就声明在这里。`std::thread t(hello);` 是关键一句：它**创建并启动**一个新线程，新线程一诞生就去执行 `hello` 这个函数。此刻程序里有了**两个**执行流——`main`（主线程）和 `t`（新线程）在并发地跑。

最后那句 `t.join();` 不能漏：它让主线程**停下来等** `t` 跑完。`join` 的字面意思是「会合」——主厨师在这里等新厨师把那道菜做完，两人「会合」后再一起往下走。如果 `main` 不等就先跑到了结尾，会出大乱子（见下一节第一个坑）。

如果想看到「输出顺序不确定」的真效果，让两个线程都打印就行：

```cpp
std::thread t(hello);             // 新线程打印「你好，来自新线程！」
std::cout << "你好，来自主线程！\n"; // 主线程同时也在打印
t.join();
```

这两句打印谁先出现在屏幕上？**不一定**——正是上一节那个演示的真实版本。

## 容易踩的坑

<Callout type="trap">
  **现象**：程序一运行就崩溃，报 `terminate called without an active exception`
  然后挂掉。**原因**：创建了 `std::thread` 却既没 `join()`（等它跑完）也没
  `detach()`（放它自己跑），线程对象析构时发现「这个厨师还没安顿好」，直接调用
  `std::terminate` 把整个程序掐了。**修法**：每个 `std::thread`
  在它析构前，必须明确二选一——要么 `t.join()` 等它结束，要么 `t.detach()`
  让它脱离自己后台跑。漏掉这一步是新手第一坑。
</Callout>

<Callout type="trap">
  **现象**：以为「主线程的话一定先打印、新线程的后打印」，照这个假设写逻辑，结果时灵时不灵。**原因**：两个线程的执行先后由操作系统调度决定，**没有任何保证**的固定顺序——这次主线程先，下次可能新线程先（就像上面那个演示每跑一次都不同）。**修法**：永远不要假设线程间的执行顺序。需要「A
  一定先于
  B」时，得用同步手段（`join`、后续章节的互斥量、条件变量等）显式约定，而不是靠运气。
</Callout>

<Callout type="trap">
  **现象**：两个线程都去改同一个全局变量（同一块砧板），最后那个变量的值莫名其妙、每次还不一样。**原因**：这就是
  <Term def="多个线程同时读写同一份共享数据、且至少有一个在写时，最终结果取决于它们碰巧谁先谁后——顺序一变结果就变，像两个厨师同时改同一张订单。这是并发编程最经典的 bug，本书后续章节专门讲怎么防。">
    竞态条件
  </Term>
  ——两个厨师同时往同一张订单上写，谁的字盖了谁的全凭手快手慢。**修法**：本章先认识到「共享数据
  + 同时写 =
  危险」即可；具体怎么给共享数据「上锁」保护，是后续「线程间共享数据」章节的主题。
</Callout>

## 小结

- **并发**是多个任务都在推进（单核靠飞快切换造出「同时」的错觉），**并行**是多核真正同时动手——并行一定并发，并发不一定并行
- 用并发图两件事：**性能**（任务并行 + 数据并行）和**关注点分离**（各管一摊、界面不卡）
- **进程**间内存隔离（两家餐厅），**线程**共享同一进程的代码/数据/堆但各有独立的栈（一家餐厅的多个厨师）
- `std::thread t(f);` 起一个新线程去跑 `f`；`t.join()` 让主线程等它跑完再继续
- 两个线程的执行先后**没有固定顺序**，谁先谁后由系统调度临时决定——别靠它写逻辑

**练习 1（改代码型）** 把下面这个单线程程序改成双线程：让一个新线程去打印 `"我是新厨师"`，主线程打印 `"我是主厨师"`，并保证程序正常退出（不崩溃）。跑几次，观察两行的先后是否每次都一样，并解释为什么。

```cpp
#include <iostream>

int main() {
    std::cout << "我是主厨师\n";

}
```

<Answer>

```cpp
#include <iostream>
#include <thread>

void cook() {
    std::cout << "我是新厨师\n";
}

int main() {
    std::thread t(cook);          // 新线程去打印「我是新厨师」
    std::cout << "我是主厨师\n";   // 主线程同时也在打印
    t.join();                     // 必须 join，否则线程析构时程序崩溃
    return 0;
}
```

两行的先后**不是每次都一样**：两个线程一起往 `cout` 打印，谁先抢到 CPU、谁先把字推上屏，由操作系统调度临时决定，没有固定顺序。`t.join()` 不能漏——它保证主线程等新线程跑完再退出，否则 `std::thread` 析构时会调用 `std::terminate` 把程序掐掉。

</Answer>

**练习 2（问答型）** 同事说：「我的电脑是单核的，所以根本没法用并发，写多线程没意义。」这句话对吗？请用「厨师 / 灶台」的话纠正他。

<Answer>

不对。**单核（一个灶台）照样能并发**——一个厨师在一个灶台上轮流切换着做几道菜，就是并发。单核做不到的只是**并行**（同一秒真的有两道菜在动手，那需要两个灶台）。而且即使在单核上，多线程依然有意义：比如让一个线程专门盯前台（界面响应）、另一个线程跑耗时计算，界面就不会卡死——这是「关注点分离」带来的好处，跟有几个核无关。

</Answer>

**练习 3（问答型）** 「两家餐厅」和「一家餐厅的两个厨师」分别对应进程还是线程？哪种「沟通」更快？为什么后者反而更容易出 bug？

<Answer>

「两家餐厅」对应**进程**——各有独立的厨房和库存（独立的内存地址空间），互相看不见。「一家餐厅的两个厨师」对应**线程**——共用同一个厨房、同一批库存（共享同一进程的代码、全局数据和堆）。线程之间沟通更快，因为它们直接读写同一份内存，不用像进程那样「打电话」传消息。但也正因为共享，两个厨师可能同时去够同一块砧板（同时读写同一份数据），引发竞态条件——这是进程因为内存隔离而天然避免、线程却必须小心防范的 bug。

</Answer>

<GlossaryItem term="并行（parallelism）">
  同一时刻真的有多件事在同时发生，靠多个 CPU
  核（或多台机器）各干各的实现。像两个厨师各占一个灶台、同一秒钟两口锅都在炒。并行一定是并发，但并发不一定并行——单核做不到并行。详见本章「并发与并行」一节。
</GlossaryItem>

<GlossaryItem term="进程（process）">
  操作系统分配资源的基本单位，拥有一整套自己独立的内存地址空间。两个进程的内存彼此隔离，谁也读不到对方的数据，要交流得专门「打电话」（进程间通信）。像两家各自独立、互相看不见后厨的餐厅。详见本章「进程与线程」一节。
</GlossaryItem>

<GlossaryItem term="线程（thread）">
  进程内部的执行单元，是 CPU
  真正调度运行的对象。同一进程里的多个线程共享这个进程的代码、全局数据和堆，但每个线程各有自己独立的栈。像同一家餐厅里共用厨房、却各拿一份备忘纸条的多个厨师。本书主要研究线程。详见本章「进程与线程」一节。
</GlossaryItem>

<GlossaryItem term="std::thread">
  C++ 标准库里代表「一个线程」的类型，声明在头文件 `<thread>` 里。写
  `std::thread t(f);`
  就会创建并启动一个新线程去执行函数 `f`。用完前必须 `join()`（等它跑完）或
  `detach()`（放它自己跑），否则程序会崩溃。详见本章「代码：你的第一个双线程程序」一节。
</GlossaryItem>

<GlossaryItem term="竞态条件（race condition）" id="竞态条件">
  多个线程同时读写同一份共享数据、且至少有一个在写时，最终结果取决于它们碰巧谁先谁后——执行顺序一变，结果就变。像两个厨师同时往同一张订单上写字，谁的盖了谁的全凭手快手慢。这是并发编程最经典的
  bug，后续章节专门讲怎么用「锁」来防。详见本章「容易踩的坑」一节。
</GlossaryItem>

</Glossary>
