## 写一半的订单，被人看了去

后厨墙上挂着一块订单板，5 号桌的菜量写在上面。现在你要把它从「3 份」改成「12 份」。可这块板子有点怪：数字得分两笔写——先擦掉十位写上「1」，再擦掉个位写上「2」。就在你写完十位、还没写个位的那一瞬，另一个厨师扭头瞄了一眼板子，看到的是「13 份」——一个你压根没打算写、谁也没真正下过的数。他照着这个乱数去备料，全乱套了。

这一章要解决的，正是「**改一个值的过程中，别让另一个人看到改了一半的样子**」。我们想要的是这样一种改动：要么对方看到的是改之前的旧数（3），要么是改之后的新数（12），**绝不会是中间那个谁都没写过的乱码**。一笔落下、要么全有、要么全无。

没有这套保证会怎样？多个厨师同时对同一块板子又读又写，迟早有人撞见「写一半」的瞬间，照着乱码干活——轻则备错料，重则整个后厨的逻辑全崩。这一章就教你怎么让这种「关键的一笔」变得不可分割、谁也插不进缝。

## 对象、内存位置、修改顺序：账目的官方台账

要讲清楚原子，得先有一套描述「内存里发生了什么」的语言，这就是 C++ 的**内存模型**。它把一切归结到「对象」和它占的「内存位置」上。

在 C++ 里，所有数据——哪怕一个 `int`、一个 `bool`——都是存在某处的<Term def="内存里一块被独立读写的存储单元。一个标量（如 int、bool、指针）占一个内存位置；相邻的一串位域可能挤在同一个内存位置里。C++ 的并发规则是按「内存位置」来界定的：两个线程同时碰同一个内存位置、且至少有一个在写，就有数据竞争。">内存位置（memory location）</Term>。你可以把每个内存位置想成订单板上的一格。多个线程并发地操作内存，本质上就是多个厨师同时去读写这些格子。

对每一个内存位置，C++ 标准保证一件极重要的事：它身上发生过的所有写入，存在一个唯一的先后顺序，叫这个对象的<Term def="针对单个对象（单个内存位置）而言：它一生中所有的写入操作，存在一个唯一确定的全序（先后排成一条线）；并且所有线程对这个对象的这条先后顺序看法完全一致。像某一条账目的「官方流水台账」——谁先改、谁后改，全店厨师查这条账时看到的顺序一模一样。注意这是单个变量的全序，不是跨多个变量之间的顺序。">修改顺序（modification order）</Term>。把它想成某一条账目的「官方流水台账」：这条账目被改了几次、谁先改谁后改，店里所有厨师回头查这条账时，看到的先后顺序**完全一致**——不会有人觉得「先改成 5 再改成 8」、另一个人却觉得「先 8 再 5」。

这里有个关键的限定必须记牢：修改顺序是**针对单独一个对象**的全序，**不是**跨多个对象之间的顺序。x 有 x 自己的台账，y 有 y 自己的台账；至于「x 的某次写」和「y 的某次写」谁先谁后，那是另一个更复杂的话题（线程间的内存序），下一章专门讲，这一章先别混进来。下面这张动画把「三个线程乱序写同一个变量 x，可对 x 的写入仍排成唯一一条全序」演给你看：

## 为什么要原子：非原子并发读写 = 未定义行为

有了内存模型的语言，就能精确地说清开篇那个「写一半被看见」的灾难了。

当两个线程同时碰同一个内存位置、且**至少有一个在写**，又**没有任何同步**把它们隔开时，就发生了<Term def="两个或更多线程并发访问同一个内存位置、其中至少有一个是写、且没有任何同步手段隔开它们。一旦发生数据竞争，C++ 标准直接判定整个程序的行为是未定义的（undefined behavior）——不是「结果随机」，而是「什么都可能发生」。">数据竞争（data race）</Term>。注意 C++ 标准对数据竞争的态度极其强硬：一旦发生，**整个程序的行为就是未定义的（undefined behavior）**。不是「结果不确定但至少不崩」，而是「**编译器有权假设它永远不发生**，于是什么都可能发生」——可能读到乱码、可能崩溃、可能把别处的逻辑也带歪。

为什么后果这么严重？因为很多类型的读写在底层**根本不是一步完成的**。一个 64 位的值在某些平台上要分两次内存访问来写。写到一半时若被另一个线程读走，读到的就是<Term def="读取一个正在被写、且写入分多步完成的变量时，读到「一部分是新值、一部分还是旧值」拼起来的结果——一个谁都没真正写过的乱码。比如 64 位变量分高 32 位、低 32 位两步写，读者在两步之间读，就拿到「高新 + 低旧」的撕裂值。非原子的并发读写就可能撕裂。">撕裂读（torn read）</Term>：高位是新值、低位还是旧值拼在一起，一个谁都没写过的乱数——正是开篇那个「13 份」。

<Term def="不可分割的操作：从任何其他线程的角度看，它要么完全没发生、要么已经全部完成，绝不会被观察到「做了一半」的中间状态。C++ 里用 std::atomic 类型提供原子操作。一个原子操作天然不会和别的原子操作构成数据竞争。">
  原子操作（atomic operation）
</Term>
就是来根治这件事的。「原子」取「不可再分」之意：一个原子读或原子写，从别的线程看，**要么完全没发生、要么已经全部完成**，绝不会被撞见「做了一半」。就像规定那笔订单改动「要么全写上、要么完全没动」。更关键的是：原子操作之间**不构成数据竞争**——把共享变量声明成原子类型，多线程并发读写它就不再是未定义行为。下面这张对照动画，把「非原子→撕裂」和「原子→完整」并排演一遍，这也是本章的核心

## 原子类型家族：从最简的 atomic_flag 说起

C++ 标准库把原子能力都放在 `<atomic>` 头文件里。先认识其中最简单的一个。

<Term def="C++ 里最基础、且标准保证一定无锁的原子类型。它只有两个状态（已置位 / 未置位），只支持两个操作：test_and_set（原子地置位并返回它之前是不是已置位）和 clear（清位）。功能极简，常用来搭一个最朴素的自旋锁。">
  std::atomic_flag
</Term>
是最朴素的原子类型：它只有「置位 /
未置位」两种状态，只支持两个动作——`test_and_set`（原子地把它置位，并返回它**之前**是不是已经置位）和
`clear`（清位）。别看它简陋，正因为它一定无锁，可以用它搭一个最简单的自旋锁（后面代码会演示）。

更常用的是 `std::atomic<bool>`：比 `atomic_flag` 好用得多，能直接读、直接写、还能做后面要讲的 CAS。它和 `atomic_flag` 是这个家族里专门处理「真假开关」的两员。

## 更宽的原子家族与 is_lock_free

家族远不止开关。`std::atomic<T*>` 是**原子指针**（原子地读写一个指针，还支持原子地让指针前后挪）。原子整型（如 `std::atomic<int>`、`std::atomic<long>`）除了普通读写，还提供 `fetch_add`、`fetch_sub` 这类「原子地加一个数、并返回加之前的旧值」的操作——这正是多线程安全计数器的标准做法。更一般地，`std::atomic<T>` 对任何<Term def="一种类型，它的对象可以仅靠「逐字节拷贝内存」就完成复制（没有自定义拷贝逻辑、没有虚函数、没有需要特殊处理的成员），且没有非平凡的析构。简单的内置类型和只含这类成员的结构体都满足。只有可平凡复制的类型才能放进 std::atomic。">可平凡复制（trivially copyable）</Term>的类型 `T` 都能用（比如一个只含几个 `int` 的小结构体）。

但这里有个**必须知道**的真相：`std::atomic<T>` **不保证一定是无锁的**。对于硬件原生支持原子操作的小类型（如 `int`、指针），它确实编译成无锁的 CPU 指令；可对于较大或硬件不直接支持的类型，标准库**会偷偷用一把内部锁来模拟原子语义**——语义上仍然「原子」（行为正确），但性能就不是无锁的了。要查清楚某个原子类型在你的平台上到底是不是真无锁，用它的成员函数 <Term def="std::atomic 的成员函数 is_lock_free()（或编译期常量 is_always_lock_free），返回这个原子类型在当前平台上是不是真用无锁指令实现。返回 true 表示无锁；false 表示标准库用了一把内部锁来模拟原子语义——语义仍正确，但不是无锁。">is_lock_free</Term>：

  被标准**强制**保证无锁；其余原子类型是不是无锁，取决于类型大小和目标平台——较大的
  `T` 很可能被库用内部锁模拟。要无锁就先用 `is_lock_free()` （或编译期的

## compare_exchange：看一眼是不是我以为的数，是才改

原子家族里还有一个核心操作，它是后续无锁编程的基石，这里先建立概念、ch6 / ch8 再深入。这就是 <Term def="compare-and-swap（比较并交换），C++ 里写作 compare_exchange_weak / compare_exchange_strong。它在一个原子操作里做：比较原子变量的当前值是否等于 expected——相等就把它换成 desired 并返回 true（成功），不等就把当前真实值写回 expected 并返回 false（失败，调用方据此重试）。是大多数无锁算法的核心原语。weak 版允许「伪失败」（明明相等也偶尔返回 false），所以通常配重试循环用；strong 版不会伪失败。">CAS（比较并交换）</Term>。

它的两个成员函数是 `compare_exchange_weak` 和 `compare_exchange_strong`，签名形如 `compare_exchange_weak(expected, desired)`。一句话讲清它在干嘛：**看一眼账上还是不是我以为的那个数（`expected`），是才把它改成新数（`desired`），不是就把账上真实的数记下来、回头重看一次**。整个「比较 + 交换」在一个原子操作里一气呵成，别的线程绝不会看到中间态——又是「要么全做、要么全不做」。下面这张动画把它的成功支和失败支分开演：

## 代码：把原子操作一段段写出来

### atomic_flag 搭一个最简自旋锁

`std::atomic_flag` 只有 `test_and_set` / `clear` 两招，却刚好够搭一个最朴素的锁。`test_and_set` 原子地置位并返回**之前**的状态——这正是「抢锁」的关键：

```cpp
#include <atomic>

class SpinlockMutex {
    std::atomic_flag flag = ATOMIC_FLAG_INIT; // 必须这样初始化为「未置位」

public:
    void lock() {
        // test_and_set 返回旧值：旧值为 true 说明锁已被别人占着，就原地空转重试
        while (flag.test_and_set(std::memory_order_acquire)) {
            // 自旋：什么都不做，反复试，直到抢到锁（旧值为 false）
        }
    }

    void unlock() {
        flag.clear(std::memory_order_release); // 清位 = 放锁
    }
};
```

`test_and_set` 把「读旧值 + 置位」打包成一个原子操作，所以两个线程同时抢锁时，只有一个会拿到「旧值是 false」（抢到），另一个拿到「旧值是 true」（继续自旋）——绝不会两个都以为自己抢到了。这就是原子操作做并发同步的最小例子。

### 原子整型计数器：fetch_add

多线程安全地给一个计数器加一，靠的不是 `++counter`，而是原子的 `fetch_add`：

```cpp
#include <atomic>

std::atomic<int> counter{0}; // 原子整型，初值 0

int bump() {
    // fetch_add 原子地「加 1 并返回加之前的旧值」——整个加法不可分割
    int old = counter.fetch_add(1, std::memory_order_relaxed);
    return old; // 返回这次加之前的值
}
```

`fetch_add(1)` 把「读—加—写回」三步压成一个原子操作，无论多少线程同时调用，每次加 1 都不会丢。这正是原子整型相对普通 `int` 的价值——后面误区一节会讲，为什么 `counter = counter + 1` 即使 `counter` 是原子的也仍然不安全。

### 查一查到底是不是无锁：is_lock_free

把变量声明成 `std::atomic` 不代表它就无锁。运行期用 `is_lock_free()` 查清楚：

```cpp
#include <atomic>
#include <iostream>

struct Big { // 一个较大的结构体
    long a, b, c, d;
};

int main() {
    std::atomic<int> ai{0};
    std::atomic<Big> ab{};
    // 小类型几乎一定无锁；大结构体很可能被库用内部锁模拟
    std::cout << "atomic<int>  无锁? " << ai.is_lock_free() << "\n";
    std::cout << "atomic<Big>  无锁? " << ab.is_lock_free() << "\n";
    return 0;
}
```

`is_lock_free()` 返回 `true` 表示这个原子类型在当前平台真用无锁指令实现，`false` 表示库偷偷加了把内部锁。语义两种情况都正确，但性能差别很大——性能敏感处务必先查。

### CAS 重试循环的骨架

CAS 几乎总是配一个重试循环用（尤其 `weak` 版可能伪失败）。这是它最典型的形态——把一个原子变量原子地翻倍：

```cpp
#include <atomic>

void atomic_double(std::atomic<int>& x) {
    int expected = x.load();             // 先读一眼当前值作为「我以为的数」
    // 相等就换成 expected*2 返回 true（退出循环）；
    // 不等（被别人改过）就把真实值回填进 expected、返回 false，循环再试一次
    while (!x.compare_exchange_weak(expected, expected * 2)) {
        // 失败时 expected 已被更新为真实的当前值，下一轮用它重新算 desired
    }
}
```

循环体里**什么都不用写**：因为失败时 `compare_exchange_weak` 已经替你把 `expected` 刷成了真实值，`while` 的条件 `expected * 2` 下一轮自动用新值重算。`weak` 版偶尔会「明明相等也返回 false」（伪失败），但反正在循环里，再试一次即可——这正是它通常比 `strong` 版更高效的搭配方式。

## 容易踩的坑

<Callout type="trap">
  **现象**：以为「凡是 `std::atomic<T>` 都是无锁的、性能必然顶配」，在性能敏感的热路径上塞了个
  `std::atomic<BigStruct>`，结果跑起来反而慢。**原因**：只有 `std::atomic_flag`
  被标准强制无锁；较大的 `T` 很可能被库用**一把内部锁**模拟原子语义——语义对，但其实是带锁的。**修法**：用
  `is_lock_free()`（或编译期 `is_always_lock_free`）查清楚再用；需要无锁就把数据拆小、用硬件原生支持的小类型。
</Callout>

<Callout type="trap">
  **现象**：以为「把变量加上 `volatile` 就线程安全了」，于是用 `volatile int`
  当多线程共享计数器，结果照样出数据竞争和乱值。**原因**：C++ 里 `volatile`
  只表示「这个变量可能被程序之外的因素改动（如硬件寄存器），别优化掉对它的读写」，它**既不保证原子性、也不防数据竞争**——`volatile
  ≠ atomic`。**修法**：多线程共享的同步变量一律用 `std::atomic`，别拿 `volatile`
  凑数；`volatile` 是给内存映射 I/O 之类场景用的，与并发无关。
</Callout>

<Callout type="trap">
  **现象**：变量已经是 `std::atomic<int>` 了，却写 `atomicVar = atomicVar + 1`
  来累加，多线程下计数还是会丢。**原因**：这一句其实是**三步**——原子地 load
  出旧值、在普通寄存器里加 1、再原子地 store 回去；三步之间别的线程能插进来，整体**不是**一个原子操作。**修法**：要原子地累加，用专门的读改写原子操作
  `atomicVar.fetch_add(1)`（或 `++atomicVar`），它把「读—加—写」压成一个不可分割的原子操作。
</Callout>

## 小结

- **原子操作**不可分割：别的线程要么看到它完全没发生、要么已全部完成，绝不会撞见「写一半」的中间态——就像那笔「要么全写上、要么完全没动」的订单改动
- 每个对象有自己的**修改顺序**：它的所有写入排成唯一一条全序，所有线程看法一致（注意是**单变量**全序，不是跨变量的内存序）
- 两个线程非原子地并发读写同一个变量、且至少一个在写 = **数据竞争** = **未定义行为**；底层分多步的写被读到一半就成「**撕裂读**」乱码
- 原子家族：`std::atomic_flag`（最简、强制无锁）、`std::atomic<bool>`、原子指针、原子整型（`fetch_add`）、`std::atomic<可平凡复制 T>`；但**不是都无锁**，用 `is_lock_free()` 查
- **CAS**（`compare_exchange_weak/strong`）在一个原子操作里「比较 + 交换」：相等才改、不等就回填真实值重试——无锁编程的基石

<Exercises>

**练习 1（解释/判断型）** 同事说：「两个线程非原子地读写同一个 `int`，最坏也就是读到一个旧值或新值，结果不确定罢了，又不会出大事，干嘛非得用 `std::atomic`？」这句话对吗？请用本章的「数据竞争 / 撕裂读 / 未定义行为」纠正他。

<Answer>

不对，而且错得很危险。两个线程并发读写同一个内存位置、至少一个在写、又没有同步，这就是**数据竞争**，C++ 标准对它的判定不是「结果不确定但安全」，而是**整个程序未定义行为**——编译器有权假设它根本不发生，于是什么都可能发生：可能读到旧值或新值，但也**可能读到「撕裂值」**（写分多步完成时，读者撞在中间，拿到「一部分新、一部分旧」拼起来的、谁都没写过的乱码），甚至可能让程序崩溃或把别处逻辑带歪。所以「最坏只是读到旧值或新值」这个前提就是错的——非原子并发读写连这点都不保证。修法：把共享变量声明成 `std::atomic<int>`，原子操作不可分割、且彼此不构成数据竞争，读到的永远是一个完整的值（要么全旧、要么全新）。

</Answer>

**练习 2（改代码型）** 下面这个多线程计数器有 bug：即使把 `counter` 改成了原子类型，并发自增时计数仍然会丢。指出 bug，并改成正确的原子自增。

```cpp
std::atomic<int> counter{0};

void worker() {
    for (int i = 0; i < 1000; ++i) {
        counter = counter + 1; // ❌ 看似用了 atomic，其实不安全
    }
}
```

<Answer>

bug 在 `counter = counter + 1`：它**不是**一个原子操作，而是**三步**——原子地 `load` 出旧值、在普通寄存器里 `+1`、再原子地 `store` 回去。三步之间别的线程可以插进来：两个线程可能都读到同一个旧值 5，各自算出 6 再写回，于是两次自增只让计数涨了 1，丢了一次。把变量声明成原子，只保证**单独的** load 和 store 各自原子，并不能把「读—加—写」三步粘成一个原子操作。

修法：用专门的读改写原子操作 `fetch_add`（或等价的 `++`）：

```cpp
std::atomic<int> counter{0};

void worker() {
    for (int i = 0; i < 1000; ++i) {
        counter.fetch_add(1); // ✓ 原子地「读—加—写」一气呵成，绝不丢
    }
}
```

`fetch_add(1)` 把整个自增压成一个不可分割的原子操作，无论多少线程同时调用，每一次 +1 都不会丢。

</Answer>

**练习 3（问答型）** 用「订单板」的比喻，分别解释：(a) 为什么 `std::atomic_flag` 能搭一个锁，而普通 `bool` 不行？(b) 为什么 `std::atomic<T>` 不保证一定无锁，该怎么查？

<Answer>

(a) 搭锁的关键动作是「抢」：谁能原子地「看一眼锁是不是空的，是空的就立刻占上」，谁就抢到。`std::atomic_flag` 的 `test_and_set` 恰好把「读旧状态 + 置位」打包成**一个不可分割的原子操作**，并返回置位前的旧值——所以两个线程同时抢，只有一个拿到「旧值是未置位」（抢到），另一个拿到「旧值是已置位」（继续自旋）。普通 `bool` 的「读一下、发现是空、再写成占用」是分开的几步，两个线程可能都读到「空」、都以为自己抢到了，锁就形同虚设。就像订单板上那格如果不能「看一眼+占上」一笔完成，两个厨师会同时以为这桌归自己。

(b) `std::atomic<T>` 只承诺**语义**上的原子（行为正确），不承诺**实现**上无锁。对硬件原生支持的小类型（`int`、指针）它编译成无锁指令；但对较大或硬件不直接支持的 `T`，标准库会**偷偷加一把内部锁**来模拟原子语义——就像那一格数字太大、一笔写不下，店里只好临时上了把小锁让人轮流写。要查清在你的平台上到底是不是真无锁，调 `is_lock_free()`（或编译期常量 `is_always_lock_free`）：返回 `true` 才是真无锁。

</Answer>

<Glossary>

<GlossaryItem term="内存位置（memory location）">
  内存里一块被独立读写的存储单元。一个标量（`int`、`bool`、指针等）占一个内存位置。C++
  的并发规则是按「内存位置」界定的：两个线程同时碰同一个内存位置、且至少有一个在写，就有数据竞争。像订单板上的一格。详见本章「对象、内存位置、修改顺序」一节。
</GlossaryItem>

<GlossaryItem term="修改顺序（modification order）">
  针对**单个对象**而言：它一生中所有的写入操作，存在一个唯一确定的全序（排成一条先后线），且所有线程对这条先后看法**完全一致**。像某条账目的「官方流水台账」——谁先改、谁后改，全店厨师查到的顺序一模一样。注意这是单个变量的全序，**不是**跨多个变量之间的顺序（那是下一章的内存序）。详见本章「对象、内存位置、修改顺序」一节。
</GlossaryItem>

<GlossaryItem term="数据竞争（data race）">
  两个或更多线程并发访问同一个内存位置、其中至少一个是写、且没有任何同步隔开它们。一旦发生，C++
  标准直接判定**整个程序的行为是未定义的**——不是「结果随机」，而是「什么都可能发生」（读到乱码、崩溃、把别处逻辑带歪都有可能）。详见本章「为什么要原子」一节。
</GlossaryItem>

<GlossaryItem term="撕裂读（torn read）">
  读取一个正在被写、且写入要分多步完成的变量时，读到「一部分是新值、一部分还是旧值」拼起来的结果——一个谁都没真正写过的乱码。比如
  64 位变量分高 32 位、低 32 位两步写，读者卡在两步之间就拿到「高新 +
  低旧」的撕裂值。非原子的并发读写就可能撕裂。详见本章「为什么要原子」一节。
</GlossaryItem>

<GlossaryItem term="原子操作（atomic operation）">
  不可分割的操作：从任何其他线程看，它要么完全没发生、要么已经全部完成，绝不会被观察到「做了一半」的中间态。像那笔「要么全写上、要么完全没动」的订单改动。C++
  用 `std::atomic`
  类型提供原子操作，且原子操作之间天然不构成数据竞争。详见本章「为什么要原子」一节。
</GlossaryItem>

<GlossaryItem term="std::atomic_flag">
  C++ 里最基础、且标准**强制保证一定无锁**的原子类型。它只有「已置位 /
  未置位」两个状态，只支持两个操作：`test_and_set`（原子地置位并返回它之前是不是已置位）和
  `clear`（清位）。功能极简，常用来搭一个最朴素的自旋锁。详见本章「原子类型家族」一节。
</GlossaryItem>

<GlossaryItem term="可平凡复制（trivially copyable）">
  一种类型，它的对象仅靠「逐字节拷贝内存」就能完成复制（没有自定义拷贝逻辑、没有虚函数、没有需要特殊处理的成员），且析构也平凡。简单内置类型和只含这类成员的结构体都满足。只有可平凡复制的类型才能放进
  `std::atomic<T>`。详见本章「更宽的原子家族」一节。
</GlossaryItem>

<GlossaryItem term="is_lock_free">
  `std::atomic` 的成员函数 `is_lock_free()`（以及编译期常量
  `is_always_lock_free`），返回这个原子类型在当前平台上是不是**真用无锁指令**实现。`true`
  表示无锁；`false`
  表示库用了一把内部锁来模拟原子语义——语义仍正确，但不是无锁、性能不同。详见本章「更宽的原子家族与
  is_lock_free」一节。
</GlossaryItem>

<GlossaryItem term="CAS（比较并交换）">
  compare-and-swap，C++ 里写作 `compare_exchange_weak` /
  `compare_exchange_strong`。它在一个原子操作里：比较原子变量的当前值是否等于
  `expected`——相等就换成 `desired` 并返回 `true`（成功），不等就把当前真实值写回
  `expected` 并返回
  `false`（失败，调用方据此重试）。像「看一眼账上还是不是我以为的那个数，是才改、不是就把真实值记下来重来」。是大多数无锁算法的核心原语；`weak`
  版允许偶尔「伪失败」，所以通常配重试循环用。详见本章「compare_exchange」一节。
</GlossaryItem>

</Glossary>
