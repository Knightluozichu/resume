## 为什么写下了代码，电脑就能懂？

你敲下一段 C++ 代码，点运行，屏幕上就出现了一行 "Hello, World!"。这中间其实经过了一条**工厂流水线**——你的代码是「原料」，经过几道工位一道道加工，最后从流水线另一头出来一个能双击就跑的「成品」。

这条流水线有**四道工位**：第一道把原料里杂七杂八的标注清理干净、展开折叠好的部分；第二道把人类能读的文字翻译成机器"方言"；第三道把方言逐字译成机器能认的二进制编码；第四道把分散的零件拼成一个完整的、能直接跑的程序。

这一章就带你走一遍这条流水线的四道工位，同时写出你人生中第一个 C++ 程序。没有这条流水线会怎样？你写的代码永远只是一堆人类能读的文字，电脑一个字也不认识。

每一行都在说什么？下面逐一看。

### main 函数：程序从这进门

<Term def="每个 C++ 程序都有的一个特殊函数——它标记了程序从哪里开始执行。操作系统运行你的程序时，第一件事就是找到 main。">main 函数</Term> 是整个程序的**入口**。你写再多代码，操作系统启动你的程序时都只会钻进 `main` 这个大门。`int` 表示它最后要交回一个整数，「交给操作系统」——`return 0` 就是在说「一切正常，我跑完了」。如果把 `return 0` 删掉？现代编译器一般会自己帮你补一句，但规范上每一扇大门走了就要把"我没事"的纸条留下来。

它处在整个流程的**起点**：你的所有代码都经过它开始执行，它是流水线前那道「开工」的指令。

### 头文件：声明"我能用什么"

代码第一行 `#include <iostream>` 是什么？它是一句 <Term def="用 #include 引入的文件，里面提前写好了别人帮你实现的功能声明。你 include 了它，就能直接调用里面的东西，不用自己从头写。">头文件</Term>。`iostream` 这个头文件里写好了 `cin`（从键盘读输入）和 `cout`（往屏幕写输出）的声明。你不 include 它就直接写 `cout`，编译器会告诉你「我不认识 cout 是什么东西」。

它处在流水线的**第一道工位（预处理）**：预处理器看到 `#include`，就把那个文件里的所有文字**原样粘贴**到你这行所在的地方。之后你再引用 `cout`，编译器才知道「哦，原来这个东西在 iostream 里定义过」——预处理这一步之后继续往下走。

<Term def="程序设计里「一个操作」这样一个独立的单元。C++ 里多数语句以分号 ; 结尾，表示「这句话说完了」。">语句</Term> `std::cout << "Hello, World!" << std::endl;` 是本文真正干活的一句——把 `"Hello, World!"` 这段文字**送进** cout 输出流，再补一个换行 `std::endl`，于是屏幕上就出字了。`std::` 是 <Term def="给名字加一个归属前缀，避免不同库里的同名词打架。std 就是「标准库的命名空间」，里面的 cin、cout、endl 都在这个空间里。">命名空间</Term>——可以理解成「住在 std 这个门牌号下的 cout」。在它前面加上 `std::`，就是在告诉编译器「我用的是 std 空间里的那个 cout，不是别的」。

它和 main 函数里余下的语句，在编译流程中对应**第二道和第三道工位**：编译器先把这行 C++ 语法解析成汇编指令，汇编器再把它变成二进制机器码。

### 什么是「流」——cin 和 cout 在干什么

`cout` 和 `cin` 都是一种叫 <Term def="数据像水一样从一个源头流向一个目的地的通道。写 cout << 是把数据送进输出流，读 cin >> 是从输入流里往外取数据。">流（stream）</Term> 的东西。想象两条管道：`cout` 是通往屏幕的那根水管，你用 `<<` 把文字往里一塞，它就顺着管道流到控制台窗口显示出来。反过来，`cin` 是从键盘那边过来的水管，你用 `>>` 往外抽数据，像用吸管从杯子里吸水。

<Term def="<< 运算符在 C++ 里被重载用来写 cout 时做输出操作符：把右边的东西送进左边的输出流。它和位运算的 << 是同一个符号，但 C++ 的运算符重载技术让它在不同类型上表现不同行为。虽然初学时不用深究「重载」概念，但要知道 cout << 只是 << 的一种特殊用法。">输出运算符（<<）</Term> 把右边的数据推向左边那条流管道，`>>` 则相反。它们背后有**缓冲区**暂存，数据并不是一个字一个字立刻上屏——等缓冲区满了或遇到 `endl`，一整批一起推过去，减少来回奔波。下面这张图把两条管道的方向画出来：

<IOStreamDiagram />

流在整个流程里不归属某一道工位——它是**运行时**才工作的概念。编译流水线只是造出了程序的「外壳」；等到你双击运行，cin/cout 这套管道才真正开始流动。

### 注释：给自己和队友留话

代码里双斜线 `//` 开头的那一行、或者 `/*` 和 `*/` 裹起来的那一段，叫 <Term def="代码里写给程序员看的文字，编译器完全忽略。// 是单行注释（从它开始到本行末尾都被忽略），/* */ 是多行注释（裹起来的部分都被忽略）。">注释</Term>。它们写给未来看这行代码的人——包括三个月后的你自己。编译器在**第二道工位（编译）**的开头就把注释剥光了，它们一个字都不会进最终的二进制程序，所以想写多少写多少。

`//` 只能管到行末；`/* */` 能跨行包住一大段。比较：

```cpp
// 这是一个单行注释——只到这一行末尾
std::cout << "hi"; // 也可以写在语句后面

/* 这是一个多行注释——
   可以跨好几行，
   裹住一大片 */
```

注释不占用运行时任何资源，因为它们走完预处理就从代码里消失了——在第一道工位，`/* */` 形式的注释有时甚至被预处理器直接替换成空格。

## 动手：跟着编译器走一遍流水线

现在你知道了每行代码在说什么，下面打开这条流水线，一步步看你的代码怎么变成能跑的程序。下面这个分步演示把一个 `hello.cpp` 从源码走完四道工位——**每切一步，上面的流程图会高亮当前工位，下方说明这一步发生了什么**。

> 猜一猜：哪一道工位最耗时？是把头文件展开、还是把 C++ 翻译成汇编？动手切一切，看看每一步的解释再说你的答案。

<Stepper>
  <Step title="① 预处理：把 #include、#define 都展开">
    <CompilationPipelineDiagram step={1} />
    预处理器先打开你的 `hello.cpp`。它做三件事：① 见到 `#include <iostream>`，就把整个 iostream 文件内容原样粘贴进来（可能有几千行）；② 见到 `#define` 宏就把所有用到的地方替换掉；③ 删掉所有注释。输出的是一个「展开过的纯净源代码」——后缀名 `.i`。你可以理解成：把原料里折叠的、注解的部分全摊平，得到一张干干净净的底稿。
  </Step>
  <Step title="② 编译：把 C++ 翻译成汇编语言">
    <CompilationPipelineDiagram step={2} />
    编译器拿到预处理后的 `.i` 文件，开始真正的「翻译」工作：做语法检查（括号对上了吗？分号没忘吧？），把 C++ 高级语法翻译成汇编语言——一种贴近机器指令但仍然用助记符表述的文字。输出文件后缀 `.s`。这就是编译器真正的角色：把人类好读的语言翻译成机器能理解的"方言"。如果语法有错，就会在这一步报错停下。
  </Step>
  <Step title="③ 汇编：把汇编助记符变成二进制机器码">
    <CompilationPipelineDiagram step={3} />
    汇编器把 `.s` 文件里的每一条汇编助记符一对一地翻译成二进制机器码，生成一个 `.o`（目标文件）。这一阶段几乎是一一对应：`mov` 翻译成某个二进制操作码，寄存器名翻译成编号。它也可以把你代码里定义的函数打包成一块块的机器码段。出来的 `.o` 还是零件，不是能直接跑的东西——它还差最后一步。
  </Step>
  <Step title="④ 链接：把零件拼成可执行文件">
    <CompilationPipelineDiagram step={4} />
    链接器把你上一步编译出来的 `.o` 文件，连同你用到的标准库（比如 `iostream` 的实现）和可能其他 `.o` 文件，拼在一起。`cout` 的实现在标准库里——你只 include 了它的声明，真正可跑的机器码在标准库那边的 `.o` 文件里。链接器把所有人对 `cout` 的引用都指向同一个地址，最终输出一个能双击就跑的**可执行文件**。
  </Step>
</Stepper>

<Callout type="tip">
  这四步通常一条 `g++` 命令就全包了：`g++ hello.cpp -o hello`。
  但知道这四道工位各在做什么，以后遇到报错才知道该往哪找——预处理错误（头文件找不到）、
  编译错误（语法/类型错）、链接错误（undefined reference）。
</Callout>

## 代码速查：一行命令走完流水线

不用 IDE 点按钮，直接在终端里敲一条命令，就能从头到尾跑完上面那四道工位。以 Linux/macOS 上的 `g++` 为例：

```bash
# 一条命令 = 预处理 + 编译 + 汇编 + 链接
g++ hello.cpp -o hello

# 运行你刚造出来的程序
./hello
# 输出：Hello, World!
```

如果你好奇中间产物的样子，可以分步走：

```bash
# 只预处理，看展开的头文件有多长
g++ -E hello.cpp -o hello.i

# 只编译成汇编，看 C++ 语句被翻译成什么
g++ -S hello.cpp -o hello.s

# 只汇编成目标文件（二进制，人类不可读）
g++ -c hello.cpp -o hello.o
```

在 Visual Studio 下，按 Ctrl+F5 运行；Xcode 里点 Product → Run。IDE 背后做的事完全一样。

## 容易踩的坑

<Callout type="trap">
  **现象**：编译报错 `undeclared identifier 'cout'` 或 `'cout' was not declared in this scope`。
  **原因**：忘记 `#include <iostream>`，或者忘了写 `std::` 前缀 / `using namespace std;`。编译器不认识 cout 是什么东西。
  **修法**：在文件最开头加 `#include <iostream>`，然后写 `std::cout`（推荐），或者在 main 之后加一句 `using std::cout;`。
</Callout>

<Callout type="trap">
  **现象**：程序编译通过了，双击却没窗口闪一下就没了，什么输出都看不到。**原因**：程序确实运行了、也输出了一行 Hello World 到控制台，但控制台窗口在输出后立刻关了——这是在 Windows 下双击 .exe 的常见行为。**修法**：在 `return 0;` 之前加一句 `std::cin.get();`（让程序等你敲一个回车再退出），或者直接从终端命令行里运行程序。
</Callout>

<Callout type="trap">
  **现象**：写了一个多行注释 `/* ... */`，编译器报一堆莫名其妙的错误。**原因**：注释的开始符号 `/*` 之后忘了写结束符号 `*/`，
  编译器把后面的代码全部当成注释吞掉了——它以为你还在注释里面。**修法**：检查 `/*` 后面一定有配对的 `*/`，用编辑器的高亮/折叠功能直接看哪些部分被识别为注释。
</Callout>

## 小结

- C++ 源码经过**四道工位**变成可执行文件：预处理→编译→汇编→链接
- `int main()` 是程序入口，`return 0` 告诉操作系统「一切正常」
- `#include` 在预处理阶段把头文件内容粘贴进来，让你能使用 `cin`/`cout`
- `cout <<` 把数据推进输出流送到屏幕，`cin >>` 从输入流取出键盘输入——背后有缓冲区暂存
- 注释 `//` 和 `/* */` 写给程序员看，编译器在处理时完全忽略它们

<Exercises>

**练习 1（改代码型）** 把上面 Hello World 程序里的 `"Hello, World!"` 改成 `"你好，C++！"`，再加一行 `std::cout << "这是我的第一个程序。" << std::endl;`。在 `return 0;` 之前加一句 `std::cin.get();` 让窗口停住。跑起来，解释你加的每一句各干了什么。

<Answer>

改完后的程序：

```cpp
#include <iostream>

int main() {
    std::cout << "你好，C++！" << std::endl;
    std::cout << "这是我的第一个程序。" << std::endl;
    std::cin.get();  // 等用户敲回车再退出
    return 0;
}
```

`"你好，C++！"` 和 `"这是我的第一个程序。"` 是两段被送进 cout 的文本，各占一行。`std::cin.get()` 让程序暂停、等你敲一下回车才继续跑到 `return 0`——在直接双击运行 .exe 的场景下，这样窗口就不会一闪就没了。

</Answer>

**练习 2（问答型）** 编译流水线的四道工位如果出一道错，分别会长什么样？各举一个例子。

<Answer>

① 预处理错：`#include` 的文件不存在，报 `fatal error: 'xxx.h' file not found`。② 编译错：忘写分号或括号不配对，报 syntax error。③ 汇编错：很少单独出错，一般被编译器打包在编译阶段一起报。④ 链接错：用了别处定义的函数但没链接进实现，报 `undefined reference to 'xxx'`。

</Answer>

**练习 3（问答型）** 把 `#include <iostream>` 删掉，程序还能编译吗？把这行注释掉试试（用 `//`），把你看到的错误信息抄下来，解释编译器为什么报这个错。

<Answer>

不能编译。用 `// #include <iostream>` 注释掉后，编译器报 `use of undeclared identifier 'cout'` 或 `'cout' was not declared in this scope`。原因是：`cout` 的声明写在了 `<iostream>` 头文件里；不 include 它，编译器读到 `cout` 时在它认识的名字表里找不到这个名字，于是拒绝继续往下翻译。

</Answer>

<Glossary>

<GlossaryItem term="main 函数">
  C++ 程序的入口。操作系统运行你的程序第一件事就是钻进 `main`，从它开始一行一行往下跑。`int` 表示它会交回一个整数给操作系统，`return 0` 就是说「正常结束了」。详见本章「main 函数」一节。
</GlossaryItem>

<GlossaryItem term="头文件">
  用 `#include` 拉进来的文件，里面写好了别人为你准备好的东西的**声明**。比如 `<iostream>` 里声明了 `cin` 和 `cout`——你不是自己写它们怎么工作，而是 include 后直接用。它在预处理阶段被「粘贴」进你的源码。详见本章「头文件」一节。
</GlossaryItem>

<GlossaryItem term="语句">
  程序里一个独立操作单元。多数 C++ 语句以分号 `;` 结束，表达「这句话说完了，可以执行下一句」。`int x = 1;`、`cout << "hi";` 都是语句。
</GlossaryItem>

<GlossaryItem term="命名空间">
  一群相关名字的「门牌号」。`std` 是 C++ 标准库住的那个门牌号——`std::cout` 意思是「std 门牌号下的那个 cout」。加上门牌号，不同库里即使起了同样的名字也不会打架。详见本章「头文件」一节。
</GlossaryItem>

<GlossaryItem term="流（stream）">
  把数据想象成水，流就是它的通道。`cout` 是通向屏幕的「输出流管道」，用 `<<` 把数据往里一塞就顺管流到控制台。`cin` 是从键盘过来的「输入流管道」，用 `>>` 往外抽数据。管道两头有缓冲池暂存，满了或遇到 `endl` 才一口气推过去。详见本章「什么是流」一节。
</GlossaryItem>

<GlossaryItem term="输出运算符（<<）">
  `<<` 符号在 `cout << x` 里不是「向左移位」，而是被重新赋予了含义：把右边的东西推进左边那根流管道。同理 `cin >> x` 的 `>>` 是「从左边管道里拉东西出来给右边」。二者都是对同一个符号赋予了与人读的直觉一致的语义。详见本章「什么是流」一节。
</GlossaryItem>

<GlossaryItem term="注释">
  代码里写给**程序员**看的文字，编译器完全忽略。`//` 是单行注释（本行到此完），`/* */` 是多行注释（中间全忽略）。给三个月后的自己留句话，别全靠记忆力。详见本章「注释」一节。
</GlossaryItem>

<GlossaryItem term="编译">
  把写给人看的 C++ 源码，一步步翻译成机器能跑的二进制程序的过程。广义包括预处理→编译（狭义）→汇编→链接四步，狭义单指其中「C++→汇编」这一步。本章 §5 的 Stepper 动画是广义编译流水线的全程分步演示。
</GlossaryItem>

</Glossary>
