## 谁该挡住谁：流水线上的一道「比远近」工序

想象一条往画布上贴贴纸的流水线：场景里每个物体被拆成无数小色块，一块接一块地送上传送带，轮到谁就把谁贴到画布对应的位置。问题来了——如果一个近处的杯子和一面远处的墙，恰好要贴到画布同一个点上，先送到的那块会先贴，后送到的又盖上去……那最后留在画布上的，到底该是杯子还是墙？

如果完全听传送带的「谁后到谁说了算」，结果就乱了：墙明明在杯子后面，却可能因为「碰巧后送上来」而盖住了杯子，近的反被远的挡住，画面穿帮。这一章要解决的就是：**怎么让流水线自动认准「近的该挡住远的」，不管它们谁先谁后送上来**。

办法是在传送带末端加一道质检工序：每块色块贴之前，先报一下「我离镜头多远」，和这个位置上**已经贴着**的那块比一比——更近才准贴、更远就丢掉。有了这道工序，三维物体的前后遮挡就全自动正确了；没有它，你的场景就只是一堆按上色顺序胡乱叠起来的色块。

## 深度缓冲：每个像素偷偷存了一个「远近」

那道质检工序要能「比远近」，前提是画布上每个像素都记着「我现在贴的这块，离镜头多远」。存这份远近账本的地方，就是 <Term def="显存里一块和颜色缓冲一样大的内存，为每个像素额外存一个 0~1 的「深度值」（离镜头的远近）。屏幕显示的是颜色缓冲，而深度缓冲只在幕后记账，供深度测试比较用。常见精度为 24 位。">深度缓冲</Term>（depth buffer，又叫 z-buffer）。

它和你上一篇认识的颜色缓冲是「孪生兄弟」：尺寸一样大、一一对应每个像素，只不过颜色缓冲存的是「这个像素什么颜色」，深度缓冲存的是「这个像素现在贴着的色块有多远」。屏幕显示的是颜色缓冲；深度缓冲不直接显示，只在幕后默默记账。它通常是 24 位精度，足够细地区分远近。

每个像素记的那个数，就是 <Term def="深度缓冲里每个像素存的那个数，范围 0~1，表示该像素当前色块离镜头的远近——约定 0 最近、1 最远（落在近平面记 0、落在远平面记 1）。片段着色器里可以用内建变量 gl_FragCoord.z 读到当前片段的深度值。">深度值</Term>（depth value）——一个 0 到 1 的数，约定 **0 表示最近、1 表示最远**。下面这张图把「一个像素 = 颜色 + 深度两份数据」，以及「靠比深度让近物盖住远物」画在一起：

<DepthBufferDiagram />

为什么要专门存这份账？因为只有记下每个像素「当前贴的有多远」，新来的色块才有得比。它在管线里的位置紧贴着颜色缓冲：每写一个像素的颜色，旁边就配一个深度，供下一块色块过来时核对。

## 深度测试：新色块和账本比一比，近的才准贴

有了账本，那道质检工序就能干活了。<Term def="GPU 在每个片段算完颜色、准备写进颜色缓冲之前做的一道检查：把这个片段的深度值和深度缓冲里该像素已存的深度值比一比，按深度函数判定——通过才写入颜色并更新深度缓冲，不通过就直接丢弃这个片段。它让三维场景的前后遮挡自动正确。">深度测试</Term>（depth test）做的事，一句话：**新片段要写进某像素前，先拿它的深度值和深度缓冲里该像素已存的深度值比，更近才写、更远就丢**。

具体地，对每一个片段：读出它的深度值 → 和深度缓冲里同位置那个数比较 → 若「更近」（默认即深度值更小）就通过，把颜色写进颜色缓冲、并把这个更小的深度值更新进深度缓冲；否则不通过，整个片段被丢弃，颜色缓冲和深度缓冲都不动。这样无论物体以什么顺序送进管线，留在屏幕上的永远是每个像素处最近的那块。

它在管线里的位置很关键：深度测试发生在**片段着色器之后**（要先有颜色和深度才能测），在片段被真正写入颜色缓冲**之前**。下面把「一个片段过这道工序」的完整判定逐步拆开看：

## 深度函数：那条「谁算通过」的规则可以换

第 ② 步里的「更小才通过」并不是写死的，它由 <Term def="决定深度测试「怎样算通过」的比较规则，用 glDepthFunc 设置。默认 GL_LESS（新深度更小才通过）。还有 GL_LEQUAL（更小或相等）、GL_GREATER（更大才通过）、GL_ALWAYS（永远通过，等于关掉测试）等八种算子，换一个就改变遮挡判定。">深度函数</Term>（depth function）决定，用 `glDepthFunc(...)` 设置。默认是 `GL_LESS`——「新深度严格更小才通过」，这也是绝大多数场景要的「近的盖远的」。

但你可以换成别的算子来玩花样。常用的几个：

| 深度函数     | 含义（新片段深度 vs 缓冲深度）       |
| ------------ | ------------------------------------ |
| `GL_LESS`    | 更小才通过（**默认**，近的胜出）     |
| `GL_LEQUAL`  | 更小或相等才通过                     |
| `GL_GREATER` | 更大才通过（反过来：远的胜出）       |
| `GL_EQUAL`   | 相等才通过                           |
| `GL_ALWAYS`  | 永远通过（等于没做测试，谁后写谁赢） |
| `GL_NEVER`   | 永远不通过（什么都画不出来）         |

把它设成 `GL_ALWAYS`，就退回到 §2 那个「谁后画谁赢」的混乱世界——这正好可以拿来验证深度测试到底有没有在起作用（见 §7）。八个算子里还有 `GL_NOTEQUAL` / `GL_GEQUAL`，用法同理，按需查表即可。

## 深度值的精度：为什么近处看得清、远处糊

到这儿还有个反直觉的真相要点破：深度值在 0~1 之间**不是均匀分布**的。它是 <Term def="深度缓冲里的深度值随真实距离不是均匀变化的，而是与 1/z 成正比——绝大部分精度集中在离镜头很近的范围，越远精度越粗。这是透视投影的副产物，好处是近处物体的遮挡判定特别精确。">非线性深度</Term>（non-linear depth）：它和真实距离的倒数 `1/z` 成正比，结果就是**绝大部分精度被堆在了离镜头很近的地方**，越往远，同样一段真实距离能分到的深度值越少、越粗。

下面这张图把这件事画得很直白——把深度值 0→1 等间隔取刻度、标回真实距离轴，你会看到近平面附近刻度挤成一簇、远平面附近稀稀拉拉：

<DepthPrecisionDiagram />

为什么要这么设计？因为我们通常更在乎近处物体之间的遮挡是否精确（它们占了画面大头），远处糊一点无所谓。这套非线性映射是透视投影「免费附送」的副产物，具体公式 §4 会拆开讲。**但它也埋了个雷**：远处深度值挨得太近，两个远处的面稍微一靠近就会分不出谁前谁后——这正是下一节 Z-fighting 的温床。

顺带一提，正因为深度是非线性的，你若想在着色器里直接把 `gl_FragCoord.z` 当灰度输出来做 <Term def="把片段的深度值当成灰度颜色输出到屏幕，用来「看见」深度缓冲。直接输出非线性的 gl_FragCoord.z 会发现画面几乎全白（精度堆在近处）；先线性化再输出，才能看到从近到远平滑的灰度渐变。">深度可视化</Term>（depth visualization），会发现画面几乎一片惨白——因为非线性把值全挤到了接近 1 的高位。想看到从近到远平滑的灰阶，得先把它**线性化**。这两种输出的差别，就是下面 Demo 让你亲手切换的。

## 深度冲突：两个面贴太近，谁也压不住谁

最后一个绕不开的现象。当两个面片**几乎共面**（深度差小到深度缓冲分辨不出）时，会出现 <Term def="两个表面靠得极近、深度值小到深度缓冲分辨不出谁前谁后时，渲染出来同一片区域里两个面交替胜出、形成一道道交错闪烁的撕裂条纹的现象。又称深度冲突。摄像机一动还会闪。常见于贴在墙上的海报、地面上的贴花等共面情形。">深度冲突</Term>（Z-fighting）：同一片区域里，哪个面通过深度测试反复横跳，渲染成一道道交错的撕裂条纹，视角一动还会闪烁。

<ZFightingDiagram />

成因正是上一节那个雷：深度值在远处（或两面太近时）精度不够，两个面的深度值被四舍五入成了同一个数，深度测试只能「随机」二选一，于是斑驳撕裂。怎么治？§7 会给三种实用修法。

## 公式：深度为什么非线性，以及怎么线性化回去

本节两件事：写出深度的非线性映射、给出把它**线性化**还原成真实距离的反算公式。先约定符号：

- $z$：片段在**视图空间**里离镜头的真实距离（沿镜头朝向，正值）
- $\text{near}$：近平面距离（视锥体最近的那张面，坐标系统章见过）
- $\text{far}$：远平面距离
- $z_\text{depth}$：写进深度缓冲的那个 $0\sim1$ 的非线性深度值
- $z_\text{ndc}$：标准化设备坐标（NDC）里的深度，范围 $-1\sim1$

**非线性映射**。透视投影会把视图空间的 $z$ 按下面这种「与 $1/z$ 成正比」的方式压进 $0\sim1$：

$$
z_\text{depth} = \frac{1/z - 1/\text{near}}{1/\text{far} - 1/\text{near}}
$$

这个式子在说：深度值不是直接拿真实距离 $z$ 来线性缩放，而是拿它的**倒数** $1/z$ 来缩放。因为 $1/z$ 在 $z$ 小（近）时变化剧烈、在 $z$ 大（远）时变化迟缓，所以近处一点点距离变化就能让 $z_\text{depth}$ 明显挪动（精度高），远处挪很大距离 $z_\text{depth}$ 才动一点（精度低）——这就是「近密远疏」的来源，也对上了上面那张精度图。

**线性化反算**。深度可视化时，我们想把这个非线性的 $z_\text{depth}$ 还原回正比于真实距离的线性值。第一步，把 $0\sim1$ 的深度值映射回 $-1\sim1$ 的 NDC 深度：

$$
z_\text{ndc} = 2 \cdot z_\text{depth} - 1
$$

这个式子在说：深度缓冲存的是 $0\sim1$，而 NDC 约定 $-1\sim1$，所以乘 2 减 1 把区间拉伸平移过去——纯粹的区间换算。第二步，套透视投影的逆运算，从 NDC 深度反解出视图空间的真实距离 $z$：

$$
z = \frac{2 \cdot \text{near} \cdot \text{far}}{\text{far} + \text{near} - z_\text{ndc} \cdot (\text{far} - \text{near})}
$$

这个式子在说：把上一步得到的 $z_\text{ndc}$ 代进去，就能解出这个片段在视图空间里的**真实距离** $z$（结果落在 $\text{near}\sim\text{far}$ 之间）。最后若想再画成 $0\sim1$ 的灰度，把 $z$ 除以 $\text{far}$ 归一即可。下面 Demo 里的「线性化」开关，切的就是「直接显示 $z_\text{depth}$」与「跑完这两步再显示」的区别。

## 动手：把深度画成灰度，亲眼看非线性

## 代码对照：开启深度测试这几行

深度测试的代码出奇地短——绝大部分活 GPU 自动干了，你只要「开开关、每帧清账本」。先看最关键的两件事：**启用深度测试** + **每帧清深度缓冲**。

<CodeTabs>
  <Tab label="C++ / OpenGL 3.3">

```cpp
// 初始化时开一次：默认是关的，不开就没有遮挡判定
glEnable(GL_DEPTH_TEST);

// 渲染循环里每帧开头：颜色缓冲 + 深度缓冲一起清
glClear(GL_COLOR_BUFFER_BIT | GL_DEPTH_BUFFER_BIT);
```

  </Tab>
  <Tab label="TypeScript / WebGL2">

```ts
// 初始化时开一次：默认是关的，不开就没有遮挡判定
gl.enable(gl.DEPTH_TEST);

// 渲染循环里每帧开头：颜色缓冲 + 深度缓冲一起清
gl.clear(gl.COLOR_BUFFER_BIT | gl.DEPTH_BUFFER_BIT);
```

  </Tab>
</CodeTabs>

两端逻辑一字对应：`glEnable(GL_DEPTH_TEST)` 在初始化时开一次；`glClear` 用按位或把颜色和深度两个缓冲一起清——清深度就是把账本重置成「最远」（1.0），这样新一帧任何东西都能先写进去。注意这里的 API 差异：

<Callout type="warn">
  API 差异：桌面是全局函数 + 大写常量 `glEnable(GL_DEPTH_TEST)`；WebGL2
  是上下文方法 + 上下文上的常量 `gl.enable(gl.DEPTH_TEST)`。清缓冲的
  `GL_DEPTH_BUFFER_BIT` 同理写成
  `gl.DEPTH_BUFFER_BIT`。**最容易漏的是每帧清深度缓冲**——只清颜色不清深度，上一帧的深度账本会残留，遮挡逐帧出错。
</Callout>

想换比较规则、或只读深度不写，再加这两行。`glDepthFunc` 换的就是 §3 那张表里的算子；`glDepthMask(GL_FALSE)` 让深度缓冲变只读（仍参与比较，但通过的片段不更新它）——画半透明物体时常用，避免它们互相挡掉。

<CodeTabs>
  <Tab label="C++ / OpenGL 3.3">

```cpp
glDepthFunc(GL_LESS);      // 设比较规则（默认就是 GL_LESS，可改 GL_LEQUAL 等）

glDepthMask(GL_FALSE);     // 深度缓冲只读：参与比较但通过的片段不写回深度
// …画完需要只读深度的物体后，记得 glDepthMask(GL_TRUE) 恢复可写…
```

  </Tab>
  <Tab label="TypeScript / WebGL2">

```ts
gl.depthFunc(gl.LESS); // 设比较规则（默认就是 LESS，可改 gl.LEQUAL 等）

gl.depthMask(false); // 深度缓冲只读：参与比较但通过的片段不写回深度
// …画完需要只读深度的物体后，记得 gl.depthMask(true) 恢复可写…
```

  </Tab>
</CodeTabs>

`depthMask(false)` 只在深度测试已启用时才有意义。改完比较规则或只读状态后记得复原，否则会污染后续物体的绘制。完整可运行源码将在章末提供（高级 OpenGL 篇基建完成后）。

## 容易踩的坑

<Callout type="trap">
  **现象**：3D
  场景里物体的前后遮挡全乱——远处的墙盖住了近处的杯子，画面随相机移动还跳来跳去。**原因**：忘了
  `glEnable(GL_DEPTH_TEST)`（默认是关的），管线退回「谁后画谁赢」，纯按绘制顺序叠色。**修法**：初始化时
  `glEnable(GL_DEPTH_TEST)` 开一次，并确保**每帧**都 `glClear(... |
  GL_DEPTH_BUFFER_BIT)` 清深度缓冲。
</Callout>

<Callout type="trap">
  **现象**：两个贴在一起的面（如墙上的海报、地上的贴花）边缘出现一道道交替闪烁的撕裂杂斑，相机一动就闪——即
  **Z-fighting（深度冲突）**。**原因**：两面几乎共面，深度差小于深度缓冲能分辨的精度，深度测试只能在它们之间「随机」二选一。**修法**：三选一或并用——①别把两物体摆得**完全共面**，给一点点偏移；②把**近平面
  near
  设得尽量远一点**（精度别浪费在贴脸的极近处）；③用**更高精度的深度缓冲**（如 32
  位，牺牲一点性能换精度）。
</Callout>

<Callout type="trap">
  **现象**：把 `gl_FragCoord.z`
  当灰度直接输出做深度可视化，结果画面几乎**一片惨白**，根本看不出远近层次。**原因**：深度值是**非线性**的，精度全堆在近处，可视范围内的值被挤到接近
  1 的高位，肉眼分不出。**修法**：先按 §4
  公式把深度**线性化**还原成真实距离再归一输出——上面 Demo
  把「线性化」开关打开，就是这个效果。
</Callout>

## 小结

- **深度缓冲**和颜色缓冲一样大，为每个像素存一个 0~1 的**深度值**（0 最近、1 最远）
- **深度测试**在片段着色器之后、写颜色之前：新片段更近（默认 `GL_LESS`）才写并更新深度，否则丢弃——遮挡自动正确，和画的先后无关
- 用 `glEnable(GL_DEPTH_TEST)` 开启、每帧 `glClear` 带上 `GL_DEPTH_BUFFER_BIT` 清账本；`glDepthFunc` 换比较规则
- 深度值是**非线性**的（与 `1/z` 成正比），精度堆在近处；要看清远近得先**线性化**还原
- 两面几乎共面会 **Z-fighting**（撕裂闪烁）：别共面 / 拉远 near 平面 / 用更高精度深度缓冲

<Exercises>

**练习 1（改 Demo 代码 / 观察题）** 在上面的深度可视化 Demo 里，把「线性化」开关在关、开之间来回切几次，并把 `near` 拖到最小（0.1）、`far` 拖到最大（50）。回答：① 非线性档下三个圆为什么灰度都偏白、彼此难分？② 把 `near` 调到最小后，画面的灰度对比是更明显还是更糊？为什么？

<Answer>

① 非线性档直接显示 `gl_FragCoord.z` 那种与 `1/z` 成正比的深度值。因为精度全堆在近处，三个圆（哪怕伪深度跨了 0.2~0.92）映射出来的值都被挤到了接近 1 的高位区间，差距很小，所以灰度都偏白、彼此难分。打开线性化后，深度被还原成正比于真实距离的线性值，灰阶才在 near~far 间均匀拉开。

② `near` 调到最小后，对比反而更**糊**（尤其非线性档下）。因为 near 越小，越多精度被浪费在贴脸的极近处，留给可视范围内物体的精度就更少——这也正是 §7 里「把 near 设得远一点能缓解 Z-fighting」的同一个道理：别让宝贵精度浪费在没人看的极近处。

</Answer>

**练习 2（问答题）** 有人说「深度测试就是按物体离镜头的远近排个序，远的先画、近的后画盖上去」。这个说法对吗？深度测试到底是不是靠「控制绘制顺序」来实现遮挡的？

<Answer>

不对。深度测试**不依赖也不改变**绘制顺序。它是**逐片段**地比深度缓冲里已存的深度值：每个片段写入前都核对一次，更近才写、更远就丢，无论这个片段属于哪个物体、那个物体是第几个画的。所以哪怕你先画近的杯子、再画远的墙，墙的片段也会因为深度更大而被丢弃，杯子照样在前面。「远的先画、近的后画」是没有深度测试时（或画半透明物体时）才需要手动排序的做法，和深度测试的自动逐片段比较是两回事。

</Answer>

**练习 3（问答题）** 你做了一张「墙 + 贴在墙上的海报」，海报和墙严丝合缝地共面，结果海报边缘满是闪烁的撕裂条纹。这是什么现象？给出至少两种不改硬件就能缓解的办法，并各说一句为什么管用。

<Answer>

这是 **Z-fighting（深度冲突）**：海报和墙几乎共面，深度差小到深度缓冲分辨不出，深度测试在两者间反复横跳。至少两种缓解：

- **给海报一点点偏移**，让它离墙微微靠前（别完全共面）——这样两面的深度值就能被区分开，不再争同一个深度。
- **把近平面 `near` 设得远一点**——非线性深度的精度都堆在近处，near 越大、整个可视范围（含这堵墙）分到的精度越足，远处也更不容易冲突。

（第三种是换更高精度的深度缓冲，如 32 位，但那要动缓冲配置、牺牲一点性能。）

</Answer>

<Glossary>

<GlossaryItem term="深度缓冲">
  显存里一块和颜色缓冲一样大的内存，为每个像素额外存一个 0~1
  的「深度值」（离镜头的远近）。它和颜色缓冲是孪生兄弟：颜色缓冲存「什么颜色」、深度缓冲存「多远」。屏幕显示颜色缓冲，深度缓冲只在幕后记账供深度测试比较，常见精度
  24 位。又叫 z-buffer。详见本章「深度缓冲」一节。
</GlossaryItem>

<GlossaryItem term="深度值">
  深度缓冲里每个像素存的那个数，范围 0~1，约定 **0 最近、1 最远**（落在近平面记
  0、远平面记 1）。片段着色器里用内建变量 `gl_FragCoord.z`
  能读到当前片段的深度值。详见本章「深度缓冲」一节。
</GlossaryItem>

<GlossaryItem term="深度测试">
  GPU
  在每个片段算完颜色、写进颜色缓冲**之前**做的一道检查：把该片段的深度值和深度缓冲里同位置已存的深度值比一比，按「深度函数」判定——通过才写颜色并更新深度缓冲，不通过就丢弃这个片段。它让三维场景的前后遮挡自动正确，和绘制先后无关。详见本章「深度测试」一节。
</GlossaryItem>

<GlossaryItem term="深度函数">
  决定深度测试「怎样算通过」的比较规则，用 `glDepthFunc` 设置。默认
  `GL_LESS`（新深度更小才通过，近的胜出）；还有
  `GL_LEQUAL`（更小或相等）、`GL_GREATER`（反过来远的胜出）、`GL_ALWAYS`（永远通过=等于关掉测试）等八种算子。详见本章「深度函数」一节。
</GlossaryItem>

<GlossaryItem term="非线性深度">
  深度值随真实距离**不是均匀变化**的，而是与 `1/z`
  成正比——绝大部分精度集中在离镜头很近的范围，越远精度越粗。它是透视投影的副产物，好处是近处遮挡判定特别精确，代价是远处易出深度冲突。详见本章「深度值的精度」一节与
  §4。
</GlossaryItem>

<GlossaryItem term="线性化深度">
  把非线性的深度值还原成正比于真实距离的线性值的过程：先把 0~1 的深度映射回 -1~1
  的 NDC
  深度，再套透视投影的逆运算解出视图空间里的真实距离。深度可视化时必须先线性化，画面才能看出从近到远平滑的灰度渐变。详见本章「深度值的精度」一节与
  §4。
</GlossaryItem>

<GlossaryItem term="深度可视化">
  把片段的深度值当成灰度颜色输出到屏幕，用来「看见」深度缓冲长什么样。直接输出非线性的
  `gl_FragCoord.z`
  会发现画面几乎全白（精度堆在近处）；先线性化再输出，才能看到从近到远平滑的灰阶。详见本章「深度值的精度」一节。
</GlossaryItem>

<GlossaryItem term="深度冲突">
  两个表面靠得极近、深度值小到深度缓冲分辨不出谁前谁后时，同一片区域里两个面交替胜出、渲染成一道道交错闪烁的**撕裂条纹**的现象（又称
  Z-fighting）。摄像机一动还会闪。常见于贴在墙上的海报、地面贴花等共面情形。三种治法：别共面
  / 拉远近平面 / 用更高精度深度缓冲。详见本章「深度冲突」一节与 §7。
</GlossaryItem>
