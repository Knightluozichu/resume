## 为什么 URP 是移动端的正确选择

Unity 有两套渲染管线。旧的那套叫 Built-in，功能全面但像个什么都想干的万能工具箱——桌面端够用，到了移动端就开始拖后腿。新的那套叫 URP（通用渲染管线），是专门为「一个画面同时跑在手机、平板、游戏主机上」设计的。

不用 URP 会怎样？你的移动端项目会发现 Draw Call 降不下去、Shader 改个参数就要重新提交一堆数据、想加个自定义后处理还得从头摸索管线扩展点。URP 把这些坑填平了——它把 Shader 数据批量提交（SRP Batcher）、把画质参数统一管理（URP Asset）、把管线扩展整成插件式（Renderer Feature）。

这一章不是让你背 URP 每个设置项，而是建立一条清晰的思维线：架构长什么样 → 数据怎么批量走 → 配置怎么调 → 前向还是延迟，最后选哪个。读完你就能看懂 URP 文档，知道每个选项在公式里的位置。

## 第一个概念：URP vs Built-in——架构级差异

<Term def="Unity 的可编程渲染管线（Scriptable Render Pipeline）家族中的通用管线，专为跨平台（移动/桌面/主机/Web）设计，替代旧 Built-in 管线。">URP</Term> 和旧 Built-in 管线最根本的区别不在画质——而在**管线本身是不是可拆可改的**。

Built-in 管线像一个黑盒：材质参数（颜色、纹理、Shader 关键字）每次切换材质都要从 CPU 重新打包发送给 GPU，每个不同材质的物体都产生一次 <Term def="每次 GPU 切换渲染状态（Shader/Material/纹理绑定），CPU 都需要发送一组新的管线设置命令；SetPass 次数是衡量 CPU→GPU 通信开销的核心指标。">SetPass Call</Term>。100 个不同材质的物体 = 100 次 SetPass——CPU 端累得半死，GPU 等着接活干。

URP 把这个问题从根上改了：

<UrpVsBuiltinDiagram />

- **SRP Batcher（核心武器）**：Shader 的「大参数」（常量缓冲 CB）按统一布局存，兼容同一 Shader 变体的不同材质只需提交参数内容、不用重建整个管线状态。几百个不同颜色的物体 → 同一批提交。
- **Renderer Feature**：URP 管线不是黑盒，提供了插件式扩展点（`RenderObjects`、`RenderPassEvent`），后处理效果、自定义 Pass 都以 Renderer Feature 形式插入，不碰管线源码。
- **Render Graph（Unity 6 新）**：URP 内部把 Pass 间的资源依赖（RT 创建/复用/释放）自动化管理，减少显式临时 RT 分配和手动清理。

总结一句：Built-in 是「每次换个材质就把整个桌面清空重铺」，URP 是「桌子不动、只换上面的东西」。

## 第二个概念：SRP Batcher 是怎么做到的

SRP Batcher 不是 GPU Instancing 的替代，它是另一种机制——两者经常被混淆，但它们解决的是不同层面的问题：

<SrpBatcherFlowDiagram />

- **GPU Instancing**：1 个 Draw Call 画 N 个**完全相同**的网格+材质。前提是网格和材质一模一样（草、子弹、相同 Prefab 的物体群）。
- **SRP Batcher**：N 个**不同材质**但**同一 Shader 变体**的物体，共享一次管线状态设置。前提是 Shader 兼容 SRP Batcher（URP/HDRP 的 Lit/Simple Lit/Unlit 等内置 Shader 默认兼容）。

> 打个比方：GPU Instancing 是一张蓝图复印 N 份同一张纸；SRP Batcher 是同一台打印机、不停换不同颜色纸——不用重新装墨盒（管线状态），只换纸（材质参数）。

SRP Batcher 生效的前提：Shader 必须声明 `CBUFFER_START(UnityPerMaterial)` 把所有逐材质属性包进去。URP 内置 Shader 都这么写了，自定义 Shader 如果照做也能享受 Batcher 加速。

下面是一个兼容 SRP Batcher 的自定义 Shader 片段：

```hlsl
CBUFFER_START(UnityPerMaterial)
    float4 _BaseColor;
    float _Metallic;
    float _Smoothness;
    // 所有逐材质属性和纹理声明都放这里
CBUFFER_END
```

Shader 声明了这个后，URP 的 SRP Batcher 就能把不同材质实例的 `_BaseColor`/`_Metallic` 等参数打包进同一个常量缓冲布局、批量提交到 GPU，不用重新设置管线状态。

<Callout type="tip">
  验证 SRP Batcher 是否生效：打开 Frame Debugger，切换一帧查看每个 Draw Call——如果
  SRP Batcher 激活，你会看到 `SRP Batch` 标签而不是 `SetPass
  Call`。影响 Batcher 的常见因素：Shader 不兼容（非 URP/HDRP 或未声明
  UnityPerMaterial CB）、使用了 MaterialPropertyBlock（会中断 Batcher）。
</Callout>

## 第三个概念：URP Asset——画质与性能的中控台

URP 的所有质量/性能参数集中在一个东西里：<Term def="URP 的核心配置资产（.asset 文件），集中管理渲染比例、MSAA、阴影距离、后处理、中间纹理格式等所有画质与性能参数。">URP Asset</Term>。你不需要分别跑到 Project Settings、Quality Settings、Graphics Settings 里各改各的——URP Asset 就是一个中控台。

<UrpAssetConfigDiagram />

关键参数速查：

| 参数 | 作用 | 移动端建议 |
|------|------|-----------|
| **Render Scale** | 缩放 eye buffer 分辨率 | 0.7–0.85（GPU bound 时优先降此项） |
| **MSAA** | 多重采样抗锯齿 | 2x 或关（移动 Tile GPU 对 MSAA 敏感） |
| **Shadow Distance** | 阴影可见范围（米） | 30–50m，大于此距离不渲阴影 |
| **Cascade Count** | 方向光阴影级联层数 | 2（移动端远距阴影可关） |
| **HDR** | 浮点颜色缓冲 | 关（移动端 HDR 增加带宽，画面提升有限） |
| **Depth/Color Texture** | 是否暴露 `_CameraDepthTexture`/`_CameraColorTexture` | 仅在自定义效果需要时开启 |

同一项目可以有多个 URP Asset（如 `URP-High.asset`、`URP-Medium.asset`、`URP-Low.asset`），在 Quality Settings 里对应 High/Medium/Low 三档——这就是画质分级的基础设施。

## 第四个概念：前向 vs 延迟——该怎么选

<Term def="逐物体逐光源计算光照的渲染方式：每个物体被每盏灯照亮一次，光照计算嵌套在物体渲染循环内——移动端默认选择。">前向渲染</Term> 和 <Term def="先把场景几何信息（位置/法线/反照率）渲进多张 G-Buffer、再单独对每个像素逐光源计算光照的渲染方式——光源多时效率高，但对带宽和 G-Buffer 格式要求高。">延迟渲染</Term> 是两条不同的光照计算路，没有绝对对错，只有场景适配问题：

```csharp
// 在 URP Asset 的 Inspector 面板中切换
// 前向渲染（移动端推荐）：每盏额外灯成本 = 物体片元数 × 灯数
//   → 适合：场景灯少（一般 ≤4 盏实时灯）、移动端、VR
// 延迟渲染（高端平台）：每盏额外灯成本 = 屏幕像素数 × 灯数
//   → 适合：多灯光场景（几十盏）、PC/主机、光照计算 >> 带宽成本
```

<Callout type="warn">
  延迟渲染在移动端不推荐——G-Buffer 需要多张高精度 RT（位置/法线/反照率/镜面），对移动
  GPU 的 tile memory 和带宽是沉重负担。移动端老老实实选前向。如果场景确实需要几十盏动态灯
  ——优先考虑烘焙 Lightmap 或 Light Probe 替代实时灯，而不是切延迟。
</Callout>

## Adaptive Probe Volumes：Unity 6 的新探针方案

<Term def="Unity 6 中替代旧 Light Probe Group 的光照探针系统：自动在场景中布设 3D 探针体积网格，按需细分（靠近几何体更密），减少手动布置工作量并改善动态物体的间接光照采样质量。">Adaptive Probe Volumes</Term> 是 Unity 6 URP 引入的新光照方案，替代了旧的 Light Probe Group 手动布置方式：

<Callout type="tip">
  Adaptive Probe Volumes 的核心改进：自动细分——不用再手动在场景里摆几百个 Probe
  球体。在 Lighting 窗口里创建 Adaptive Probe Volume，Unity
  自动根据场景几何密度生成探针网格（墙边密、空旷处疏），动态物体进入不同区域自动采样最近探针。
  配合 Lightmap 烘焙使用，可大幅减少实时灯数量。
</Callout>

    void Start()

## 容易踩的坑

<Callout type="trap">
  **现象**：项目切到 URP 后 Draw Call 没减少，和 Built-in 时差不多。**原因**：SRP Batcher 缩短的是 SetPass Call（管线状态切换）而非 Draw Call 数本身——如果看 Frame Debugger 发现 SetPass 从 200 降到 20 但 Draw Call 还是 500，Batching 已经在工作了，你盯错指标了。**修法**：看 SetPass 数量而非 Draw Call；合批（GPU Instancing/Static Batching）才降 Draw Call 数。
</Callout>

<Callout type="trap">
  **现象**：在 URP Asset 里开了 HDR，移动端帧率掉了一半。**原因**：HDR 需要 RGBA16F 浮点格式的颜色缓冲，数据量是普通 RGBA8 的 2 倍——移动 GPU 的 tile memory 和带宽压力直接翻倍。**修法**：移动端关掉 HDR（大多数手机屏幕也不支持真正的 HDR 显示）；如果确实需要 HDR 效果（如 Bloom），优先考虑伪 HDR 后处理技巧替代。
</Callout>

<Callout type="trap">
  **现象**：自定义 Shader 死活不进入 SRP Batcher，Frame Debugger 里始终显示 SetPass。**原因**：Shader 没声明 `CBUFFER_START(UnityPerMaterial)` 逐材质常量缓冲、或者使用了 `MaterialPropertyBlock`（中断 Batcher 的最常见原因）。**修法**：按本章第二个概念的代码片段重构 Shader 属性声明；避免对需要合批的物体使用 MaterialPropertyBlock——如果确实需要运行时改属性，考虑用不同的 Material 实例（SRP Batcher 支持不同材质参数但同一 Shader 变体）。
</Callout>

## 小结

- **URP ≠ 降画质版 Built-in**——它是可编程管线，SRP Batcher 批量提交材质参数，Renderer Feature 插件化扩展
- **SRP Batcher** 降 SetPass、不降 Draw Call——适合「同 Shader 不同材质参数」的海量物体场景
- **URP Asset** 是中控台：Render Scale、MSAA、Shadow Distance、HDR 四个参数是移动端调优四件套
- **前向 vs 延迟**：移动端选前向（≤4 盏实时灯 + Lightmap 烘焙），多灯场景先烘焙再考虑延迟
- **Adaptive Probe Volumes**（Unity 6）：自动布探针网格，替代手动 Probe Group

<Exercises>

**练习 1（操作题）** 打开你的 URP 项目的 Frame Debugger，找到一帧中的任意一个 Draw Call，记录它的 SetPass Call 数量和 SRP Batch 标签。然后关掉 Graphics Settings 里的 SRP Batcher 重新 Build，再看同一帧的 SetPass Call 数——对比前后的变化。

<Answer>

操作步骤：Window → Analysis → Frame Debugger → Enable，点击一帧展开渲染顺序。在左侧树中找到带 `Draw Mesh` 标签的条目，右侧面板会显示 `SRP Batch`（如果 Batcher 激活）或 `SetPass Call`。记录数量后，Edit → Project Settings → Graphics → 取消勾选 `SRP Batcher`，重新 Play 并 Frame Debugger 再截同一帧，对比 SetPass Call 变化。合理预期：SRP Batcher 激活时 SetPass 大幅减少（从数百到几十）。

</Answer>

**练习 2（问答型）** 一个场景里有 200 棵树，每棵树用不同颜色的同一材质（Lit Shader）。在这个场景中，SRP Batcher 和 GPU Instancing 哪个更有效？为什么？

<Answer>

SRP Batcher 更有效。因为 200 棵树材质不同（颜色不同），GPU Instancing 要求网格和材质完全相同才能一次画多份——不同颜色意味着不同材质实例，Instancing 不生效。SRP Batcher 正好解决这个场景：同一 Shader 变体（Lit）、不同材质参数（不同颜色），通过统一的常量缓冲布局批量提交，大幅减少 SetPass。

</Answer>

**练习 3（选型题）** 你正在做一个移动端室内场景项目：一个房间里有 6 盏实时点光源 + 窗外方向光。分析你为什么不该在移动端切到延迟渲染来优化这 6 盏灯——并给出替代方案。

<Answer>

移动端选延迟渲染不划算：G-Buffer（至少 4 张 RT）每帧消耗的 tile memory 和带宽远超这 6 盏灯在前向渲染下的额外开销。替代方案：①把 6 盏灯里大部分改成 Baked（Lightmap 烘焙），只留 1–2 盏关键灯做实时；②用 Light Probe 给动态物体补齐间接光照；③考虑 Adaptive Probe Volumes（Unity 6）自动布探针。对于移动端室内场景，前向 + 烘焙 + 探针是标准解法。

</Answer>

<Glossary>

<GlossaryItem term="URP">
  Unity 可编程渲染管线家族中的「通用管线」，专为跨平台设计（手机/平板/PC/主机/Web）。替代旧 Built-in 管线，内置 SRP Batcher、Renderer Feature 插件机制、Render Graph 自动资源管理。详见本章「第一个概念」。
</GlossaryItem>

<GlossaryItem term="SRP Batcher">
  URP/HDRP 的核心批量提交技术：把兼容同一 Shader 变体的不同材质的参数（颜色/纹理/数值）按统一常量缓冲布局打包，减少 CPU→GPU 管线状态切换的次数（降 SetPass 而非 Draw Call）。详见本章「第二个概念」。
</GlossaryItem>

<GlossaryItem term="SetPass Call">
  每次 GPU 切换渲染状态（换 Shader 或材质）时，CPU 重建管线设置的那次指令通信。SetPass 数量是 CPU 端渲染开销的关键指标——降 SetPass 就是降 CPU 负担。详见本章「第一个概念」。
</GlossaryItem>

<GlossaryItem term="URP Asset">
  URP 管线所有质量/性能参数的集中配置文件（.asset 后缀）。Render Scale/MSAA/Shadow Distance/Cascade Count/HDR 等都在这里改。同一项目可有多个 Asset 对应不同画质档位。详见本章「第三个概念」。
</GlossaryItem>

<GlossaryItem term="Renderer Feature">
  URP 的管线扩展插件机制：不碰管线源码就能在渲染流程中插入自定义 Pass（如自定义后处理、覆盖渲染、深度/模板操作），在 URP Asset 的 Renderer Data 中添加。详见本章「第一个概念」。
</GlossaryItem>

<GlossaryItem term="前向渲染">
  逐物体×逐光源计算光照：每个物体被场景中每盏实时灯各照一次。移动端和 VR 默认选择——灯少时效率高，灯多时成本暴增（物体片元数×灯数）。详见本章「第四个概念」。
</GlossaryItem>

<GlossaryItem term="延迟渲染">
  先把场景的几何信息（位置/法线/反照率）渲进多张 G-Buffer，再对每个屏幕像素逐光源算一次光照。多灯光场景效率极高（光照成本=屏幕像素数×灯数，与物体数无关），但 G-Buffer 带宽负担大，移动端不推荐。详见本章「第四个概念」。
</GlossaryItem>

<GlossaryItem term="Adaptive Probe Volumes">
  Unity 6 替代旧 Light Probe Group 的方案：自动根据场景几何生成 3D 探针体积网格，靠近几何体自动细分更密。动态物体进入区域后自动采样最近探针获取间接光照。详见本章「Adaptive Probe Volumes」节。
</GlossaryItem>

</Glossary>
