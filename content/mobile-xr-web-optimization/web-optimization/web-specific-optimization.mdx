## 为什么把 Unity 带到浏览器

十年前，在浏览器里跑 3D 游戏是个笑话——Flash 都卡得不行，别说 3A 级渲染。2023 年，你在任何一个 Chromium 内核的浏览器里打开 itch.io，能直接玩到用 Unity 6 构建的全 3D 游戏——不需要下载、不需要安装、不需要高端显卡。

这一切依赖三样东西：**WebGL 2.0**（浏览器里的 GPU API）、**WebAssembly**（接近原生的 C++ 运行速度）、以及 **Unity 的 IL2CPP Web 构建管线**——这套组合把游戏引擎编译成一个 `.wasm` 二进制 + JavaScript 胶水代码，在浏览器的沙箱里直接运行。

代价也有：Wasm 的 CPU 性能约是原生 C++ 的 60-80%、WebGL 2 没有 Compute Shader（也无法用最新的 GPU 功能）、浏览器对内存和线程的访问有严格的安全限制。**Web 专项优化就是在这个受限的沙箱里，尽力逼近原生体验。**

## 1. WebAssembly 2023——从"能用"到"好用"

**WebAssembly 的进化不是渐进的——2023 年的四个核心特性让它从"能跑"跨越到"接近好用"。**

<WebAssemblyFeatureDiagram />

```csharp
// Unity 6 中确认 Wasm 特性开启的方法：
// 在 PlayerSettings → WebGL → Player → Other Settings →
//   Enable WebAssembly SIMD（默认开启）
//   Enable WebAssembly 2023（Unity 6 默认开启）
//
// 构建后可通过浏览器 Console 验证：
// console.log(Module.wasmFeatures);
// 输出应包含：simd, bulk-memory, threads
```

<Term def="SIMD 128：一条 CPU 指令同时处理 4 个 32 位浮点数——向量加法/乘法/点乘一次做完，数学库直接受益 2-4 倍加速">SIMD 128</Term> 对 Unity 游戏的影响最大：物理引擎的向量运算、蒙皮动画的矩阵乘法、粒子系统——这些以前要被编译成一串串标量指令的工作，现在一条 SIMD 指令搞定。

<Term def="Bulk Memory：Wasm 的批量内存操作指令集——memcpy/memset 由单条指令完成而非逐字节循环，AssetBundle 解压和场景加载瞬间加速 3-5 倍">Bulk Memory</Term> 则解决了 Web 端最慢的一环——加载。AssetBundle 从服务器下载后需要解压进 Wasm 线性内存——以前是 JavaScript 逐字节拷过去，Bulk Memory 用 C 级原生指令直接搬整个内存块——3-5 倍提速。

## 2. Chrome DevTools——Web 端的 Unity Profiler

Unity Profiler 在 Web 端不可用——浏览器的安全沙箱阻止了引擎建立网络 socket 来传输 Profiler 数据。你的工具是**浏览器自带的开发者工具**。

```javascript
// 在浏览器 Console 中可手动触发 Unity 内部的一些性能数据
// 前提：Build 时勾选了 Development Build
// unityInstance 是全局的 Unity 游戏实例

// 获取当前帧时间（毫秒）
console.log(Module['_getFrameTime']());

// 或者通过 Unity 的 SendMessage 获取自定义数据
// C# 端：
public class WebPerfReporter : MonoBehaviour {
    void Update() {
        float frameTime = Time.unscaledDeltaTime * 1000f;
        // 把数据通过 Application.ExternalCall 推到 JS
        Application.ExternalCall("onFrameTimeUpdate", frameTime);
    }
}
```

**Chrome DevTools Performance 面板操作流程：**

1. 打开页面 → F12 → Performance 面板
2. 点击录制按钮（圆点）→ 等待游戏运行 5-10 帧 → 停止
3. 在时间线中找到 **F 标记**（frame）——hover 看每帧耗时
4. 展开 Main 线程火焰图——紫红色块是 Wasm 代码执行（`wasm-to-js`），黄色块是 JS 脚本，绿色块是 GPU 等待
5. Memory 面板 → 录制堆快照 → 检查 Wasm Memory 和 JS Heap 是否持续增长

<Callout type="tip">
  **看懂火焰图**：DevTools 把 Wasm 函数名渲染成了 C++ mangled 格式（`__Z10MyFunctionv` 之类）。可以用 Chrome 的 "demangle" 扩展或 Wasm dissasembler 还原成可读的函数名——或者简单地在 C# 中给函数起名字足够独特以便搜索。
</Callout>

## 3. Web 纹理格式与内存

Web 端没有 ASTC/ETC2 硬件解码器——纹理压缩格式只能选浏览器 WebGL 实现直接支持的：

```csharp
// Unity Texture Importer → Platform-specific Overrides → WebGL
//
// 推荐格式：
// DXT5 (RGBA) / DXT1 (RGB) — 桌面 Chrome/Firefox/Edge 全部支持
// ETC2 (RGB) / ETC2_EAC (RGBA) — 移动浏览器 Android/iOS Chrome 支持
//
// WebP 纹理暂不支持——WebGL 不认 WebP 作为 GPU 纹理格式
//
// Build 后的纹理内存用量：
// 1024×1024 RGBA DXT5 = 512KB（1/8 原大小）
// 1024×1024 RGB  DXT1 = 512KB（1/8 原大小）
```

<Callout type="warn">
  **Web 端纹理内存限制**：iOS Safari 的 WebGL 内存上限约 1GB——但这包含 Wasm 堆 + JS 堆 + GPU 纹理。一张完全不压缩的 4K 贴图（4096×4096 RGBA）就是 64MB——游戏里拉 20 张就撞墙。务必用 DXT/ETC2 压缩所有 Web 纹理，且对 >1024 的纹理用 Mipmap（引入额外 1/3 内存，但采样性能省很多）。
</Callout>

## 4. AssetBundle 远程加载——Web 端必须做

APK 可以 150MB+，Web 游戏首次加载必须 ≤ 50MB（否则用户等不起）。解决方案就是 **AssetBundle 按需从服务器加载**。

```csharp
// Web 端 AssetBundle 加载必须用 UnityWebRequest（不能用 LoadFromFile）
using UnityEngine.Networking;
using System.Collections;

public class WebAssetLoader : MonoBehaviour
{
    public string bundleURL = "https://cdn.example.com/bundles/mylevel";

    IEnumerator Start()
    {
        using var request = UnityWebRequestAssetBundle.GetAssetBundle(bundleURL);

        // Web 端设置超时（网络比本地 IO 不可靠得多）
        request.timeout = 30;

        yield return request.SendWebRequest();

        if (request.result != UnityWebRequest.Result.Success)
        {
            Debug.LogError($"下载失败: {request.error}");
            yield break;
        }

        var bundle = DownloadHandlerAssetBundle.GetContent(request);

        // 加载 Bundle 中的资源
        var prefab = bundle.LoadAsset<GameObject>("MyPrefab");
        Instantiate(prefab);

        // 记得卸载 Bundle（但不卸载已实例化的资源）
        bundle.Unload(false);
    }
}
```

<Callout type="trap">
  **CORS 头缺失**：Web 游戏从 `cdn.example.com` 加载 AssetBundle 时，该服务器必须返回 `Access-Control-Allow-Origin` 头——不然浏览器出于安全策略直接拒绝加载。本地测试时 `localhost` 加载本地文件通常不受限，但一部署到生产就挂——**提前确认 CDN 配置 CORS 头。**
</Callout>

## 5. 浏览器兼容性清单——三座大山

**① WebGL 2.0 支持**：Chrome 56+（2017 年 1 月）、Firefox 51+（2017 年 1 月）、Safari 15+（2021 年 9 月）。任何不支持 WebGL 2 的浏览器完全无法运行 Unity Web 游戏——但 2025 年往后这个限制几乎只影响 IE 11（已退役）。

**② SharedArrayBuffer 和跨域隔离**：Unity 多线程依赖 `SharedArrayBuffer`——它需要服务器返回两个 HTTP 响应头：

```
Cross-Origin-Opener-Policy: same-origin
Cross-Origin-Embedder-Policy: require-corp
```

缺这两个头 → `SharedArrayBuffer` = `undefined` → Unity 回退为单线程 → 性能差 50%+。

**③ iOS Safari 的 AudioContext 限制**：iOS Safari 不允许在用户触摸屏幕前播放音频（`AudioContext` 处于 `suspended` 状态）。Unity Web 必须监听第一个 `touchend` 事件并调用 `audioContext.resume()`——否则游戏全程静音。Unity 6 的默认 WebGL 模板已处理此限制。

<Callout type="warn">
  **别忘了移动浏览器**：你的 Web 游戏可能在桌面 Chrome 里跑 60FPS，在 iPhone Safari 里掉到 15FPS。iOS 的 WebGL 性能限制更多：无 SIMD（直到 iOS 16.4）、内存上限更低、散热降频更快。**必须用真机测。**
</Callout>

## 容易踩的坑

<Callout type="trap">
  **现象**：Web 游戏在本地 `localhost` 测试正常，部署到服务器后 Wasm 文件加载失败——控制台提示 "unsupported MIME type"。**原因**：服务器没有配置 `.wasm` 文件的 MIME 类型。**修法**：服务器配置中添加 `application/wasm` MIME 类型映射。对 Nginx：`types { application/wasm wasm; }`；对 S3/CloudFront：上传时指定 Content-Type。
</Callout>

<Callout type="trap">
  **现象**：Web 游戏加载后每 3-5 秒出现一个规律性的帧率尖刺，标记为 `idle`。**原因**：浏览器的 JavaScript GC 在主线程上的 stop-the-world 回收——Unity 的 IL2CPP 生成的 Wasm 代码不会触发 GC，但 C# 对象的分配会映射到 JS 堆上的创建/销毁。**修法**：减少 C# 的堆分配——用对象池代替 Instantiate/Destroy、Strcut 代替 class（值类型不在堆上）、PlayerSettings → Strip Engine Code 剥离未用代码→减少 Wasm 体积也减少 JS 胶水代码的 GC 压力。
</Callout>

<Callout type="trap">
  **现象**：`UnityWebRequestAssetBundle.GetAssetBundle` 下载完成后 `LoadAsset` 返回 null。**原因**：AssetBundle 文件本身的版本或平台不匹配——Web 端要用 WebGL 平台下的 AssetBundle（`BuildTarget.WebGL`）。**修法**：确保 AssetBundle 是针对 WebGL 构建的。在 AssetBundle Browser 工具中选择 Build Target = WebGL 重新打包。
</Callout>

## 小结

- WebAssembly SIMD 128 加速向量/矩阵运算 2-4×、Bulk Memory 加速 AssetBundle 解压 3-5×——Unity 6 默认开启
- Chrome DevTools Performance 面板是 Web 端的唯一 Profiler——学会看 F 标记、火焰图和 Wasm 代码占用
- Web 纹理用 DXT5/DXT1（桌面）和 ETC2（移动浏览器），iOS Safari WebGL 内存上限 ~1GB
- AssetBundle 远程用 `UnityWebRequest` 加载（禁 `LoadFromFile`）——服务器必配 CORS 头
- 兼容性三座山：WebGL 2 浏览器支持 + COOP/COEP 头开启 SharedArrayBuffer + iOS Safari AudioContext 解锁

**练习 1（改代码型）** 为一个 Unity Web 项目添加运行时帧时间监控：每 60 帧（1 秒）把平均帧时间通过 `Application.ExternalCall` 发送到浏览器的 `console.log`。在 DevTools Performance 面板中交叉验证这两个数据是否一致。

<Answer>
```csharp
using System.Collections.Generic;

public class FrameTimeTracker : MonoBehaviour
{
    Queue<float> frameTimes = new();
    const int reportInterval = 60;

    void Update()
    {
        frameTimes.Enqueue(Time.unscaledDeltaTime * 1000f);
        if (frameTimes.Count > reportInterval)
            frameTimes.Dequeue();
    }

    IEnumerator ReportLoop()
    {
        while (true)
        {
            yield return new WaitForSeconds(1f);
            float avg = 0;
            foreach (var t in frameTimes) avg += t;
            avg /= frameTimes.Count;
            Application.ExternalCall("console.log", $"Avg frame: {avg:F1}ms");
        }
    }
}
```
</Answer>

**练习 2（问答型）** 为什么 Web 端的 `SharedArrayBuffer` 需要特殊的 HTTP 响应头？这两个头各自起什么作用？

<Answer>
`Cross-Origin-Opener-Policy: same-origin` 把当前页面和最顶层的浏览上下文绑定在同一个 origin 下——防止跨域页面通过 `window.open` 访问 SharedArrayBuffer。`Cross-Origin-Embedder-Policy: require-corp` 要求页面上所有跨域资源（图片、脚本等）必须明示允许被嵌入——防止恶意页面把你的资源加载进自己的页面后利用 SharedArrayBuffer 做侧信道攻击。两个头合在一起 = 浏览器信任你这个页面是"隔离安全的"——才允许使用共享内存。
</Answer>

**练习 3（实现型）** 写一个 WebGL 平台的资源管理器：用 `UnityCache`（IndexedDB 缓存）实现 AssetBundle 的本地缓存——首次下载后存入缓存，下次直接从缓存加载，只在服务器版本号更新时重新下载。

<Answer>
```csharp
IEnumerator LoadBundleCached(string bundleName, uint version)
{

    {
        Debug.LogError($"加载失败: {request.error}");
        yield break;
    }

    var bundle = DownloadHandlerAssetBundle.GetContent(request);
    // 使用 bundle...
}
// Unity 6 的 UnityWebRequestAssetBundle 已内置 IndexedDB 本地缓存
// 只需提供 version 参数——版本号变化时自动失效缓存并重下载
```
</Answer>

<Glossary>

<GlossaryItem term="WebAssembly（Wasm）">一种二进制指令格式，在浏览器里以接近原生速度运行 C/C++/Rust 编译产生的代码。Unity IL2CPP 把 C# 编译成 Wasm——是现代 Web 3D 游戏的关键。《类比》：浏览器是操作系统，Wasm 是它的 exe 格式。

<GlossaryItem term="SIMD 128">"单指令多数据"——一条 CPU 指令同时处理 4 个 float。向量加法 (x1+x2,y1+y2,z1+z2,w1+w2) 原本需要 4 条指令，SIMD 一条搞定——Unity Mathematics 包直接受益。

<GlossaryItem term="Bulk Memory">Wasm 的批量内存操作：`memory.copy`（一次性搬一个内存块到另一位置）、`memory.fill`（一次性填充一块内存）——替代逐字节循环。AssetBundle 解压从 JS 字节拷贝变 Wasm 内存块传输——3-5 倍加速。

<GlossaryItem term="CORS（Cross-Origin Resource Sharing）">浏览器的跨域安全机制——A 域名的网页加载 B 域名的资源时，B 必须明确说"允许 A 来拿"。CDN 上放的 AssetBundle 必须返回 `Access-Control-Allow-Origin` 头——否则浏览器拒绝加载。

<GlossaryItem term="SharedArrayBuffer">让 Web Workers 和主线程共享同一块内存——Unity 多线程依赖它。浏览器出于安全（Spectre 漏洞）默认禁用——需要 COOP/COEP 头显式授权。

</Glossary>
