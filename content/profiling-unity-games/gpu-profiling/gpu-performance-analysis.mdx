## 为什么需要 GPU 分析

你的 CPU 跑得很轻松——每帧只花了 5ms——但帧率还是到了瓶颈。CPU 在悠闲地喝茶，GPU 却在疯狂加班。看不到 GPU 侧的数字，你只能对着 60 帧到突然 25 帧的诡异现象干瞪眼。

没有 GPU 分析会怎样？你会把所有问题都归咎于 CPU——花大量时间优化逻辑代码，结果帧率一动都不动。你甚至不知道根本原因是：场景里一个透明粒子特效的层叠层数太多，导致同一像素被反复绘制了十几次（Overdraw），GPU 的填充率被撑爆了——这种问题改 CPU 代码永远改不好。

这一章教你看懂 GPU 侧的性能数据。从 Draw Call 到 Fill Rate 到 Memory Bandwidth，GPU 的瓶颈和 CPU 的瓶颈长得完全不一样——你得用不同的眼睛去看。

## 第一个概念：GPU Profiler——看 GPU 在忙什么

<Term def="Unity Profiler 中专门记录 GPU 侧绘制耗时的模块：显示每帧 GPU 花费的时间总量、每个渲染阶段的耗时分解（不透明物体/透明物体/后处理/阴影等），与 CPU Usage 中的 Rendering 类别是「配套对」——CPU Rendering 是「准备命令」，GPU Usage 是「执行命令」。">GPU Usage Profiler</Term>
是 GPU 分析的入口。它和 CPU Usage 里的 Rendering 类别是「上下楼」的关系：CPU 侧的 Rendering 条是「给 GPU 发命令的准备时间」（构建命令缓冲区、提交数据到驱动），GPU Usage 显示的是「GPU 真的花在画图上的时间」——后者才决定你能不能按时交帧。

<GPUProfilerOverviewDiagram />

GPU Profiler 的关键指标：

- **GPU ms**：当前帧 GPU 花的总毫秒数。GPU bound 时这个值会接近帧预算（60 FPS → 16.6ms）。
- **渲染阶段分解**：不透明物体（Opaque）、透明物体（Transparent）、后处理（Post-Processing）、阴影贴图（Shadow Maps）——每个阶段有独立的耗时条。如果透明物体阶段特别长，说明透明层叠太多或者透明材质 Shader 太复杂。
- **Draw Call / Batches / SetPass**：渲染命令提交的统计数字，在 GPU Profiler 底部单独列出——这些数字在下一节单独展开。

GPU Profiler 在整个渲染管线中的位置是**管线的尾巴**：它衡量的是从顶点着色器 → 光栅化 → 像素着色器 → 输出到帧缓冲这一整套 GPU 硬件执行的时间。而 CPU 侧的 Rendering 衡量的是「驱动准备阶段」。两者加在一起才是从「说画什么」到「真的画出来」的完整路径。

## 第二个概念：Draw Call、Batches 与 SetPass——渲染命令的三层结构

这三个数字是 GPU Profiler 里最常被盯着的三个指标——但理解错一个、优化方向就会跑偏。按从大到小的层次来讲：

<DrawCallExplanationDiagram />

<Term def="CPU 向 GPU 发送的一次「画东西」命令：每一条命令包含要画的网格引用、材质引用、以及变换矩阵。Draw Call 本质是 CPU-GPU 通信的次数——Draw Call 越多，驱动和 CPU 的准备开销越大。">Draw Call</Term>
是 CPU 告诉 GPU「画这个」的次数。一条 Draw Call = 一个网格的一段索引区间 + 一个材质 + 一组变换。Draw Call 多本身不一定让 GPU 跑得慢——GPU 执行 Draw Call 是很快的——但**每一条 Draw Call 都需要 CPU 侧准备命令、验证状态、提交到驱动**。上千条 Draw Call 意味着 CPU 侧花了大量时间在这上面——你看到的表象是 GPU 侧「没什么事干」，实际是 CPU 来不及给 GPU 喂命令。

<Term def="Unity 渲染统计里代表「渲染命令批次数」的指标：多个使用相同材质和变换条件的 Draw Call 可以被批处理（Batch）为一次底层 API 调用——Batches = 真正发到驱动的绘制调用数量。">Batches</Term>
是 Unity 在 CPU 侧做完了静态/动态合批（Batching）之后的**实际命令数量**。如果 Draw Call 是 500 但 Batches 是 200——说明有 300 个 Draw Call 被合批了，Unity 把它们合并成了 200 条实际命令。Batches 比 Draw Call 更接近「驱动到底收到了多少条命令」。**优化目标不是 Draw Call = 0，而是 Batches 在目标平台的合理范围内**（PC 放宽到几百都行，移动端最好压到 50 以内）。

<Term def="Unity 渲染统计里代表「材质切换次数」的指标：每调用一次 `Graphics.SetRenderTarget` 或切换一次 Shader Pass，SetPass 计数 +1。SetPass 高 = 材质很杂，GPU 频繁切换着色器程序——合批的最大障碍。">SetPass</Term>
是最容易被忽视但最关键的数字。SetPass = 材质 / Shader 的切换次数——每次切换 Shader 或 Render Pass，GPU 都要刷新管线状态（换着色器程序、换采样器、换混合模式……），这套刷新操作本身开销不低。想合批？前提是同一个 Shader、同一个材质参数——SetPass 越高，说明你的材质种类越分散，合批的空间越小。

三者关系一句话总结：**Draw Call 多 → Unity 自动合批 → 看 Batches 是否下降；Batches 还是高 → 看 SetPass——SetPass 高说明材质 / Shader 种类太杂，合批失败**。

## 第三个概念：Fill Rate 与 Memory Bandwidth——GPU 的「吞吐量瓶颈」

如果 Draw Call 和 SetPass 都很正常（Batches 在几十以内、SetPass 在个位数），但 GPU ms 还是很高——那么瓶颈在别的地方：

<Term def="GPU 每秒能向屏幕写入的像素总量（通常是 ROP —— 渲染输出单元的能力上限）。当场景中每像素被反复遮盖绘制多次（Overdraw）或屏幕上有大量全屏后处理 Pass 时，Fill Rate 会成为瓶颈。">Fill Rate</Term>
和
<Term def="GPU 与显存之间的数据传输带宽（GB/s）。当纹理采样次数过多、RT 分辨率太高、或频繁读写显存（如 Compute Shader / UAV）超过显存带宽上限时，GPU 会瓶颈在 Memory Bandwidth。移动端 GPU 的带宽相对 PC 低很多，是移动端最常见的 GPU 瓶颈。">Memory Bandwidth</Term>
是 GPU 的两种「物理瓶颈」——不是你代码写得烂，是硬件有上限。

<FillRateBandwidthDiagram />

Fill Rate 问题通常表现为：你降低场景的几何复杂度（少画几个物体），帧率完全没变化。因为这个瓶颈不在「画多少东西」而在「每个像素画多少次」——根源是 Overdraw（多个透明层叠在同一像素上重复绘制）或多重全屏后处理。移动端 Tile-Based GPU 对 Fill Rate 尤其敏感——Mali 和 Adreno 在大量透明叠加的场景下 Fill Rate 很容易成为第一瓶颈。

Memory Bandwidth 问题表现为：场景里有大量高质量的纹理（4K 甚至 8K）、或者单个材质用了很多张纹理采样（Normal Map + Metallic Map + Height Map + Detail Map 一起上）、或者 Compute Shader 频繁读写 Buffer。**移动端显存带宽通常是 PC 的 1/4 到 1/10**——在 PC 上不痛不痒的纹理采样，到了手机上就是带宽杀手。

如何判断是哪一种瓶颈？简单但有效的实战技巧：**逐帧降低画质分辨率（比如从 1080p 降到 540p），如果 GPU ms 大幅下降就是 Fill Rate 瓶颈；如果几乎不变就是 Memory Bandwidth 瓶颈**。因为分辨率只影响 Fill Rate 的计算量，不影响纹理采样次数和显存读写量。

## 第四个概念：Frame Debugger——一帧一画

<Term def="Unity 的逐 Draw Call 帧渲染查看器：暂停一帧后，能逐条遍历每一条 Draw Call 的渲染结果（在 Game 窗口实时预览），同时显示每个 Draw Call 用的是什么 Shader、什么网格、什么纹理、渲染目标是什么——排查渲染顺序、合批失败、意外 Draw Call 源头的神器。">Frame Debugger</Term>
是 GPU Profiler 的补充武器。Profiler 告诉你「这一帧 GPU 花了 8ms、有 300 个 Draw Call」——但它不告诉你**这 300 个 Draw Call 是谁发的、发的顺序有没有问题、为什么有个物体画出来看不见**。Frame Debugger 给你这些答案。

打开方式：Window → Analysis → Frame Debugger → 点击 Enable ——它会暂停当前帧，然后给你一个「Draw Call 逐条步进器」：

<FrameDebuggerWorkflowDiagram />

按顺序看四条信息：
1. 当前 Draw Call 在 Game 窗口的渲染结果（实时预览到这步为止的画面）
2. 这个 Draw Call 是哪个 Shader 的哪个 Pass（Shader Properties 全部列出来）
3. 用的是什么网格（Mesh）和什么纹理（Textures）
4. 渲染目标是什么（Render Target：到底是画到屏幕、画到阴影贴图、还是画到后处理 RT）

<p style={{marginTop: "16px"}}>Frame Debugger 最能帮你发现的三类问题：</p>

- **意外 Overdraw**：明明是个不透明物体，画完又被一个透明物体盖住——在 Frame Debugger 里按步进，看到同样区域被反复绘制。
- **合批失败根因**：明明材质一样，为什么没合批？Frame Debugger 里看两条相邻 Draw Call 的「Why this draw call can't be batched with the previous one」提示，告诉你具体差在哪。
- **无效 Draw Call**：某些不需要的 UI 元素或被遮挡的物体在发画图命令，但画出来看不见——Frame Debugger 的逐条预览帮你一眼揪出来。

## 实战：一帧 GPU 瓶颈排查

前面学了四个概念——GPU Profiler 模块、Draw Call/Batches/SetPass、Fill Rate vs Bandwidth、Frame Debugger。现在把这些串成一次完整的 GPU 排查流程：

<Stepper>
  <Step title="① 打开 GPU Profiler，看总 GPU ms">
    运行游戏到问题场景，打开 Profiler → 切到 GPU Usage 模块。找到 GPU ms 超过帧预算的那一帧（比如目标 16.6ms@60FPS，GPU 却跑了 22ms）。记下具体的 GPU ms 值——这就是你要缩减的空间量。


  </Step>
  <Step title="② 看渲染阶段分解，找到最宽的条">
    展开这一帧的渲染阶段分解——Opaque、Transparent、Post-Processing、Shadows……锁定耗时最长的那个阶段。如果 Transparent 占了 15ms 而其他阶段加起来不到 7ms，问题就在透明渲染。如果 Post-Processing 特别长，可能是全屏 Bloom/DOF 太重或开了太多后处理 Pass。


  </Step>
  <Step title="③ 看 Draw Call/Batches/SetPass 数字">
    在 GPU Profiler 底部看这三个数字。如果 SetPass 达到几十、Batches 好几百——说明材质太杂，需要做材质合批（Material Merging）或推 SRP Batcher。如果 SetPass 和 Batches 都很低但 GPU ms 还高——填空率测试。


  </Step>
  <Step title="④ 填充分辨率测试（填空率 vs 带宽判断）">
    去 Project Settings → Quality → 把当前画质档的分辨率缩放从 1.0 降到 0.5（分辨率减半）。重新跑一次同一帧——如果 GPU ms 接近减半（如 22ms → 11ms），瓶颈是 Fill Rate，考虑减少透明叠加和全屏后处理。如果 GPU ms 几乎不变，瓶颈是 Memory Bandwidth——检查纹理分辨率和采样次数。


  </Step>
  <Step title="⑤ 用 Frame Debugger 找具体来源">
    确定了瓶颈类型之后，打开 Frame Debugger。如果是 Fill Rate 问题，逐条步进找重复绘制的区域；如果是 Bandwidth 问题，逐个看 Draw Call 用了哪些纹理、多大的分辨率——找那些 4K 的非关键纹理，降回 1K 或 512。


  </Step>
</Stepper>

> 猜一猜：在步骤④中，你把分辨率降到 50% 后 GPU ms 从 22ms 降到 20ms——降了 2ms。这意味着 Fill Rate 是瓶颈吗？答案在 §八练习里自测。

## 关键代码：SRP Batcher 与调试用的 GPU 计时

### 启用 SRP Batcher——材质合批的终极方案

如果你的项目用的是 URP 或 HDRP，Unity 提供的 <Term def="Scriptable Render Pipeline Batcher：URP/HDRP 下的一种批处理机制——它不要求材质参数完全一致，只要 Shader 变体相同就能合批，比传统 Static/Dynamic Batching 的限制宽松得多。Draw Call 数量不变，但 SetPass 大幅下降。">SRP Batcher</Term> 是最简单、收益最大的 GPU 优化：

```csharp
// 在 URP Pipeline Asset 上勾选 SRP Batcher（编辑器操作）
// 或用代码在全局设置中启用：

using UnityEngine;
using UnityEngine.Rendering.Universal;

// PipelineAsset.asset 里把 m_UseSRPBatcher 设为 true
// 或在启动时用代码确认状态：
void Start()
{
    bool batcherEnabled =
        GraphicsSettings.currentRenderPipeline is UniversalRenderPipelineAsset urp
        && urp.useSRPBatcher;

    Debug.Log($"SRP Batcher enabled: {batcherEnabled}");
}
```

SRP Batcher 的核心逻辑是：**把同一 Shader 的所有物体的属性数据（world matrix、颜色值、材质参数等）打包成一个大的 GPU Buffer，一次上传，然后 GPU 通过索引来取各自的数据**——省掉了「每个物体都要单独设置材质参数并提交一次」的 SetPass 开销。传统管道是「一个物体 → 设参数 → 提交 → 画 → 下一个物体」，SRP Batcher 是「上传所有物体的参数 → 连续画所有物体」。

<Callout type="tip">
  SRP Batcher 不减少 Draw Call 数量——只减少 SetPass。所以你会发现启用后 Draw Call 数没变，Batches 可能也没变，但 GPU ms 显著下降——因为「切换材质」的驱动/GPU 开销被消灭了。用 GPU Profiler 验证效果只看 GPU ms 就行，别盯着 Draw Call 数不动。
</Callout>

### 用 GPU 计时器确认优化效果

CPU 侧的 `ProfilerMarker` 很好用，但你想直接在 C# 里知道**GPU 侧**跑一个特定的绘制命令花了多久——可以用 `Graphics` 类的 GPU 计时器：

```csharp
using UnityEngine;
using UnityEngine.Profiling;
using UnityEngine.Rendering;

public class GPUTimingTest : MonoBehaviour
{
    CustomSampler m_GpuSampler;

    void Start()
    {
        // 创建一个与 GPU 侧同步的 Sampler（需要平台支持）
        m_GpuSampler = CustomSampler.Create("MyPass.GPU",
            true);  // true = 使用 GPU timing（Recorder）
    }

    void LateUpdate()
    {
        m_GpuSampler.Begin();
        // ... 中间是你想测量的渲染代码或 Graphics.DrawXxx 调用 ...
        m_GpuSampler.End();
    }
}
```

<Callout type="warn">
  不是所有平台都支持 GPU 计时器。Android 上部分 Mali GPU 和较老的 Adreno 驱动可能不支持 `CustomSampler` 的 GPU timing 模式——你会看到 GPU 时间一直是 0。此时回到 GPU Profiler 看模块的 Timeline，或者在 Build Settings 里启用 `Graphics Jobs` 并确认目标设备的 GPU 驱动版本。
</Callout>

## 容易踩的坑

<Callout type="trap">
  **现象**：Draw Call 数看起来很低（不到 50），以为渲染没问题。结果真机 GPU ms 还是 18ms——各种后处理一开直接 25ms。**原因**：Draw Call 只反映 GPU 要画多少东西，不反映每个东西画起来有多贵——一个全屏后处理（如 Bloom + Color Grading + Vignette）本质上只用了两条 Draw Call（一个 Blit 到 RT，一个 Blit 回屏幕），但每个像素要跑几百条 Shader 指令——Draw Call 少 ≠ GPU 不忙。**修法**：用 GPU Profiler 看渲染阶段分解——关掉后处理 Pass 后 GPU ms 降了多少，差值就是后处理的开销。填充率问题从分辨率测试入手判断。
</Callout>

<Callout type="trap">
  **现象**：照着官方文档开了 SRP Batcher，Draw Call 数字没变，以为没生效。**原因**：SRP Batcher 的设计目标是降低 SetPass 切换开销，不是减少 Draw Call 数量——Draw Call 数几乎不变是正常的。如果盯着 Draw Call 数字看，你会觉得「开了和没开一样」。**修法**：验证 SRP Batcher 是否生效的两个信号——① Profiler 里 GPU ms 下降（SetPass 切换开销被消除）；② Frame Debugger 里每个 Draw Call 会显示 "SRP Batch" 标记而不是 "SetPass Call"。拿这两个指标判断，别拿 Draw Call 数。
</Callout>

<Callout type="trap">
  **现象**：Editor 里 GPU Profiler 显示 GPU ms 只有 4ms，真机却是 16ms。**原因**：Editor 里 Unity 用你开发机的独立显卡（PC 上的 RTX 或 Apple Silicon 上的 Metal GPU），而真机是移动 GPU（Adreno / Mali / Apple A 系列）。桌面 GPU 的 Fill Rate 和 Memory Bandwidth 通常是移动 GPU 的 5–10 倍，Editor 跑不动在真机上一定跑不动，但 Editor 跑得动在真机上不一定跑得动。**修法**：GPU 性能一切结论必须在目标设备上实测。没有真机？先对着 Editor 里的 GPU Profiler 做相对对比（改前 vs 改后），比例关系有一定参考价值，但绝对数字不可信。
</Callout>

## 小结

- **GPU Profiler** 衡量 GPU 侧的真实绘制耗时——CPU Rendering 是「准备命令」，GPU Usage 是「执行命令」
- **Draw Call → Batches → SetPass** 三层递进：Draw Call 可合批、Batches 要看实际命令数、SetPass 高说明材质杂
- **Fill Rate** 瓶颈用分辨率测试判断：降分辨率后 GPU ms 等比例下降 → Fill Rate 瓶颈
- **Memory Bandwidth** 瓶颈降分辨率无效——要查纹理分辨率和采样次数，移动端带宽通常是 PC 的 1/4
- **Frame Debugger** 逐 Draw Call 步进排查渲染顺序、Overdraw、合批失败根因和无效 Draw Call

<Exercises>

**练习 1（问答型）** SRP Batcher 启用后 Draw Call 数没变——是没生效吗？写出你用来验证 SRP Batcher 是否生效的两个方法。

<Answer>
不是没生效——SRP Batcher 的设计目标是降低 SetPass 切换开销而不是减少 Draw Call 数量。验证方法一：GPU Profiler 里 GPU ms 是否下降（SetPass 切换开销被消灭）。验证方法二：Frame Debugger 里逐条看 Draw Call 标记，SRP Batcher 生效后每条 Draw Call 会显示 "SRP Batch" 标注而不是 "SetPass Call" 标注。
</Answer>

**练习 2（操作题）** 打开你项目的 GPU Profiler，找一帧 GPU ms 最高的数据。执行三步分析并记录：①GPU ms 是多少，哪个渲染阶段（Opaque/Transparent/PostProcess/Shadows）占最多？②SetPass 是多少，若超过 10，在 Frame Debugger 里找第一条 SetPass Call 的「Why not batched」提示并记录原因。③把画质缩放降到 50%，GPU ms 变化了多少？根据结果判断是 Fill Rate 瓶颈还是 Memory Bandwidth 瓶颈。

<Answer>
此题为实际操作题，答案取决于你的项目。规范回答格式：①GPU ms: XX，占最多的阶段: XX。②SetPass: XX，Frame Debugger 提示: "This object has a different material / shader keywords / ..."。③降分辨率后 GPU ms 从 XX 变为 XX，变化幅度 XX——如果等比例下降则 Fill Rate 瓶颈，如果几乎不变则 Memory Bandwidth 瓶颈。
</Answer>

**练习 3（问答型）** 为什么在 Editor 里测 GPU 性能不可靠？写出至少两个具体原因。

<Answer>
原因一：Editor 用你开发机上的桌面 GPU（独立显卡或 Apple Silicon），而真机是移动 GPU（Adreno/Mali 等），桌面 GPU 的 Fill Rate 和 Memory Bandwidth 通常是移动 GPU 的 5 倍以上，Editor 跑得动不代表真机跑得动。原因二：Editor 进程本身在跑 Scene View、Inspector 等 UI 渲染，这些也会占用 GPU 资源——Editor 里看到的 GPU 时间是「你的游戏+编辑器开销」的总和，不是纯净的游戏 GPU 时间。
</Answer>

</Exercises>

<Glossary>

<GlossaryItem term="GPU Usage Profiler">
  Unity Profiler 中专门记录 GPU 侧绘制耗时的模块。显示每帧 GPU 总耗时以及按渲染阶段（不透明物体、透明物体、后处理、阴影）的分解耗时。是 CPU Rendering 的「楼下」——CPU 准备命令（Rendering 条），GPU 执行命令（GPU Usage）。详见本章「第一个概念」。
</GlossaryItem>

<GlossaryItem term="Draw Call">
  CPU 向 GPU 发送的一次「画东西」命令。每一条 Draw Call 包含网格数据、材质引用和变换矩阵。Draw Call 多 = CPU 准备命令的开销大——它是 CPU-GPU 通信的频率指标，不是 GPU 画图快慢的指标。详见本章「第二个概念」。
</GlossaryItem>

<GlossaryItem term="Batches">
  Unity 做完静态/动态合批之后实际发到驱动的绘制调用数。300 个 Draw Call 可能被合批成 150 个 Batches。Batches 比 Draw Call 更接近「驱动收到了多少条命令」。详见本章「第二个概念」。
</GlossaryItem>

<GlossaryItem term="SetPass">
  材质/Shader Pass 的切换次数。每换一次 Shader 或 Render Pass，SetPass+1。SetPass 高 = 材质种类太杂，合批的最大障碍。SRP Batcher 的主要作用就是降低 SetPass。详见本章「第二个概念」。
</GlossaryItem>

<GlossaryItem term="Fill Rate">
  GPU 每秒能向屏幕写入的像素总量上限。当一个像素被反复绘制多次（Overdraw）或有大量全屏后处理时，Fill Rate 会成为瓶颈。降分辨率后 GPU ms 等比例下降 = Fill Rate 瓶颈。详见本章「第三个概念」。
</GlossaryItem>

<GlossaryItem term="Memory Bandwidth">
  GPU 与显存之间的数据传输带宽。高分辨率纹理采样、Compute Shader 大量读写 Buffer 时容易成为瓶颈。降分辨率后 GPU ms 基本不变 = Memory Bandwidth 瓶颈。移动端比 PC 低很多（通常是 1/4 – 1/10），是移动端最常见的 GPU 瓶颈。详见本章「第三个概念」。
</GlossaryItem>

<GlossaryItem term="Frame Debugger">
  Unity 的逐 Draw Call 帧渲染查看器——暂停一帧后能逐条步进每一条 Draw Call，实时预览渲染结果，并查看当前 Draw Call 的 Shader、网格、纹理、渲染目标。排查 Overdraw、合批失败根因、意外 Draw Call 的神器。详见本章「第四个概念」。
</GlossaryItem>

<GlossaryItem term="Overdraw">
  同一像素被绘制了多次（多个物体或透明层叠加在同一像素位置）。第一次绘制是必须的，后续的覆盖绘制就是浪费——这些额外的像素着色器运行消耗的是 Fill Rate。移动端 GPU 对 Overdraw 敏感度高。详见本章「第三个概念」。
</GlossaryItem>

<GlossaryItem term="SRP Batcher">
  Scriptable Render Pipeline Batcher：URP/HDRP 下的一种批处理机制。不要求材质参数完全一致、只要 Shader 变体相同就能合批——把同一 Shader 的所有物体属性打包成一个大 Buffer 一次上传，GPU 按索引取各自数据。效果是 SetPass 大幅下降、GPU ms 下降，但 Draw Call 数量不变。详见本章「关键代码」节。
</GlossaryItem>

</Glossary>
