## 你的手机为什么这么烫

你正在地铁上玩半小时手游，突然手游变卡了、手机后背烫得握不住、电量从 80% 掉到了 40%——这不是游戏代码写崩了，是大量消耗的能量正在从你的手掌流失。

每一帧画面背后是一套数字流水线在跑：芯片计算颜色、显存放图片、屏幕按节拍刷新。这套流水线跑得快画面流畅，但能量也消耗得飞快——就像工厂把传送带加速，电表也跟着猛转。

这一章教你管理这套流水线的速度：不是永远全速冲刺，而是学会在需要时加速、在空闲时踩刹车——让游戏不卡的同时，电池多撑一倍的时间。

## 第一个概念：帧率越高，功耗不是只翻一倍

<Term def="屏幕每秒刷新的画面数量。30FPS 即每秒 30 张画面、60FPS 即每秒 60 张。帧率翻倍意味着 CPU/GPU 单位时间处理像素量翻倍，但功耗因芯片频率的非线性关系往往翻 2–4 倍。">帧率</Term>决定了你给这条流水线多紧的节奏。从 30FPS 切到 60FPS，像素吞吐量正好翻一倍——但 <Term def="芯片在执行渲染计算时所消耗的电能速率。帧率越高、渲染分辨率越高、芯片频率越高，功耗越大。移动端功耗直接决定电池续航和发热程度。">功耗</Term>翻的不止一倍。

原因在于芯片频率和功耗是非线性的关系：GPU 核心在低频下工作省电效率高，但一旦需要赶上更紧的帧时间（60FPS 里每帧只剩约 16ms，CPU 最多只能花 12ms 在这帧上），芯片电压必须抬高、频率随之飞涨——功耗约随频率的三次方增长。帧率翻倍 → 每帧预算减半 → 频率必须提 30%–50% → 功耗翻了 2–4 倍。

<PowerFrameRateDiagram />

下面这张图就是帧率三档的全景对比：30FPS 留给每帧约 33ms 空闲——CPU/GPU 有大量时间降频「偷懒」、省电效果最好；60FPS 把每帧预算压到 ~16ms——芯片必须满速运转、发热和耗电剧烈上升。还有第三种选择——不是死扣 30 或 60，而是按场景需求切换：

<Term def="根据场景负载自动切换目标帧率的策略：战斗场景切 60FPS（需要精确操作）、探索场景切 30FPS（省电）、菜单场景可更低。Unity 的 Adaptive Performance 包提供了系统级支持。">动态帧率</Term>就是为这种场景设计的：战斗状态（玩家在匹配操作）切高帧率确保手感流畅，探索跑图时降到 30FPS 省电，菜单/背包打开时再降到 30 以下——全程只在玩家能感知到的地方用高帧率。

## 第二个概念：Render Scale——画少一点，省一大块

<Term def="渲染分辨率与屏幕实际分辨率的比例系数。设 0.8 即每边渲染原 80% 的像素数，总像素 = 0.8² = 0.64，GPU 只需处理 64% 的像素，大幅省电且能对抗温控降频。">Render Scale</Term>是移动端功耗治理最直接、最有效的单一调整项——比优化任何 Shader 都见效快。

为什么这么有效？GPU 在渲染一帧时的第一大工作就是给屏幕上的每一个像素写颜色——这叫像素填充。屏幕有 1080×2400 ≈ 260 万像素，GPU 每帧要把这 260 万个像素点按你的 Shader 逻辑全算一遍。你把 Render Scale 从 1.0 降到 0.8 —— GPU 处理的像素数立刻从 260 万降到 0.8×0.8×260万 ≈ 166 万，减少了 36% 像素量。GPU 工作量直降，频率就可以调低，功耗明显下降。

在 Unity 里调 Render Scale 只需要一行代码：

```csharp
// 把渲染分辨率降为屏幕的 70%（每边像素数 × 0.7）
// 总像素 = 0.7 × 0.7 ≈ 49%，GPU 工作量减半
Screen.SetResolution(
    Mathf.RoundToInt(Screen.currentResolution.width * 0.7f),
    Mathf.RoundToInt(Screen.currentResolution.height * 0.7f),
    true   // 保持全屏
);
```

或者在 URP 项目的 Quality 设置里直接拉 `Render Scale` 滑块——不需要改一行代码，Editor 里调一下立刻生效。配合自适应画质调整（检测到设备温度升高→主动降到 0.7；温度正常→恢复到 1.0），这就是最温控友好的功耗策略。

## 第三个概念：TargetFrameRate 与 VSync——两位帧率管控者

帧率不是靠「猜」来控制的——Unity 给了你两个互相关联的开关来精确限制每帧产出频率。

<Term def="Unity 引擎级别的帧率目标 API（Application.targetFrameRate）。设为 60 时引擎尝试每帧在 ~16ms 内完成并控制帧率不超过 60。与 VSync 同时设置时有冲突规则。">TargetFrameRate</Term>是软性上限：设 `Application.targetFrameRate = 30` 后引擎会尽量让你跑不到 30 帧——但它只是尽力而为，GPU 画快了有可能超过去。

<Term def="屏幕硬件发出的垂直同步信号，60Hz 屏幕约每 16.67ms 发出一次。开启 VSync 后 GPU 必须等收到信号才能交换渲染完成的画面到屏幕——防止画面撕裂，但帧率被锁定为刷新率的整数分频。">VSync</Term>是硬性节拍器：屏幕每刷新一次就发一次垂直同步信号（60Hz 屏幕 ≈ 每 16.67ms 一次），GPU 必须等收到这个信号才能把画好的画面换到屏幕上——不等就强行换会导致画面从中间撕裂（上半张是上一帧、下半张是这一帧）。

<VsyncFrameBudgetDiagram />

两套机制各有各的用场：

- **VSync 用于稳定节奏场景**：`QualitySettings.vSyncCount = 1`（每帧等一个 VSync = 60FPS）、`= 2`（每两个 VSync 才换一帧 = 30FPS）——帧间隔绝对均匀，适合画面敏感的动作游戏。
- **TargetFrameRate 用于动态调整场景**：先把 `QualitySettings.vSyncCount = 0` 完全关掉 VSync，再用 `Application.targetFrameRate` 在不同场景里设不同值——战斗切 60、探索切 30、菜单切 20。

两者同时设置时的规则：VSync Count > 0 会直接覆盖 TargetFrameRate——因为硬同步信号决定了帧交换时刻，引擎的软目标在 VSync 开启时被忽略。所以你如果要做动态帧率，必须先关掉 VSync。

```csharp
// 先关 VSync 以允许手动控制帧率
QualitySettings.vSyncCount = 0;

// 在不同场景切帧率
void OnEnterBattle() => Application.targetFrameRate = 60;
void OnEnterExplore() => Application.targetFrameRate = 30;
void OnEnterMenu() => Application.targetFrameRate = 20;
```

## 第四个概念：设备发热不是你的错——但你要能应对

<Term def="手机 SoC 温度超过安全阈值后，系统固件强制降低 CPU/GPU 工作频率的保护机制。此时即使应用渲染逻辑能在 16ms 内跑完，因芯片频率被强制压在低位，实际渲染远慢于正常、帧率显著下降。">温控降频</Term>是移动开发里躲不开的现实：手机玩游戏时间一长，芯片温度升高，系统为了保护硬件不烧坏，直接把 CPU/GPU 频率强制压下来——你写的代码再高效也没用，硬件拒绝了你的请求，每帧时间因为频率低而被拉长，帧率硬生生掉下来。

Unity 的 <Term def="Unity 的官方包（Adaptive Performance），可读取设备温度状态、当前性能等级、温控警告等系统级指标，并在过热告警触发时让应用自主降画质以保护帧率。支持 Samsung GameSDK 和 Android 原生 API。">Adaptive Performance</Term>包就是为这种场景准备的——它给你提供了读取系统温度、性能等级、温控状态的 API，让你在设备将热未热时就先下手为强：

```csharp
using UnityEngine;
using UnityEngine.AdaptivePerformance;

public class ThermalManager : MonoBehaviour
{
    IAdaptivePerformance m_Ap;

    void Start()
    {
        // 检查设备是否支持 Adaptive Performance
        if (AdaptivePerformanceGeneralSettings.Instance == null)
            return;
        m_Ap = Holder.Instance;
        if (m_Ap == null || !m_Ap.Active)
            return;

        // 注册温控状态变化回调
        m_Ap.Thermal.ThermalStatusChanged += OnThermalChanged;
    }

    void OnThermalChanged(ThermalMetrics metrics)
    {
        // WarningLevel: 0=NoWarning / 1=ThrottlingImminent / 2=Throttling
        switch (metrics.WarningLevel)
        {
            case WarningLevel.ThrottlingImminent:
                // 温度即将超标，主动提前降配保帧率
                Screen.SetResolution(1920, 1080, true);  // 降到 1080p
                LODManager.SetMaxLOD(2);                 // 切粗 LOD
                break;
            case WarningLevel.Throttling:
                // 已经降频了——进一步收紧各项开销
                Screen.SetResolution(1280, 720, true);   // 降到 720p
                QualitySettings.shadows = ShadowQuality.Disable;
                PostProcessingManager.Disable();
                break;
            case WarningLevel.NoWarning:
                // 温度恢复正常——恢复画质
                Screen.SetResolution(2400, 1080, true);
                LODManager.SetMaxLOD(0);
                QualitySettings.shadows = ShadowQuality.All;
                PostProcessingManager.Enable();
                break;
        }
    }

    void OnDestroy()
    {
        if (m_Ap != null && m_Ap.Active)
            m_Ap.Thermal.ThermalStatusChanged -= OnThermalChanged;
    }
}
```

核心理念：不是在设备已经发烫降频后才去补救（那时候硬件频率已经被按死了），而是当你读到「温度即将超标」的告警时，在降频发生之前就主动降低开销、减少发热——这样频率就能保持在正常水平，帧率不受影响。

## 第五个概念：QualitySettings 分级——一套配置适配所有档位

<Term def="Unity 中管理画质档位的全局配置 API。Project Settings → Quality 面板预设若干档位（Fastest/Fast/Simple/Good/Beautiful/Fantastic），每档独立控制阴影质量、纹理质量、抗锯齿、VSync、像素光数量等。运行时用 SetQualityLevel(index) 切换。">QualitySettings</Term>是 Unity 给每个画质档位预设好的一切渲染参数的集合——不是只有「高画质」和「低画质」两个极端，而是几组预先配好的、按钮点一下就全部切换的档次。

移动端通常设 3–4 档，每档该关什么有标准答案：

| 档位 | 定位 | 阴影 | 后处理 | 纹理分辨率 | Render Scale |
|------|------|------|--------|------------|--------------|
| Low | 中低端机/省电模式 | 关闭 | 关闭 | 512 | 0.7 |
| Medium | 中端机/日常模式 | 低质量 | Bloom 仅 | 1024 | 0.85 |
| High | 旗舰机/性能模式 | 全开 | 全效果 | 2048 | 1.0 |
| Ultra | 超旗舰/极佳画质 | 全开+软影 | 全效果+SSAO | 4096 | 1.0 |

在 Unity 里预设好这些档位后，检测到设备性能等级就切到对应档位：

```csharp
// 根据设备性能等级选画质档位
int SelectQualityLevel(PerformanceLevel perf)
{
    return perf switch
    {
        PerformanceLevel.Low    => 0,  // QualitySettings.names[0] = "Low"
        PerformanceLevel.Medium => 1,  // "Medium"
        PerformanceLevel.High   => 2,  // "High"
        _                       => 2,
    };
}

void ApplyQualityForDevice()
{
    int level = SelectQualityLevel(GetDevicePerformanceLevel());
    QualitySettings.SetQualityLevel(level, applyExpensiveChanges: true);
    // applyExpensiveChanges: true = 立即触发纹理重加载等耗时操作
    // 仅在场景加载阶段调用，不要在游戏过程中调
}
```

<Term def="设备在当前温控/电量状态下的实时可用性能等级（PerformanceLevel）：系统根据芯片型号、当前温度、电池电量综合评估出的设备性能可用余量，反映的是当下实际可用的算力而非纸面规格。" id="term-adaptive-performance">性能等级</Term>——注意：不是设备售价（「这是旗舰机开 Ultra」），而是**当下这台设备在当前温度和电量下的实际可用算力余量**。一台旗舰机在电量只有 5% 且已经发烫的时候，实际可用性能可能级别只能跑 Medium 档——系统级的 Adaptive Performance 会真实反映这个数据。

## 关键代码：分档省电策略的完整实现

### Render Scale + TargetFrameRate 联动省电

这是移动端功耗优化最常用的一段组合——Render Scale 和帧率目标一起调整：

<CodeTabs>
  <Tab label="✓ 完整：分场景动态调整">

```csharp
using UnityEngine;

public class PowerManager : MonoBehaviour
{
    // 根据场景类型切省电策略
    public void ApplyPowerProfile(GameScene scene)
    {
        // 先关闭 VSync 以允许手动控帧
        QualitySettings.vSyncCount = 0;

        switch (scene)
        {
            case GameScene.Combat:
                // 战斗：流畅优先
                Application.targetFrameRate = 60;
                SetRenderScale(1.0f);
                break;
            case GameScene.Exploration:
                // 探索：平衡省电
                Application.targetFrameRate = 30;
                SetRenderScale(0.85f);
                break;
            case GameScene.Menu:
                // 菜单：省电优先
                Application.targetFrameRate = 20;
                SetRenderScale(0.7f);
                break;
        }
    }

    void SetRenderScale(float scale)
    {
        int w = Mathf.RoundToInt(
            Screen.currentResolution.width * scale);
        int h = Mathf.RoundToInt(
            Screen.currentResolution.height * scale);
        Screen.SetResolution(w, h, true);
    }
}
```

### Adaptive Performance 温控保护骨架

上面那段是项目骨架——不是每个项目都需要这么详细，但关键三行（注册回调、读温控状态、降配策略）必不可少：

```csharp
// 最小可用温控保护（3 行核心逻辑）
m_Ap.Thermal.ThermalStatusChanged += (metrics) =>
{
    if (metrics.WarningLevel == WarningLevel.ThrottlingImminent)
        Screen.SetResolution(1280, 720, true);   // 马上降到 720p
    else if (metrics.WarningLevel == WarningLevel.NoWarning)
        Screen.SetResolution(2400, 1080, true);  // 恢复全分辨率
};
```

<Callout type="tip">
  每个项目在接入 Adaptive Performance 后都应该至少测一次完整的「从常温跑游戏到温度触发降频」的流程（用 Perfdog 全程录制电流/温度/帧率三条曲线）：确认温度告警触发时机是否合理、降配策略是否真的压住了温度上升、温度恢复后画质是否正常回来。
</Callout>

{/* 7. 常见误区（spec §七）：A 型 ≥2 个；固定 现象→原因→修法 格式。此处提供 3 个。 */}

## 容易踩的坑

<Callout type="trap">
  **现象**：设了 `Application.targetFrameRate = 30`，但设备温度仍然一路上涨，Perfdog 显示 GPU 频率仍高居不下。**原因**：只降了 CPU 这边的目标帧率，GPU 的 Render Scale 仍为 1.0——GPU 还在按全分辨率渲染像素，它的工作量没有变化，频率就不会降下来。**修法**：降帧率同时必须降 Render Scale（`Screen.SetResolution`）——让 CPU 和 GPU 两边的负载一起下降才能显著降低功耗。只降 CPU 不减 GPU 负载 = 功耗省了个寂寞。
</Callout>

<Callout type="trap">
  **现象**：在 120Hz 屏幕（如 iPad Pro）上设 `Application.targetFrameRate = 60`，实际跑出来的却是接近 120FPS，功耗远超预期——为什么 60 没生效？**原因**：高刷新率设备上系统默认走最高刷新率 `VSync Count = 0 或 1`（120Hz 屏幕即 120FPS）。Unity 的 `targetFrameRate = 60` 在某些平台上被系统忽略——因为系统级刷新率本身是 120Hz，你不显式限制 VSync 等级，GPU 就按每标刷新来换帧。**修法**：在高刷设备上需要同时设 `Application.targetFrameRate = 60` **并**用 `Screen.SetResolution(w, h, true, 60)` 的第五个参数（Unity 2022+）来显式限制刷新率，或设 `QualitySettings.vSyncCount = 2`（每两个 VSync 换一帧=60FPS）。
</Callout>

<Callout type="trap">
  **现象**：QualitySettings 预设了 4 档画质，但切到 Low 档后对比 High 档功耗几乎没降——GPU 频率曲线几乎一样高。**原因**：Low 档只调了纹理精度、阴影、抗锯齿这些**不决定像素填充量**的参数——这些参数确实省了 GPU Shader 的计算量，但 Fill Rate（每帧要填充的像素总数）一毛没变。手机屏幕是 1080×2400，GPU 每一帧还是要填满 260 万像素——这才是功耗的大头。**修法**：Low 档必须降 Render Scale（如 0.7），让 GPU 总像素量直降 50%——这是唯一能显著拉低 GPU 频率的调整项。
</Callout>

{/* 8. 小结与练习（spec §八）：小结 ≤5 条 bullet × ≤40 字；与 §一目标一一呼应；A 型不加推导/实现题。 */}

## 小结

- **帧率 ×2 ≠ 功耗 ×2**：芯片频率与功耗的非线性关系导致功耗翻 2–4 倍
- **Render Scale 是第一抓手**：降 1 个比例系数 → 像素量呈平方下降 → GPU 大幅省电
- **TargetFrameRate + VSync**：两个开关一个软一个硬，动态帧率先关 VSync 再设 TargetFrameRate
- **Adaptive Performance**：在设备发烫之前就感知、提前降配防降频——不是等频率被按下去再补救
- **QualitySettings 分档**：关键看 Render Scale 和阴影——只调纹理和抗锯齿省不了多少功耗

<Exercises>

**练习 1（改代码型）** 在你的 Unity 项目中加入一个 `PowerManager` 脚本，实现战斗场景切换 60FPS + Render Scale 1.0、探索场景切换 30FPS + Render Scale 0.85、菜单场景切换 20FPS + Render Scale 0.7 的分场景策略。跑 Perfdog 录两次（各 5 分钟）：第一次用「全程 60FPS + Render Scale 1.0」、第二次用你的分场景策略——对比两次的耗电量和 GPU 温度曲线。

<Answer>
核心代码见 §六「分场景动态调整」Tab。测试预期：分场景策略在 5 分钟内大部分时间处于探索/菜单场景，GPU 频率和温度应明显低于全程 60FPS 的对照组。如果两者差异不大——检查你的 Render Scale 是否在 `SetResolution` 调用后生效（在 Perfdog 里看屏幕分辨率是否真的变了）。
</Answer>

**练习 2（问答型）** Step③ 已经降到 30FPS 且 Render Scale 0.8 ——Perfdog 显示 GPU 温度仍在涨，为什么？

<Answer>
降了帧率和 Render Scale 都不等于温度不涨——只是涨的速度变慢了。原因可能是：①场景本身在持续加载资源（纹理/网格流入 Native 内存），内存带宽占用让芯片持续饱和→温度继续上升。②降了 CPU/GPU 的工作量但没降它们在芯片内部的频率——芯片在低频高负载状态下功耗曲线不一定线性下降，长时间稳在高负载区间仍发热。③系统后台有其他进程在消耗资源（如录制自身 Perfdog），增加了芯片总体功耗。要彻底停涨温度需要进一步收紧频率（目标帧率降到 20 甚至 15）+ 主动等待温度回落（例如暂停渲染 1-2 秒）。
</Answer>

**练习 3（问答型）** 为什么 120Hz 屏幕上设 `targetFrameRate = 60` 可能不生效？写出具体原因和两层修复方案。

<Answer>
原因：高刷新率设备上系统级默认 VSync 走 120Hz，`targetFrameRate` 是个软目标——在 VSync 面前被忽略。修复方案一：用 `Screen.SetResolution(w, h, true, 60)` 的第五个参数（Unity 2022+）显式要求系统把刷新率限制到 60Hz。修复方案二：设 `QualitySettings.vSyncCount = 2`（每两个 120Hz VSync 换一帧=60FPS）——这是硬同步，不会像 TargetFrameRate 一样被系统忽略。
</Answer>

</Exercises>

<Glossary>

<GlossaryItem term="帧率">
  屏幕每秒刷新的画面张数。30FPS 每帧约 33ms 预算（含系统和开销），60FPS 每帧约 16ms。帧率翻倍 = CPU/GPU 单位时间工作量翻倍——但因芯片频率需提升才能赶上更紧的帧预算、功耗涨 2–4 倍。详见本章「第一个概念」。
</GlossaryItem>

<GlossaryItem term="功耗">
  芯片在运行渲染工作时消耗的电能——表现为电池掉电速度和发热程度。帧率越高、Render Scale 越大、芯片频率越高，功耗越大。移动端功耗上限是电池容量和散热能力——超限 = 系统强制降频甚至杀进程。详见本章「第一个概念」和「第四个概念」。
</GlossaryItem>

<GlossaryItem term="动态帧率">
  按场景需求自适应切换目标帧率的优化策略。核心逻辑：只在玩家能感知的流畅场景用高帧率（战斗 60FPS），在不需要高帧率的场景降到 30FPS 或更低（省电）。Unity Adaptive Performance 包支持系统级动态帧率感知。详见本章「第一个概念」。
</GlossaryItem>

<GlossaryItem term="Render Scale">
  渲染分辨率与屏幕物理分辨率的比例系数。设 0.8 = 每边渲染原 80% 像素，总像素量 0.8² = 0.64，GPU 只需处理 64% 像素量。移动到端省电最直接有效的手段——Fill Rate 直降，GPU 频率随之降低。Unity 里用 `Screen.SetResolution` 或 URP Quality 设置里的 Render Scale 滑块调整。详见本章「第二个概念」。
</GlossaryItem>

<GlossaryItem term="VSync">
  显示器发出的垂直同步信号。60Hz 屏幕每 16.67ms 发一次——开 VSync 时 GPU 必须等信号到才能换帧（画面完整无撕裂），关 VSync 时 GPU 画完立刻换帧（可能画面撕裂）。`QualitySettings.vSyncCount` 控制等待几个 VSync 信号换一帧。详见本章「第三个概念」。
</GlossaryItem>

<GlossaryItem term="TargetFrameRate">
  Unity 的引擎级帧率目标 API：`Application.targetFrameRate`。设 60 时引擎控制帧率 ≤60，但这是软目标——在 VSync 开启时被覆盖。适用于关了 VSync 后手动调帧——战斗 60、探索 30、菜单 20。详见本章「第三个概念」。
</GlossaryItem>

<GlossaryItem term="温控降频">
  手机 SoC 在温度超过安全阈值时系统固件强制降低 CPU/GPU 频率的保护机制——此时渲染代码再高效也没用，因为硬件被限频了。Unity Adaptive Performance 包提供 API 可以在降频发生前就检测到温度预警、主动降画质保护帧率。详见本章「第四个概念」。
</GlossaryItem>

<GlossaryItem term="Adaptive Performance">
  Unity 官方包：可读取设备当前的温度状态、性能等级、温控告警等系统级实时指标。核心理念：在温控降频发生之前就接获告警、主动降低渲染开销——避免被硬件强制降频后帧率崩盘。支持 Samsung GameSDK 和 Android 原生温控 API。详见本章「第四个概念」。
</GlossaryItem>

<GlossaryItem term="QualitySettings">
  Unity 管理画质档位的全局 API：在 Project Settings → Quality 里预设若干档位（每档独立控制阴影、纹理、抗锯齿、VSync、像素光数量等），运行时用 `QualitySettings.SetQualityLevel(index)` 一键切换整组配置。移动端建议至少分 Low / Medium / High 三档，Low 档必须降 Render Scale。详见本章「第五个概念」。
</GlossaryItem>

<GlossaryItem term="Perfdog（性能狗）">
  腾讯出品的移动端性能测试工具。无需 Development Build，通过系统级采集（adb shell dumpsys / Xcode instruments）直接读设备的 FPS、CPU/GPU 频率、温度、电流、功耗曲线——这些 Unity Profiler 看不到。与 Unity Profiler 互补：Perfdog 看系统级能耗和温控状态，Unity Profiler 看引擎内部热点。详见本章 Stepper 流程。
</GlossaryItem>

</Glossary>
