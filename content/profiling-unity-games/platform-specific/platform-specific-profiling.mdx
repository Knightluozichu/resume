## 每台机器都在演自己的戏

你辛辛苦苦优化好了一个场景——PC 上跑得飞快，帧率稳在 120。可玩家反馈说手机端卡成幻灯片，还有人说 Web 端加载要等半分钟。

道理很简单：每台设备都是一台不同的「舞台」——PC 有独立显卡、大内存；手机靠一块小芯片连 CPU 和 GPU 挤在一起散热；游戏机硬件完全固定但工具链封闭；眼镜戴在脸上对帧率比手机还敏感；网页端跑在浏览器沙箱里，能用的资源最小最受限。

**你没有一种工具能通吃这五个舞台。** 每个平台各有一套数据采集方式，换平台不看数据、靠猜——百分之百会出问题。这一章就是帮你把五个平台的工具链和标志性套路梳理清楚。

## 知识点一：五平台差异总览

先把五个平台的标志性工具和典型瓶颈画在一张图上：

<PlatformProfilingOverviewDiagram />

再拆开看每个平台的独特之处：

- **Console（PS5 / Xbox / Switch）**：硬件完全固定，调试工具由厂商提供（SN Systems / PIX / NintendoTargetManager）。<Term def="一种调试时通过 USB/网络把 PC 上的分析工具连接至运行中的游戏机的方式，无需游戏机本地安装完整工具链。">远程调试</Term> 是标配——PC 上跑一套 Profiler 工具，通过网线接到开发机上的游戏。帧预算极度严格（PS5/Xbox Series 通常要求 60 FPS @ 16.67ms 预算内有足够的 CPU/GPU 并行空间）。
- **PC**：最开放的平台。Unity Profiler 直接本地跑；RenderDoc 可以逐帧截取、逐 Draw Call 回放；显卡厂商也各有关键工具（NVIDIA NSight / AMD Radeon GPU Profiler / Intel GPA）。

- **Mobile（Android / iOS）**：**省电比绝对值更重要**。Unity Profiler 要通过 ADB（Android）或 USB（iOS）远程连；第三方工具 Perfdog 不需要连 Unity，直接跑在手机上抓系统级数据（CPU/GPU/温度/FPS 同时看）。**最大的坑**：开发机上测的数据和手机上测的数据往往不在一个量级——移动端必须实机验证。

- **XR（VR / MR 眼镜）**：帧率就是体验生命线——低于 72 FPS 就可能导致晕动症。所有 XR 平台都强制 VSync 且帧预算极紧（Quest 的 TimeWarp 管道在 CPU 侧必须赶在 vBlank 前完成）。工具链上 Oculus 用 OVR Metrics Tool，OpenXR 依赖各 SDK 内置的 Performance HUD。

- **Web（WebGL）**：运行在浏览器沙箱里，无法调用系统 Profiler。唯一靠谱的工具是 Chrome DevTools 的 Performance 面板 + Memory 面板。与原生最大的区别：受浏览器主线程调度干扰，帧时间图中几乎总能看到「不是你的代码在跑，而是浏览器 GC 在跑」的尖刺。

## 知识点二：Unity Profiler 远程连接（Android / iOS）

Unity 自带 Profiler 是所有平台都能用的入口工具。PC 上直接打开即可，移动端需要先建立连接。

### Android 端连接步骤

1. 在 Unity 里勾选 **Build Settings → Development Build** 和 **Autoconnect Profiler**
2. 用 USB 连接手机，确认 `adb devices` 能看到设备
3. 手机打开 **USB 调试**
4. Build & Run 到手机上
5. Unity 的 Profiler 窗口（Window → Analysis → Profiler）会自动检测设备并连接

```csharp
// 如果要在代码里控制 Profiler 状态（只在 Development Build 下有效）
using UnityEngine.Profiling;

// 开启 CPU Profiler 采样
Profiler.enabled = true;

// 只在你关注的帧范围内记录
Profiler.BeginSample("MyCustomMarker");
// ... 你需要测量的代码
Profiler.EndSample();
```

<Callout type="warn">
  Android 连接失败的常见原因（按概率排序）：① USB 线只支持充电不支持数据传输；
  ② 没开 Development Build；③ ADB 未授权（手机上弹窗点了拒绝）；
  ④ 手机和 PC 不在同一局域网（WiFi 模式）。
</Callout>

### iOS 端连接步骤

iOS 连接比 Android 多一个门槛——需要 Xcode 作为中转：

```csharp
// iOS 上 Unity Profiler 的额外要求
// 1. Build Settings 中 Target SDK 设为 "Device SDK"
// 2. 在 Xcode 中打开导出的 .xcodeproj
// 3. 用 USB 连 iPhone，Xcode 选中设备
// 4. Build & Run，Unity Profiler 会自动发现设备
```

<Callout type="tip">
  iOS 端 Profiler 连接的标志性问题是**不显示设备**——
  通常是因为 Xcode 未信任开发者证书或手机未解锁。确保 iPhone 亮屏且已解锁再用
  USB 连接。
</Callout>

## 知识点三：第三方工具——各平台的不二之选

Unity Profiler 是第一步，但要看到 GPU 的逐帧细节、抓系统级功耗温度、或是 Web 端的内存快照——必须靠第三方工具。

### RenderDoc：逐帧截取与逐 Draw Call 回放

<Term def="开源 GPU 帧调试器，能在 PC 上截取任意一帧的完整渲染状态——所有 Draw Call、所有 RenderTexture 中间结果、所有 Shader 输入/输出——逐步骤回放。">RenderDoc</Term> 是 GPU 调试的瑞士军刀。它通过注入渲染管线来截取一帧的全部状态，你可以单独查看某一时刻某个 RenderTexture 里画的是什么，逐个 Draw Call 回放、看每步加减了什么。

适合平台：**PC、Console（开发机）**。不适合 Mobile（需要 root 或特殊驱动签名）。

```csharp
// 在 Unity 代码里嵌入 RenderDoc 集成：
// 按钮按一次就自动截取当前帧并保存为 .rdc 文件
if (Input.GetKeyDown(KeyCode.F12)) {
    // RenderDoc 集成通常在构建中以插件形式调用
    // 等价命令行：renderdoccmd capture -w <进程名>
    Debug.Log("RenderDoc capture triggered (需要 RenderDoc 处于运行状态)");
}
```

### Perfdog：移动端的最强系统级监视器

<Callout type="tip">
  不需要修改 Unity 工程！Perfdog 是一个独立 app，装在手机上后台跑，
  同时记录 FPS、CPU/GPU 占用率、功耗（mW）、电池电流（mA）、温度——全部实时。
  导出 CSV 后可以在 PC 端分析历史趋势。
</Callout>

适合平台：**Android、iOS**。尤其适合排查间歇性卡顿——因为它在后台持续记录，不漏掉任何一帧。

### Xcode Instruments：iOS 专属系统底层工具

iOS 开发绕不开它——可以追踪线程调度、内存分配调用栈、Metal/GPU 性能计数器。

适合平台：**iOS 独占**。

### Chrome DevTools：WebGL 的唯一出路

WebGL 游戏跑在浏览器里，无法调 Unity Profiler。Chrome DevTools 的 Performance 面板能录帧时间线、Memory 面板能拍堆快照——这是 Web 端唯二靠谱的分析入口。

## 知识点四：平台配置清单

一个项目导出到五个平台时，至少应该配好这些设置。下面是一个逐步骤对照清单：

<Stepper>
  <Step title="Console 平台配置">
    ① 设好帧预算（PS5/Xbox Series：16.67ms；Switch：33.33ms 手持模式）。
    ② 关闭 Console 的 VSync（由系统层面已强制）。
    ③ 用厂商工具（SN Systems / PIX）锁定 GPU Profile 模式。
    ④ 用 Burst Compiler + Jobs 系统榨 CPU 的 SIMD 并行。
  </Step>
  <Step title="PC 平台配置">
    ① QualitySettings 里开最高档，默认 targetFrameRate = -1（让玩家自行限帧）。
    ② 如果有 RenderDoc 依赖，构建时带上 Developer Build 模式。
    ③ 用 `SystemInfo.graphicsDeviceType` 区分 Vulkan / DX12 / Metal 后端。
  </Step>
  <Step title="Mobile 平台配置">
    ① 强制 targetFrameRate = 30（见第 6 章）。
    ② 关闭 MSAA、把 Render Scale 拉到 0.7。
    ③ 关闭 VSync。
    ④ 加上温控退让策略。
  </Step>
  <Step title="XR 平台配置">
    ① `Application.targetFrameRate` = 平台刷新率（Quest 72/80/90/120，Vision Pro 90/96/100）。
    ② 强制开启 Single-Pass Instanced 渲染（每眼不再重复算一帧）。
    ③ 打开 Foveated Rendering（只把视点中心区域高分辨率渲染）。
    ④ 调 TimeWarp 管线：CPU 侧必须在 vBlank 前完成提交。
  </Step>
  <Step title="Web 平台配置">
    ① 关闭 Development Build（WebGL 上严重影响大小和速度）。
    ② 用 `PlayerSettings.WebGL.emscriptenArgs` 加 `-s ASSERTIONS=0` 祛除断言检查。
    ③ 加载完成后用 Chrome DevTools → Performance 录制 5 秒帧时间线，找尖刺。
  </Step>
</Stepper>

## 平台检测与自动配置

跨平台项目中，第一个要写的代码就是平台检测和配置切换。下面是三个场景的核心实现。

### 场景一：构建时排除不兼容代码

```csharp
using UnityEngine;

public class PlatformSetup : MonoBehaviour {
    void Awake() {
#if UNITY_STANDALONE || UNITY_EDITOR
        // PC/Editor：开全特效
        QualitySettings.SetQualityLevel(5);
        Application.targetFrameRate = -1;
#elif UNITY_ANDROID || UNITY_IOS
        // 移动端：按 GPU 自动选档
        QualitySettings.SetQualityLevel(
            SystemInfo.graphicsDeviceName.Contains("Adreno 7") ? 3 : 1
        );
        Application.targetFrameRate = 30;
#elif UNITY_WEBGL
        // Web：最低画质、强制 30FPS
        QualitySettings.SetQualityLevel(0);
        Application.targetFrameRate = 30;
#elif UNITY_XBOXONE || UNITY_PS5 || UNITY_GAMECORE_SCARLETT
        // Console：最高画质、让平台管理帧率
        QualitySettings.SetQualityLevel(5);
#endif
    }
}
```

<Callout type="warn">
  预处理指令 `#if UNITY_ANDROID` 只在**构建时**生效，编辑器里这段代码不会跑。
  如果你要在编辑器里模拟移动端行为，使用 `Application.isMobilePlatform` 做运行时判断。
</Callout>

### 场景二：Profiler API 自定义标记

```csharp
using UnityEngine.Profiling;

public class ProfilerMarkers : MonoBehaviour {
    void Update() {
        Profiler.BeginSample("Game.Update.AI");
        UpdateAI();  // 把 AI 耗时单独标出来
        Profiler.EndSample();

        Profiler.BeginSample("Game.Update.Physics");
        Physics.Simulate(Time.deltaTime);
        Profiler.EndSample();

        Profiler.BeginSample("Game.Update.Render");
        // 可嵌套标记，方便在 Profiler 里定位瓶颈函数
        Profiler.BeginSample("Game.Update.Render.Shadows");
        RenderShadows();
        Profiler.EndSample();
        Profiler.EndSample();
    }
}
```

<Callout type="tip">
  `Profiler.BeginSample` / `EndSample` 是零成本开销的——在非 Development Build
  中编译器会直接剔除这些调用。所以正式发布的包不会因为这些标记变慢。
</Callout>

### 场景三：平台后端检测

```csharp
// 不同平台默认的图形 API 后端不同，甚至可以强制切换
using UnityEngine.Rendering;

if (SystemInfo.graphicsDeviceType == GraphicsDeviceType.Vulkan) {
    Debug.Log("跑在 Vulkan 上——移动端/PC 均支持");
} else if (SystemInfo.graphicsDeviceType == GraphicsDeviceType.Metal) {
    Debug.Log("跑在 Metal 上——iOS/macOS 独占");
} else if (SystemInfo.graphicsDeviceType == GraphicsDeviceType.Direct3D12) {
    Debug.Log("跑在 DX12 上——Windows 独占");
} else if (SystemInfo.graphicsDeviceType == GraphicsDeviceType.OpenGLES3) {
    Debug.Log("跑在 GLES3 上——WebGL/部分 Android 回退");
}
```

## 容易踩的坑

<Callout type="trap">
  **现象**：PC Profiler 里帧时间只有 8ms（轻松 120 FPS），部署到手机上变成 45ms（掉到
  22 FPS）。**原因**：PC 的独立显卡和手机的集成 GPU 性能差距巨大——不是在「线性缩放」，
  而是在跨越硬件代际。移动 GPU 对透明材质、多 Pass Shader、MSAA 的惩罚比 PC 重得多。
  **修法**：关闭后处理（Post-Processing）中的 Bloom/DOF；减少 Shader Pass 数；
  关闭 MSAA 改用 FXAA/TAA；用 URP 代替内置管线。**永远在目标设备上实机验证**。
</Callout>

<Callout type="trap">
  **现象**：WebGL 版本的帧时间图中总有规律性尖刺，CPU Profiler 显示标记为
  `(idle)` 而非你的代码。**原因**：浏览器的主线程会被 JavaScript GC、样式计算、
  DOM 事件处理等系统任务抢占——这些不是 Unity 的问题，是浏览器调度的固有开销。
  **修法**：减小堆分配、避免每帧创建新对象（触发高频 GC）；降低目标帧率到 30；
  关闭 `PlayerSettings.WebGL.exceptionSupport`（改为 None 可以省掉异常处理开销）。
</Callout>

<Callout type="trap">
  **现象**：Android 上 Profiler 连上了，但 Profiler 窗口里 CPU Usage 一直为 0 或
  数据完全不变。**原因**：Development Build 里打开了 Autoconnect Profiler 却没开
  **"Record"**（Profiler 窗口左上角的录制按钮）。Unity Profiler 连上≠在录制——必须手动点录制。
  **修法**：点击 Profiler 窗口的红色录制按钮，或者 Window → Analysis → Profiler
  的 Record 开关。
</Callout>

## 小结

- 五个平台各有一套工具链：Console = 厂商工具 + RenderDoc；PC = Unity Profiler + RenderDoc；Mobile = Unity Profiler 远程 + Perfdog；XR = OVR Metrics + SDK Perf HUD；Web = Chrome DevTools
- Unity Profiler 远程连接 Android 需要 ADB + USB 调试 + Development Build；iOS 还需要 Xcode 中转
- 第三方工具各有专长：RenderDoc 看 GPU 帧细节；Perfdog 看移动端系统级功耗温度；Xcode Instruments 看 iOS 底层；Chrome DevTools 看 Web 端
- 每个平台都有自己的「非你代码」瓶颈源的标志特征——Web 的浏览器 GC、Mobile 的温控降频、Console 的帧预算硬限制
- 预处理指令 + 运行时检测配合使用，才能让同一份代码在五个平台上编译通过并正确运行

```csharp
        }
#endif

    }
}
```

</Answer>

**练习 3（问答型）** 为什么 WebGL 不能用 Unity Profiler，而必须用
Chrome DevTools？

<Answer>

WebGL 游戏运行在浏览器沙箱里——它能访问的 API 只有 Web 标准规定的那几项（WebGL
2.0、Web Audio 等）。Unity Profiler 需要和引擎内部通信（通常是网络 socket），
这在浏览器安全模型下是禁止的。Chrome DevTools 是浏览器自带的开发者工具，
能直接看到 JavaScript 调用栈、帧时间线和内存分配，是 Web 端唯一能用的性能分析通道。

</Answer>

</Exercises>

<Glossary>

<GlossaryItem term="远程调试">
  把 PC 上运行的分析工具通过网络或 USB 连接到一台正在运行游戏的设备上，
  读取它的性能数据。游戏机上叫「DevKit 调试」，手机上就是 ADB 或 Xcode 连接。
</GlossaryItem>

<GlossaryItem term="RenderDoc">
  一个免费开源的 GPU 帧调试器。可以截取游戏中任意一帧的完整渲染状态——
  包括所有 Draw Call、中间 RenderTexture、Shader 输入输出——然后逐步骤回放，
  是 PC/Console 端 GPU 调试的瑞士军刀。
</GlossaryItem>

<GlossaryItem term="Perfdog">
  移动端独立性能监视 App。不需修改 Unity 工程，装在手机上后台持续记录
  FPS、CPU/GPU 百分比、功耗（mW）、电池电流（mA）、温度——用于排查
  间歇性卡顿问题，可导出 CSV 历史数据离线分析。
</GlossaryItem>

<GlossaryItem term="Xcode Instruments">
  Apple 为 iOS/macOS 提供的系统级性能分析工具套件。能追踪线程调度、
  Metal GPU 性能计数器、内存分配调用栈、耗电归属。iOS 性能瓶颈定位必备。
</GlossaryItem>

<GlossaryItem term="Chrome DevTools">
  Chrome 浏览器自带的开发者工具。Performance 面板记录帧时间线和 JavaScript
  调用栈，Memory 面板抓取堆内存快照——是 WebGL 游戏唯二可用的性能分析入口。
</GlossaryItem>

<GlossaryItem term="TimeWarp">
  一种 XR 渲染管线中的独占技术——在 CPU 端渲染完成、但实际显示之前，
  再根据头部姿态的最新追踪数据「微调」画面。如果 CPU 没在 vBlank
  前完成，TimeWarp 就会卡住，导致画面撕裂或抖动。
</GlossaryItem>

</Glossary>
