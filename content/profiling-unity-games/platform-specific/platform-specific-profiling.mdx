## 一台设备一个性格

你在开发机上写好游戏了，电脑上跑 60 帧稳稳的。推到手机上装好打开——帧率不到 20。不是你的代码写错了，是你写代码的那台电脑和玩家手里的手机完全是两个世界里的设备。

电脑有独立显卡、几十 GB 内存、强力散热风扇；手机只有一小块芯片贴在玻璃后盖上——没风扇、只有巴掌大的散热区域。更复杂的是：连电脑用的工具到了手机上根本打不开了——手机的调试端口不一样、浏览器和游戏主机上的更不一样。每换一个平台就像到了一个说着不同语言的城市——你必须学会用这个平台的工具说话。

这一章教你五套平台的工具语言：电脑、手机、游戏主机、头戴设备和网页——每个平台的工具有什么、怎么把调试线接过去、怎么读出性能数据。读完你会发现不同平台的性能分析不是「同样的操作搬到不同机器上」——而是每一步都不一样。

{/* 3. 概念讲解（spec §三）：C 实战型可把「是什么」压缩至一段、篇幅让给第六节代码对照。

## 第二个概念：把 Profiler 搬到手机上——Android 和 iOS 的远程连接

在 <Term def="一种内置 Profiler 连接支持、但不做 Release 级优化的特殊构建版本。语法接近 Release 构建但允许 Editor 的 Profiler 远程连接，用于在真机上做性能分析。">Development Build</Term> 上你已经勾了 <Term def="Development Build 的一项构建设置：勾选后设备运行时会自动搜索并连接同局域网内的 Editor Profiler，省去手动输入 IP 地址的步骤。">Autoconnect Profiler</Term>，但 Android 和 iOS 的底层连接方式完全不同——不是「都插根线就行」。

<RemoteProfilerDiagram />

**Android** 走 ADB 端口转发：把真机 Android 设备和电脑之间的调试数据通过 USB 线或 WiFi 用一个特定端口（默认 55000）打通。插入 USB 线后在终端跑一条命令就把设备的数据流导到 Editor 了：

```bash
# Android：ADB 端口转发——把设备上的 Profiler 端口
# 映射到 PC 的 55000 端口
adb forward tcp:55000 tcp:55000
```

之后在 Unity Editor 的 Profiler 窗口顶部的 Active Profiler Target 下拉里选择 `AndroidPlayer`——Profiler 数据开始传输。

如果 Autoconnect 失败，手动输入设备 IP：

```bash
# 查看设备 IP（在 WiFi 连接下）
adb shell ip addr show wlan0

# Profiler 窗口 → Active Profiler Target → 输入
# 192.168.x.x:55000
```

**iOS** 走和 Android 完全不同的路径：iOS 设备必须通过一台 Mac 电脑中转——用 USB 线连 Mac、在 Xcode 的 Devices 窗口里确保设备「可被发现」，Unity Editor 同一台 Mac 上的 Profiler 会自动发现这个设备。iOS 设备无法直接和 Windows Editor 建立 Profiler 连接。

<Callout type="warn">
  两个设备不能在同一个 Profiler 窗口同时连接——一个 Profiler 窗口只能同时监听一个 Target 设备。如果要同时对比两台设备的数据，每台各录一段 .data 文件后导入 Profile Analyzer 做离线对比。
</Callout>

连接后的第一步验证——在 Profiler 窗口看顶部帧率是否和真机上一致、CPU Usage 色条是否在变化。确认数据在流了再开始真正的分析。

## 第三个概念：三个独立工具——RenderDoc、Xcode Instruments、Perfdog

Unity Profiler 告诉你「这帧 GPU 花了 14ms」——但不告诉你这 14ms 花在哪几个 Draw Call 上、每个 Draw Call 的 Shader 在做什么、CPU 在同时跑什么。这些是 Platform-Specific 工具的领地。

<Term def="开源的跨平台 GPU 帧调试器（支持 Vulkan/D3D/Metal/GLES）。捕获单帧所有 Draw Call、查看每个 Draw Call 的输入/输出纹理、Shader 编译结果、资源绑定——逐 Draw 分析 GPU 热点。">RenderDoc</Term> 是通用的 GPU 帧逐帧分析工具——Android Vulkan/GLES、PC DirectX/Vulkan、甚至 Switch 都支持（iOS 用 Xcode 自带的 GPU Frame Capture 代替）。工作流是：用 Unity Profiler 发现 GPU Bound → 用 RenderDoc 捕获那一帧 → 逐 Draw Call 展开看 GPU Duration、输入纹理、Shader 汇编代码。

```bash
# RenderDoc 捕获步骤（以 Android Vulkan 为例）
# ① 在 Unity 中 Build Android Development Build（勾选 Vulkan）
# ② 在 RenderDoc 的 Launch Application 里指向 apk
# ③ 设备上运行游戏、RenderDoc 里按 F12 捕获一帧
# ④ 展开 Event Browser 找到 GPU Duration 最高的 Draw Call
```

每个慢 Draw Call 展开后可以看：输入纹理（检查分辨率是否过大）、Shader 汇编（检查采样次数和分支）、输出 Render Target 的最终效果——这是 Unity Profiler 完全看不到的深度。

<Term def="Xcode 内置的性能分析套件：提供系统级 CPU 能耗等级、GPU 能耗报告、Metal 时间线、整个进程的物理内存压力（PSS/RSS）和温度状态——Unity Profiler 看不到的系统外部指标。">Xcode Instruments</Term> 是 iOS 端必备的补充——Unity Profiler 看引擎内部的 CPU/GPU 模块，Instruments 看系统外部的整体功耗、内存压力和温度状态。配合使用能回答：「引擎内部 GPU 只有 8ms，为什么系统说你的 GPU 已经热得快降频了？」

<Term def="腾讯出品的移动端性能测试工具：通过系统级采集直接读 FPS、CPU/GPU 频率、温度、电流、功耗曲线——无需 Development Build，Release 包直接测。与 Unity Profiler 互补。">Perfdog（性能狗）</Term> 不需要 Development Build——Release 构建的包就能测，更接近玩家真实体验。Perfdog 看 FPS 曲线、CPU/GPU 频率、温度、电流和功耗——这些 Unity Profiler 完全看不到。

三个工具各看一层——RenderDoc 逐 Draw 看 GPU 内部、Xcode Instruments 看系统级功耗和内存压力、Perfdog 看整体温度和耗电。一个完整的平台性能分析流程通常是三把刀并用。

## 第四个概念：Web 平台——浏览器里找瓶颈

WebGL 构建运行在浏览器的沙箱里——Unity Editor 无法通过常规网络协议与浏览器内的 WebAssembly 运行时建立 Profiler 连接。但浏览器自带的开发者工具就是一套强大的性能分析工具——不需要额外安装任何东西。

<Term def="Chrome 浏览器自带的性能分析面板：录制 Web 页面的帧时间线（FPS、JS/WASM 调用耗时、GPU 光栅化时间）——是 WebGL 游戏性能分析的核心工具。">
  Chrome DevTools Performance 面板
</Term>是 WebGL 游戏分析的唯一入口：

1. 打开 Chrome → 按 F12 打开 DevTools → 切到 **Performance** 标签
2. 点 Record（圆形按钮）→ 在 WebGL 游戏里正常操作 5-10 秒 → 点 Stop
3. 看 FPS 曲线（绿色横线）、JS/WASM 调用火焰图（每个函数的 Self Time）、GPU 光栅化耗时

```javascript
// JavaScript Console 里手动打标记——在时间线上标关键节点
// 在 Unity 中通过 Debug.Log 输出到 Console，配合 Performance 时间线找对应区间
console.time("BattleLoop");
// ... 战斗逻辑 ...
console.timeEnd("BattleLoop");
```

<Callout type="tip">
  DevTools 的 **Memory** 面板可以抓堆快照（Heap Snapshot）做前后对比——在进入场景前抓一份、场景退出后抓一份，看哪些对象没被回收。这就是 WebGL 版的「Memory Profiler 快照对比」，最适用于排查 Web 端的内存泄漏。
</Callout>

**Rendering 面板**（DevTools → 三个点 → More Tools → Rendering）是 WebGL 端另一个重要工具——可以实时看到 FPS 仪表、Paint Flashing（高亮每帧重绘区域）、Layer Borders（看哪些元素被 GPU 合成层）。

## 第五个概念：平台配置清单——上线前必须逐项检查

<Stepper>
  <Step title="① 在 Build Settings 启用对应平台的 Development Build">
    File → Build Settings → 选目标平台（Android / iOS / PC Standalone / WebGL）→ 勾选 Development Build。Console 平台（PS5/Xbox/Switch）需要在对应 SDK 里配置 DevKit 连接，不走 Unity 的 Build Settings。
  </Step>
  <Step title="② 确认真机 Profiler 连接方式并验证数据流通">
    Android：USB ADB 端口转发或 WiFi 同网段 → Profiler 窗口选 AndroidPlayer → 检查 CPU Usage 色条是否在变。iOS：USB 连 Mac → Xcode Devices 确认可发现 → Profiler 窗口选 iOSPlayer。Web：打开 DevTools Performance 面板 → Record 看时间线。
  </Step>
  <Step title="③ 检查各平台的帧预算是否在目标范围内">
    不同平台的帧预算完全不同：PC 通常 60FPS（16.67ms）、移动端可 30/60FPS（16/33ms）、XR 必须 ≥72FPS（≤13.9ms）、Console 需锁定在目标帧率且每帧必须在预算内（认证强制要求）。用 Profiler 看每帧的 CPU Total + GPU Total 时间，确认加起来在预算范围内。
  </Step>
  <Step title="④ 跑一次同场景、同操作的全平台基线录制并存档 .data">
    在所有目标平台上各跑一次相同的操作（如「进场景 → 跑 5 秒 → 放技能」），每台平台录一份 .data 文件存档。这在后续做「同场景下不同平台的性能对比」时是唯一可靠的比对基准——不同平台的数字不具有天然的可比性（PC 的 10ms 和移动端的 10ms 含义完全不同）。
  </Step>
</Stepper>

## 实战：五平台一次完整的 Profiling 流程

> 猜一猜：你在 PC 上开发 Unity 游戏、发现 GPU Time 很高——下一步用什么工具深入分析 GPU 瓶颈？答案在 §八练习里自测。

<Stepper>
  <Step title="① 发现瓶颈：从 Unity Profiler 到平台工具">
    在 PC 上用 Unity Profiler 录一段 GPU Bound 场景的数据——发现 Rendering 模块的 GPU 时间占整帧 60%。记住这一帧的帧号——下一步 RenderDoc 会精确捕获这一帧。
  </Step>
  <Step title="② GPU 深度分析：RenderDoc 捕获单帧">
    用 RenderDoc 连接 Unity 进程（PC 端直接 Launch Application 指向 .exe），在相同场景下按 F12 捕获一帧。展开 Event Browser 按 GPU Duration 降序——找到 Duration 最高的 3 个 Draw Call。展开每个慢 Draw：看 Fragment Shader 的汇编（是否有大量纹理采样、复杂分支）、输入纹理分辨率（是否不必要的大图）、输出 Render Target 格式（是否浮点 RT 带来额外带宽）。
  </Step>
  <Step title="③ 移动端验证：Perfdog 看温度与能耗">
    优化完 GPU 热点后同步推 Mobile 包——用 Perfdog 录 10 分钟对比优化前后的 GPU 频率曲线、温度曲线和耗电量。PC 上 GPU 时间从 12ms 降到 7ms——在移动端可能表现为：GPU 频率从 500MHz 降到 350MHz，芯片温度降了 5°C，10 分钟省了 200mAh 电量。这些数字 Unity Profiler 拿不到，只有 Perfdog 能告诉你。
  </Step>
  <Step title="④ WebGL 端录制性能帧时间线">
    Web 端打开 Chrome DevTools Performance 面板，在相同场景录 5 秒——看 FPS 曲线是否平稳、有无明显的 Long Task（>50ms 的黄色块）打断帧率。有 Long Task 就展开看调用栈定位是哪段 WASM 调用/GC 暂停导致的。Memory 面板抓前后快照对比——检查场景切换后是否存在 Managed 对象未回收的情况。
  </Step>
</Stepper>

## 关键代码：平台自动检测与工具链路配置

### Android ADB 连接自动检查脚本

每次插设备连 Profiler——如果每次都要手动敲 `adb forward` 然后输 IP 再去 Profiler 窗口选——重复操作非常低效。下面这段 Editor 脚本自动做设备检测和端口转发：

<CodeTabs>
  <Tab label="✓ Editor 脚本自动 ADB 连接">

```csharp
using UnityEditor;
using System.Diagnostics;

public static class AndroidProfilerConnector
{
    [MenuItem("Tools/Profiler/Auto Connect Android")]
    public static void AutoConnect()
    {
        // ① 检查 ADB 是否可用
        var check = Process.Start(new ProcessStartInfo
        {
            FileName = "adb",
            Arguments = "devices",
            RedirectStandardOutput = true,
            UseShellExecute = false,
            CreateNoWindow = true,
        });
        string output = check.StandardOutput.ReadToEnd();
        check.WaitForExit();

        // ② 确认至少有一台设备在线
        if (!output.Contains("\tdevice"))
        {
            EditorUtility.DisplayDialog("No Device",
                "没有检测到 Android 设备——请用 USB 连接并开启调试", "OK");
            return;
        }

        // ③ 端口转发：把设备的 Profiler 端口接到 PC 55000
        Process.Start("adb", "forward tcp:55000 tcp:55000")
              .WaitForExit();

        // ④ 提示用户在 Profiler 窗口选 AndroidPlayer
        EditorUtility.DisplayDialog("Connected",
            "ADB 端口已转发。请在 Profiler 窗口顶部 Active Target 选 AndroidPlayer",
            "OK");
    }
}
```
  </Tab>
  <Tab label="✗ 手动重复步骤（低效）">

```bash
# 每次都要手动敲这 4 条命令
adb devices                          # 检查设备在不在
adb forward tcp:55000 tcp:55000     # 端口转发
adb shell ip addr show wlan0        # 查 IP（仅 WiFi）
# 然后去 Profiler 窗口手动选 Target 输 IP——太慢了
```
  </Tab>

### 平台自识别与工具选择器

当项目同时发布多个平台时——在代码里写一套工具选择逻辑，根据当前构建目标平台自动选对工具链路：

```csharp
using UnityEngine;
#if UNITY_EDITOR
using UnityEditor;
#endif

public static class PlatformProfilingHelper
{
    public enum Platform { PC, Android, iOS, Console, Web }

    // 根据编译目标自动判断当前平台
    public static Platform CurrentPlatform
    {
        get
        {
#if UNITY_STANDALONE_WIN || UNITY_STANDALONE_OSX || UNITY_STANDALONE_LINUX
            return Platform.PC;
#elif UNITY_ANDROID
            return Platform.Android;
#elif UNITY_IOS
            return Platform.iOS;
#elif UNITY_PS5 || UNITY_GAMECORE_XBOXSERIES || UNITY_SWITCH
            return Platform.Console;
#elif UNITY_WEBGL
            return Platform.Web;
#else
            return Platform.PC;
#endif
        }
    }

    // 获取当前平台推荐的 GPU 分析工具
    public static string RecommendedGpuTool()
    {
        return CurrentPlatform switch
        {
            Platform.PC      => "RenderDoc 或 NSight Graphics（N卡）",
            Platform.Android => "RenderDoc（Vulkan/GLES）",
            Platform.iOS     => "Xcode GPU Frame Capture（Metal）",
            Platform.Console => "平台 SDK 自带工具（PIX/Razor）",
            Platform.Web     => "Chrome DevTools Rendering 面板",
            _                => "Unity Profiler Rendering 模块",
        };
    }

    // 获取当前平台推荐的整体性能分析方案
    public static string RecommendedPipeline()
    {
        return CurrentPlatform switch
        {
            Platform.PC => "Unity Profiler → RenderDoc → Profile Analyzer",
            Platform.Android => "Perfdog（系统级）→ Unity Profiler Remote → RenderDoc",
            Platform.iOS => "Xcode Instruments → Unity Profiler → Metal GPU Frame Capture",
            Platform.Console => "DevKit Profiler → SDK 专用 GPU 工具",
            Platform.Web => "Chrome DevTools Performance → Memory → Rendering",
            _ => "Unity Profiler",
        };
    }
}
```

## 容易踩的坑

<Callout type="trap">
  **现象**：Android 连接了 USB 线、跑了 `adb forward`、Profiler 窗口也选了 AndroidPlayer——但等了半分钟没有任何数据过来。**原因**：最常见原因：①没有勾选 `Development Build` 和 `Autoconnect Profiler` 两个都勾；②Android 的防火墙/权限没有开 INTERNET；③ADB 本身没识别到设备（`adb devices` 结果显示 offline）。**修法**：先用 `adb devices` 确认设备状态是 `device`（不是 offline）→ 在 Build Settings 里确认 Dev Build + Autoconnect 都已勾（一个不能少）→ 检查 AndroidManifest 是否有 `<uses-permission android:name="android.permission.INTERNET"/>` → 最后 `adb forward tcp:55000 tcp:55000` 再试。
</Callout>

<Callout type="trap">
  **现象**：用 RenderDoc 捕获了一帧，展开看每个 Draw Call 的 GPU Duration 都很短——加起来却远小于 Unity Profiler 里显示的 GPU 总时间。**原因**：RenderDoc 只捕获了应用自己的 GPU 命令——而 Unity Profiler 的 GPU Total 包含了**整个 GPU 时间轴**（包括系统合成、其他进程占用 GPU、VSync 等待等不在你 Draw Call 列表里的 GPU 时间）。RenderDoc 的数字是「你的 Draw Call 们在 GPU 上花了多久」——Profiler 的数字是「这帧的 GPU 一共工作了多久」。**修法**：两块数字一起看——RenderDoc 精准定位你的 Draw Call 中的热点、Profiler 告诉你 GPU 在帧里是否还有你控制不了的外部等待时间。如果 Profiler 的 GPU time >> RenderDoc 的 Draw Call 总和 ≈ 存在你控制不了的外部 GPU 等待。
</Callout>

<Callout type="trap">
  **现象**：Chrome DevTools Performance 面板显示 WebGL 游戏的 FPS 很高（60 稳定），但玩家反馈说「时不时画面会定住半秒」。**原因**：Performance 面板默认看的滚动平均帧率会抹平偶发的长停顿。这种偶发的停顿（200–500ms）在几秒钟的录制里可能只出现一次，但玩家会明显感知到画面突然冻结。**修法**：在 Performance 面板上放大时间轴、检查有没有超过 50ms 的黄色 Long Task 条块——如果有，展开看调用栈定位是 `WASM 编译`、`GC 暂停` 还是某帧突然计算量激增。同时去 Memory 面板抓 Heap Snapshot——看是否在停顿瞬间分配了大量临时对象触发 GC。
</Callout>

C 型 +1 道独立实现题。 */}

## 小结

- **五个平台五种连法**：PC 直连 / Android ADB / iOS 必须过 Mac / Console DevKit / Web 浏览器 DevTools
- **Unity Profiler 定位到瓶颈类别**——**平台专用工具逐帧深度分析**：RenderDoc 逐 Draw 看 GPU、Instruments 看系统功耗、Perfdog 看温度和电流曲线
- **WebGL 没有 Profiler 端口**——Chrome DevTools Performance + Memory + Rendering 三大面板代替
- 跨平台上线前必须**同场景、同操作、全平台各录基线**归档——PC 的 10ms 和 Mobile 的 10ms 含义完全不同
- **平台配置清单**：Dev Build 勾选 → 真机连接验证 → 帧预算达标确认 → 全平台基线录制

**练习 1（操作题）** 在你的项目里做一次跨平台 Profiling：①在 PC Standalone 上录一段同场景的 Unity Profiler 数据；②在 Android 设备上用 Development Build 连接 Profiler 录同一段场景；③对比两者的 CPU Usage 总时间——在同场景下差异是多少？记录下来并解释为什么会有这个差异。

<Answer>
操作步骤：按 §五 Stepper 的第一步在 PC 上录、第二步在 Android 上录。差异原因通常：PC 的 CPU 是 x86 高频大核、IL2CPP 生成的 x86 代码直接跑在桌面 CPU 上；Android 的 CPU 是 ARM 架构、频率受限于移动设备的热限制——同样一段 C# 脚本编译成 ARM IL2CPP 后运行效率可能只有 PC 的 1/3 甚至更低。另外 PC 直连 Editor 的 Profiler 几乎无连接开销，Android 的 USB/WiFi Profiler 连接会引入额外 CPU 开销（约 1–3ms 每帧的数据序列化+传输）。
</Answer>

**练习 2（操作题·C 型实现）** 写一个 Editor 脚本——`Tools/Profiler/Platform Check` 菜单项，点击后在 Console 输出：①当前构建目标平台 ②当前平台推荐的 GPU 分析工具 ③当前平台推荐的完整 Profiling 工具链路。参考 §六的 `PlatformProfilingHelper` 类。

<Answer>
参考 §六的第二个代码块 `PlatformProfilingHelper`。扩展需求：加一个 `[MenuItem]` 方法调用 `RecommendedGpuTool()` 和 `RecommendedPipeline()`，用 `Debug.Log` 输出到 Console。
</Answer>

**练习 3（问答型）** 你在 PC 上开发 Unity 游戏、Unity Profiler 发现 GPU Time 很高——下一步用什么工具深入分析 GPU 瓶颈？为什么不用别的工具？

<Answer>
用 RenderDoc 捕获一帧逐 Draw 看 GPU 热点。选择 RenderDoc 而不是别的工具是因为：RenderDoc 是唯一跨平台（PC/Android/Switch）、能捕获单帧所有 Draw Call 并查看每个 Draw 的输入纹理/Shader 编译结果/资源绑定的通用 GPU 帧调试工具。Unity Profiler 只能告诉你「GPU 花了多少毫秒」，RenderDoc 能告诉你「哪个 Draw 吃了最多 GPU」和「为什么」（纹理太大？Shader 太复杂？还是带宽不够？）。如果是 NVIDIA 显卡也可用 NSight Graphics——看 GPU 硬件计数器更全——但 RenderDoc 更通用。
</Answer>

<Glossary>

<GlossaryItem term="Development Build">
  内置 Profiler 连接支持的特殊构建版本：语法接近 Release 但保留 Profiler 数据端口让 Editor 远程连接。Build Settings → Development Build 勾选即可。所有真机平台的 Profiling 必须基于 Development Build——Release 构建不带 Profiler 连接模块。详见本章「第二个概念」。
</GlossaryItem>

<GlossaryItem term="Autoconnect Profiler">
  Development Build 的便捷设置：勾选后真机游戏启动时自动搜索并连接同一局域网内的 Editor Profiler，省去手动输入 IP。底层是多播网络发现 + 自动握手，解决「设备重启后 IP 变了要重新输」的烦恼。详见本章「第二个概念」。
</GlossaryItem>

<GlossaryItem term="ADB 端口转发">
  通过 `adb forward tcp:55000 tcp:55000` 把 Android 设备的 Profiler 端口映射到 PC 的同一端口——让 Unity Editor 能直接跟 Android 设备通信。是 Android 远程 Profiler 连接的第一步（先转发端口、再在 Profiler 窗口选设备）。详见本章「第二个概念」和「§六 ADB 代码」。
</GlossaryItem>

<GlossaryItem term="RenderDoc">
  开源的跨平台 GPU 帧调试器——捕获单帧所有 Draw Call 并按 GPU Duration 排序。展开每个 Draw 能看到顶点/片段 Shader 汇编、输入纹理分辨率、输出 Render Target 效果、资源绑定表——逐像素追查 GPU 瓶颈在哪个具体 Draw Call 上。Unity Profiler 只给总 GPU Time，RenderDoc 给逐 Draw Call 拆解。详见本章「第三个概念」。
</GlossaryItem>

<GlossaryItem term="Xcode Instruments">
  Xcode 内置的 iOS 性能分析套件——看系统级 CPU/GPU/Metal 能耗报告、物理内存统计、温度和电池数据。这些 Unity Profiler 全局看不到的指标在 Instruments 里展开是对应的——配合使用能完整定位 iOS 平台性能问题。详见本章「第三个概念」。
</GlossaryItem>

<GlossaryItem term="Perfdog（性能狗）">
  腾讯出品的移动端性能测试工具——直接通过系统级采集获取 FPS、CPU/GPU 频率和电流曲线。不需要 Development Build——Release 包就能测，更贴近玩家体验。与 Unity Profiler 互补：Perfdog 看系统级能耗和温控状态，Unity Profiler 看引擎内部热点函数。详见本章「第三个概念」。
</GlossaryItem>

<GlossaryItem term="Chrome DevTools Performance 面板">
  Chrome 浏览器自带的性能分析面板：录制 WebGL 游戏的帧时间线（FPS、JS/WASM 调用耗时、GPU 光栅化时间）。由于 WebGL 运行在浏览器沙箱内无法由 Editor 直连 Profiler，Performance 面板是 WebGL 的唯一入口工具。详见本章「第四个概念」。
</GlossaryItem>

<GlossaryItem term="帧预算">
  目标帧率对应的单帧最大可承受时间：60FPS → 16.67ms 总预算（含 CPU+GPU+系统开销），30FPS → 33.33ms，90FPS → 11.11ms。任何平台 Profiling 的第一步都不是找热点——而是确认帧 CPU+GPU 总时间在当前平台的帧预算之内。详见本章「平台配置清单」的第三步。
</GlossaryItem>

</Glossary>
