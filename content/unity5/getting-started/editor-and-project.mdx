## 没有引擎，做一个游戏要从哪开始？

想做一个游戏：屏幕上要画出角色、角色要会因为重力往下掉、玩家按键要能控制它、还得把模型和音乐这些素材读进来。这些事每一件都很硬核——光是「把一个三维模型画到屏幕上」就能写上几千行底层代码。如果全都从零自己写，你可能一年都做不出一个能跑的小游戏。

Unity 就是来替你扛下这些脏活的。它像一座**已经盖好的片场**：灯光、摄影机、地板、物理规律（东西会往下掉）都给你备齐了，你不用自己造摄影机、自己算重力。你要做的，是当导演——把演员和道具摆进场、告诉他们该怎么演。

这一章就带你认识这座片场的「控制台」：Unity 的编辑器界面长什么样、五个主要窗口各管哪一摊事、一个项目的文件是怎么组织的，以及做游戏时你会一遍遍重复的那条工作流。没有这些，你打开 Unity 只会看到一堆面板，不知道从哪下手。

## Unity 到底替你做了什么

<Term def="一套帮你把游戏跑起来的现成软件框架——渲染画面、模拟物理、处理输入、管理资源这些底层活儿它都包了，你只管做玩法。">
  游戏引擎
</Term>
是一套已经写好的「游戏底层设施」。具体来说，它至少替你包办了四件大事：

- **渲染**：把三维模型、贴图、灯光算成屏幕上一帧帧的画面（这是最重、最难自己写的一块）。
- **物理**：让物体受重力下落、互相碰撞、反弹——你只要说「这东西有重量」，引擎自动算它怎么动。
- **输入**：把键盘、鼠标、手柄、触摸屏的信号收上来，变成你代码里能读的「玩家按了跳跃键」。
- **资源管理**：把模型、贴图、声音这些素材读进内存、在合适的时机加载和释放。

把这些交给引擎，你就能把精力全花在**玩法**上——这关怎么设计、角色跳多高、敌人怎么追你。Unity 是目前最流行的游戏引擎之一，你跟它打交道的窗口，就叫 <Term def="Unity 那个图形化的操作界面——你在里面摆放对象、调整属性、点击 Play 测试游戏。它不写代码也能搭出大半个游戏。">编辑器</Term>。

## 编辑器的五大窗口，各管一摊

打开 Unity，你会看到一堆面板。别慌——它们的标准布局就五块，每一块对应片场里的一个角色。下面这张图是你认识整个界面的「地图」：

<EditorLayoutDiagram />

逐个说这五个窗口在干什么：

### Hierarchy（层级）：场上有谁

<Term def="列出当前场景里所有对象的清单窗口，呈树状（对象可以有子对象）。它回答「这一场里都有谁」。">
  Hierarchy
</Term>
是**当前这一场的演员名单**。你这关里有什么——主角、摄影机、灯光、地面、敌人——全列在这儿，一行一个。它只管「场上有谁」，不管这些东西长什么样、在哪。对象之间还能有父子关系（比如「角色」下面挂着「武器」），所以它呈树状。

### Scene（场景）与 Game（游戏）：导演的视角 vs 观众的视角

这两个窗口常常并排或用标签切换，最容易搞混，但分工很清楚：

- **Scene 窗口**是**导演摆位的视角**。你在这里的三维空间里自由地拖动、旋转、缩放对象，把布景一件件摆好。你能从任意角度看、能飞到场景背后去——这是**给你这个制作者用的**工作台，玩家永远看不到它。
- **Game 窗口**是**摄影机取景**，显示的是**玩家真正会看到的画面**——也就是游戏里那台摄影机拍到的东西。点了 Play 之后，游戏就在这个窗口里跑起来。

一句话记牢：**Scene 是你摆位用的，Game 是玩家看的**。

### Inspector（检视面板）：选中谁就显示谁的资料

<Term def="显示并编辑「当前选中对象」全部属性的面板——选中谁，它就显示谁的资料卡，所有数值都能在这里改。">
  Inspector
</Term>
是**被选中那个演员的资料卡**。你在 Hierarchy 里点中一个对象，Inspector
立刻显示出它的全部信息：它在哪个位置、有多大、挂着哪些功能、每个功能的参数是多少——而且都能直接在这里改。**选中变了，Inspector
显示的内容就跟着变**。它是你调整一切细节的地方。

### Project（工程）：所有资源的仓库

<Term def="显示项目里全部资源文件的窗口，对应硬盘上的 Assets 文件夹——模型、贴图、脚本、声音都堆在这里，是你的素材库。">
  Project
</Term>
窗口是**道具仓库**。你为这个游戏准备的所有素材——三维模型、贴图、声音、脚本、动画——都堆在这里。注意它和
Hierarchy 的区别：**Project 是「仓库里有哪些素材」，Hierarchy
是「这一场用上了哪些」**。素材要先从仓库（Project）拖进场景（Hierarchy），才会出现在游戏里。

## 一个 Unity 项目里都有什么文件

当你新建一个 Unity 项目，它会在硬盘上生成一个文件夹，里面有几个固定的子目录。你只需要关心一个，其余的离它们远点：

```text
MyGame/
├── Assets/            ← 你的资源根目录：模型、贴图、脚本、场景全放这里
│   ├── Scenes/        （你建的场景文件 .unity）
│   ├── Scripts/       （你写的 C# 脚本 .cs）
│   └── Textures/      （贴图等素材）
├── Library/           ← Unity 自动生成的缓存，别手动碰
├── ProjectSettings/   ← 项目配置，用编辑器菜单改，别手动碰
├── Packages/          ← 包依赖清单
└── Temp/ / Logs/      ← 临时文件，可随时删
```

唯一属于你的地盘是 <Term def="Unity 项目里存放你全部资源的根文件夹；Project 窗口显示的就是它的内容。放进 Assets 的东西 Unity 才会识别和导入。">Assets</Term> 文件夹——**Project 窗口显示的就是 `Assets` 里的内容**。你的模型、脚本、场景文件全放在这里（可以再分子文件夹整理）。放进 `Assets` 的素材，Unity 才会自动导入、生成预览、让你能在编辑器里用。

`Library` 是 Unity 根据 `Assets` 自动生成的**缓存**（导入后的素材、编译结果等），删了它 Unity 会重新生成，所以它不该进版本管理、更不要手动编辑。`ProjectSettings` 存项目级配置，要改请走编辑器的菜单，而不是去手动改文件。记住一句话：**只动 `Assets`，其余交给 Unity**。

## 场景、对象、组件：一层套一层

游戏不是一锅乱炖，它是有层级的。一个 <Term def="一个关卡、一个界面或一个可独立加载的「房间」。游戏由若干场景组成，运行时一次通常加载一个场景。文件后缀是 .unity。">场景（Scene）</Term> 就是一个**关卡或一个界面**（主菜单是一个场景，第一关是另一个场景）。场景里装的是一个个 <Term def="Unity 场景里最基本的「东西」——角色、摄影机、灯光、地面都是 GameObject。它本身只是个空壳，真正的本事靠挂在它身上的组件提供。">GameObject</Term>，也就是「演员和道具」。

但 GameObject 本身只是个**空壳**——它默认什么都不会，连「在哪个位置」这种事都得靠别人告诉它。真正给它本事的，是挂在它身上的 <Term def="挂在 GameObject 上、给它赋予某种能力的功能模块。位置靠 Transform 组件、外观靠 Renderer 组件、玩法逻辑靠脚本组件——一个 GameObject 可以挂很多个组件。">组件（Component）</Term>。组件就像**给演员加的技能或装备**：挂上「Transform 组件」它才有了位置和朝向，挂上「Renderer 组件」它才画得出来，挂上你写的「脚本组件」它才有了玩法逻辑。

所以整条链是：**Project（仓库里的资源）→ 拖进 → Scene（场景）→ 里面是一个个 GameObject → 每个 GameObject 挂着若干 Component**。下面这个动画把这条链一步步点亮，跟着走一遍你就懂了 Unity 是怎么把一切组织起来的：

## 预告：一个脚本长什么样

前面说「挂脚本」给 GameObject 加玩法逻辑——脚本到底长什么样？在 Unity 里右键新建一个 C# 脚本，默认会得到下面这段（细节后续章节会专门讲，这里先混个眼熟）：

```csharp
using UnityEngine;

// 这个类就是一个「脚本组件」，名字要和文件名一致
public class PlayerController : MonoBehaviour
{
    // Start：这个对象第一次「上场」时调用一次，用来做初始化
    void Start()
    {
        Debug.Log("Player 上场了！");
    }

    // Update：游戏运行时每一帧都会调用一次，玩法逻辑大多写在这
    void Update()
    {
        // 这里以后会写：读玩家输入、移动角色……
    }
}
```

三个要点先记住：① 这个类继承自 `MonoBehaviour`——只有继承了它的脚本，才能作为**组件**挂到 GameObject 上。② `Start()` 在对象首次启用时**只跑一次**，适合做初始化。③ `Update()` 在游戏运行时**每一帧都跑**（一秒可能几十上百次），是写玩法逻辑的主战场。把这段脚本挂到一个 GameObject 上、点 Play，控制台就会打印那句「Player 上场了！」——这就是「挂脚本」的全过程。

## 容易踩的坑

<Callout type="trap">
  **现象**：在 Scene 窗口里把对象摆得好好的，切到 Game
  窗口却看不到、或位置不对。**原因**：把 Scene（你摆位用的导演视角）和
  Game（玩家通过摄影机看到的画面）搞混了——Scene
  里看得到不等于摄影机拍得到。**修法**：记住 Game
  窗口显示的是**场景里那台摄影机**拍到的内容；东西看不到就去调摄影机的位置和朝向，或确认对象在摄影机的取景范围内。
</Callout>

<Callout type="trap">
  **现象**：点了 Play，在运行中把某个属性调得正合适，点 Stop
  后发现改动全没了、变回原样。**原因**：Play（运行模式）下对场景所做的修改都是**临时**的，只为让你方便测试；Stop
  退出运行模式时，场景会**整体还原**到点 Play
  之前的样子。**修法**：要永久生效的改动，必须在**编辑模式**（没点 Play
  时）做；运行中调出的好数值，记下来、Stop 之后再填回去。
</Callout>

<Callout type="trap">
  **现象**：辛苦做的模型/脚本放进了项目文件夹，Project
  窗口里却怎么都看不到、用不了。**原因**：素材没放进 `Assets`
  文件夹（放到了项目根目录或 `Library` 等其它目录）——Unity 只导入 `Assets`
  里的东西。**修法**：把所有资源放进 `Assets`（或它的子文件夹）。直接拖进
  Project 窗口最稳妥，Unity 会自动放对位置并完成导入。
</Callout>

## 小结

- Unity 是**游戏引擎**，替你包办渲染、物理、输入、资源管理，让你专注做玩法
- 编辑器五大窗口：**Hierarchy**（场上有谁）、**Scene**（导演摆位）、**Game**（玩家视角）、**Inspector**（选中对象的资料卡）、**Project**（资源仓库）
- 项目里只动 **`Assets`**（你的资源根），`Library` / `ProjectSettings` 等交给 Unity，别手碰
- 层级一层套一层：**Project 资源 → Scene 场景 → GameObject → Component 组件**
- 工作流是「编辑 ⇄ 运行」循环；**Play 模式下的修改 Stop 后会丢**，永久改动要在编辑模式做

<Exercises>

**练习 1（问答型）** 用自己的话说清楚：Scene 窗口和 Game 窗口的区别是什么？为什么需要两个？

<Answer>

Scene 窗口是**给制作者（你）用的导演摆位视角**：可以从任意角度自由观察场景、拖动和编辑对象，玩家永远看不到它。Game 窗口显示的是**玩家真正会看到的画面**，也就是场景里那台摄影机拍到的内容。需要两个是因为：你摆位、编辑时需要能自由移动视角看清每个角落（Scene 干这个），但你也要随时确认「玩家最终看到的效果」对不对（Game 干这个）。点 Play 后，游戏就在 Game 窗口里跑起来。

</Answer>

**练习 2（问答型）** 一个新建的 GameObject「本身什么都不会」，那它怎么才有位置、能被画出来、能响应玩法逻辑？这背后是 Unity 的什么设计？

<Answer>

靠给它挂**组件（Component）**。GameObject 本身只是个空壳，能力全由挂在它身上的组件提供：挂 **Transform** 组件才有位置/旋转/缩放，挂 **Renderer** 组件才画得出来，挂你写的**脚本**组件才有玩法逻辑。这背后是 Unity 的「组合」设计——不是靠继承造一个个庞大的对象，而是把功能拆成一块块组件，需要什么本事就挂什么组件。所以在 Inspector 里看到的那一串东西，就是这个 GameObject 挂着的所有组件。

</Answer>

**练习 3（验证型）** 在 Play 模式下把某个物体的某个属性（比如位置）改一改，再点 Stop。说出你预期会发生什么，并解释为什么 Unity 要这样设计。

<Answer>

预期：Stop 之后，刚才在运行中改的属性会**还原**回点 Play 之前的值，改动丢失。Unity 这样设计是因为 Play 模式的本意是**让你安全地测试**——你可以在运行中随便折腾、调参数、试效果，不用担心把精心搭好的场景弄乱，因为 Stop 一退出就自动恢复原状。代价就是：运行中调出的好数值不会自动保留，要永久生效必须在编辑模式（没 Play 时）重新设置。

</Answer>

</Exercises>

<Glossary>

<GlossaryItem term="游戏引擎">
  一套帮你把游戏跑起来的现成软件框架。渲染画面、模拟物理（重力/碰撞）、处理玩家输入、管理素材这些底层脏活它都包了，你只管设计玩法。Unity
  就是一个游戏引擎。详见本章「Unity 到底替你做了什么」一节。
</GlossaryItem>

<GlossaryItem term="编辑器">
  Unity 那个图形化的操作界面——你在里面摆放对象、调属性、点 Play
  测试游戏，不写一行代码也能搭出大半个游戏。本章把它比作数字片场的「控制台」。详见本章「编辑器的五大窗口」一节。
</GlossaryItem>

<GlossaryItem term="Hierarchy">
  「层级」窗口，列出**当前场景里所有对象**的清单（呈树状，对象可以有子对象）。它回答「这一场上都有谁」。注意它只列场上用到的，不等于仓库里全部素材。详见本章「Hierarchy」一节。
</GlossaryItem>

<GlossaryItem term="Inspector">
  「检视面板」，显示并编辑**当前选中对象**的全部属性——选中谁就显示谁的资料卡，所有数值都能在这里改。选中变了，它显示的内容就跟着变。详见本章「Inspector」一节。
</GlossaryItem>

<GlossaryItem term="Project">
  「工程」窗口，显示项目里**全部资源文件**，对应硬盘上的 `Assets`
  文件夹——模型、贴图、脚本、声音都堆在这里，是你的素材仓库。素材要先从这儿拖进场景才会出现在游戏里。详见本章「Project」一节。
</GlossaryItem>

<GlossaryItem term="Assets">
  Unity 项目里存放你**全部资源**的根文件夹；Project
  窗口显示的就是它的内容。只有放进 `Assets`（或其子文件夹）的东西，Unity
  才会识别并导入。做项目时基本只动这个文件夹。详见本章「一个 Unity
  项目里都有什么文件」一节。
</GlossaryItem>

<GlossaryItem term="场景（Scene）">
  一个**关卡、一个界面或一个可独立加载的「房间」**（主菜单是一个场景，第一关是另一个场景）。游戏由若干场景组成，运行时通常一次加载一个，文件后缀
  `.unity`。详见本章「场景、对象、组件」一节。
</GlossaryItem>

<GlossaryItem term="GameObject">
  Unity 场景里最基本的「东西」——角色、摄影机、灯光、地面都是
  GameObject。它本身只是个空壳，什么都不会；真正的本事靠挂在它身上的**组件**提供。详见本章「场景、对象、组件」一节。
</GlossaryItem>

<GlossaryItem term="组件（Component）">
  挂在 GameObject 上、给它某种能力的功能模块，像「给演员加的技能/装备」。挂
  Transform 才有位置，挂 Renderer 才画得出来，挂脚本才有玩法逻辑——一个
  GameObject 可以挂很多组件。详见本章「场景、对象、组件」一节。
</GlossaryItem>

<GlossaryItem term="Play 模式">
  点编辑器顶部的 Play 按钮后进入的「运行模式」——游戏真的跑起来，你能在 Game
  窗口里实际操作测试。再点一次（Stop）就回编辑模式。**关键坑**：Play
  模式下对场景的修改是临时的，Stop 后会全部还原。详见本章「容易踩的坑」一节。
</GlossaryItem>

</Glossary>
