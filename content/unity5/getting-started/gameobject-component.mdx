## 一个空演员，怎么变成会演戏的角色？

上一章我们把 Unity 编辑器当成一间数字片场，场上的角色、道具叫 GameObject。但有个问题：一个刚被叫上场的「演员」，其实什么都不会——他不会动、不会说话、甚至你都看不见他。光把演员摆上台，戏是演不起来的。

那怎么让他有本事？办法是给他**发技能卡**：发一张「会武打」的卡，他就能打；发一张「会唱歌」的卡，他就能唱。要哪种本事，就发哪张卡，随时能加、能换。演员本人始终是那个空壳，真正让他「能演」的是身上那一叠技能卡。

这一章就讲 Unity 是怎么用这套「空壳演员 + 技能卡」的思路来搭建一切的：为什么 GameObject 本身是空的、技能卡（组件）如何一项项给它赋能、以及 Unity 为什么偏偏选了这条路而不是更"传统"的做法。想不通这一点，你会一直纳闷「为什么我新建的物体在屏幕上看不到」。

## GameObject：一个什么都不会的空壳

<Term def="Unity 场景里最基本的「东西」——角色、摄影机、灯光、地面都是 GameObject。它本身只是个空壳容器，没有任何行为，真正的本事靠挂在它身上的组件提供。">
  GameObject
</Term>
是 Unity
场景里最基本的单位，但它有个反直觉的特点：**它本身什么行为都没有**。它不知道自己长什么样、不会动、也不会响应任何事——它只是一个**空壳、一个容器**，作用是「持有」一组能力模块，并给自己起个名字、占个位。

这听起来很奇怪：那它有什么用？答案是——GameObject 的价值不在于它"是什么"，而在于你**往它身上挂了什么**。一个空 GameObject 就像片场里一个还没领到任何技能卡的演员：你能给他起名、点名让他上场，但他暂时一身本事都没有。

## 组件：一张张给演员赋能的技能卡

让 GameObject「长出本事」的，是

<Term def="挂在 GameObject 上、给它赋予某种能力的功能模块，像给演员加的技能卡或装备。一个 GameObject 可以同时挂很多个组件，每个负责一种能力。">
  组件（Component）
</Term>
。组件就是那一张张技能卡：挂上一张，GameObject
就多一项本事；挂上另一张，又多一项。关键在于——**一个 GameObject
可以同时挂很多个组件**，它最终的能力，就是身上所有组件能力的总和。

具体来说：挂上一个负责「位置」的组件，它才知道自己站在世界的哪个角落；挂上一个负责「绘制」的组件，它才画得出来、你才看得见它；挂上你写的「逻辑」组件，它才会响应玩家、动起来。下面这个动画，把一个灰色空壳逐步挂上组件、一项项点亮能力的过程演给你看：

看完这个动画，你应该能体会到那句话的分量：**GameObject 是空壳，组件才是本事**。所以第一章末尾那个练习的答案也清楚了——一个新建的空 GameObject 之所以可能在屏幕上看不到，正是因为它还没挂上负责「画出来」的组件。

## 组合优于继承：Unity 为什么这样设计

你可能会想：让一个对象「会飞 + 会射击」，直接写一个「会飞会射击的敌人」类不就行了？这正是传统面向对象的做法——靠**继承**层层派生类。但它有个致命问题，叫**类爆炸**。

Unity 走的是另一条路，叫

<Term def="一种设计取向：把功能拆成一块块可复用的组件，要哪种能力就给对象挂上对应组件，而不是靠类的继承去层层堆叠能力。它能避免「类爆炸」，且能力可在运行时自由增删。">
  组合优于继承
</Term>
。下面这张对比图把两条路并排放在一起，一眼就能看出差别：

<CompositionVsInheritanceDiagram />

左边传统继承：想要「会飞 + 会射击」的敌人，你得专门造一个 `FlyingShootingEnemy` 子类把两种能力合进来。一旦再加「会潜水」「会隐身」……每多一种能力，能力的**组合数**就翻倍，子类数量呈 $2^n$ 指数膨胀，而且每个子类的代码很难复用。

右边 Unity 组合：永远只有**一个 GameObject**，能力被拆成一张张可勾选的组件（`FlyMovement`、`Shooter`、`Health`……）。要哪种本事就勾上哪个组件，自由拼装，再也不用为能力组合去造新类。这就是 Unity 整个模型的灵魂——也是为什么它叫「组件模型」。

## Transform：每个 GameObject 都必有、删不掉

在所有组件里，有一个特别——它是**唯一一个每个 GameObject 都强制自带、且无法移除**的组件，那就是

<Term def="记录并控制 GameObject 位置（Position）、旋转（Rotation）、缩放（Scale）的组件，相当于它在世界里的「站位牌」。每个 GameObject 必有且不可移除。">
  Transform
</Term>
。它记录这个对象的**位置、旋转、缩放**——用片场的话说，就是每个演员自带的一块「站位牌」，写着他站哪、朝哪、个头多大。

为什么唯独它不可删？因为「我在世界里的哪个位置」是任何对象都绕不开的基本信息：哪怕一个纯逻辑、看不见的对象，父子层级和坐标变换也都依赖它。所以 Unity 规定：GameObject 一被创建，就自带一个 Transform，你能改它的值，但删不掉它。其它组件（画得出、能碰撞……）则都是**可选**的，可挂可不挂、可随时移除。

## 常见组件，以及怎么把它们挂上去

除了必有的 Transform，下面这几个是你最常打交道的组件：

- <Term def="负责把 GameObject 用网格和材质画到屏幕上的组件（最常见的是 MeshRenderer）。没有它，物体在 Game 窗口里就看不见。">
    Renderer
  </Term>
  （如 MeshRenderer）：决定它**怎么被画出来**——没有它，物体就是隐形的。
- <Term def="给 GameObject 一个碰撞体积的组件（如 BoxCollider、SphereCollider），让它能被撞到、能踩在上面，参与碰撞检测。">
    Collider
  </Term>
  （如 BoxCollider）：给它一个**碰撞体积**，能被撞到、能踩在上面。
- <Term def="把 GameObject 交给物理引擎模拟的组件——挂上后它会受重力下落、被力推动、与其它物体真实地碰撞反弹。">
    Rigidbody
  </Term>
  ：交给**物理引擎**接管——受重力下落、被力推动、真实地碰撞反弹。
- <Term def="你用 C# 写的、继承自 MonoBehaviour 的类，作为组件挂到 GameObject 上以添加自定义玩法逻辑，并获得 Start、Update 等生命周期回调。">
    脚本组件（MonoBehaviour）
  </Term>
  ：你写的**玩法逻辑**——只有继承了 MonoBehaviour 的脚本才能当组件挂上去。

## 代码里怎么操作组件：GetComponent 与 AddComponent

「挂组件」不只在编辑器里点点点，运行时也能用代码操作。两个最常用的 API：`GetComponent<T>()` 取得 GameObject 上**已经挂着的**某个组件，`AddComponent<T>()` 在运行时**动态挂上**一个新组件。常见配合是「先取，没有再加」：

```csharp
using UnityEngine;

// 这个脚本本身也是一个组件（继承 MonoBehaviour），挂在某个 GameObject 上
public class EnsureRigidbody : MonoBehaviour
{
    void Start()
    {
        // ① GetComponent<T>()：取自身这个 GameObject 上已挂的 Rigidbody 组件
        Rigidbody rb = GetComponent<Rigidbody>();

        // ② 取不到会返回 null —— 说明还没挂这个组件
        if (rb == null)
        {
            // ③ AddComponent<T>()：运行时动态给 GameObject 挂上一个 Rigidbody
            rb = gameObject.AddComponent<Rigidbody>();
        }

        // ④ 现在 rb 一定有效，可以放心读写它的属性了
        rb.mass = 2f;
    }
}
```

逐行看：① `GetComponent<Rigidbody>()` 在「我」这个 GameObject 身上找 Rigidbody 组件；② 关键是它**找不到时返回 `null`**，所以拿到引用后习惯先判空；③ 确实没挂时，用 `gameObject.AddComponent<Rigidbody>()` 当场补一个——这等价于在 Inspector 里点 Add Component；④ 这之后 `rb` 必定指向一个真实组件，访问它的 `mass` 等属性才安全。记住这条因果链：**组件得先挂上，代码才取得到**。

## 容易踩的坑

<Callout type="trap">
  **现象**：在场景里新建一个空 GameObject，摆好位置，运行后 Game
  窗口里却什么都看不到。**原因**：空 GameObject 默认只带一个
  Transform，**没有任何负责「画出来」的组件**——引擎没东西可画。**修法**：给它挂上
  MeshRenderer（配 MeshFilter 提供网格），或直接用内置的 Cube /
  Sphere（它们出生就带这两个组件）。记住：没有 Renderer 就没有画面。
</Callout>

<Callout type="trap">
  **现象**：想把某个 GameObject 上的 Transform
  组件删掉（觉得用不到位置），却发现它的右键菜单里没有「Remove
  Component」。**原因**：Transform 是**每个 GameObject
  必有、不可移除**的组件——位置/旋转/缩放是任何对象都绕不开的基本信息。**修法**：别想着删它；你只能改它的值（或把对象的位置设成你要的）。可删的是
  Renderer、Collider 这些可选组件，Transform 不在其列。
</Callout>

<Callout type="trap">
  **现象**：分不清「这是个 GameObject 还是个 Component」，或以为「脚本 =
  GameObject」。**原因**：把**容器**和**挂在容器上的能力**混为一谈了。**修法**：记牢分工——你在场景里摆放、命名、选中的是
  **GameObject（演员）**；你写的脚本、Transform、Renderer 都是挂在它身上的
  **Component（技能卡）**。一个 GameObject 可挂多个 Component，但 Component
  不能脱离 GameObject 独立存在。
</Callout>

## 小结

- **GameObject 是空壳容器**，本身什么行为都没有；它的价值在于你往它身上挂了什么
- **组件（Component）是技能卡**：挂一个加一项能力，一个 GameObject 可挂多个，能力是所有组件之和
- Unity 用**组合优于继承**：能力拆成可复用组件、按需拼装，避开了传统继承的「类 $2^n$ 爆炸」
- **Transform 是唯一必有、不可移除**的组件，记录位置/旋转/缩放；其余（Renderer/Collider/Rigidbody/脚本）都可选
- 加组件：编辑器里点 **Add Component**，或代码里 `AddComponent<T>()`；取已挂组件用 `GetComponent<T>()`（取不到返回 `null`）

**练习 1（问答型）** 用「空演员 + 技能卡」的比喻，解释为什么「一个新建的空 GameObject 在 Game 窗口里看不到」。它至少要挂上什么，才会被画出来？

<Answer>

空 GameObject 就像一个刚上场、还没领任何技能卡的演员——他有名字、占个位（自带 Transform 提供位置），但没有任何「让自己被看见」的本事。要让它出现在画面里，至少得给它挂上一个 **Renderer**（最常见是 MeshRenderer，配合 MeshFilter 提供网格数据）。MeshRenderer 负责把网格用材质画到屏幕上；没有它，引擎就没有任何东西可画，所以物体是隐形的。直接用内置的 Cube、Sphere 等对象也行——它们出生就自带 MeshFilter + MeshRenderer。

</Answer>

**练习 2（问答型）** 「组合优于继承」相比传统继承，好在哪里？请举一个「会飞 + 会射击的敌人」的例子说明。

<Answer>

传统继承做「会飞 + 会射击的敌人」，得专门造一个 `FlyingShootingEnemy` 子类把两种能力合进来；再加「会潜水」「会隐身」，子类数量会随能力种类呈 $2^n$ 指数膨胀（类爆炸），代码也难复用。组合则只有**一个 GameObject**，把能力拆成独立组件（`FlyMovement`、`Shooter`、`Diving`……），要哪种本事就勾上哪个组件。好处：①组件可在不同对象间复用；②新增能力只是新增一个组件，不会催生大量子类；③运行时还能用 `AddComponent`/`Destroy` 动态增删能力。

</Answer>

**练习 3（改 Demo / 代码型）** 本章「组件组合动画」按 Transform → MeshRenderer → Collider → PlayerScript 的顺序点亮能力。如果让你给这个动画**再加一个 Rigidbody 组件**（让对象「会受重力下落」），你会把它放在哪一步、点亮什么能力文案？另外，用 `GetComponent`/`AddComponent` 写一行：取自身的 Rigidbody，没有就加一个。

<Answer>

合理的位置是放在 **Collider 之后**（先有碰撞体积，再交给物理引擎模拟才有意义），点亮的能力文案可以写「**会下落 / 受物理**」——对应 Rigidbody 让对象受重力、被力推动、真实碰撞反弹。代码上「取不到就加」可以写成：

```csharp
// 取自身的 Rigidbody，没有就加一个
Rigidbody rb = GetComponent<Rigidbody>() ?? gameObject.AddComponent<Rigidbody>();
```

`GetComponent<Rigidbody>()` 取不到时返回 `null`，`??` 空合并运算符就退而用 `AddComponent<Rigidbody>()` 当场补挂，最终 `rb` 一定有效。

</Answer>

  都必有、且不可移除**——位置是任何对象都绕不开的基本信息。详见本章「Transform：每个
  GameObject 都必有、删不掉」一节。
</GlossaryItem>

<GlossaryItem term="Renderer">
  负责把 GameObject 用网格和材质**画到屏幕上**的组件（最常见的是
  MeshRenderer）。没有它，物体在 Game
  窗口里就是隐形的。这正是「新建空对象看不到」的原因。详见本章「常见组件，以及怎么把它们挂上去」一节。
</GlossaryItem>

<GlossaryItem term="Collider">
  给 GameObject 一个**碰撞体积**的组件（如
  BoxCollider、SphereCollider），让它能被撞到、能踩在上面、参与碰撞检测。是可选组件，按需挂。详见本章「常见组件，以及怎么把它们挂上去」一节。
</GlossaryItem>

<GlossaryItem term="Rigidbody">
  把 GameObject
  交给**物理引擎**模拟的组件——挂上后它会受重力下落、被力推动、与其它物体真实地碰撞反弹。需要物理表现的对象才挂，纯静态物体通常不挂。详见本章「常见组件，以及怎么把它们挂上去」一节。
</GlossaryItem>

<GlossaryItem term="脚本组件（MonoBehaviour）">
  你用 C# 写的、继承自 **MonoBehaviour** 的类，作为组件挂到 GameObject
  上以添加自定义**玩法逻辑**。只有继承了 MonoBehaviour
  的脚本才能当组件挂上去，并获得 `Start`、`Update`
  等生命周期回调。详见本章「常见组件，以及怎么把它们挂上去」一节。
</GlossaryItem>

</Glossary>
