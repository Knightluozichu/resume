# 章节内容规范（chapter-spec）

> 模板（chapter-template.mdx）是骨架，本文件是法律。
> 总监 review 时逐节对照本规范打 pass / fail，任一节 fail，`draft` 不得置 `false`。
> 量化约束（字数/条数/行数）未来由 lint 脚本机器检查——本规范即"章节 lint skill"的种子。

## 〇、章节类型泛化（先定型，再写作）

每章动笔前必须先归入一个类型，类型决定各段式的强度：

| 类型 | 特征 | 典型章节 | 主 Demo 形态 |
|---|---|---|---|
| **A 概念型** | 讲清"是什么/在哪环节" | OpenGL 简介、你好窗口、深度测试 | `PipelineViz`（看流程） |
| **B 数学型** | 核心是推导和几何直觉 | 变换、坐标系统、摄像机 | `MathViz`（拖参数看几何） |
| **C 实战型** | 核心是代码实现 | 你好三角形、纹理、模型加载 | `ShaderDemo`（改 shader 看渲染） |
| **D 对比型** | 核心是方案差异与取舍 | Phong vs Blinn-Phong、阴影算法对比 | `CompareSlider`（左右对比） |

各段式强度矩阵（**重** = 本章重心；**必** = 必须有；**选** = 按需；**省** = 可省略）：

| 段式 | A 概念 | B 数学 | C 实战 | D 对比 |
|---|---|---|---|---|
| 1 学习目标 | 必 | 必 | 必 | 必 |
| 2 直觉引入 | 必 | 必 | 必 | 必 |
| 3 概念讲解 | **重** | 必 | 必 | 必 |
| 4 数学推导 | 省 | **重** | 选 | 选 |
| 5 交互 Demo | 必 | 必 | 必 | 必 |
| 6 代码对照 | 选 | 选 | **重** | 必 |
| 7 常见误区 | ≥2 个 | ≥2 个 | ≥3 个 | ≥2 个 |
| 8 小结练习 | 必 | 必+推导题 | 必+实现题 | 必+选型题 |
| 9 出处声明 | 必 | 必 | 必 | 必 |
| 10 名词解释 | 必 | 必 | 必 | 必 |

### 动笔前置：知识点清单（先行，硬要求）

每章**动笔前**先产出一份「知识点清单」（可写在 PR/任务卡或章节顶部注释），逐条列出本章涉及的全部知识点 / 专业名词，并为每条标注：

1. **viz 形态**：图示 / 动画（Stepper）/ 交互（Slider·CompareSlider·DemoStage）/ 图片比喻（Figure）/ 代码对照——**每个知识点至少配一种**，难啃的（管线、矩阵、坐标变换等）**必须用动画掰碎**，不能只堆文字
2. **是否进名词解释**：陌生专业名词一律进章末 `<Glossary>`，正文首现处用 `<Term>` 高亮
3. **覆盖核对**：review 时对照此清单，确认没有知识点被纯文字一笔带过（漏配 viz = 打回）

这是「每章吃透每个知识点」的执行抓手：先把知识点摊开高亮，再逐个决定怎么直观讲，而不是写完一篇文字再补图。

## 一、学习目标

**目的**：让读者带着验收标准开始读。

**约束**
- 2–3 条，每条以可检验动词开头：能解释 / 能修改 / 能推导 / 能实现 / 能回答
- 禁用不可检验动词：了解、熟悉、掌握、理解
- 最后一条必须是一个具体的检验问题（读完能当场自测）

**泛化**
- A 型必含「能解释 ___ 在渲染管线中的位置」
- B 型必含一条「能推导 / 能手算 ___」
- C 型必含一条「能改出 ___ 效果」
- D 型必含一条「能说出什么场景用哪个」

## 二、直觉引入

**目的**：在零术语前提下让读者知道"这章解决什么问题"。

**约束**
- ≤3 段，每段 ≤120 字
- **零术语**——专业名词最早只能出现在第三节；违例 = 打回
- 必须回答两个问题：这章解决什么问题？没有它会怎样？
- 允许至多一张示意图

**泛化**
- 类比只从四个库中选：①空间与地图（坐标/变换类）②工厂流水线（管线/流程类）③相机与摄影（观察/投影类）④光与材质（光照/PBR 类）
- 同一 section 内类比保持一致，禁止中途换类比体系

## 三、概念讲解

**目的**：建立准确的概念模型。

**约束**
- 一个小节只讲一个概念；讲不完就拆小节
- 术语首现 = `<Term def="一句话释义">术语</Term>` 高亮（行内强调 + 悬浮/聚焦可见释义），并在正文给出展开；同一术语同时进章末名词解释（§十）。让小白在阅读流中即点即懂，不必跳出去查
- 每个概念至少配一张图示，或一个指向第五节 Demo 的引用
- 单小节 ≤500 字；全章新概念 ≤5 个，超出 = 章节必须拆分

**泛化**
- 讲解顺序固定：是什么 → 为什么需要 → 在管线中的位置
- C 实战型可把"是什么"压缩至一段，篇幅让给第六节

## 四、数学推导

**目的**：让公式可推、可算、可被几何直觉支撑。

**约束**
- 每个 display 公式后必须紧跟一句人话翻译（"这个式子在说……"）
- 推导不跳步；跳步 = 打回
- 节首列符号表：本节首次出现的每个符号一行
- display 公式 ≤8 个，超出拆小节
- 矩阵一律列主序（与 GLSL 一致），并显式注明

**泛化**
- B 数学型必须配 `MathViz` 交互（拖动输入、实时看几何输出）
- 无数学的章节整节省略，frontmatter 标 `math: false`
- 禁止为凑结构硬写数学

## 五、交互 Demo

**目的**：本站灵魂——把"读懂"变成"玩懂"。

**约束**
- Demo 前必有一句"猜一猜：把 ___ 调大会发生什么？"
- 可调 uniform 控件 ≤5 个（更多 = 认知过载，拆 Demo）
- 必有重置按钮；初始参数必须开箱即展示本章核心效果
- 加载用骨架屏，首帧目标 <1s
- 移动端竖屏可用，或显式降级（录屏视频 + 文字说明）
- WebGL2 不可用时必须有兜底（截图 + 说明），不许白屏

**泛化**
- 主 Demo 形态按章节类型选（见类型矩阵）；每章 1 个主 Demo，辅 Demo ≤2 个
- 所有 Demo 共用统一容器组件（DESIGN.md 组件气质节）

## 六、代码对照

**目的**：原书 C++/OpenGL 与本站 WebGL2/TS 互为镜像，读者两边都能落地。

**约束**
- 单个代码块 ≤30 行；两个代码块禁止直接相邻，之间必须有解说文字
- C++ Tab 与 WebGL2 Tab 必须是**同一逻辑段**的两种实现，禁止错位
- API 差异点（如 `glGenBuffers` vs `gl.createBuffer`、`#version 330 core` vs `#version 300 es`）首次出现用 Callout 标注
- 完整可运行源码以章末链接提供，正文只放讲解段

**泛化**
- C 实战型逐段讲解，覆盖 ≥80% 关键代码
- A/B 型只贴最能说明概念的 1–3 段
- D 对比型用并排 diff 突出差异行

## 七、常见误区

**目的**：把读者必然会踩的坑提前踩掉。

**约束**
- 2–4 个；固定三行格式：**现象 →** **原因 →** **修法**
- 每个 ≤150 字
- 至少 1 个配错误效果图，或 Demo 的"错误模式"开关（让读者亲眼看翻车）

**泛化**
- 选坑优先级：原书该章高频疑问 > 自己踩过的 > 合理推测
- D 对比型必须含一个"场景选错方案"的坑

## 八、小结与练习

**目的**：闭环——回收学习目标，留下手感。

**约束**
- 小结 ≤5 条 bullet，每条 ≤40 字
- 小结必须与第一节学习目标一一呼应；有目标未被回收 = 打回
- 练习 2–3 题，必含 1 道"改 Demo 代码"题
- 答案放折叠组件内，默认收起

**泛化**
- B 型 +1 道推导题；C 型 +1 道独立实现题（给目标效果图）；D 型 +1 道场景选型题

## 九、出处声明

**目的**：CC BY-NC 4.0 合规，对原作者的尊重。

**约束**
- 必须存在，位置固定章末
- 内容固定四要素：原文链接 + 中文译版链接 + CC BY-NC 4.0 声明 + "本文为改编重写"
- 由 `<Attribution>` 组件统一渲染，禁止手写变体

**泛化**
- 无。本节不允许任何变化。

## 十、名词解释（Glossary）

**目的**：小白对陌生专业名词一脸懵——章末给一份"人话词典"，把本章每个专业名词用最直白的话再讲一遍，消除阅读门槛。

**约束**
- 位置固定章末（出处声明之前或之后均可，建议出处之前）
- 由 `<Glossary>` 统一渲染，内含若干 `<GlossaryItem term="...">人话释义</GlossaryItem>`
- **覆盖**：本章正文首现、用 `<Term>` 高亮过的专业名词，**必须**在此有对应词条（高亮与词条一一对应，漏一个 = 打回）
- 释义面向**零基础**：一句话能懂的大白话 + 必要时一句类比，避免用更难的术语解释术语；可附"详见本章 ___ 节"指引
- 术语按正文出现顺序或字母/拼音排序，保持稳定

**泛化**
- 难懂的核心名词（如管线阶段、齐次坐标、法线），词条里可嵌/链到一个 `<Stepper>` 或 `<Figure>` 把它动画掰碎，不只给文字定义
- 纯入门的常识词不必收录；只收"小白可能卡壳"的专业名词

## 总监 Review 流程

1. 按章节类型调出强度矩阵，确认该有的段式都在
2. 逐节对照上述约束打 pass / fail，打回时注明节号与违反条款
3. 全 pass → 批准 `draft: false`
4. 同一约束被打回 3 次以上 → 说明约束不合理或缺工具，修订本规范而非硬扛
